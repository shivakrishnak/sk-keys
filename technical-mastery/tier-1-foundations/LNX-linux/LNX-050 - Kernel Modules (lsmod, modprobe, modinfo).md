## TL;DR

Kernel modules are loadable kernel extensions - device drivers, filesystem
drivers, network protocols - that can be added/removed without rebooting.
`lsmod` lists loaded modules. `modprobe module_name` loads a module AND
its dependencies. `modinfo module_name` shows module details. `rmmod`
unloads. Modules live in `/lib/modules/$(uname -r)/`. Auto-load at boot:
list in `/etc/modules-load.d/*.conf`. Blacklist problematic modules:
add to `/etc/modprobe.d/blacklist.conf`. `modprobe -r` unloads including
unused dependencies.

### The Problem This Solves

**Problem 1**: A new server arrives with a NIC model the kernel doesn't
support. The driver exists as a kernel module but isn't loaded. `modprobe
igb` loads the Intel Gigabit Ethernet driver without rebooting - NIC
immediately available.

**Problem 2**: A nouveau (open-source NVIDIA) driver conflicts with the
proprietary NVIDIA driver. Adding `nouveau` to the blacklist in
`/etc/modprobe.d/` prevents it from loading at boot, allowing the NVIDIA
driver to load instead.

**Problem 3**: A network bridge module (`br_netfilter`) must be loaded before
Docker/Kubernetes network configuration. Without it, overlay networking
fails. Adding it to `/etc/modules-load.d/k8s.conf` ensures it loads at
boot automatically.

### Textbook Definition

**Kernel module (loadable kernel module, LKM)**: An object file containing
kernel code that can be dynamically loaded into and removed from the running
kernel. Extends kernel functionality without requiring a monolithic kernel
with all features compiled in. Commonly used for: device drivers, filesystem
drivers (ext4, xfs), network protocols (ip_tables), and Linux Security
Modules (apparmor, selinux).

**Module dependencies**: Some modules require others to be loaded first
(e.g., `kvm_intel` requires `kvm`). `modprobe` handles dependency resolution
automatically by reading `modules.dep` (generated by `depmod`). `insmod`
does NOT resolve dependencies (lower level tool).

**Module parameters**: Modules accept parameters at load time to configure
behavior (e.g., `modprobe usbhid quirks=0x04F2:0x0111:0x40`). Parameters
can also be set in `/etc/modprobe.d/*.conf` as `options module_name param=value`.

### Understand It in 30 Seconds

```bash
# === Listing modules ===
lsmod                      # list loaded modules (name, size, used-by)
lsmod | grep ext4          # check if ext4 module is loaded
lsmod | wc -l              # count loaded modules

# === Module information ===
modinfo ext4               # show module details
modinfo -F filename ext4   # just the filename
modinfo -F description ext4  # just the description
modinfo -F depends ext4    # dependencies

# === Loading modules ===
modprobe ext4              # load module + dependencies (recommended)
insmod /path/to/module.ko  # load by file path, no dependency resolution
modprobe -v ext4           # verbose: show what is being loaded

# === Unloading modules ===
rmmod ext4                 # remove module (fails if in use)
modprobe -r ext4           # remove + unused dependencies

# === Module parameters ===
modprobe usbcore autosuspend_delay_ms=1000  # set parameter at load time
# Set persistent options in /etc/modprobe.d/:
echo "options usbcore autosuspend_delay_ms=1000" \
    > /etc/modprobe.d/usb.conf

# === Auto-load at boot ===
# Add module name (one per line) to a .conf file:
cat > /etc/modules-load.d/mymodules.conf << 'EOF'
br_netfilter
overlay
ip_vs
EOF
# These load at boot via systemd-modules-load.service

# === Blacklisting modules ===
# Prevent module from loading (e.g., conflicting driver):
cat > /etc/modprobe.d/blacklist-nouveau.conf << 'EOF'
blacklist nouveau
options nouveau modeset=0
EOF
# Then regenerate initramfs:
update-initramfs -u        # Debian/Ubuntu
# dracut --force           # RHEL/CentOS

# === Module location ===
ls /lib/modules/$(uname -r)/         # all modules for current kernel
ls /lib/modules/$(uname -r)/kernel/  # organized by subsystem
find /lib/modules/$(uname -r) -name "*.ko" | grep intel  # find Intel modules

# === Dependency information ===
modprobe --show-depends ext4         # show what would be loaded
cat /lib/modules/$(uname -r)/modules.dep | grep ext4  # raw dep file
depmod -a                            # regenerate modules.dep (after adding module)

# Verify a module is available without loading:
modprobe --dry-run br_netfilter
echo $?   # 0 = module exists, non-zero = not found
```

### First Principles

**Module compilation and loading:**
```
Module source (.c) -> gcc (with kernel headers) -> .ko file

.ko file structure (ELF format):
  - init_module() function: runs when module loads
  - cleanup_module() function: runs when module unloads
  - MODULE_LICENSE: "GPL" or "Proprietary" (affects kernel taint)
  - MODULE_DESCRIPTION, MODULE_AUTHOR
  - MODULE_VERSION
  - Symbol table: exported symbols (functions other modules can use)
  - Relocation info: how to link into kernel address space

When modprobe loads a module:
1. Reads /lib/modules/$(uname -r)/modules.dep -> resolve deps
2. For each dep (and the module itself):
   a. insmod: reads .ko file into memory
   b. Kernel verifies module signature (if Secure Boot)
   c. Kernel verifies kernel ABI compatibility (vermagic)
   d. Applies kernel address relocations
   e. Calls module's init_module() function
   f. Module registers its hooks (interrupt handlers, device ops)
3. Module is now active in kernel

Why modules over built-in:
  - Smaller kernel: don't compile everything in
  - Faster boot: only load drivers for present hardware
  - Dynamic: load/unload without reboot
  - Easier development: rebuild module without rebuilding kernel

Why some things are built-in (not modules):
  - Core VFS: must be available before any filesystem
  - Core process scheduler: fundamental, always needed
  - Some security modules: must be present at kernel init time
```

**lsmod output explained:**
```
Module          Size    Used by
br_netfilter    24576   1
bridge         172032   1 br_netfilter
nf_conntrack   172032   4 xt_conntrack,nf_nat,nf_conntrack_netlink,bridge

Column 1: Module name

```

### Thought Experiment

Setting up a server for Kubernetes networking (requires specific modules):

```bash
# Kubernetes requires these kernel modules for networking:
# br_netfilter, overlay, ip_vs, ip_vs_rr, ip_vs_sh, ip_vs_wrr

# Step 1: Check what's loaded:
lsmod | grep -E "br_netfilter|overlay|ip_vs"

# Step 2: Load immediately for current session:
modprobe br_netfilter
modprobe overlay
modprobe ip_vs
modprobe ip_vs_rr
modprobe ip_vs_wrr
modprobe ip_vs_sh

# Step 3: Verify:
lsmod | grep -E "br_netfilter|overlay|ip_vs"
# Also verify kernel parameters exposed by br_netfilter:
ls /proc/sys/net/bridge/
# bridge-nf-call-iptables  bridge-nf-call-ip6tables

# Step 4: Make persistent (survive reboot):
cat > /etc/modules-load.d/k8s.conf << 'EOF'
br_netfilter
overlay
ip_vs
ip_vs_rr
ip_vs_wrr
ip_vs_sh
EOF

# Step 5: Set sysctl parameters exposed by modules:
cat > /etc/sysctl.d/k8s.conf << 'EOF'
net.bridge.bridge-nf-call-iptables = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward = 1
EOF
sysctl --system   # apply now

# Step 6: Verify all required modules on next boot:
# (After reboot, check again)
lsmod | grep -E "br_netfilter|overlay|ip_vs"

# Step 7: Diagnose if a module won't load:
modprobe -v ip_vs 2>&1          # verbose errors
dmesg | grep -i "ipvs\|ip_vs"   # kernel log for load errors
modinfo ip_vs                   # verify module exists for this kernel version
find /lib/modules/$(uname -r) -name "ip_vs.ko*"  # find the .ko file
```

### Mental Model / Analogy

```
The Linux kernel is like a Swiss Army knife:

Core kernel = the main blade (always present, always ready)
  - Process scheduling
  - Memory management
  - Core VFS

Kernel modules = the attachments (add when needed):
  - ext4.ko = the can opener (load when you need ext4 filesystem)
  - nvidia.ko = the magnifying glass (load for NVIDIA GPU)
  - br_netfilter.ko = the scissors (load for container networking)
  - kvm_intel.ko = the saw (load for running virtual machines)

lsmod = "which attachments are currently attached?"
modprobe = "click in attachment with all its required adapters"
  (modprobe handles: "this attachment needs THAT adapter first")
rmmod/modprobe -r = "detach attachment (with now-unneeded adapters)"
modinfo = "read the label on the attachment box"
/etc/modules-load.d/ = "attachments to pre-attach at opening"
/etc/modprobe.d/blacklist = "attachments I never want attached"

insmod = "forcefully clip on attachment by hand" (no adapter check)
  (dangerous: might not have required adapters = kernel crash/oops)

lsmod "Used by" count = how many tools are using this attachment
  (can't detach: 2 tools using it = can't remove while in use)
```

### Gradual Depth - Five Levels

**Level 1:**
`lsmod` (see loaded), `modprobe name` (load), `modinfo name` (inspect).
Modules live in `/lib/modules/$(uname -r)/`. Persistent loading: add to
`/etc/modules-load.d/myconf.conf`. Blacklisting: add to
`/etc/modprobe.d/blacklist.conf`. Always use `modprobe` not `insmod` (handles
dependencies).

**Level 2:**
`depmod -a` regenerates dependency map (needed after adding a .ko manually).
Module parameters: `options module param=value` in `/etc/modprobe.d/`.
`/proc/sys/` parameters exposed by modules (available after module loads).
`sysctl` parameters that modules expose. Kernel taint: loading proprietary
modules taints the kernel (`cat /proc/sys/kernel/tainted`).

**Level 3:**
`vermagic`: each module is compiled for a specific kernel version+config.
A module from a different kernel version won't load (ABI safety). DKMS
(Dynamic Kernel Module Support): automatically recompiles modules for new
kernels (used by NVIDIA, VirtualBox, ZFS). Out-of-tree modules (not in
mainline kernel): DKMS handles them. Module signing: kernel compiled with
`CONFIG_MODULE_SIG` requires modules to be signed. Unsigned modules with
Secure Boot enabled = load failure.

**Level 4:**
Module loading internals: `load_module()` in kernel/module.c. ELF loading,
symbol resolution, relocation, `mod_verify_sig()`, `do_init_module()`.
Module aliasing: `modalias` sysfs attribute on devices (e.g., `pci:v..d..`)
triggers `udevd` to call `modprobe modalias` for automatic driver loading.
`udevadm test /sys/bus/pci/devices/...` simulates udev matching. Kernel
module parameter sysfs interface: `/sys/module/MODULE/parameters/PARAM`.
Can modify parameters at runtime for modules that support it.

**Level 5:**
eBPF programs: the modern alternative to kernel modules for observability
and networking. eBPF code runs in the kernel safely-sandboxed without being
a kernel module. `bpftool prog list` shows loaded eBPF programs. For custom
drivers (hardware) or filesystem drivers: kernel modules are still required.
Rust for Linux (kernel 6.1+): kernel modules can now be written in Rust
(safer memory model than C). First Rust driver merged: PHY driver. `CONFIG_RUST`
kernel config option. Kernel module development with `kunit`: unit testing
framework for kernel code.

### Code Example

**BAD - module loading mistakes:**
```bash
# BAD 1: Using insmod instead of modprobe:
insmod kvm_intel    # FAILS if kvm module not already loaded
# ERROR: insmod: ERROR: could not insert module kvm_intel.ko:
#   Unknown symbol in module

# GOOD: modprobe handles dependencies:
modprobe kvm_intel
# modprobe loads kvm first (dependency), then kvm_intel

# BAD 2: Adding module to /etc/modules instead of /etc/modules-load.d/:
echo "br_netfilter" >> /etc/modules
# /etc/modules is Debian-specific and deprecated on systemd systems
# Works on some distros, not others

# GOOD: Use /etc/modules-load.d/ (works on all systemd distros):
echo "br_netfilter" > /etc/modules-load.d/k8s-network.conf

# BAD 3: Forgetting to regenerate initramfs after blacklisting:
cat > /etc/modprobe.d/blacklist.conf << 'EOF'
blacklist nouveau
EOF
reboot    # nouveau MIGHT still load from initramfs!
# The initramfs has its own copy of modprobe configs

# GOOD: regenerate initramfs after modprobe.d changes:
cat > /etc/modprobe.d/blacklist-nouveau.conf << 'EOF'
blacklist nouveau
options nouveau modeset=0
EOF
update-initramfs -u   # Debian/Ubuntu - includes new blacklist in initramfs
# dracut --force       # RHEL/CentOS
reboot                 # now nouveau is reliably blacklisted

# BAD 4: Removing a module that is in use:
rmmod xfs              # ERROR: rmmod: ERROR: Module xfs is in use
# xfs filesystem is mounted, you cannot remove the driver
# Would need to unmount all XFS filesystems first
```

**GOOD - module management for production:**
```bash
#!/bin/bash
# module-verify.sh: verify required modules are loaded or loadable

REQUIRED_MODULES=(
    "br_netfilter"
    "overlay"
    "ip_vs"
    "ip_vs_rr"
    "ip_vs_wrr"
    "ip_vs_sh"
    "nf_conntrack"
)

echo "=== Checking Required Kernel Modules ==="
all_ok=true

for mod in "${REQUIRED_MODULES[@]}"; do
    if lsmod | grep -q "^${mod}"; then
        echo "LOADED:   $mod"
    elif modprobe --dry-run "$mod" &>/dev/null; then
        echo "AVAILABLE (not loaded): $mod"
        # Load it:
        modprobe "$mod"
        echo "  -> Loaded $mod"
    else
        echo "MISSING:  $mod  (not available for kernel $(uname -r))"
        all_ok=false
    fi
done

if $all_ok; then
    echo ""
    echo "All required modules are available"
else
    echo ""
    echo "ERROR: Some modules are missing. Check kernel version."
    echo "Kernel: $(uname -r)"
    echo "Module path: /lib/modules/$(uname -r)/"
    exit 1
fi

# Show module info for audit:
echo ""
echo "=== Module Details ==="
for mod in "${REQUIRED_MODULES[@]}"; do
    echo "--- $mod ---"
    modinfo -F description "$mod" 2>/dev/null || echo "(no description)"
    modinfo -F version "$mod" 2>/dev/null || echo "(no version)"
done
```

### Common Misconceptions

| Misconception | Reality |
|--------------|---------|
| "`insmod` and `modprobe` do the same thing" | Key difference: dependency handling. `insmod module.ko` loads exactly that one module with no dependency resolution - if dependencies aren't loaded, it fails with "Unknown symbol in module". `modprobe module_name` reads the dependency file (`modules.dep`) and loads all required dependencies first. Always use `modprobe` for normal use; `insmod` is for developers testing a newly compiled `.ko` file. |
| "Blacklisting a module in /etc/modprobe.d/ always prevents loading" | The main blacklist file only prevents automatic loading (from udev/modalias). A manual `modprobe blacklisted_module` CAN still load it. And critically: if the module is baked into the initramfs, it may load from there BEFORE the main filesystem's modprobe.d is read. Fix: regenerate initramfs AFTER adding the blacklist entry (`update-initramfs -u` or `dracut --force`). |
| "Kernel modules can do anything in the kernel" | Modules run in kernel space and can do essentially anything the kernel can do (access all memory, all hardware). But: modules with `MODULE_LICENSE("GPL")` can use internal kernel symbols. Proprietary modules can only use symbols explicitly exported for non-GPL use. Modules also taint the kernel if they're proprietary (`cat /proc/sys/kernel/tainted` - bit 0 = proprietary module). Tainted kernel = kernel developers won't debug it. |
| "lsmod shows all kernel functionality" | No. Core kernel features can be compiled DIRECTLY into the kernel (built-in), not as modules. `cat /boot/config-$(uname -r) | grep CONFIG_EXT4` shows: `=y` means built-in (always present, doesn't appear in lsmod), `=m` means loadable module, `=n` means not compiled. `lsmod` only shows modules (loadable, currently loaded). |
| "Module removal is always safe" | Removing a module while it's in use can cause kernel oops or panic. The kernel reference counting (the "Used by" count in lsmod) prevents `rmmod` from removing in-use modules. But `modprobe -f` or `rmmod` with `--force` can bypass this. A corrupted or buggy `cleanup_module()` can also panic the kernel. In production: never force-remove in-use modules. |

### Failure Modes & Diagnosis

**Module fails to load - version mismatch:**
```bash
# Symptom: modprobe fails with "version magic" error
modprobe mymodule
# FATAL: Module mymodule not found in directory /lib/modules/5.15.0-76-generic
# OR: modprobe: ERROR: could not insert 'mymodule':
#     Exec format error  (version magic mismatch)

# Diagnosis:
modinfo mymodule.ko | grep vermagic
# vermagic:  5.15.0-69-generic SMP mod_unload modversions
# Current kernel:
uname -r
# 5.15.0-76-generic
# MISMATCH: module built for .69, running .76

# This happens after kernel update: DKMS modules need recompilation

# Check DKMS status:
dkms status
# Might show: nvidia, 535.104.05, 5.15.0-69-generic, x86_64: installed
# The module exists for old kernel, not new kernel

# Rebuild for current kernel:
dkms autoinstall           # rebuild all DKMS modules for current kernel
# OR: for specific module:
dkms install nvidia/535.104.05 -k $(uname -r)

# After rebuild:
modprobe nvidia
lsmod | grep nvidia

# Prevention: when running kernel updates, dkms autoinstall should
# run automatically (systemd dkms service). Check if it ran:
journalctl -u dkms 2>/dev/null || dmesg | grep dkms
```

### The Surprising Truth

Kernel modules are NOT isolated from the kernel - they run in the SAME
address space as the kernel with FULL kernel privileges. There is no sandbox.
A bug in a kernel module (like a null pointer dereference) WILL crash the
entire kernel (kernel panic). This is fundamentally different from userspace
applications, where a segfault only kills the process. This is exactly why
the Linux kernel developers are extremely careful about what modules are
accepted into the mainline kernel. The Rust language was introduced into the
kernel (Linux 6.1, December 2022) specifically because Rust's ownership model
prevents entire classes of memory safety bugs that cause kernel panics in C
modules. The first Rust driver merged was a PHY (physical layer) network
driver. Google's Android team has been one of the biggest advocates for Rust
in the kernel - they found that memory safety bugs account for ~70% of Linux
kernel CVEs (security vulnerabilities). eBPF (covered in LNX-073) is an
alternative approach: eBPF programs are sandboxed by a verifier before
running, making it impossible to crash the kernel or access arbitrary memory.
eBPF is increasingly replacing traditional modules for observability and
networking.

### Interview Deep-Dive

**Foundational:**
Q: What is a kernel module and why does Linux use them instead of having everything compiled into the kernel?
A: A kernel module (loadable kernel module, LKM) is an object file containing kernel code - device drivers, filesystem drivers, network protocols - that can be dynamically loaded into and removed from a running Linux kernel without rebooting. Why modules instead of monolithic: (1) Smaller kernel: a Linux kernel compiled with every possible driver for every hardware would be gigabytes in size. Modules allow shipping a kernel binary that fits in memory (typical: 5-20 MB compressed) and loading only what's needed. (2) Faster boot: only load drivers for hardware that's actually present. The initramfs starts a udev daemon that detects hardware and loads appropriate modules automatically via `modalias` matching. (3) No reboot for new hardware: add a USB device, udev detects it, `modprobe usb_storage` loads automatically. No reboot. (4) Easier development: build a new driver as a module, load it with `modprobe`, test it, fix bugs, reload. No kernel rebuild required. (5) Core features ARE built-in: the process scheduler, memory manager, core VFS must be compiled directly into the kernel (`=y` in .config) because they're needed before any module can load. `cat /boot/config-$(uname -r) | grep CONFIG_EXT4` shows `=m` (module) or `=y` (built-in). `lsmod` shows currently loaded modules. `modprobe -l` lists all available modules.

**Intermediate:**
Q: How does Linux automatically load the correct kernel module when you connect a USB device?
A: Automatic device-module binding is a collaboration between the kernel, sysfs, and udevd: (1) Hardware detected: kernel's USB subsystem detects new device. (2) Kernel creates sysfs entry: `/sys/bus/usb/devices/DEVICE/`. (3) Kernel computes `modalias`: a string describing the device's identity. For USB: `usb:v04F2p0111d...` (vendor ID, product ID, device class, etc.). Written to `/sys/bus/usb/devices/DEVICE/modalias`. (4) Kernel sends uevent: emits a `KOBJ_ADD` uevent to udevd with the `MODALIAS` environment variable set to this string. (5) udevd processes the uevent: udev rules can match on `MODALIAS` and trigger `modprobe`. The default udev rule (in `/lib/udev/rules.d/80-drivers.rules`) is essentially: `ENV{MODALIAS}=="?*", RUN{builtin}+="kmod load $env{MODALIAS}"`. (6) modprobe uses the alias: `modules.alias` file (generated by `depmod`) maps modalias patterns to module names. For example: `alias usb:v04F2p0111d*dc*dsc*dp*ic*isc*ip*in* usbhid`. `modprobe` finds this match and loads `usbhid`. (7) Module probes the device: the module's `probe()` function is called with the device. It checks if it supports this specific device and configures it. Result: device is now accessible (appears as `/dev/input/eventN`, `/dev/sdb`, etc.).

**Expert:**
Q: A Kubernetes worker node upgrade fails because CNI plugins stop working after kernel update. Walk through the diagnosis and explain the role of kernel modules in container networking.
A: Kubernetes container networking depends on kernel modules that provide network namespaces, virtual Ethernet, bridging, and load balancing. Typical modules required: `br_netfilter` (netfilter hooks in the bridge layer - allows iptables to filter bridged traffic), `overlay` (overlayfs for container filesystems), `ip_vs` + `ip_vs_rr/rr/sh/wrr` (IPVS for kube-proxy's load balancing in IPVS mode), `nf_conntrack` (connection tracking for NAT/iptables), `veth` (virtual Ethernet pairs for pod networking). Diagnosis after kernel update: (1) Check which modules are loaded: `lsmod | grep -E "br_netfilter|overlay|ip_vs"`. Missing modules = networking will fail. (2) Check if modules exist for new kernel: `find /lib/modules/$(uname -r) -name "br_netfilter*"`. If module files don't exist: new kernel was installed without the module package. (3) For RHEL/CentOS: `rpm -q kernel-modules-$(uname -r)` should be installed. (4) For Ubuntu: `dpkg -l linux-modules-$(uname -r)`. (5) Install missing module packages: `apt install linux-modules-$(uname -r)` or `yum install kernel-modules`. (6) Load required modules: `modprobe br_netfilter overlay ip_vs ip_vs_rr`. (7) Ensure persistent: verify `/etc/modules-load.d/k8s.conf` contains required modules. (8) Sysctl parameters: `br_netfilter` exposes `/proc/sys/net/bridge/bridge-nf-call-iptables`. If module not loaded, this sysctl parameter doesn't exist, and Kubernetes pre-flight check will fail. Set via `/etc/sysctl.d/k8s.conf`. (9) After fixing: `kubeadm reset` and `kubeadm init` (or just restart kubelet and check `kubectl get nodes`). Prevention: use DKMS for any out-of-tree CNI kernel components; include kernel module verification in the node upgrade runbook.
