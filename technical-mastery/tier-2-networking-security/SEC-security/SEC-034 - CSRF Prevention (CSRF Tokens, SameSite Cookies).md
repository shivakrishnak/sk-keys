⚡ TL;DR - CSRF prevention has two main approaches. Modern
applications typically use BOTH for defense in depth:

**1. SameSite Cookie Attribute (modern, recommended):**
`Set-Cookie: session=...; SameSite=Lax` - browser doesn't
send the cookie with cross-site requests (except top-level
GET navigations). This is why CSRF dropped from OWASP 2021
Top 10. SameSite=Lax covers 90%+ of CSRF scenarios.
SameSite=Strict prevents EVEN top-level navigation cross-site.
SameSite=None (required for cross-site embeds) requires
Secure and needs explicit CSRF tokens.

**2. CSRF Tokens (traditional, still needed for legacy or SameSite=None):**
The synchronizer token pattern: server generates a random
token per session, embeds it in forms/headers, validates it
on state-changing requests. Browser cannot read the CSRF token
from evil.com (SOP prevents cross-origin reads), so attacker
cannot replicate it. An XSS attack can steal the CSRF token,
which is why CSRF tokens don't help if XSS exists.

**JWT in headers (not cookies) = no CSRF:** If authentication
uses `Authorization: Bearer <JWT>` (not cookies), CSRF doesn't
apply - browsers don't auto-send custom headers cross-origin.

### 🔥 The Problem This Solves

**CSRF EXISTS BECAUSE BROWSERS AUTO-SEND COOKIES:**
When a browser makes any request to example.com, it
automatically includes all cookies for example.com -
regardless of which page initiated the request. Evil.com
can host a page that silently sends a POST to bank.com.
If the user is logged into bank.com: the bank cookie rides
along automatically. Bank.com sees a valid authenticated
request. The transfer happens. The user never knew.

**WHY "JUST CHECK ORIGIN/REFERER" IS INSUFFICIENT:**
Origin and Referer headers can be absent (privacy settings,
redirects) and were historically forgeable in some browsers.
CSRF tokens and SameSite cookies are more reliable because
they don't depend on the browser sending optional headers.

**THE EVOLUTION:**
OWASP 2013: CSRF in Top 10. OWASP 2017: CSRF in Top 10.
OWASP 2021: CSRF dropped out. Why? SameSite cookies.
When SameSite=Lax or Strict is the default (modern browsers
now default to Lax when unspecified): the attack prerequisite
(browser auto-sending cookies cross-site) is removed.
CSRF tokens remain necessary for applications using
SameSite=None or supporting older browsers.

### 📘 Textbook Definition

**CSRF Prevention:** Techniques that prevent forged
cross-site requests from being accepted as legitimate.

**Synchronizer Token Pattern (CSRF Tokens):**
Server generates a cryptographically random, unguessable
token per user session. The token is embedded in every
HTML form (hidden field) or response header (for AJAX).
Server validates the token on every state-changing request.
Since the token is tied to the session and generated by
the server: an attacker on evil.com cannot know the token
(SOP prevents cross-origin reads). A forged request without
the token → rejected.

**Double-Submit Cookie Pattern:**
An alternative where the CSRF token is set as a cookie AND
required as a request header or body parameter. Since evil.com
cannot read cookies from bank.com (SOP), evil.com cannot
read the CSRF cookie value and therefore cannot submit
the correct CSRF header. Note: vulnerable if the application
has a subdomain XSS vulnerability (subdomain can set cookies
on the parent domain).

**SameSite Cookie Attribute:**
An attribute on `Set-Cookie` that controls when the browser
sends a cookie with cross-origin requests:

- `SameSite=Strict`: Cookie never sent in cross-site requests.
  Even clicking a link FROM another site TO your site
  doesn't include the cookie. Maximum security, worse UX
  (users clicking links from emails/other sites appear logged out).

- `SameSite=Lax`: Cookie sent only for top-level navigation
  (e.g., clicking a link) with safe methods (GET). NOT sent
  for cross-site sub-resources (images, iframes), forms, fetch,
  XHR, or unsafe methods (POST, PUT, DELETE). Covers 90%+ of CSRF.
  Modern browser default when SameSite is unspecified.

- `SameSite=None`: Cookie always sent cross-site. Required
  for third-party embeds (payment iframes, analytics), SSO.
  MUST be combined with `Secure`. REQUIRES CSRF tokens when used.

### ⏱️ Understand It in 30 Seconds

**One line:**
SameSite=Lax on session cookies prevents browsers from
including cookies with cross-origin POST requests - eliminating
the main CSRF attack vector. For applications requiring
SameSite=None: add CSRF tokens (random per-session value
in form + validated on server).

**One analogy:**
> CSRF tokens are like a secret handshake in a members-only
> club. To make a transaction at the club (state-changing request):
> you need the secret handshake (CSRF token) that only members
> who are CURRENTLY IN the club know. A spy (attacker) at another
> club (evil.com) cannot overhear the secret handshake (SOP blocks
> cross-origin reads). When they try to initiate a transaction:
> they don't know the handshake → rejected. SameSite cookies
> are the equivalent of: the club only serves members who
> walked in through the front door (first-party navigation),
> not members who were carried in by an outside agent (cross-site request).

### 🔩 First Principles Explanation

**Why SameSite=Lax covers most CSRF:**

```
CSRF ATTACK PREREQUISITE:
  1. Browser auto-sends session cookie with cross-origin request
  2. Server trusts any request with a valid session cookie

WITHOUT SAMESITE:
  User on evil.com:
    <form action="https://bank.com/transfer" method="POST">
      <input name="to" value="attacker">
      <input name="amount" value="10000">
    </form>
    <script>document.forms[0].submit();</script>
  
  Browser: POST /transfer → includes bank.com session cookie!
  Bank.com: valid session → executes transfer.
  CSRF attack successful.

WITH SAMESITE=LAX:
  Browser: this is a cross-site POST request.
    SameSite=Lax says: don't send cookie for cross-site
    sub-requests, fetch, XHR, or form POSTs.
  Browser: sends POST WITHOUT session cookie.
  Bank.com: no session cookie → 401 Unauthorized.
  CSRF attack FAILS.

WHAT SAMESITE=LAX STILL ALLOWS (and why it's "lax"):
  1. Top-level navigation via GET: clicking a link to bank.com
     from anywhere sends the cookie (expected browser behavior)
  2. Only for GET requests: safe-method navigations
  
  CSRF via GET request: attacker still needs to make users
    navigate to a specific URL. This is a real (but harder) attack:
    <img src="https://bank.com/transfer?to=attacker&amount=1000">
    Fix: NEVER perform state changes in GET handlers.
    GET requests must be idempotent and safe.

SAMESITE=STRICT covers the GET-navigation case too:
  The session cookie is not sent even for top-level link clicks.
  Users clicking links from emails/other sites appear logged out.
  Best for: banking, high-security applications.
  Trade-off: users must re-authenticate after following external links.

JWT IN AUTHORIZATION HEADER = NO CSRF:
  CSRF requires cookies (auto-sent by browser).
  If auth uses: Authorization: Bearer <JWT> (a custom header),
  Browser does NOT auto-send custom headers cross-origin.
  Cross-origin requests from evil.com cannot set the Authorization header.
  Therefore: SPA using JWT in Authorization header is NOT vulnerable to CSRF.
  No CSRF tokens needed.
  This is why stateless JWT APIs commonly skip CSRF protection.
```

### 🧪 Thought Experiment

**SCENARIO: Choosing the right CSRF prevention for different architectures**

```
ARCHITECTURE 1: Traditional server-rendered web app
  Auth: session cookie (HttpOnly)
  Forms: HTML forms that submit with cookies
  
  CSRF Risk: HIGH. Classic CSRF target.
  
  SOLUTION:
    1. Add SameSite=Lax to session cookie:
       Set-Cookie: session=...; HttpOnly; Secure; SameSite=Lax
    2. Add CSRF token to all state-changing forms:
       <input type="hidden" name="csrf_token" value="{{ csrf_token }}">
    3. Validate csrf_token on all POST/PUT/DELETE handlers
    
  DEFENSE IN DEPTH: Both SameSite and CSRF tokens.
    If SameSite fails (old browser, bug): CSRF token catches it.
    If CSRF token implementation has a bug: SameSite catches it.

ARCHITECTURE 2: SPA + REST API (JWT auth)
  Auth: JWT in Authorization header (not cookie)
  Frontend: React/Vue/Angular SPA
  
  CSRF Risk: NONE for the JWT-authenticated endpoints.
    Reason: browser can't auto-send custom headers cross-origin.
    evil.com cannot set Authorization header in cross-origin requests.
  
  SOLUTION: No CSRF tokens needed for JWT-in-header auth.
    Focus on: XSS prevention (XSS can read JWT if in localStorage),
    CORS configuration (control which origins can read API responses).
  
  EXCEPTION: If the SPA also uses cookies for anything
    (refresh token cookie, analytics, CSRF cookie itself):
    those cookies may need SameSite protection.

ARCHITECTURE 3: Traditional app + cross-site payment widget
  Auth: session cookie
  Payment widget: third-party iframe from payment.com
    needs to call bank.com API with bank.com session cookie
  
  CSRF Risk: HIGH (session cookie needed cross-site for payment widget)
  
  SOLUTION:
    Session cookie: SameSite=None; Secure (required for cross-site)
    CSRF protection: REQUIRED. SameSite=None means no SameSite protection.
    Add CSRF tokens to all state-changing endpoints.
    Payment widget: generates CSRF token, sends in custom header.
    Bank API: validates CSRF token from custom header.
    CORS: configure allow-origin for payment.com + allow-credentials.
```

### 🧠 Mental Model / Analogy

> CSRF protection has two models:
> 
> SameSite cookies = don't deliver letters (cookies) to a
> forged return address (cross-site request). The post office
> (browser) checks: "did this request originate from the
> destination domain?" If no: the cookie stays in the mailbox.
> The attacker's forged letter arrives empty-handed.
> 
> CSRF tokens = a secret password on the letter itself.
> The recipient (server) requires the correct password
> (CSRF token) on any letter requesting action. The attacker
> can forge the letter structure but cannot know the password
> (can't read it from the server - SOP blocks it).
> 
> Both are independent defenses. SameSite removes the attacker's
> ability to send authenticated requests. CSRF tokens invalidate
> requests that lack the server-generated proof. Using both
> is defense in depth: if one layer has a bug, the other catches it.

### 📶 Gradual Depth - Five Levels

**Level 1 - What it is (anyone can understand):**
CSRF happens when a bad website tricks your browser into
sending actions to a bank/social media/other site without
you knowing. Your session cookie goes along automatically.
Prevention: add SameSite=Lax to your session cookie (modern
browsers do this automatically). This tells the browser:
"don't automatically include this cookie when requests come
from other websites." No cookie included = bank sees it
as an unauthenticated request = rejected.

**Level 2 - How to use it (junior developer):**
Add `SameSite=Lax` to your session cookie. Most frameworks
support this. For forms: add `{% csrf_token %}` (Django),
`csrf_field()` (Laravel), or implement a hidden input with
a server-generated random token. Validate the token on
the server for every POST/PUT/DELETE. If using JWT in
Authorization header: no CSRF tokens needed.

**Level 3 - How it works (mid-level engineer):**
SameSite=Lax protects against:
- Form POSTs from evil.com: no cookie sent.
- Fetch/XHR from evil.com: no cookie sent.
- Image/iframe sub-resources: no cookie sent.

SameSite=Lax does NOT protect against:
- GET-based state changes (fix: never use GET for state changes)
- 307 redirect attacks (browser follows redirect, same method)
- All browsers updated: SameSite support is now universal in modern
  browsers, but old browsers may not support it.

CSRF token double-submit: cookie value == request header value.
Both must be from the same domain (SOP prevents evil.com from
reading the cookie to submit the matching header). Vulnerable
if attacker can SET a cookie on your domain (cookie injection
via subdomain XSS or misconfigured domain scope).

**Level 4 - Why it was designed this way (senior/staff):**
CSRF tokens were the primary defense for 15 years because
SameSite didn't exist. SameSite=Lax became the browser
default in Chrome 80 (Feb 2020), Safari 12 (2018), Firefox 79 (2020).
Browsers defaulting to Lax solved CSRF for applications
that didn't explicitly set SameSite. The OWASP 2021 removal
of CSRF from the Top 10 reflects this browser-level fix.
CSRF tokens remain necessary for: applications that set
SameSite=None (legacy SSO, payment integrations), APIs
that accept cross-origin credentialed requests by design,
and defense-in-depth for high-security applications.

**Level 5 - Mastery (distinguished engineer):**
Login CSRF: a specific variant where the attacker forces
a victim to log in as the attacker's account. The victim
then uses the application while authenticated as the attacker.
The attacker can see the victim's activity. Protection:
CSRF token on the LOGIN form itself (not just state-changing
actions). Custom CSRF headers (e.g., `X-Requested-With: XMLHttpRequest`):
browsers don't send custom headers in cross-origin simple
requests. This is not a complete defense (HTML5 features
can manipulate some headers) but adds a layer for legacy
APIs. Checking `Content-Type: application/json` provides
similar protection (simple requests can't set this content
type cross-origin) but is also not complete. Rely on tokens
or SameSite, not header checking alone.

### ⚙️ How It Works (Mechanism)

**CSRF token validation flow:**

```
CSRF TOKEN LIFECYCLE:

INITIALIZATION (on session creation):
  Server: generate csrf_token = secrets.token_urlsafe(32)
  Store: session.csrf_token = csrf_token
  
FORM RENDERING (GET /form):
  Server embeds in HTML:
  <form method="POST" action="/transfer">
    <input type="hidden" name="csrf_token"
           value="{{ csrf_token }}">
    ...form fields...
  </form>
  
  Also accessible for AJAX:
  <meta name="csrf-token" content="{{ csrf_token }}">

AJAX REQUEST (JavaScript):
  const csrfToken = document.querySelector(
    'meta[name="csrf-token"]'
  ).content;
  
  fetch('/api/transfer', {
    method: 'POST',
    headers: {
      'X-CSRF-Token': csrfToken,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify(transferData),
  });

SERVER VALIDATION (on every POST/PUT/DELETE):
  def validate_csrf(request):
    # Get token from form or header
    submitted = (request.form.get('csrf_token') or
                 request.headers.get('X-CSRF-Token'))
    expected = session.get('csrf_token')
    
    if not submitted or not expected:
        abort(403)
    
    # CONSTANT-TIME COMPARISON (prevents timing attacks)
    if not hmac.compare_digest(submitted, expected):
        abort(403)

WHAT ATTACKER CANNOT DO:
  evil.com → POST /transfer { csrf_token: ??? }
  Attacker cannot know the CSRF token because:
    1. They can't read bank.com's HTML (SOP blocks cross-origin reads)
    2. They can't read bank.com's cookies (SOP)
    3. They can't read the meta tag value (SOP)
  Any submitted CSRF token → validation fails → 403 Forbidden.
```

### 💻 Code Example

**CSRF protection implementation in Python FastAPI:**

```python
import secrets
import hmac
from fastapi import FastAPI, Request, HTTPException, Form
from fastapi.responses import HTMLResponse
from fastapi.templating import Jinja2Templates

app = FastAPI()
templates = Jinja2Templates(directory="templates")

# Session storage (use Redis in production)
sessions: dict = {}

def get_or_create_csrf_token(session_id: str) -> str:
    """Get or create CSRF token for the session."""
    if session_id not in sessions:
        sessions[session_id] = {}
    if "csrf_token" not in sessions[session_id]:
        sessions[session_id]["csrf_token"] = secrets.token_urlsafe(32)
    return sessions[session_id]["csrf_token"]

def validate_csrf_token(session_id: str, submitted_token: str) -> bool:
    """
    Validate CSRF token using constant-time comparison.
    Constant-time prevents timing attacks on the comparison.
    """
    if not session_id or session_id not in sessions:
        return False
    expected = sessions[session_id].get("csrf_token")
    if not expected or not submitted_token:
        return False
    # hmac.compare_digest: constant-time comparison
    return hmac.compare_digest(expected, submitted_token)

@app.get("/transfer-form", response_class=HTMLResponse)
async def get_transfer_form(request: Request):
    session_id = request.cookies.get("session_id", "")
    csrf_token = get_or_create_csrf_token(session_id)
    
    return templates.TemplateResponse(
        "transfer.html",
        {"request": request, "csrf_token": csrf_token}
    )

@app.post("/transfer")
async def post_transfer(
    request: Request,
    to_account: str = Form(...),
    amount: float = Form(...),
    csrf_token: str = Form(...)
):
    session_id = request.cookies.get("session_id", "")
    
    # CSRF validation - before any state change
    if not validate_csrf_token(session_id, csrf_token):
        raise HTTPException(
            status_code=403,
            detail="CSRF token invalid or missing"
        )
    
    # Proceed with the transfer
    # ... actual transfer logic ...
    return {"status": "transfer initiated"}

# For AJAX/fetch requests: validate X-CSRF-Token header
@app.post("/api/transfer")
async def api_transfer(request: Request):
    session_id = request.cookies.get("session_id", "")
    csrf_token = request.headers.get("X-CSRF-Token", "")
    
    if not validate_csrf_token(session_id, csrf_token):
        raise HTTPException(status_code=403, detail="CSRF invalid")
    
    body = await request.json()
    # ... process transfer ...

# Set-Cookie with SameSite for defense in depth:
# SameSite=Lax: main defense
# CSRF token: backup defense + protection for SameSite=None cases

# GOOD: Session cookie configuration
def set_session_cookie(response, session_id: str):
    response.set_cookie(
        "session_id",
        session_id,
        httponly=True,       # No JavaScript access
        secure=True,         # HTTPS only
        samesite="lax",      # Primary CSRF protection
        max_age=3600,        # 1 hour expiry
        path="/",
    )

# BAD: Missing SameSite and HttpOnly
# response.set_cookie("session_id", session_id)
```

### ⚖️ Comparison Table

| Defense | CSRF Protection | Applies To | Limitations |
|:---|:---|:---|:---|
| **SameSite=Lax** | Strong (most scenarios) | All cookie-based auth | Doesn't protect SameSite=None, GET-based state changes |
| **SameSite=Strict** | Strongest | Highest security apps | Worse UX: links from emails/other sites don't carry session |
| **CSRF Tokens** | Strong | All cookie-based auth | Requires token generation, storage, validation in every form/handler |
| **Double Submit Cookie** | Medium-strong | APIs without server-side state | Vulnerable to subdomain XSS (can set cookies) |
| **JWT in Auth header** | N/A (immune to CSRF) | REST APIs with JWT | Only if JWT not in cookies - if JWT IS in cookie: same CSRF risk |
| **Custom Request Header** | Partial | AJAX only | Not complete for all HTML5 requests |
| **Origin/Referer check** | Weak | - | Headers can be absent, historically forgeable |

### ⚠️ Common Misconceptions

| Misconception | Reality |
|:---|:---|
| CORS headers prevent CSRF | CORS controls whether JavaScript can READ cross-origin responses. CSRF is about the server ACTING on a cross-origin request. The browser sends the request (with cookies) before checking CORS. The CSRF action occurs at the server BEFORE the browser checks if JavaScript can read the response. A CORS policy of `Access-Control-Allow-Origin: *` does not prevent CSRF - in fact, `*` cannot even be combined with credentials, so CORS with credentials has nothing to do with preventing the initial request from being sent. CSRF requires its own defense: CSRF tokens or SameSite cookies. |
| HTTPS prevents CSRF | HTTPS encrypts traffic between browser and server. CSRF doesn't require reading the traffic. The browser sends the HTTPS request with cookies and the server processes it - HTTPS doesn't affect whether the browser auto-includes cookies in cross-origin requests. CSRF is a browser behavior issue, not a transport encryption issue. HTTPS prevents network interception; CSRF bypasses authentication by abusing the browser's cookie-sending behavior. Both protections are needed independently. |

### 🚨 Failure Modes & Diagnosis

**CSRF token implementation errors:**

```
ERROR 1: CSRF token not tied to session
  Bug: global CSRF token (same for all users)
  Attack: attacker generates their own CSRF token from any
    account, uses it in a forged request for a victim.
  Fix: CSRF token MUST be: unique per user session,
    cryptographically random, server-generated.

ERROR 2: CSRF token in URL (leaked in logs/Referer)
  Bug: ?csrf_token=... in URL query string
  Fix: CSRF token ONLY in:
    - Hidden form field (POST body)
    - Custom request header (X-CSRF-Token)
    - Response cookie (for double-submit pattern)
    Never in URL parameters.

ERROR 3: Skipping CSRF validation on "harmless" actions
  "This endpoint only reads data, no need for CSRF."
  But GET-based state changes exist. Logout-CSRF is real:
    <img src="https://bank.com/logout"> from evil.com
    Logs out the user. DoS attack, not data theft.
    But still a CSRF attack.
  Policy: validate CSRF on all state-changing requests.
    If in doubt about whether it changes state: validate.

ERROR 4: Non-constant-time comparison
  Bug: if csrf_token == expected_token: ...
  Timing attack: Python str == compares character by character.
    String that matches first N characters takes longer
    than a completely wrong string. Attacker measures timing
    to enumerate token character by character.
  Fix: hmac.compare_digest(submitted, expected) always.

TESTING CSRF PROTECTION:
  1. Open bank.com. Get logged-in session cookie.
  2. Open attacker HTML page:
     <form action="https://bank.com/transfer" method="POST">
       <input name="amount" value="9999">
       <input name="to" value="attacker">
     </form>
     <script>document.forms[0].submit();</script>
  3. Open HTML page in browser. Does the transfer execute?
     If YES: CSRF vulnerability (no protection or broken protection).
     If NO (403/401): CSRF protection working.
  4. Also test: remove CSRF token from legitimate form,
     submit. Server should return 403.
```

### 💎 Transferable Wisdom

**Reusable Engineering Principle:**
"Browser-enforced defaults are more reliable than developer-remembered controls."
SameSite=Lax as a browser default (Chrome 80) reduced CSRF
risk more than 20 years of developer education about CSRF tokens.
This illustrates: making the secure behavior the default
at the platform/infrastructure level eliminates vulnerability
classes more effectively than requiring developers to explicitly
implement the control. When designing security controls:
prefer browser defaults, framework defaults, and platform
defaults that are secure. Make the insecure option the one
requiring explicit configuration. CSRF tokens required the
developer to REMEMBER to implement them; SameSite required
the developer to REMEMBER to opt OUT of security.

### 💡 The Surprising Truth

The CSRF attack is almost 25 years old (first described ~2000),
has been in OWASP Top 10 since 2007, and was a top vulnerability
for over a decade. It was removed from OWASP 2021 primarily
because browsers fixed it by defaulting to `SameSite=Lax`.
A browser behavior change - not better developer education,
not better frameworks - solved one of the most persistent
web security vulnerabilities. The lesson: systemic platform
fixes are exponentially more effective than per-developer
security training. Browser vendors, framework authors, and
platform engineers who build security into defaults protect
millions of developers who don't remember to implement the
control themselves. Invest in changing defaults, not just
in security education. Education teaches what to do;
defaults ensure it's done automatically.

Does CORS protect against CSRF?**

*Why they ask:* The CORS/CSRF confusion is one of the most
common security misunderstandings. Tests both CSRF knowledge
and understanding of browser security model.

*Strong answer includes:*
- Primary defense: SameSite=Lax on session cookies. Modern browsers
  default to this. Prevents cookies from being sent with cross-origin
  POST/fetch/XHR requests. Cover the most common CSRF attack vectors.
  Doesn't require any form changes.
- Supplementary: CSRF tokens. Still needed for: SameSite=None apps
  (third-party embeds), older browsers, defense in depth.
  Implementation: server-generated random token per session,
  embedded in forms and AJAX headers, validated on every state-changing
  request using constant-time comparison.
- JWT in Authorization header: no CSRF risk. Browsers don't auto-send
  custom headers cross-origin. Pure REST APIs using bearer tokens are immune.
- CORS does NOT prevent CSRF. The key distinction: CORS controls
  if JavaScript can READ the response to a cross-origin request.
  CSRF doesn't need to read the response - the attack is the server
  ACTING on the forged request. The forged request is sent and
  processed BEFORE the browser enforces CORS on the response.
  A bank transfer doesn't need to read the response to succeed.
- Practical test: create an HTML file from a different origin that
  submits a form to the target. If the action succeeds without CSRF token:
  CSRF vulnerability exists. If it fails (403 or missing/invalid cookie):
  protection is working.
