## ⚡ TL;DR

Kubernetes observability = **metrics** (Prometheus + Grafana), **logs** (Fluentd/Vector → Elasticsearch/Loki), and **traces** (OpenTelemetry + Jaeger/Tempo). The kube-prometheus-stack is the de facto standard. Key metrics: Pod CPU/memory utilization, request rate, error rate, latency (RED method). Control plane: etcd, API server, scheduler metrics.

---

## 🔥 Problem This Solves

Without observability: "Why is the cluster slow?" → blind troubleshooting. With metrics, you see: all nodes at 90% CPU, HPA scaled to max, memory throttling errors. With logs: the specific error message causing 500s. With traces: the exact service in your microservices chain adding 800ms latency.

---

## 📘 Textbook Definition

Kubernetes observability encompasses the collection, storage, and visualization of telemetry data (metrics, logs, traces) generated by cluster components and workloads. It enables understanding of system behavior, performance characteristics, and failure modes through inspection of external outputs rather than internal state.

## 🔩 First Principles

- **Metrics Server**: in-cluster component; provides `kubectl top`; powers HPA/VPA
- **Prometheus**: pull-based metrics collection; scrapes `/metrics` endpoints
- **ServiceMonitor** CRD: tells Prometheus what to scrape
- **Grafana**: dashboards from Prometheus data
- **AlertManager**: handles alerts from Prometheus (dedup, route, silence)
- **Loki**: log aggregation (like Prometheus but for logs); indexed by labels
- **OpenTelemetry**: vendor-neutral tracing/metrics/logs SDK + collector

## 🧪 Thought Experiment

At 2am: PagerDuty fires "p99 latency > 2s". Grafana dashboard: API gateway p99 latency spike. Trace view: 95% of the latency is in `payment-service`. Logs: payment-service getting `connection refused` to external payment API. Check: network policy? No. Firewall? Yes - IP block applied by security team. Resolution in 10 minutes because metrics + traces + logs pointed directly to the cause.

## 🧠 Mental Model / Analogy

Observability is the **cockpit instrumentation** of your cluster. Metrics = gauges and dials (CPU, memory, request rate). Logs = flight recorder (what happened and when). Traces = flight path (the journey of each request through microservices). Without instruments, you're flying blind.

## 📶 Gradual Depth

**Level 1 - Beginner**: Install kube-prometheus-stack. Access Grafana for dashboards. Use `kubectl top pods` for resource usage.

**Level 2 - Practitioner**: Configure ServiceMonitors for your apps. Add custom metrics in your app (Micrometer for Spring Boot). Set up AlertManager rules for critical alerts (OOMKill, HPA maxed out, disk pressure).

**Level 3 - Advanced**: RED method: Rate, Errors, Duration (latency) - the three golden signals per service. USE method for infrastructure: Utilization, Saturation, Errors. PromQL: `rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])` = error rate.

**Level 4 - Expert**: Recording rules: pre-compute expensive queries. TSDB (Prometheus time-series DB) retention and compaction. Thanos/Cortex/VictoriaMetrics for long-term, multi-cluster metrics storage. OpenTelemetry Collector: fan-in from multiple sources, fan-out to multiple backends. Cilium Hubble: L3/L4/L7 network flow observability without application instrumentation. eBPF-based continuous profiling (Parca, Pyroscope): CPU profiling without code changes.

### kube-prometheus-stack Components

```
kube-prometheus-stack installs:
  1. Prometheus Operator + CRDs (ServiceMonitor,
    PrometheusRule)
  2. Prometheus (collects metrics)
  3. Alertmanager (handles alerts)
  4. Grafana (visualization, default dashboards for K8s)
  5. kube-state-metrics (K8s object metrics: Pod count,
    Deployment status)
  6. node-exporter (node metrics: CPU, mem, disk, network)
  7. Prometheus Adapter (custom metrics API for HPA)
```

### Critical K8s Alerts

```yaml
# PrometheusRule
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: kubernetes-alerts
spec:
  groups:
    - name: kubernetes
      rules:
        - alert: PodCrashLooping
          expr: increase(
              kube_pod_container_status_restarts_total[15m]) > 5
          for: 0m
          labels:
            severity: critical
          annotations:
            summary: "Pod {{ $labels.pod }} is crash looping"

        - alert: HPAMaxedOut
          expr: kube_horizontalpodautoscaler_status_current_replicas == \
            kube_horizontalpodautoscaler_spec_max_replicas
          for: 5m
          labels:
            severity: warning
          annotations:
            summary: "HPA {{ $labels.hpa }} is at maximum replicas"

        - alert: PersistentVolumeUsageHigh
          expr: (kubelet_volume_stats_used_bytes / kubelet_volume_stats_capacity_bytes) > 0.85
          for: 5m
          labels:
            severity: warning
```

### Spring Boot Metrics

```java
// micrometer + prometheus (spring-boot-actuator)
// pom.xml:
// <dependency>
//   <groupId>io.micrometer</groupId>
//   <artifactId>micrometer-registry-prometheus</artifactId>
// </dependency>

// Auto-exposes: /actuator/prometheus
// Include in application.properties:
management.metrics.export.prometheus.enabled=true
management.endpoints.web.exposure.include=prometheus,health,info

// Custom metrics
@Service
public class PaymentService {
    private final Counter paymentCounter;
    private final Timer paymentTimer;

    PaymentService(MeterRegistry registry) {
        this.paymentCounter = Counter.builder("payments.processed")
            .tag("status", "success")
            .register(registry);
        this.paymentTimer = Timer.builder("payments.duration")
            .register(registry);
    }

    public PaymentResult process(Payment payment) {
        return paymentTimer.record(() -> {
            // ... process payment
            paymentCounter.increment();
            return result;
        });
    }
}
```

### PromQL Key Queries

```promql
# Pod CPU utilization %
100 * sum(rate(container_cpu_usage_seconds_total{
  container!="",namespace="production"}[5m]))
by (pod) /
sum(kube_pod_container_resource_requests{
  resource="cpu",namespace="production"})
by (pod)

# HTTP error rate by service
rate(http_requests_total{status=~"5..",namespace="production"}[5m])
/
rate(http_requests_total{namespace="production"}[5m])

# Memory usage vs limit
container_memory_working_set_bytes{namespace="production"}
/
kube_pod_container_resource_limits{resource="memory",
    namespace="production"}

# Deployment desired vs ready
kube_deployment_status_replicas_ready
/
kube_deployment_spec_replicas
```

## 🔄 E2E Flow: Alert to Resolution

```
Prometheus: evaluates `PodCrashLooping` every 15s
  → increase(restarts) > 5 for pod payment-service-xyz
  → Alert fires → sends to Alertmanager

Alertmanager:
  → Deduplicates (same alert in last 5 min? skip)
  → Routes to #payments Slack channel
  → Routes to PagerDuty for on-call

On-call engineer:
  → Opens Grafana: payment-service namespace
  → Pod restarts graph: 3 restarts in 5 minutes
  → Logs: "OutOfMemoryError: Java heap space"
  → Action: kubectl patch deployment increase memory limits
  → Deploy (via GitOps PR)
  → Monitor: restarts stop
  → Resolve PagerDuty alert
```

## ⚠️ Common Misconceptions

| Misconception                         | Reality                                                                 |
| ------------------------------------- | ----------------------------------------------------------------------- |
| "kubectl top = Prometheus"            | kubectl top uses Metrics Server (short-term); Prometheus = long-term    |
| "More metrics = better observability" | Cardinality matters: too many labels = Prometheus OOM                   |
| "Logs are enough"                     | Logs without metrics = slow MTTR; traces needed for distributed systems |
| "Grafana stores metrics"              | Grafana visualizes; Prometheus stores (or Thanos/Cortex for long-term)  |

## 🚨 Failure Modes

| Failure                    | Symptom                        | Fix                                                                |
| -------------------------- | ------------------------------ | ------------------------------------------------------------------ |
| Prometheus TSDB full       | Prometheus restarts; data loss | Increase storage; reduce retention; add Thanos                     |
| High cardinality           | Prometheus OOM                 | Avoid high-cardinality labels (user_id, request_id)                |
| ServiceMonitor not scraped | App metrics missing            | Check Prometheus target UI; verify ServiceMonitor selector matches |
| AlertManager silenced      | No alerts during incident      | Check silences; set silence expiry                                 |

## 🧠 Think About This

The most overlooked Kubernetes observability dimension is **Kubernetes events**. Events show: OOMKilled Pods, failed scheduling attempts, image pull errors, probe failures, horizontal scaling decisions - all timestamped and queryable. By default events are deleted after 1 hour. Configure `event-ttl` on the API server, or use `kube-event-router` to send events to your log aggregation system. Events are often the fastest path to root cause analysis because they directly describe what Kubernetes is doing internally, whereas Pod logs only show what the application is doing.
