# 057 — DFS

⚡ TL;DR — DFS explores a graph by diving as deep as possible along each path before backtracking, using a stack (or recursion) to track the exploration frontier.

### 🔥 The Problem This Solves

WORLD WITHOUT IT:
You need to detect whether a dependency graph has a cycle (circular imports in Python packages, circular Maven dependencies). You have the graph as an adjacency list. You could try all possible paths, but that is exponential. You could use BFS, but BFS only tracks distance — it has no notion of "currently on the active path" that is needed to see if you revisit a node on the same chain.

THE BREAKING POINT:
Cycle detection requires knowing whether you are currently *inside* a path to a node or whether you have merely *visited* it in a previous independent exploration. These are different states. BFS processes nodes level-by-level and does not maintain the "current active path" — it cannot distinguish between "already visited via another route" and "visited as an ancestor in the current chain."

THE INVENTION MOMENT:
Use a stack. Commit fully to exploring one branch before backtracking. While exploring forward, maintain a "currently active" set. If you encounter a node already in the active set, you are back at an ancestor — a cycle exists. After fully exploring a branch, remove it from the active set (backtrack). This recursive plunge-and-backtrack strategy is exactly why **DFS** was created.

### 📘 Textbook Definition

**Depth-First Search (DFS)** is a graph traversal algorithm that explores as far as possible along each branch before backtracking. It uses an explicit stack (iterative form) or the call stack (recursive form). DFS runs in `O(V + E)` time and `O(V)` space. DFS classifies edges into tree edges, back edges (cycles in directed graphs), forward edges, and cross edges — enabling cycle detection, topological sorting, and strongly connected component identification.

### ⏱️ Understand It in 30 Seconds

**One line:**
Go deep down one path completely before trying any other path.

**One analogy:**
> Imagine exploring a cave system by always following the left wall. You go as deep as the cave allows, marking every room you enter. When you hit a dead end, you backtrack to the last unmarked fork and try the next branch. You eventually visit every room by committing fully to each direction before giving up.

**One insight:**
The power of DFS is not in finding shortest paths (it doesn't). It is in **edge classification** — understanding the structure of the graph. A back edge (from a descendant to an ancestor in the DFS tree) proves a cycle exists. Forward edges and cross edges reveal graph properties that BFS cannot expose. DFS is the right algorithm when you need to reason about the *shape* of the graph, not just distances.

### 🔩 First Principles Explanation

CORE INVARIANTS:
1. A node has exactly **three states**: unvisited, in-progress (on the current active path), and completed.
2. DFS visits each node exactly once, achieving `O(V+E)` time.
3. When DFS first visits a node u and then recursively visits neighbour v, the edge u→v is a **tree edge**. If v is already in-progress (on the stack), u→v is a **back edge** and reveals a cycle.

DERIVED DESIGN:
The three-state model is the minimal abstraction that exposes cycle detection. Two-state (visited/unvisited) is sufficient for connectivity and path finding, but cannot distinguish a back edge from a cross edge — both connect to an already-visited node. The in-progress color (GRAY in CLRS notation) is what enables the distinction.

**DFS tree and timestamps:**
Classical DFS assigns two timestamps per node: `disc[v]` (discovery time, when first visited) and `fin[v]` (finish time, when all descendants are processed). For any edge u→v in a DFS:
- `disc[u] < disc[v] < fin[v] < fin[u]` → tree/forward edge
- `disc[v] < disc[u] < fin[u] < fin[v]` → back edge (v is ancestor of u → CYCLE)
- `fin[v] < disc[u]` → cross edge

THE TRADE-OFFS:
Gain: O(V+E) complexity; exposes complete graph structure including cycles, SCCs, topological order.
Cost: Does not guarantee shortest paths; recursion-based DFS risks `StackOverflowError` for deep graphs (depth > ~10,000 in JVM).

### 🧪 Thought Experiment

SETUP:
Directed graph: A→B, B→C, C→A, A→D. Does this graph have a cycle?

WHAT HAPPENS WITHOUT DFS (using BFS):
BFS from A discovers: level 1 = {B, D}, level 2 = {C}. All nodes visited. BFS has no back edge concept — C is already "visited" when A tries to revisit it, but BFS cannot tell whether C→A creates a cycle or C merely has an outgoing edge to an already-visited node.

WHAT HAPPENS WITH DFS:
Start at A (GRAY). Visit B (GRAY). Visit C (GRAY). C's neighbour is A — A is GRAY (in-progress, on the current path). This is a back edge. Cycle detected: A→B→C→A. DFS terminates with "cycle found." BFS would never expose this because it does not track the currently-active path.

THE INSIGHT:
A back edge exists in a directed graph if and only if there is a cycle. A back edge can only be detected by tracking "currently in the recursion stack" — which requires the in-progress state. This is DFS's unique capability over BFS.

### 🧠 Mental Model / Analogy

> DFS is like reading a book with footnotes. You start reading the main text. When you hit a footnote marker, you immediately flip to the footnote and follow *its* footnotes recursively — diving as deep as the chain goes. Only after you finish a complete footnote chain do you return to where you left off and continue the main text.

"Main text paragraph" → current node being processed
"Footnote" → adjacent unexplored neighbour
"Following footnote chain" → recursive DFS call
"Returning to main text" → backtracking after recursive call returns
"Marking a footnote as already read" → setting node state to COMPLETED

Where this analogy breaks down: Real footnote chains are typically acyclic. DFS must detect and handle cycles — if a footnote pointed back to the main text, the reader would loop forever without the cycle-detection mechanism.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
DFS explores a graph by following one path as far as possible, then backing up and trying another path. Like solving a maze by always going forward until you hit a wall, then backtracking to the last fork.

**Level 2 — How to use it (junior developer):**
To implement DFS recursively: call `dfs(node)`, mark node as visited, then recursively call `dfs(neighbour)` for each unvisited neighbour. To iteratively implement it: push source onto a stack, then loop: pop a node, process it, push unvisited neighbours. For cycle detection, use a three-color scheme: WHITE (unvisited), GRAY (in-progress), BLACK (completed). Finding a GRAY node during traversal means a cycle exists.

**Level 3 — How it works (mid-level engineer):**
DFS produces a DFS forest — a collection of DFS trees (one per connected component for undirected graphs, or per root for directed). Edge classification (tree, back, forward, cross) comes from comparing `disc[]` and `fin[]` timestamps. Topological sort is the reverse order of `fin[]` timestamps — nodes finishing last come first in topological order. Tarjan's SCC algorithm uses DFS with a stack and low-link values to find all strongly connected components in O(V+E).

**Level 4 — Why it was designed this way (senior/staff):**
DFS was designed to expose the *recursive structure* of graphs. The DFS tree makes this explicit: every undirected graph is a tree (with back edges representing cycles) when viewed through DFS. This perspective enables correctness proofs for SCC algorithms: Kosaraju's algorithm runs DFS twice (once on original graph, once on transposed graph in reverse finish order), and the correctness relies on the fact that the first DFS captures the "sink SCC first" finish order. Recognising that DFS on the transposed graph peels off SCCs in source-to-sink order is a deep insight into graph structure.

### ⚙️ How It Works (Mechanism)

**Three-color DFS (for cycle detection):**
```
┌────────────────────────────────────────────┐
│ DFS traversal states                       │
│                                            │
│  Node states:                              │
│    WHITE = unvisited                       │
│    GRAY  = in current path (on stack)      │
│    BLACK = fully explored                  │
│                                            │
│  dfs(u):                                   │
│    color[u] = GRAY   ← entering            │
│    for each v in neighbors(u):             │
│      if color[v] == GRAY:                  │
│        → CYCLE DETECTED (back edge u→v)   │
│      if color[v] == WHITE:                 │
│        dfs(v)                              │
│    color[u] = BLACK  ← fully explored     │
└────────────────────────────────────────────┘
```

**Topological sort via DFS finish order:**
```
topSort = []
dfs(u):
  visited[u] = true
  for each v in neighbors(u):
    if not visited[v]: dfs(v)
  topSort.prepend(u)  ← add on exit

// topSort order after all nodes visited =
// valid topological ordering
```

**DFS iterative (explicit stack):**
```java
void dfsIterative(int src,
    Map<Integer, List<Integer>> graph) {
    Deque<Integer> stack = new ArrayDeque<>();
    Set<Integer> visited = new HashSet<>();
    stack.push(src);
    while (!stack.isEmpty()) {
        int node = stack.pop();
        if (visited.contains(node)) continue;
        visited.add(node);
        process(node);
        // push in reverse order to maintain
        // left-to-right traversal order
        List<Integer> nbrs =
            graph.getOrDefault(node, List.of());
        for (int i=nbrs.size()-1; i>=0; i--)
            stack.push(nbrs.get(i));
    }
}
```

Note: The iterative DFS does not faithfully reproduce the three-color semantics of recursive DFS. For cycle detection requiring GRAY tracking, recursive DFS is cleaner; alternatively, track state explicitly in the stack frame (push a "finish" marker alongside each node).

### 🔄 The Complete Picture — End-to-End Flow

NORMAL FLOW:
```
Graph G (directed or undirected) + source
→ Initialise visited/color arrays
→ [DFS ← YOU ARE HERE]
  → Recurse deep into first neighbour
  → Backtrack when path exhausted
  → Try next unvisited neighbour
  → Collect finish order on exit
→ Output: visited order, cycle flag,
  finish[] timestamps, DFS tree
```

FAILURE PATH:
```
Deep graph (path length 100,000)
→ Recursive DFS exceeds JVM default stack
→ StackOverflowError
→ Fix: use iterative DFS with explicit stack,
  or increase thread stack via new Thread(r,
  "dfs-thread").setDaemon(false) with
  custom stackSize
```

WHAT CHANGES AT SCALE:
For graphs with V=10⁸ nodes (web crawl, social network), recursive DFS is impractical. Iterative DFS with explicit stack is required. The visited set must be memory-efficient — a `BitSet` for integer node IDs uses 1 bit per node (12.5 MB for 10⁸ nodes) vs 32 bytes per node for a `HashSet` (4 GB). For distributed graphs, graph-parallel systems (Apache Giraph, GraphX) implement DFS-derived algorithms differently due to network traversal costs.

### 💻 Code Example

**Example 1 — DFS cycle detection (directed graph):**
```java
enum Color { WHITE, GRAY, BLACK }

boolean hasCycle(int n,
    Map<Integer,List<Integer>> graph) {
    Color[] color = new Color[n];
    Arrays.fill(color, Color.WHITE);
    for (int i = 0; i < n; i++) {
        if (color[i] == Color.WHITE) {
            if (dfsCycle(i, graph, color))
                return true;
        }
    }
    return false;
}

boolean dfsCycle(int u,
    Map<Integer,List<Integer>> graph,
    Color[] color) {
    color[u] = Color.GRAY;
    for (int v : graph.getOrDefault(u,
                                    List.of())) {
        if (color[v] == Color.GRAY)
            return true; // back edge = cycle
        if (color[v] == Color.WHITE)
            if (dfsCycle(v, graph, color))
                return true;
    }
    color[u] = Color.BLACK;
    return false;
}
```

**Example 2 — Topological sort via DFS:**
```java
List<Integer> topologicalSort(int n,
    Map<Integer,List<Integer>> graph) {
    boolean[] visited = new boolean[n];
    Deque<Integer> result = new ArrayDeque<>();

    for (int i = 0; i < n; i++) {
        if (!visited[i])
            dfsTopoSort(i, graph, visited, result);
    }
    return new ArrayList<>(result);
}

void dfsTopoSort(int u,
    Map<Integer,List<Integer>> graph,
    boolean[] visited,
    Deque<Integer> result) {
    visited[u] = true;
    for (int v : graph.getOrDefault(u,List.of())) {
        if (!visited[v])
            dfsTopoSort(v,graph,visited,result);
    }
    result.addFirst(u); // prepend on exit
}
```

**Example 3 — DFS for connected components:**
```java
int countComponents(int n, int[][] edges) {
    List<List<Integer>> adj = new ArrayList<>();
    for (int i = 0; i < n; i++)
        adj.add(new ArrayList<>());
    for (int[] e : edges) {
        adj.get(e[0]).add(e[1]);
        adj.get(e[1]).add(e[0]); // undirected
    }

    boolean[] visited = new boolean[n];
    int components = 0;
    for (int i = 0; i < n; i++) {
        if (!visited[i]) {
            dfsComponent(i, adj, visited);
            components++;
        }
    }
    return components;
}

void dfsComponent(int u,
    List<List<Integer>> adj,
    boolean[] visited) {
    visited[u] = true;
    for (int v : adj.get(u))
        if (!visited[v])
            dfsComponent(v, adj, visited);
}
```

### ⚖️ Comparison Table

| Algorithm | Stack | Shortest Path | Cycle Detection | Topological Sort | Best For |
|---|---|---|---|---|---|
| **DFS** | Call stack / explicit | No | Yes | Yes | Traversal, cycle detect, topo sort, SCC |
| BFS | Queue | Yes (unweighted) | Indirect | No | Shortest hop path, level-order |
| Topological Sort (Kahn's) | Queue (in-degree) | No | Yes (detects) | Yes | Dependency ordering with explicit cycle check |
| Iterative DFS | Explicit stack | No | Complex | Yes | Large graphs where recursion depth overflows |

How to choose: Use DFS for cycle detection, topological sort, SCC, and path existence. Use BFS when shortest hop-count path is needed.

### ⚠️ Common Misconceptions

| Misconception | Reality |
|---|---|
| DFS finds the shortest path | DFS finds A path, not the shortest. It commits to the first branch and may return a longer path before the shortest |
| Iterative DFS with a stack is identical to recursive DFS | Iterative DFS does not automatically reproduce GRAY tracking for cycle detection. Cycle detection requires either recursive DFS or explicit tracking of "currently on stack" in the iterative version |
| Marking visited on push is the same as on pop in DFS | Unlike BFS, marking on pop for DFS can cause nodes to be pushed multiple times but processed once — technically correct but wastes stack space. Marking on push is preferred |
| DFS always detects cycles for undirected graphs | For undirected graphs, any visited neighbour that is not the parent in the DFS tree proves a cycle. But you must skip the parent edge, or every tree edge would falsely appear as a back edge |
| DFS time complexity is O(V²) | DFS using adjacency lists is O(V+E). Using adjacency matrices it is O(V²) due to scanning all V potential neighbours per node |

### 🚨 Failure Modes & Diagnosis

**1. StackOverflowError on deep recursive DFS**

Symptom: `java.lang.StackOverflowError` for graphs with path length > ~5,000–10,000 nodes.

Root Cause: JVM default thread stack is ~512KB. Each DFS frame consumes ~100 bytes. Deep graphs exhaust the stack before finishing.

Diagnostic:
```bash
# Check stack size:
java -XX:+PrintFlagsFinal -version | grep ThreadStackSize
# Typical output: ThreadStackSize = 512 (KB)
```

Fix: Switch to iterative DFS with an explicit `Deque<Integer>` stack, or launch DFS in a new thread with increased stack size:
```java
new Thread(null, () -> dfsRecursive(src),
    "deep-dfs", 64*1024*1024).start();
```

Prevention: Always use iterative DFS for production code where graph depth is unbounded.

---

**2. Cycle detection false negative in undirected graphs**

Symptom: DFS reports "no cycle" for an undirected graph that clearly has a cycle.

Root Cause: In undirected graphs, every edge u-v appears as both u→v and v→u. When DFS visits u, then visits v via u→v, then from v sees u via v→u — it incorrectly identifies u as a back edge even though u is just the parent of v in the DFS tree.

Diagnostic:
{% raw %}
```java
// Test on triangle: 0-1-2-0
// Correct answer: has cycle (triangle)
// Wrong answer from buggy code: no cycle
assert hasCycleUndirected(3,
    new int[][]{{0,1},{1,2},{2,0}}) == true;
```
{% endraw %}

Fix: Track the parent node and skip the parent edge:
```java
boolean dfs(int u, int parent,
            boolean[] visited, List<List<Integer>> adj) {
    visited[u] = true;
    for (int v : adj.get(u)) {
        if (v == parent) continue; // skip tree edge
        if (visited[v]) return true; // back edge
        if (dfs(v, u, visited, adj)) return true;
    }
    return false;
}
```

Prevention: Always pass and check the parent node in undirected graph DFS.

---

**3. Missing nodes in disconnected graph traversal**

Symptom: DFS only visits a subset of nodes; some components never processed.

Root Cause: DFS started from a single source node and only traverses the connected component containing that source. Disconnected graphs have multiple components.

Diagnostic:
```java
// Check visited count vs expected:
int unvisited = 0;
for (boolean v : visited)
    if (!v) unvisited++;
System.out.println("Unvisited nodes: " + unvisited);
```

Fix: Wrap DFS in an outer loop over all nodes:
```java
for (int i = 0; i < n; i++)
    if (!visited[i]) dfs(i, graph, visited);
```

Prevention: Always check whether the graph is guaranteed to be connected. If not, always use the outer loop.

### 🔗 Related Keywords

**Prerequisites (understand these first):**
- `Graph` — DFS traverses graphs; understand adjacency list and matrix representations.
- `Stack` — DFS is mechanically a stack-based traversal; the call stack in recursion IS the stack.
- `Recursion` — recursive DFS is the clearest implementation; understand call stack depth and backtracking.

**Builds On This (learn these next):**
- `Topological Sort` — DFS finish-time ordering directly produces a valid topological sort.
- `Strongly Connected Components` — Kosaraju's and Tarjan's SCC algorithms are DFS-based.
- `Backtracking` — backtracking is DFS on an implicit state-space graph with constraint pruning.

**Alternatives / Comparisons:**
- `BFS` — explores level-by-level; finds shortest unweighted paths but does not expose graph structure the way DFS does.
- `Topological Sort (Kahn's)` — BFS-based topological sort using in-degree counting; often preferred when cycle detection as a separate step is not needed.

### 🧠 Think About This Before We Continue

**Q1.** Tarjan's algorithm for finding Strongly Connected Components (SCCs) uses DFS combined with a "low-link" value — the smallest discovery timestamp reachable from a node's subtree via back edges. An SCC is identified when a node's low-link equals its own discovery time. Consider a graph where node A has edges to B and C, B has an edge back to A (forming cycle A-B-A), and C has no back edges. Trace Tarjan's algorithm step-by-step. At what point is the SCC {A,B} identified, and what happens to C's SCC entry?

**Q2.** DFS on an undirected graph classifies each edge as either a tree edge or a back edge — there are no forward edges or cross edges (unlike directed graphs). Explain why forward edges and cross edges are impossible in undirected DFS. Then consider: if you run DFS on an undirected graph and count the number of back edges, what does that count tell you about the minimum number of edges you would need to remove to make the graph acyclic (a spanning forest)?
