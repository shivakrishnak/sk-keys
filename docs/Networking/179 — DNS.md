# 179 — DNS

⚡ TL;DR — DNS (Domain Name System) is the internet's distributed phone book: it translates human-readable hostnames (google.com) to IP addresses (142.250.x.x) via a hierarchical tree of authoritative servers, using UDP port 53 with TTL-based caching to make ~99% of queries resolve from cache — and it's involved in every outbound network call your application makes.

---

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
In 1983, the internet used a single `HOSTS.TXT` file maintained by the Stanford Research Institute and manually distributed to every connected host. With 563 hosts, this was manageable. By 1987, the file had grown to hundreds of entries and was being updated daily, with every host needing to download the file regularly. At millions of hosts: impossible. You'd need to memorise 142.250.80.14 instead of "google.com."

**THE BREAKING POINT:**
A single centrally-maintained file doesn't scale: updates are delayed (file must be fetched), the file becomes enormous, a single host failure breaks all name resolution globally, and no organisation can "own" part of the namespace. The internet needed a distributed, hierarchical naming system where each organisation manages its own names.

**THE INVENTION MOMENT:**
Paul Mockapetris designed DNS in 1983 (RFC 882, 883). Key insights: (1) hierarchical namespace — split at dots: `www.example.com` = `www` under `example` under `com` under the root `.`; (2) distributed authority — `example.com` is authoritative for everything under it; (3) delegation — the root delegates `.com` to Verisign, `.com` delegates `example.com` to its owner; (4) caching — TTL (Time To Live) allows resolvers to cache answers for hours, reducing query load by 99%+.

---

### 📘 Textbook Definition

**DNS** (Domain Name System) is a hierarchical, distributed naming system that maps domain names to IP addresses (and other record types). Components: (1) **DNS namespace** — hierarchical tree rooted at `.`; (2) **Resource Records (RR)** — A (IPv4), AAAA (IPv6), CNAME (alias), MX (mail), NS (name server), TXT (text), PTR (reverse lookup), SRV (service location); (3) **Authoritative servers** — hold the definitive records for a zone; (4) **Recursive resolvers** — query the hierarchy on behalf of clients; (5) **Caching** — TTL determines how long records are cached. DNS uses UDP port 53 for queries ≤512 bytes, TCP port 53 for larger responses or zone transfers.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
DNS translates hostnames to IPs via a distributed hierarchy of name servers, cached aggressively by resolvers and clients — every application network call starts with a DNS lookup.

**One analogy:**

> DNS is like a city's directory assistance with a phone tree. You call 411 (recursive resolver). The operator doesn't know every number, so they call the city registry (root server): "Who handles .com?" The city says "call Verisign." Verisign says "call example.com's nameserver." example.com's nameserver gives you 93.184.216.34. The operator tells you and writes the number on a sticky note (cache, TTL=3600). For the next hour, the operator answers the same question from the sticky note without making phone calls.

**One insight:**
DNS is involved in **every** outbound network call your application makes. A microservice making 1,000 requests/second to `api.example.com` performs up to 1,000 DNS lookups/second — unless the application or runtime caches DNS results. Java's JVM caches DNS results for a configurable time (default: indefinitely until restart). This is why Kubernetes deploys CoreDNS per cluster.

### 🔩 First Principles Explanation

**RESOURCE RECORD TYPES:**

```
A     → hostname → IPv4 address
        example.com. 3600 IN A 93.184.216.34

AAAA  → hostname → IPv6 address
        example.com. 3600 IN AAAA 2606:2800:220:1:248:1893:25c8:1946

CNAME → alias → canonical name (cannot coexist with A at apex)
        www.example.com. 3600 IN CNAME example.com.

MX    → mail exchange (with priority)
        example.com. 3600 IN MX 10 mail.example.com.

NS    → name server for a zone
        example.com. 172800 IN NS ns1.example.com.

TXT   → arbitrary text (SPF, DKIM, domain verification)
        example.com. 3600 IN TXT "v=spf1 include:_spf.google.com ~all"

PTR   → IP → hostname (reverse DNS)
        34.216.184.93.in-addr.arpa. 3600 IN PTR example.com.

SRV   → service endpoint discovery
        _http._tcp.example.com. 3600 IN SRV 0 0 80 www.example.com.
```

**DNS HIERARCHY:**

```
Root (.)
  ├── .com (TLD — Verisign)
  │     ├── google.com (authoritative NS: ns1.google.com)
  │     ├── example.com (authoritative NS: ns1.example.com)
  │     └── ...
  ├── .org (TLD — Public Interest Registry)
  ├── .io  (TLD — ICANN)
  └── .internal (split-horizon: private DNS)
```

**TTL AND CACHING:**

```
TTL = 3600 → cache for 1 hour
TTL = 60   → cache for 60 seconds (frequent IP changes)
TTL = 86400 → cache for 24 hours (stable IPs)
TTL = 0    → do not cache (always fetch fresh)

Low TTL: enables faster propagation of IP changes
         (important before DNS migrations)
High TTL: reduces DNS load, but IP changes take longer to propagate
          (old TTL must expire everywhere before new record is seen)
```

### 🧪 Thought Experiment

**SETUP:**
You're migrating your web application from IP 93.184.216.34 to 93.184.216.100. Current TTL: 86400 (24 hours). How long until all users see the new IP?

**ANALYSIS:**
Some resolvers cached your A record at different times in the last 24 hours. A resolver that cached it 1 second before the change will keep the old IP for ~24 hours. A resolver that cached it 23 hours before will refresh in 1 hour. Maximum propagation delay = current TTL value = 24 hours.

**BEST PRACTICE:**
1 week before migration: reduce TTL to 60 seconds. Wait for old TTL (86400s) to expire everywhere — all caches now have 60s TTL. Make the change. New IP propagates in 60 seconds. After migration is stable: raise TTL back to 86400.

**LESSON:**
"DNS propagation" delay is entirely determined by the TTL in place **before** the change. Changing TTL at the moment of migration doesn't help — you needed to lower it 24+ hours earlier.

### 🧠 Mental Model / Analogy

> DNS is a distributed phone book with a lookup cascade. Your book (local cache) → ask local librarian (recursive resolver) → librarian asks city hall (root) → city hall says "ask the county for .com" → county says "ask the city for google.com" → city gives the number. The librarian writes the number on a card that expires in N hours (TTL). Next time: hand you the card directly. This works for billions of lookups because 99%+ hit caches, and the hierarchy distributes load across the entire world.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
DNS translates website names (google.com) to IP addresses (142.250.x.x) because humans can't remember numbers but computers need numbers to send data. It's like a phonebook for the internet. When you type a URL, your computer asks a DNS server "what's the IP for this name?" before it can connect.

**Level 2 — How to use it (junior developer):**
In code: `socket.getaddrinfo('example.com', 80)` performs DNS resolution. In HTTP clients it's automatic. Cache awareness: once resolved, the IP is cached locally until TTL expires. JVM DNS: Java caches DNS forever by default (controlled by `networkaddress.cache.ttl` security property). Set to 60 in production to pick up IP changes. Kubernetes: pods use CoreDNS for internal service discovery — `service-name.namespace.svc.cluster.local`. Diagnose: `nslookup`, `dig`, `host` commands. `dig google.com` shows A records, TTL, authoritative server.

**Level 3 — How it works (mid-level engineer):**
DNS negative caching (NXDOMAIN — "name does not exist"): cached for the TTL of the SOA record's minimum TTL field. DNSSEC: signs DNS records cryptographically to prevent cache poisoning. DNS over HTTPS (DoH) and DNS over TLS (DoT): encrypt DNS queries to prevent ISP monitoring. Split-horizon DNS: same hostname resolves to different IPs depending on where you query from (internal network → private IP, external → public IP). AWS Route 53 uses split-horizon for internal load balancing. Anycast DNS: the same IP (e.g., 1.1.1.1) is announced from multiple PoPs globally — BGP routes your query to the nearest one.

**Level 4 — Why it was designed this way (senior/staff):**
DNS's hierarchical, delegated design enables internet-scale decentralisation: no single organisation controls all names. The "zone cut" mechanism (NS records delegating authority to child zones) allows any organisation to manage its own subdomain without coordination with the parent. DNS's caching design reflects the insight that most lookups are for popular names: 80% of queries resolve from cache. The Kaminsky DNS cache poisoning attack (2008) exploited DNS's use of sequential transaction IDs: an attacker could race to provide a forged response before the legitimate resolver. The fix: random source ports + random transaction IDs (5^16 entropy). DNSSEC (cryptographic signing) provides a stronger fix but has only ~3% deployment despite 15+ years of work, due to operational complexity.

### ⚙️ How It Works (Mechanism)

```bash
# DNS lookup with full details
dig google.com
# Shows: A records, TTL, authoritative/recursive flags

# Trace full resolution path (from root)
dig +trace google.com

# Query specific DNS server
dig @8.8.8.8 google.com

# Reverse DNS lookup
dig -x 8.8.8.8

# Check all record types
dig google.com ANY

# Check TTL remaining (already cached)
dig google.com | grep -i ttl

# Check SOA (authoritative info)
dig SOA google.com

# Lookup MX records (email)
dig MX gmail.com

# Check nameservers for a domain
dig NS google.com

# Check TXT records (SPF, DKIM, verification)
dig TXT google.com

# Low-level: what DNS server is being used
cat /etc/resolv.conf        # Linux DNS config
# nameserver 8.8.8.8        # DNS server IP

# Check /etc/hosts (local overrides, checked before DNS)
cat /etc/hosts

# Flush DNS cache (Linux systemd)
systemd-resolve --flush-caches
# Check cache stats
systemd-resolve --statistics
```

### 🔄 The Complete Picture — End-to-End Flow

```
┌────────────────────────────────────────────────┐
│  DNS Resolution: www.example.com               │
└────────────────────────────────────────────────┘

 Browser → OS DNS stub resolver
   → Check /etc/hosts: no entry
   → Check OS DNS cache: not cached
   → Query recursive resolver (e.g., 8.8.8.8)

 Recursive resolver (8.8.8.8):
   → Check resolver cache: not found
   → Query root server (e.g., a.root-servers.net)
      ← NS: .com → a.gtld-servers.net

   → Query a.gtld-servers.net
      ← NS: example.com → ns1.example.com
         TTL: 172800 (2 days)

   → Query ns1.example.com (authoritative)
      ← A: www.example.com → 93.184.216.34
         TTL: 3600 (1 hour)

   → Cache: www.example.com = 93.184.216.34, TTL=3600
   → Return 93.184.216.34 to client

 Browser → TCP connect to 93.184.216.34:443
```

### 💻 Code Example

```python
import socket
import dns.resolver  # pip install dnspython
import time

def dns_lookup_with_ttl(hostname: str) -> dict:
    """Lookup A records and TTL using dnspython."""
    result = {'hostname': hostname, 'records': []}

    try:
        answers = dns.resolver.resolve(hostname, 'A')
        result['ttl'] = answers.rrset.ttl
        result['records'] = [str(r) for r in answers]
        result['nameserver'] = str(answers.nameserver)
    except dns.resolver.NXDOMAIN:
        result['error'] = 'NXDOMAIN'
    except dns.resolver.Timeout:
        result['error'] = 'Timeout'

    return result

# Check caching behaviour
for _ in range(3):
    start = time.perf_counter()
    addr = socket.getaddrinfo('google.com', 80)
    elapsed = (time.perf_counter() - start) * 1000
    print(f"Resolved in {elapsed:.2f}ms: {addr[0][4][0]}")
    # First call: network lookup (~10-50ms)
    # Subsequent calls: OS cache (~0.1ms)

# DNS record types
for rtype in ['A', 'AAAA', 'MX', 'TXT', 'NS', 'CNAME']:
    try:
        answers = dns.resolver.resolve('google.com', rtype)
        for r in answers:
            print(f"{rtype:6s}: {r}")
    except Exception:
        pass
```

### ⚠️ Common Misconceptions

| Misconception                                    | Reality                                                                                                                                                                |
| ------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| "DNS propagation takes 48 hours"                 | It takes as long as the current TTL. With TTL=60, propagation is 60 seconds. "48 hours" assumes TTL=86400 (24h) + buffer. Reduce TTL before changes.                   |
| DNS is only for web browsing                     | DNS is used by every internet protocol: email (MX records), APIs, databases, Kubernetes services, certificate authorities, email authentication (SPF/DKIM TXT records) |
| DNS is always available                          | DNS failures cause cascading failures in all dependent services. Always use multiple DNS servers (primary + secondary) and implement retry logic with short timeouts   |
| The same hostname always resolves to the same IP | DNS can return multiple IPs (round-robin load balancing), geo-based IPs (Anycast / Route 53 latency routing), or different IPs based on the querying resolver          |

### 🚨 Failure Modes & Diagnosis

**DNS Resolution Failures Causing Service Outage**

**Symptom:**
Services that were working fail with "Name or service not known" or connection timeouts. All HTTP calls fail.

```bash
# Test basic DNS resolution
dig google.com @8.8.8.8
# If this fails: DNS server unreachable

# Check configured DNS servers
cat /etc/resolv.conf

# Test resolution with multiple servers
dig google.com @1.1.1.1   # Cloudflare
dig google.com @8.8.8.8   # Google

# Test DNS latency
time dig google.com @8.8.8.8

# In Kubernetes: test CoreDNS
kubectl exec -it <pod> -- nslookup kubernetes.default.svc.cluster.local
kubectl -n kube-system get pods -l k8s-app=kube-dns
kubectl -n kube-system logs <coredns-pod>

# Check for negative TTL (NXDOMAIN cached)
dig myservice.internal.company.com
# NXDOMAIN with TTL: wait for TTL to expire, then retry
```
