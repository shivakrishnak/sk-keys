# 181 — CDN

⚡ TL;DR — A CDN (Content Delivery Network) is a globally distributed network of edge servers that caches content near users, reducing origin server load and latency from hundreds of milliseconds to single-digit milliseconds for cached assets — with modern CDNs (Cloudflare, AWS CloudFront) also providing DDoS protection, TLS termination, WAF, and edge compute.

---

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
A user in Sydney requesting a static image from a server in Virginia experiences: DNS resolution (20ms) + TCP handshake (200ms RTT) + TLS handshake (200ms) + HTTP request/response (200ms) = 620ms for a 10KB image. Every user on the planet, regardless of location, pays the full round-trip cost to the origin. A viral tweet makes a US-based image host unreachable for Australian users (200ms RTT × thousands of requests).

**THE BREAKING POINT:**
Netflix serves 700TB/s of video globally. Without a CDN, every video stream originates from Netflix datacentres. The bandwidth cost alone would be prohibitive. More critically, a user in Mumbai watching 4K video at 25 Mbps would need a 200ms+ RTT connection to the US — causing constant buffering because the TCP connection can't sustain the required throughput (BDP problem: 25 Mbps × 0.200s = 625KB window needed for smooth streaming).

**THE INVENTION MOMENT:**
Akamai pioneered CDNs in 1998 to serve MIT's website during World Cup traffic. The insight: instead of routing all traffic to a central origin, distribute copies of content to hundreds of edge PoPs (Points of Presence) worldwide. Users are directed to the nearest PoP via DNS (return the PoP's IP closest to the querying resolver). Static content (images, CSS, JS, video) served from cache at the PoP with 1-5ms latency. Dynamic content and cache misses fetch from origin. This model reduces origin load by 95%+ for media-heavy sites.

---

### 📘 Textbook Definition

A **CDN** (Content Delivery Network) is a geographically distributed system of servers (edge nodes / PoPs) that cache and serve content closer to end users, reducing latency, improving availability, and offloading traffic from origin servers. Key functions: (1) **Content caching** — edge servers cache responses based on HTTP cache headers (Cache-Control, ETag, Last-Modified); (2) **Request routing** — DNS-based or Anycast routing directs users to the nearest PoP; (3) **Origin pull** — on cache miss, the edge fetches from origin and caches the response; (4) **TLS termination** — HTTPS terminates at the edge, reducing TLS handshake RTT; (5) **Security** — DDoS mitigation, WAF (Web Application Firewall), bot protection at the edge.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
A CDN caches your content at edge servers around the world so users get responses from a nearby server in 5-20ms instead of from your origin server in 200ms+ — and absorbs DDoS traffic before it reaches you.

**One analogy:**

> A CDN is like a chain of local convenience stores (edge nodes) stocking the most popular items from a central warehouse (origin). When you want a bottle of water, you go to the corner store (5 minutes away), not the warehouse (2 hours away). If the corner store is out of stock (cache miss), it orders from the warehouse and restocks. The warehouse ships to millions of stores, but each individual customer only needs to travel to their corner store.

**One insight:**
A CDN doesn't just accelerate static content — it also moves TLS termination to the edge. A user in Tokyo connecting to a US-based site without CDN pays 3 × 150ms RTT for TCP+TLS handshake (450ms) before the first byte. With a CDN, TLS terminates at the Tokyo PoP (3 × 5ms = 15ms), and the CDN-to-origin connection is persistent and optimised. This alone reduces TTFB (Time to First Byte) by 400ms+.

### 🔩 First Principles Explanation

**CDN ARCHITECTURE:**

```
┌──────────────────────────────────────────┐
│  CDN Request Flow                        │
└──────────────────────────────────────────┘

User (Mumbai) → DNS: "cdn.example.com?"
  Recursive resolver (Mumbai area)
  CDN authoritative DNS: "nearest PoP for this resolver"
  → Mumbai PoP IP: 104.18.x.x

User → TCP connect to Mumbai PoP (5ms RTT)
User → TLS handshake (5ms × 1.5 = 7.5ms)
User → GET /logo.png

Mumbai PoP:
  Cache HIT?  → Return logo.png immediately (HIT ratio ~95%)
  Cache MISS? → Fetch from origin (Virginia: 200ms)
               → Cache with Cache-Control: max-age=86400
               → Return to user

Total user latency (HIT):  5ms + 7.5ms + ~2ms = ~15ms
Total user latency (MISS): 5ms + 7.5ms + ~200ms + ~2ms = ~215ms
Without CDN:               150ms + 225ms + ~200ms = ~575ms
```

**CACHE CONTROL HEADERS:**

```
Cache-Control: max-age=86400, public
→ Edge caches for 86400s (24h); 'public' = OK to cache in CDN

Cache-Control: max-age=0, no-store
→ Never cache (CDN passes through to origin every time)

Cache-Control: max-age=31536000, immutable
→ Cache forever (use for versioned assets: app.abc123.js)

Vary: Accept-Encoding
→ Cache separate versions for gzip vs br vs uncompressed

ETag: "abc123"
Cache-Control: max-age=60, must-revalidate
→ After 60s, revalidate with If-None-Match: "abc123"
   → 304 Not Modified (no body transfer) if unchanged
```

**CDN ROUTING MECHANISMS:**

_DNS-based routing:_

- CDN authoritative DNS returns different IPs based on querying resolver location (EDNS Client Subnet)
- Downside: DNS TTL caching means some users get "wrong" PoP temporarily
- Used by: CloudFront, Akamai, Fastly

_Anycast routing:_

- Same IP (e.g., 1.1.1.1) announced from every PoP via BGP
- BGP naturally routes to nearest PoP by AS hop count / prefix preference
- No DNS TTL issue; PoP failover is immediate
- Used by: Cloudflare, Google (GCP CDN), DNS/NTP services

**CACHE INVALIDATION:**

```
Option 1: TTL expiry (wait for max-age)
Option 2: CDN API cache purge (immediate but costs API call)
Option 3: Cache-busting (change URL: app.v2.abc123.js)
Option 4: Surrogate keys / Cache tags (purge by tag, not URL)
```

### 🧪 Thought Experiment

**SETUP:**
Your React SPA has: `index.html` (changes on every deploy), `app.abc123.js` (content-hashed, immutable), `/api/user` (dynamic, no cache).

**CDN STRATEGY:**

_index.html:_
`Cache-Control: no-cache` (CDN revalidates every request; origin returns 304 if unchanged)
Reasoning: index.html references the hashed JS filename. When you deploy, the new index.html references `app.xyz789.js`. Users must always get the latest index.html to load the correct JS.

_app.abc123.js:_
`Cache-Control: public, max-age=31536000, immutable`
Reasoning: the hash in the filename changes on every deploy. Old hash = old content, always. CDN can cache forever. Users loading the new app.xyz789.js get a cache miss (new hash) and pull from origin once; subsequently served from edge.

_/api/user:_
`Cache-Control: no-store` or pass through
Reasoning: user-specific, sensitive data. Must not be cached at CDN.

**RESULT:**

- ~95% of traffic (JS/CSS/images) served from CDN cache
- Zero origin requests for immutable assets after first user in each PoP
- index.html always fresh (304 revalidation is cheap)
- API traffic still hits origin (correct; not cacheable)

### 🧠 Mental Model / Analogy

> A CDN is like a franchised food chain. The corporate kitchen (origin server) develops the recipes and prepares ingredients. Local franchise stores (edge PoPs) prepare the final product close to customers using those ingredients. Most customers go to their local store (fast, 5min away). Occasionally a local store runs out of an ingredient (cache miss) and orders from corporate (fetch from origin). The local store caches the restocked ingredient for tomorrow's customers. Viral demand: if a million people want the same product today, the local stores handle the load — corporate kitchen doesn't receive a million individual orders.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
A CDN stores copies of your website's files (images, videos, JavaScript) in servers around the world. When someone in Japan visits your website hosted in the US, instead of the data travelling across the Pacific, it comes from a server in Tokyo that already has a copy. This makes websites faster globally and protects against traffic spikes.

**Level 2 — How to use it (junior developer):**
Basic CDN usage: set your DNS to point `cdn.example.com` to CloudFront/CloudFlare, configure the origin (your server), and set `Cache-Control` headers in your HTTP responses. The CDN respects these headers. Key settings: cache TTL (how long the CDN caches), origin path (what URL to fetch from your server), custom headers, HTTPS certificate. For static assets: use content hashing in filenames and set long TTLs. For APIs: either bypass CDN or cache with `Cache-Control: s-maxage=60` (CDN-only cache, bypasses browser cache). Monitor: check your CDN dashboard's cache hit ratio — aim for >80% for static content.

**Level 3 — How it works (mid-level engineer):**
CDN internals: edge servers run Nginx/Varnish/custom software. On cache miss, the edge makes a request to origin using persistent TCP connections (reusing the warm TCP/TLS connection, avoiding setup overhead). CDN-to-origin uses the origin's IP directly (bypasses DNS). Cache key: usually `host + path + query string` (varies by CDN config). Vary header: `Vary: Accept-Encoding` creates separate cache entries per encoding. Cache hierarchy: some CDNs have 2-tier caching (edge PoP + regional "mid-tier" shield) to protect origin from many edge PoPs all missing the same object simultaneously. Edge compute: Cloudflare Workers, Lambda@Edge — run code at the edge PoP for personalisation, auth, request manipulation without origin roundtrip.

**Level 4 — Why it was designed this way (senior/staff):**
CDNs solve the fundamental trade-off between geographic distribution and content freshness. The CAP theorem equivalent for CDNs: you can have Consistency (fresh content) or Availability/Performance (low latency from cache), but not both at the same time for mutable content. The solution is the three-tier cache architecture: immutable assets (hash-versioned) get infinite TTL (perfect performance, no staleness); mutable but infrequently-changing assets (index.html) get short TTL + ETag revalidation (one RTT to origin for freshness check without body transfer); dynamic/sensitive content bypasses CDN (no cache). Edge compute (Workers, Lambda@Edge) represents the next evolution: instead of purely caching, CDNs now run application logic at the edge, enabling personalisation and auth with origin-level flexibility but CDN-level latency.

### ⚙️ How It Works (Mechanism)

```bash
# Check CDN headers in response
curl -I https://cdn.example.com/logo.png

# Key response headers to look for:
# X-Cache: HIT (CloudFront/Varnish — served from cache)
# X-Cache: MISS (CDN fetched from origin)
# CF-Cache-Status: HIT / MISS / BYPASS (Cloudflare)
# Age: 3600 (seconds this response has been in CDN cache)
# Cache-Control: public, max-age=86400

# Check which PoP you hit
curl -I https://www.cloudflare.com/ 2>/dev/null | grep -i "cf-ray\|server"
# CF-Ray: 7a2b3c4d-SYD (SYD = Sydney PoP)

# Test CDN performance from different locations
# (use online tools like GTmetrix, WebPageTest, Pingdom)

# Check CloudFront distribution config (AWS CLI)
aws cloudfront list-distributions
aws cloudfront get-distribution --id EDFDVBD6EXAMPLE

# Invalidate CloudFront cache (costs $0.005/path)
aws cloudfront create-invalidation \
  --distribution-id EDFDVBD6EXAMPLE \
  --paths "/*"

# Cloudflare cache purge (via API)
curl -X POST "https://api.cloudflare.com/client/v4/zones/{zone_id}/purge_cache" \
  -H "Authorization: Bearer {api_token}" \
  -H "Content-Type: application/json" \
  --data '{"purge_everything":true}'
```

### 🔄 The Complete Picture — End-to-End Flow

```
┌──────────────────────────────────────────────────────┐
│  CDN Cache Hit vs Miss Flow                          │
└──────────────────────────────────────────────────────┘

 CACHE HIT (most requests):

 User (Tokyo) → DNS → Tokyo PoP IP
 User → TCP (5ms) → TLS (7ms) → GET /logo.png
 Tokyo PoP: cache HIT
 ← 200 OK, logo.png, Age: 3600, X-Cache: HIT
 Total: ~15ms

 CACHE MISS (first user in PoP, or TTL expired):

 User (Tokyo) → DNS → Tokyo PoP IP
 User → TCP (5ms) → TLS (7ms) → GET /logo.png
 Tokyo PoP: cache MISS
 Tokyo PoP → persistent connection to origin (Virginia)
   → GET /logo.png
   ← 200 OK, logo.png, Cache-Control: max-age=86400
 Tokyo PoP: store in cache (TTL=86400)
 ← 200 OK, logo.png, X-Cache: MISS
 Total: ~215ms (but all subsequent requests: ~15ms)

 CDN SHIELD (two-tier):

 Tokyo PoP MISS → Regional PoP (Singapore) → check cache
   Singapore HIT → return to Tokyo PoP, Tokyo caches
   Singapore MISS → fetch from origin (one request per region,
                    not per PoP)
```

### 💻 Code Example

**Example — Setting cache headers in a web app:**

```python
from flask import Flask, send_file, make_response
import hashlib, os

app = Flask(__name__)

@app.route('/assets/<path:filename>')
def serve_asset(filename):
    """Serve static assets with CDN-friendly cache headers.
    Assets are content-hashed, so immutable caching is safe.
    """
    filepath = os.path.join('static', filename)

    response = make_response(send_file(filepath))

    # Immutable asset (hash in filename): cache forever
    if any(c in filename for c in ['.', '-'] and
           len(filename.split('.')[-2]) == 8):  # hash segment
        response.headers['Cache-Control'] = (
            'public, max-age=31536000, immutable'
        )
    else:
        response.headers['Cache-Control'] = (
            'public, max-age=3600'
        )

    return response

@app.route('/')
def index():
    """HTML must always be fresh (references hashed assets)."""
    response = make_response(render_index())
    response.headers['Cache-Control'] = 'no-cache'
    # CDN will revalidate; use ETag for conditional requests
    content = response.get_data()
    etag = hashlib.md5(content).hexdigest()
    response.headers['ETag'] = f'"{etag}"'
    return response

@app.route('/api/data')
def api_data():
    """API: bypass CDN or use short CDN-only cache."""
    response = make_response(get_data_json())
    # s-maxage: CDN caches for 60s; browser does not cache
    response.headers['Cache-Control'] = (
        'public, s-maxage=60, max-age=0'
    )
    return response
```

### ⚠️ Common Misconceptions

| Misconception                     | Reality                                                                                                                                                                                         |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| CDN is only for static files      | Modern CDNs cache API responses (with correct Cache-Control), terminate TLS, run edge compute, and proxy dynamic content — all reducing origin load and latency                                 |
| CDN cache invalidation is instant | Invalidation propagates to all PoPs in seconds but may take up to 1 minute. Immutable assets (hash-versioned) eliminate this problem entirely                                                   |
| CDN eliminates origin load        | On cache miss (new content, cache expired, non-cacheable), CDN fetches from origin. A cache stampede (many PoPs missing simultaneously) can spike origin load. Use CDN shield/mid-tier caching. |
| CDN is always expensive           | Cloudflare's free tier is extremely capable. AWS CloudFront costs ~$0.01/GB — for most applications, CDN cost is negligible compared to origin bandwidth/compute savings                        |

### 🚨 Failure Modes & Diagnosis

**Cache Poisoning: Incorrect Content Served from CDN**

**Symptom:**
Users see stale content after a deployment. Some users see new content, others see old.

```bash
# Check which PoP you're hitting and what version
curl -I https://cdn.example.com/app.js | grep -E "Age|X-Cache|ETag|Last-Modified"

# Force cache bypass (CDN-specific headers)
curl -H "Cache-Control: no-cache" https://cdn.example.com/app.js

# Check CloudFront headers
curl -I https://cdn.example.com/app.js | grep -i "cf-ray\|x-amz-cf"

# Invalidate specific paths
aws cloudfront create-invalidation \
  --distribution-id EDFD... \
  --paths "/app.*.js"

# Prevent recurrence: use content hashing in filenames
# app.js → app.abc123def456.js (changes on every build)
# CDN automatically caches new hash without invalidation
```

**Root Cause:** Long TTL on mutable files without cache-busting filenames. Fix: use content hashing for all JS/CSS assets; keep TTL short for HTML.
