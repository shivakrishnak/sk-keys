# 169 — UDP

⚡ TL;DR — UDP (User Datagram Protocol) is the internet's minimal transport protocol: it adds port numbers and a checksum to IP, but provides no reliability, ordering, or congestion control — offering the lowest possible latency at the cost of potential packet loss, used for DNS, video streaming, gaming, and VoIP.

---

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
TCP adds at least 1.5 RTT overhead before a single byte of data can be transferred (3-way handshake). For a DNS query, this means: 1 TCP SYN → 1 SYN-ACK → 1 ACK → 1 DNS query → 1 DNS response = 2.5 RTTs minimum, for a response that might be 50 bytes. TCP's retransmission is also counterproductive for real-time audio: if a voice packet is retransmitted 500ms later, it's too late to play — the conversation has moved on.

**THE BREAKING POINT:**
A video call is running. Your network drops 1% of packets. TCP's response: retransmit the dropped packet, hold back subsequent data while waiting. The call freezes and then jerks forward when the retransmit arrives. UDP's response: the dropped packet is gone. The codec interpolates the missing audio. The call continues smoothly with a brief artifact. Real-time delivery of newer data beats reliable delivery of older data.

**THE INVENTION MOMENT:**
David Reed designed UDP (RFC 768, 1980) as a minimal wrapper around IP that adds only multiplexing (port numbers) and optional error detection (checksum). The rest — reliability, ordering, congestion control — is left to the application. This enables applications to implement exactly the guarantees they need and no more. DNS uses UDP for single-packet query/response. Video calls use UDP because dropped frames are better than delayed frames. QUIC uses UDP as a substrate to implement a better version of TCP's features in user space.

---

### 📘 Textbook Definition

**UDP** (User Datagram Protocol) is a connectionless, unreliable, unordered transport protocol defined in RFC 768 (1980). A UDP datagram consists of a source port, destination port, length, checksum (optional for IPv4, mandatory for IPv6), and the payload. UDP makes no guarantees about: delivery (datagrams may be lost), ordering (datagrams may arrive out of order), duplication (datagrams may arrive multiple times), or congestion control (no rate adaptation). Each datagram is independent — there is no connection state. The UDP header is 8 bytes (vs 20+ for TCP).

---

### ⏱️ Understand It in 30 Seconds

**One line:**
UDP is TCP with everything stripped out except port numbers and a checksum — sending a datagram is fire-and-forget, no acknowledgement, no connection, no guarantee.

**One analogy:**

> UDP is a postcard dropped in a mailbox. No tracking, no confirmation, no guaranteed delivery. If the postcard gets lost, you'll never know. But postcards are fast and cheap — you don't need a registered mail form (TCP handshake) for "wish you were here." For a one-sentence message with an obvious response (DNS query), postcards are exactly right. For your tax return, use registered mail (TCP).

**One insight:**
"Unreliable" doesn't mean UDP is bad or broken. It means UDP puts reliability decisions in your hands. Modern applications (QUIC, WebRTC, games) build sophisticated reliability on top of UDP when they need it — but with application-specific trade-offs (e.g., "deliver the latest state, not every state").

---

### 🔩 First Principles Explanation

**CORE INVARIANTS:**

1. **No connection**: no handshake, no state — each datagram is independent.
2. **No reliability**: the network may drop, duplicate, or reorder datagrams.
3. **No congestion control**: UDP will saturate a link if the application doesn't self-limit.
4. **Preserve message boundaries**: one `sendto()` → one datagram → one `recvfrom()`. Unlike TCP streams, UDP preserves message boundaries.
5. **Best-effort checksum**: UDP checksum (optional in IPv4) covers header + payload. IP checksum covers only the IP header.

**UDP HEADER (8 bytes total):**

```
 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
|     Source      |   Destination   |
|      Port       |      Port       |
+--------+--------+--------+--------+
|                 |                 |
|     Length      |    Checksum     |
+--------+--------+--------+--------+
|          data octets ...
+--------------- ...
```

**When to choose UDP:**

| Use case             | Why UDP                                                                 | Reliability approach                           |
| -------------------- | ----------------------------------------------------------------------- | ---------------------------------------------- |
| DNS                  | Single query/response fits in one packet; response arrives or times out | Application: retry on timeout                  |
| VoIP / Video call    | Stale audio worse than missing audio                                    | Codec interpolation; no retransmit             |
| Live video streaming | Latency matters; viewer can see a glitch but not a freeze               | FEC (Forward Error Correction)                 |
| Online gaming        | State updates supersede each other; latest wins                         | Application-level sequence numbers             |
| QUIC (HTTP/3)        | Implement TCP reliability in user space                                 | QUIC implements its own ACK/retransmit         |
| NTP                  | Single packet time sync                                                 | Application: use statistical outlier rejection |
| DHCP                 | Broadcast before IP configured                                          | Application: retry                             |

**THE TRADE-OFFS:**
**Gain:** 0 RTT connection setup, 8-byte header overhead, no head-of-line blocking, application controls reliability strategy, works for broadcast/multicast (TCP is unicast only).
**Cost:** No reliability — application must handle loss; no congestion control — application can cause network congestion; message size limited to ~65KB (IP datagram limit).

### 🧪 Thought Experiment

**SETUP:**
Build a simple multiplayer game that sends player position updates 60 times per second. Should you use TCP or UDP?

**OPTION A — TCP:**

- Each position update is ACKed reliably
- Player moves → update sent → packet lost → TCP retransmits after RTO (~200ms) → subsequent updates blocked (head-of-line blocking) → player's movement freezes for 200ms then jumps forward
- Network latency: 50ms. Feels: 250ms (50ms + retransmit). Unplayable.

**OPTION B — UDP:**

- Each position update is a datagram with a sequence number
- Player moves → update sent → packet lost → next update arrives 16ms later with newer position
- Application logic: if sequence number < last received, discard (out-of-order). If gap detected, interpolate.
- 1 packet dropped out of 60: player position interpolated for 16ms. Barely noticeable.
- Network latency: 50ms. Feels: 50ms. Playable.

**APPLICATION-LEVEL RELIABILITY for UDP games:**

- Sequence numbers: discard stale updates
- Interpolation: smooth movement during packet loss
- Selective reliability: critical events (player death, score) sent with application-level ACK; position updates are unreliable

**THE INSIGHT:**
UDP forces the application to decide what level of reliability each message needs. This is more work to implement, but enables dramatically better user experience for latency-sensitive applications.

### 🧠 Mental Model / Analogy

> UDP is a loud announcement in a crowded room. You shout your message. Everyone nearby hears it (or doesn't — it's loud in here). You don't know who heard it or if they got the full message. You don't wait for confirmation. You just keep announcing. TCP, by contrast, is a direct phone call: you dial (handshake), wait for someone to pick up, speak, confirm they heard each sentence, and hang up properly. Phone calls are better for conversations. Shouting is better for real-time status updates where you don't care if someone missed one — they'll hear the next one in 16ms.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
UDP sends messages without waiting for confirmation that they arrived. It's faster than TCP because there's no back-and-forth, no setup time. The downside: messages might get lost or arrive out of order. Used for things where speed matters more than perfection: video calls, online games, and DNS lookups.

**Level 2 — How to use it (junior developer):**
UDP socket: `socket(AF_INET, SOCK_DGRAM, 0)`. No `connect()` needed (optional for convenience). Send: `sendto(sock, data, len, 0, &addr, addrlen)`. Receive: `recvfrom(sock, buf, len, 0, &src_addr, &addrlen)`. In Python: `sock.sendto(data, (host, port))` and `data, addr = sock.recvfrom(65535)`. Add a timeout: `sock.settimeout(2.0)`. One `sendto()` = one datagram; one `recvfrom()` = one datagram. No looping needed for message boundaries (unlike TCP).

**Level 3 — How it works (mid-level engineer):**
UDP datagrams are encapsulated directly in IP packets. The 8-byte UDP header is appended after the IP header. UDP preserves datagram boundaries because each datagram is a separate IP packet. Maximum payload: 65507 bytes (65535 - 8-byte UDP header - 20-byte IP header), but practical limit is the MTU (1500 bytes for Ethernet) minus headers = 1472 bytes, to avoid IP fragmentation. Larger UDP datagrams get fragmented at the IP layer — and if any fragment is lost, the entire datagram is discarded. For large payloads, either use TCP or implement application-level fragmentation. PMTUD (Path MTU Discovery) works differently for UDP — the DF bit can be set to discover path MTU and avoid fragmentation.

**Level 4 — Why it was designed this way (senior/staff):**
UDP's "unreliability" was a deliberate architectural choice reflecting the end-to-end principle: the network should not impose reliability on applications that don't need it. The internet researchers of the 1970s observed that different applications have radically different reliability requirements: DNS needs one-shot request/reply; NFS (network file system) needs reliable ordered delivery; VoIP needs low latency over reliability. Imposing TCP's reliability on all traffic would penalise applications that don't need it. UDP was designed as the "thin" transport — just ports and a checksum — letting applications layer exactly what they need. This is why modern protocols like QUIC, DTLS (Datagram TLS), and SRTP (Secure RTP for audio) are all built on UDP: they need reliability, security, or real-time semantics, but they want to implement them themselves with application-specific trade-offs.

### ⚙️ How It Works (Mechanism)

**UDP sockets in Python:**

```python
import socket

# SERVER: receive UDP datagrams
def udp_server(port: int):
    sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    sock.bind(('0.0.0.0', port))
    sock.settimeout(5.0)  # timeout for recvfrom
    print(f"UDP server listening on port {port}")
    while True:
        try:
            data, addr = sock.recvfrom(65535)
            print(f"Received {len(data)} bytes from {addr}")
            # Send response back to same addr
            sock.sendto(b"ACK: " + data[:20], addr)
        except socket.timeout:
            continue

# CLIENT: send UDP datagrams
def udp_client(host: str, port: int, message: str):
    sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    sock.settimeout(2.0)  # timeout for response

    try:
        # sendto: fire-and-forget
        sock.sendto(message.encode(), (host, port))

        # Optional: wait for response (application-level ACK)
        response, server_addr = sock.recvfrom(65535)
        print(f"Response: {response.decode()}")
    except socket.timeout:
        print("No response (datagram may be lost)")
    finally:
        sock.close()

# DNS query using UDP (manual implementation)
def dns_query(hostname: str, dns_server: str = '8.8.8.8'):
    """Send a minimal DNS A record query over UDP."""
    import struct

    # Build minimal DNS query packet
    txid = 0x1234
    flags = 0x0100  # Standard query, recursion desired
    questions = 1
    dns_header = struct.pack('!HHHHHH',
                             txid, flags, questions, 0, 0, 0)

    # Encode hostname as DNS labels
    query = b''
    for label in hostname.split('.'):
        query += bytes([len(label)]) + label.encode()
    query += b'\x00'  # root label
    query += struct.pack('!HH', 1, 1)  # Type A, Class IN

    sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    sock.settimeout(2.0)
    sock.sendto(dns_header + query, (dns_server, 53))

    response, _ = sock.recvfrom(512)
    sock.close()

    # Parse response (extract IP from answer section)
    rcode = response[3] & 0x0F
    ancount = struct.unpack('!H', response[6:8])[0]
    print(f"DNS response: RCODE={rcode}, answers={ancount}")
    return response
```

```
┌────────────────────────────────────────────────┐
│  DNS query over UDP                            │
└────────────────────────────────────────────────┘

       │
       ▼

       │

       │

       │
       ▼

       ▼

```

### 💻 Code Example

**Example — Game state synchronization over UDP:**

```python
import socket
import struct
import time
from collections import namedtuple

PlayerState = namedtuple('PlayerState', ['seq', 'x', 'y', 'timestamp'])

class GameSync:
    """UDP-based game state synchronisation.
    Uses sequence numbers to detect stale updates.
    Implements application-level ordering (not TCP ordering).
    """

    def __init__(self, local_port: int, remote_host: str, remote_port: int):
        self.sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
        self.sock.bind(('0.0.0.0', local_port))
        self.sock.setblocking(False)
        self.remote = (remote_host, remote_port)
        self.seq = 0
        self.last_received_seq = -1

    def send_state(self, x: float, y: float):
        """Send player position. No ACK expected."""
        self.seq += 1
        # Pack: seq(4B) x(4B) y(4B) timestamp(8B) = 20 bytes
        data = struct.pack('!IffQ',
                           self.seq,
                           x, y,
                           int(time.time() * 1000))
        try:
            self.sock.sendto(data, self.remote)
        except Exception:
            pass  # Drop silently — network error, try next frame

    def recv_state(self) -> PlayerState | None:
        """Receive state update. Discard stale/out-of-order."""
        try:
            data, addr = self.sock.recvfrom(1472)  # MTU-safe
        except BlockingIOError:
            return None  # No data available

        if len(data) < 20:
            return None  # Malformed packet

        seq, x, y, ts = struct.unpack('!IffQ', data[:20])

        # Discard stale: out-of-order packet arrived late
        if seq <= self.last_received_seq:
            print(f"Discarding stale packet: seq={seq} <= {self.last_received_seq}")
            return None

        self.last_received_seq = seq
        return PlayerState(seq=seq, x=x, y=y, timestamp=ts)

    def close(self):
        self.sock.close()
```

### ⚠️ Common Misconceptions

| Misconception                                | Reality                                                                                                                                                                          |
| -------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| UDP is unreliable and should be avoided      | UDP is deliberately minimal; applications that need reliability implement it themselves (QUIC, WebRTC, games) with application-specific trade-offs                               |
| UDP is always faster than TCP                | For a single large transfer, TCP with its congestion control may actually achieve similar throughput; UDP's advantage is in latency (no handshake) and flexibility               |
| UDP doesn't need a socket bind on the server | Server sockets must `bind()` to a port to receive incoming datagrams; only clients sending to a known server can omit `bind()` (OS assigns ephemeral port)                       |
| Large UDP datagrams are fine                 | UDP datagrams larger than the path MTU are fragmented by IP; if any fragment is lost, the entire datagram is discarded — effectively worse reliability than many small datagrams |
| UDP is only for real-time apps               | DNS, DHCP, NTP, SNMP, TFTP, syslog — many non-real-time protocols use UDP for simplicity and efficiency                                                                          |

### 🚨 Failure Modes & Diagnosis

**High UDP Packet Loss Causing Application Errors**

**Symptom:**
DNS timeouts, VoIP audio dropouts, or game lag spikes. `netstat -su` shows increasing `receive buffer errors`.

**Root Cause:**
Receive buffer overflow — application not reading UDP socket fast enough; kernel drops incoming datagrams when the receive buffer fills.

**Diagnostic Commands:**

```bash
# Check UDP receive buffer errors
netstat -su | grep -i "receive buffer errors\|packets dropped"

# Or from /proc
cat /proc/net/udp  # Last column: drops

# Check receive buffer size
sysctl net.core.rmem_default  # default SO_RCVBUF
sysctl net.core.rmem_max      # max SO_RCVBUF

# Increase global default
sysctl -w net.core.rmem_default=8388608   # 8MB

# Or per-socket in application:
sock.setsockopt(socket.SOL_SOCKET, socket.SO_RCVBUF, 8388608)

# Monitor drops in real time
watch -n 1 "netstat -su | grep -i 'errors\|drop'"
```

**Fix:**
Increase socket receive buffer; ensure application reads from socket faster (dedicated reader thread); use non-blocking socket with select()/poll().
