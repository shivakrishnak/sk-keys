# 191 — BGP

⚡ TL;DR — BGP (Border Gateway Protocol) is the routing protocol of the internet. It allows autonomous systems (ISPs, cloud providers, enterprises) to exchange IP prefix reachability information, enabling packets to find their way across the global internet. Every internet routing decision — including Anycast, CDN routing, and multi-homed cloud connectivity — ultimately depends on BGP.

---

### 🔥 The Problem This Solves

The internet consists of 100,000+ independently operated networks (ISPs, cloud providers, enterprises). Each network knows its own prefixes. How does a packet from a user's home network reach a server in a datacentre on another continent, passing through 10-15 intermediate networks that each need to agree on a path? BGP is the glue that connects the internet by distributing prefix reachability across all these networks.

---

### 📘 Textbook Definition

**BGP (Border Gateway Protocol):** An exterior gateway protocol (EGP) used to exchange routing information between autonomous systems (ASes) on the internet. Current version: BGP-4 (RFC 4271). Key properties: path-vector protocol (not just distance/cost, but the full AS path); policy-driven (operators can prefer, filter, manipulate routes); uses TCP port 179 (reliable delivery). Each BGP speaker knows the full AS path to reach every advertised prefix — enabling loop detection (an AS discards routes containing its own ASN in the path).

**Autonomous System (AS):** A collection of IP prefixes under a single routing policy, identified by an ASN (Autonomous System Number). Examples: Cloudflare AS13335, Google AS15169, AWS AS16509. ASNs are assigned by regional internet registries (ARIN, RIPE, APNIC).

**iBGP vs eBGP:** iBGP (internal BGP) — BGP sessions within the same AS (between routers in one network). eBGP (external BGP) — BGP sessions between different ASes (the "real" internet BGP).

---

### ⏱️ Understand It in 30 Seconds

**One line:**
BGP is the internet's routing protocol — it lets every network announce "I can reach these IP prefixes" so that packets from anywhere can find their way to the destination, hop by hop across thousands of independently-operated networks.

**One analogy:**
> BGP is like a global network of postal sorting offices, each sharing their routing directories. Each office (AS) tells its neighbours: "I can deliver mail to these zip codes (IP prefixes), and the path goes through offices X, Y, Z (AS path)." When you want to send a letter to a zip code, your local office knows the best chain of offices to route it through. An office can decline to route certain mail (BGP policy) or prefer certain routes over others (local preference).

### 🔩 First Principles Explanation

**BGP FUNDAMENTALS:**
```
AS Path (key concept):
  Packet wants to reach 8.8.8.0/24 (Google)
  
  Your ISP's BGP table entry for 8.8.8.0/24:
    Next-hop: 209.85.250.1  (peer router IP)
    AS Path: 15169           (one AS hop: Google's own AS)
    Origin: i (IGP)
    
  Alternative path through a different provider:
    Next-hop: 198.41.128.1
    AS Path: 174 15169       (2 AS hops: Cogent → Google)
    
  BGP prefers shorter AS path → routes via 15169 directly

BGP Route Selection (simplified, in priority order):
  1. Highest LOCAL_PREF (prefer customer > peer > upstream)
  2. Shortest AS_PATH length
  3. Lowest ORIGIN type (i=IGP > e=EGP > ?)
  4. Lowest MED (from neighbor)
  5. Prefer eBGP over iBGP
  6. Lowest IGP cost to next-hop
  7. Lowest router-id (tiebreaker)
```

**BGP PEERING TYPES:**
```
Transit (paid):
  Customer AS pays upstream provider for full internet routing table
  Provider: "I'll forward your traffic to anywhere on internet"
  Customer: "I'll accept your full routing table (700,000+ prefixes)"

Peering (settlement-free):
  Two networks exchange their customer routes only
  "I'll give you routes to my customers if you give me routes to yours"
  No money exchanged; at IXPs (Internet Exchange Points)
  
  IXP example: DE-CIX Frankfurt — 1000+ networks exchange traffic
  Without IXP: AS A → pay transit to reach AS B
  With IXP: AS A and AS B connect directly (no transit cost)

Customer:
  Small network connects to ISP (provider-customer relationship)
  Customer's prefixes announced to provider's full table
```

**BGP HIJACKING:**
```
Normal:
  AS13335 (Cloudflare) announces 1.1.1.0/24
  Internet routes 1.1.1.x traffic to Cloudflare

Hijack attack:
  Rogue AS announces 1.1.1.0/24 (same or more-specific prefix)
  If accepted: some internet traffic routes to rogue AS
  → traffic interception, blackholing, or impersonation

Protection: RPKI (Resource Public Key Infrastructure)
  IP address owners cryptographically sign which ASNs can announce their prefixes
  ROA (Route Origin Authorization): "AS13335 may announce 1.1.1.0/24"
  Routers validate: if announcement doesn't match ROA → reject (INVALID)
  BGP communities + ROV (Route Origin Validation) = defense against hijacking
```

**BGP IN CLOUD (AWS):**
```
AWS uses BGP for:
1. VPN (site-to-site): BGP dynamic routing over IPSec
   - Your router advertises on-prem prefixes to AWS via BGP
   - AWS VGW advertises VPC CIDRs to your router
   - Route propagation: auto-updates route tables

2. Direct Connect: dedicated link, BGP peering with AWS
   - Advertise prefixes to/from AWS

3. Transit Gateway: hub-and-spoke BGP internally in AWS
   - Multiple VPCs and VPN connections routed via TGW

Example: Direct Connect BGP config (Cisco)
  router bgp 65000
    neighbor 169.254.255.1 remote-as 7224  (AWS ASN)
    neighbor 169.254.255.1 activate
    network 192.168.0.0 mask 255.255.0.0  (advertise on-prem range)
```

### 🧪 Thought Experiment

**THE 2010 CHINA TELECOM BGP HIJACK:**
China Telecom (AS23724) accidentally (or intentionally) announced more-specific prefixes for 50,000+ IP blocks including US DoD, Dell, Apple, and others. Many global routes pointed traffic through China Telecom for 18 minutes. Impact: packet interception capability, service disruption. This is the BGP hijacking risk at scale — no cryptographic validation in BGP means anyone who can send BGP updates to a peer can redirect internet traffic. RPKI (now widely deployed) would have prevented this: the ROAs for those prefixes would mark China Telecom's announcements as INVALID, and routers with RPKI validation would reject them.

### 🧠 Mental Model / Analogy

> BGP is the world's largest bulletin board system. Each post office (AS) nails a notice: "I can deliver to ZIP codes 8.8.8.0-8.8.8.255, and the mail goes through me (AS15169) directly." Neighbouring offices read the board and forward: "I can also deliver to those ZIP codes via my neighbour AS15169." This ripples through the global postal network. Every office builds its routing table by reading all the notices, applying its own preferences (pay less, prefer shorter routes), and picking the best path. The bulletin board is never fully synchronised — it takes 30-90 seconds for a change to propagate worldwide.

---

### 📶 Gradual Depth — Four Levels

**Level 1:** BGP is the routing protocol that makes the internet work. Each network (ISP, Google, AWS) announces which IP ranges it owns. BGP routers exchange these announcements and build routing tables so that packets can travel from any network to any other network.

**Level 2:** AWS VPN uses BGP: your router peers with AWS on a private IP (169.254.x.x), exchanges routes. Your router advertises your office IP range; AWS advertises the VPC CIDR. Route tables in AWS automatically update via BGP route propagation. Key: BGP is dynamic — when you add a subnet, BGP announces it without manual route table updates.

**Level 3:** BGP attributes: AS_PATH (loop detection, path length for routing), LOCAL_PREF (internal preference — prefer customer routes over peer routes), MED (Multi-Exit Discriminator — hint to peer about preferred entry point), COMMUNITY (tags for policy: blackhole community 666:666 = DDoS blackhole). Route manipulation: AS path prepending (add your own AS to make path appear longer, reducing BGP preference — used to drain traffic from one link before maintenance). BGP convergence: after a failure, BGP withdrawals must propagate through the internet AS graph — typically 30-90 seconds for full convergence.

**Level 4:** BGP was designed in 1989 on a napkin (three engineers at a diner, literally on napkins — "three-napkin BGP"). It was designed for policy flexibility, not security — no authentication, no cryptographic origin validation. The internet is held together by trust and BGP communities. Modern defences: RPKI (Route Origin Validation), BGPsec (path signing, not widely deployed due to overhead), MANRS (Mutually Agreed Norms for Routing Security) — network operators committing to filtering and RPKI deployment. The fundamental tension: BGP needs to be flexible enough for complex routing policies but that flexibility creates attack surface.

### ⚙️ How It Works (Mechanism)

```bash
# View BGP routing information (public looking glasses)
# https://bgp.he.net/ - BGP route information
# https://lg.he.net/ - Hurricane Electric looking glass

# Check what AS owns an IP (public WHOIS)
whois 8.8.8.8 | grep -E "origin|route"
# route: 8.8.8.0/24
# origin: AS15169  ← Google's ASN

# Trace AS path to destination
traceroute -A 8.8.8.8  # -A flag shows AS numbers per hop
# 1. 192.168.1.1 (AS0)
# 2. 203.x.x.x (AS1234)  ← your ISP
# 3. 72.x.x.x (AS7018)   ← AT&T transit
# 4. 8.8.8.8 (AS15169)   ← Google

# Check RPKI validation for a prefix
# https://rpki.cloudflare.com/
# Or: https://stat.ripe.net/app/launchpad/

# AWS: check BGP session state
aws directconnect describe-virtual-interfaces \
  --query 'virtualInterfaces[*].[virtualInterfaceName,bgpPeers[*].[bgpStatus]]'

# Linux router: show BGP table (FRRouting / BIRD)
# FRRouting:
vtysh -c "show ip bgp summary"
vtysh -c "show ip bgp 8.8.8.0/24"
```

### 🔄 The Complete Picture — End-to-End Flow

```
BGP Route Propagation:

Google adds new prefix 2001:db8::/32
  → Google's routers announce to direct peers (Cloudflare, Akamai, etc.)
  → Peers accept (if policy allows), add to their BGP tables
  → Peers announce to their peers: "We can reach 2001:db8::/32 via AS15169"
  → Propagates across internet (30-90 seconds for full convergence)

Packet routing:
  User (ISP-A) → server 2001:db8::1
  ISP-A BGP table: 2001:db8::/32 via AS174 (Cogent) via AS15169 (Google)
  Router: forward to Cogent peering link
  Cogent BGP table: 2001:db8::/32 via AS15169 (direct peer)
  Router: forward to Google peering link
  Google receives packet, routes to server
```

### 💻 Code Example

```python
import subprocess
import re

def get_asn_for_ip(ip: str) -> dict:
    """Query WHOIS to find ASN and prefix for an IP address."""
    try:
        # Use RIPE NCC's whois service
        result = subprocess.run(
            ["whois", "-h", "whois.radb.net", ip],
            capture_output=True, text=True, timeout=10
        )
        output = result.stdout
        
        asn_match = re.search(r'origin:\s+(AS\d+)', output, re.IGNORECASE)
        route_match = re.search(r'route6?:\s+([^\n]+)', output, re.IGNORECASE)
        descr_match = re.search(r'descr:\s+([^\n]+)', output, re.IGNORECASE)
        
        return {
            "ip": ip,
            "asn": asn_match.group(1) if asn_match else "unknown",
            "prefix": route_match.group(1).strip() if route_match else "unknown",
            "description": descr_match.group(1).strip() if descr_match else "",
        }
    except Exception as e:
        return {"ip": ip, "error": str(e)}

# Examples
for ip in ["8.8.8.8", "1.1.1.1", "104.16.0.0"]:
    info = get_asn_for_ip(ip)
    print(f"{ip}: ASN={info.get('asn')} "
          f"prefix={info.get('prefix')} "
          f"({info.get('description', '')})")
# 8.8.8.8:   ASN=AS15169 prefix=8.8.8.0/24 (Google)
# 1.1.1.1:   ASN=AS13335 prefix=1.1.1.0/24 (Cloudflare)
# 104.16.x:  ASN=AS13335 prefix=104.16.0.0/13 (Cloudflare CDN)
```

### ⚖️ Comparison Table

| Aspect | eBGP | iBGP |
|---|---|---|
| Scope | Between different ASes | Within one AS |
| Purpose | Inter-domain routing | Distribute external routes internally |
| TTL | 1 (adjacent routers only) | 255 (any router in AS) |
| Next-hop | Updated at each AS boundary | Unchanged (loop prevention) |
| Full mesh | Not required | Required (or use route reflectors) |

---

### ⚠️ Common Misconceptions

| Misconception | Reality |
|---|---|
| BGP finds the fastest path | BGP finds the path that satisfies policy (LOCAL_PREF, customer > peer) then optimises for AS path length. Speed/latency is not a native BGP metric |
| BGP is only for internet providers | Cloud engineers use BGP for AWS VPN/Direct Connect, AWS Transit Gateway BGP routing, and Anycast services. BGP is increasingly a cloud engineer skill |
| BGP changes are instantaneous | BGP convergence takes 30-90 seconds for full internet propagation. AWS VPN BGP changes: 60-300 seconds. Plan for this in change management |

### 🚨 Failure Modes & Diagnosis

**BGP Route Leak: Full Internet Routing Table Accepted from Customer**

```bash
# Symptoms: router CPU spike, memory exhaustion
# Route leak: customer sends you their full internet table instead of just their prefixes

# Check: how many prefixes are you receiving?
vtysh -c "show ip bgp summary" | grep "State/PfxRcd"
# Should be ~5 prefixes from customer; if 700,000+ → route leak!

# Immediate fix: apply inbound prefix filter
# Only accept customer's announced prefixes (their ASN's addresses)

# FRRouting prefix-list (allow only customer's /24):
vtysh << 'EOF'
conf t
ip prefix-list CUSTOMER-IN permit 192.168.100.0/24
ip prefix-list CUSTOMER-IN deny 0.0.0.0/0 le 32

router bgp 65000
  neighbor 198.51.100.1 prefix-list CUSTOMER-IN in
EOF

# AWS BGP: limit prefixes from VPN
# Max-prefix setting in VPN connection config
# If customer advertises > N prefixes → BGP session torn down automatically

# Prevention: always apply prefix-lists AND max-prefix limits on all BGP sessions
```
