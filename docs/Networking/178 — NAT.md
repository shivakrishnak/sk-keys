# 178 — NAT

⚡ TL;DR — NAT (Network Address Translation) translates private IP addresses to public ones as packets traverse a gateway, enabling thousands of devices to share a single public IP; AWS uses NAT Gateways for outbound internet access from private subnets, and Kubernetes uses SNAT/DNAT for pod-to-service routing via kube-proxy or eBPF.

---

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
Every internet-connected device needs a globally unique IP address. IPv4 provides only ~4.3 billion addresses — far fewer than the world's devices. Without NAT, each smartphone, laptop, and IoT device would require a dedicated public IP, exhausting the IPv4 space decades ago.

**THE BREAKING POINT:**
By the mid-1990s, ISPs were running out of public IP address blocks to assign to customers. A household with 5 devices would need 5 public IPs. With millions of households, the math didn't work. IPv6 was the long-term fix, but deployable in the 1990s.

**THE INVENTION MOMENT:**
NAT (RFC 1631, 1994) allows a single public IP to be shared by many private hosts. The NAT gateway maintains a **connection tracking table**: when an internal host (192.168.1.5:54321) connects to an external server (8.8.8.8:443), the NAT gateway rewrites the source to (203.0.113.1:32000) and records the mapping. Return packets matching (203.0.113.1:32000) are translated back to (192.168.1.5:54321). From the external server's perspective, all traffic appears to come from the single public IP 203.0.113.1.

---

### 📘 Textbook Definition

**NAT** (Network Address Translation) is the process of modifying IP address information in packet headers while in transit. Types: **SNAT** (Source NAT — change source IP/port, used for outbound traffic from private to public), **DNAT** (Destination NAT — change destination IP/port, used for inbound traffic to services, e.g., port forwarding), **PAT/NAPT** (Port Address Translation / NAT overloading — many-to-one mapping using different source ports, the most common form). NAT maintains a **connection tracking table** mapping (src-IP, src-port, dst-IP, dst-port) → (translated-src-IP, translated-src-port). Stateful: the NAT gateway must see both outbound and return traffic.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
NAT is a translation layer at the network boundary: it swaps private IP addresses for public ones (SNAT) and routes return packets back to the correct private host by maintaining a connection table.

**One analogy:**

> NAT is like a company's receptionist. When any employee makes an outbound call, the switchboard replaces their internal extension (192.168.1.x) with the company's single public number (203.0.113.1) and keeps a log of which extension made which call. When a return call comes in, the switchboard looks up the log and routes it to the right extension. External callers see only the company number — not the individual employees.

**One insight:**
NAT fundamentally breaks the internet's end-to-end principle: two hosts behind different NATs cannot directly connect to each other without a rendezvous mechanism (STUN, TURN, ICE). This is why WebRTC needs ICE servers and why gaming uses relay servers — two players behind NAT can't directly open a connection to each other.

### 🔩 First Principles Explanation

**SNAT (Source NAT) — Outbound:**

```
BEFORE NAT (private → internet):
  src=192.168.1.5:54321 dst=8.8.8.8:443

NAT Gateway (203.0.113.1):
  Records: (192.168.1.5:54321, 8.8.8.8:443) → 203.0.113.1:32000
  Rewrites: src=203.0.113.1:32000

AFTER NAT:
  src=203.0.113.1:32000 dst=8.8.8.8:443

Return packet:
  src=8.8.8.8:443 dst=203.0.113.1:32000

NAT lookup: 203.0.113.1:32000 → 192.168.1.5:54321
Rewrite: dst=192.168.1.5:54321

Delivered to: 192.168.1.5:54321 (correct host)
```

**DNAT (Destination NAT) — Inbound / Port Forwarding:**

```
External client → 203.0.113.1:80 (public IP, port 80)

DNAT rule: dst 203.0.113.1:80 → 192.168.1.10:8080

NAT Gateway rewrites:
  dst=192.168.1.10:8080

Internal server at 192.168.1.10:8080 receives request.
Response: src=192.168.1.10:8080 → SNAT → src=203.0.113.1:80
```

**NAT IN CLOUD INFRASTRUCTURE:**

_AWS NAT Gateway:_

```
Private subnet (10.0.1.0/24)
  EC2 instance (10.0.1.50) → NAT Gateway (10.0.2.1) → IGW → Internet

Route table for private subnet:
  0.0.0.0/0 → nat-gateway-id

Route table for public subnet (where NAT GW lives):
  0.0.0.0/0 → igw-id
```

_Kubernetes kube-proxy (iptables) DNAT:_

```
Service ClusterIP: 10.96.0.10:80 (virtual IP)
Backing pods:      10.244.1.5:8080, 10.244.2.5:8080

kube-proxy sets up DNAT:
  dst=10.96.0.10:80 → randomly select 10.244.1.5:8080 or 10.244.2.5:8080

This is how Kubernetes services work: ClusterIP is a DNAT rule.
```

**CONNTRACK (Linux Connection Tracking):**
The kernel module tracking NAT state. `conntrack -L` shows all tracked connections. If the conntrack table fills up (default 65K entries), new connections fail. Tune: `net.netfilter.nf_conntrack_max`.

### 🧪 Thought Experiment

**SETUP:**
Two users on separate home networks (both using 192.168.1.5) try to play a peer-to-peer game.

**PROBLEM:**
Player A: 192.168.1.5 (behind NAT, public IP: 1.2.3.4)
Player B: 192.168.1.5 (behind NAT, public IP: 5.6.7.8)

Player A tries to connect to Player B's "192.168.1.5" — this goes to Player A's own machine (local network). Player A tries to connect to 5.6.7.8 (Player B's public IP) on a specific port. Player B's NAT has no DNAT rule for that port — the connection is dropped.

**NAT TRAVERSAL SOLUTIONS:**

1. **UPnP**: router automatically creates port forwarding rules for games.
2. **STUN**: both players contact a STUN server to discover their public IP:port; exchange this info; attempt direct connection (works if both NATs support endpoint-independent mapping).
3. **TURN** (relay): if direct fails, both connect to a TURN relay server; TURN proxies traffic between them.
4. **ICE** (Interactive Connectivity Establishment): WebRTC's protocol that tries all of the above in order.

### 🧠 Mental Model / Analogy

> NAT is like a P.O. Box for an entire neighbourhood. All residents use P.O. Box 1000 (the public IP) as their address. The post office (NAT gateway) maintains a private log of who's expecting which package. When mail arrives addressed to "P.O. Box 1000, for Order #32000," the post office looks up its log: Order #32000 was placed by apartment 1.5 — deliver there. No outsider knows apartment 1.5 exists. DNAT is the post office also accepting walk-in deliveries and routing them to the right apartment based on what they're delivering (destination port → internal server).

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
NAT lets many devices share one public IP address. Your home router does this — all your devices (phone, laptop, TV) use private addresses (192.168.x.x), but to the internet they all appear to come from your router's single public IP. The router keeps track of which device made which connection so it can route replies back correctly.

**Level 2 — How to use it (junior developer):**
In AWS: private subnets (no internet gateway route) need a NAT Gateway in a public subnet to reach the internet (e.g., for package downloads). NAT Gateway is one-directional: outbound only — internet cannot initiate connections to private instances. For inbound: use a load balancer or internet gateway with a public subnet. Cost: AWS NAT Gateway charges per hour + per GB. Alternative: NAT instance (cheaper, but you manage it). In Kubernetes: use `kubectl exec` to check pod connectivity; pods in private nodes need a NAT mechanism (cloud NAT gateway or kube-proxy SNAT).

**Level 3 — How it works (mid-level engineer):**
Linux NAT uses iptables/nftables with MASQUERADE (dynamic SNAT) or SNAT (static) targets. Connection tracking (`nf_conntrack`) maintains a table of all active connections. Each entry expires after idle timeout (default 30s UDP, 5 days for ESTABLISHED TCP). Conntrack table overflow is a common production issue at scale. The NAT table entry uses a 5-tuple for uniqueness: (proto, src-IP, src-port, dst-IP, dst-port). With PAT (Port Address Translation), the NAT gateway assigns a new source port to differentiate connections. Maximum simultaneous connections with 1 public IP: ~65,535 per destination IP:port combination (limited by 16-bit port range).

**Level 4 — Why it was designed this way (senior/staff):**
NAT was designed as a temporary fix for IPv4 exhaustion but has persisted for 30+ years. It violates the internet's end-to-end principle: hosts behind NAT are not directly addressable, requiring application-level workarounds (STUN/TURN/ICE for WebRTC, relay servers for gaming, HTTP proxies for filtered networks). NAT also complicates security: it hides internal topology (security advantage) but makes it impossible to track which internal host made a connection without NAT logs (compliance disadvantage). Carrier-Grade NAT (CGN/LSNAT) adds a second NAT layer at the ISP level, breaking even more protocols. IPv6 + global addressing is the only real fix, but deployment is 30+ years in progress.

### ⚙️ How It Works (Mechanism)

```bash
# View Linux NAT rules (iptables)
iptables -t nat -L -n -v
# PREROUTING: DNAT rules (inbound translation)
# POSTROUTING: SNAT/MASQUERADE rules (outbound translation)

# Set up MASQUERADE (SNAT with dynamic public IP)
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

# Set up port forwarding (DNAT)
iptables -t nat -A PREROUTING -i eth0 -p tcp --dport 80 \
  -j DNAT --to-destination 192.168.1.10:8080

# Check conntrack table
conntrack -L
conntrack -L --proto tcp --state ESTABLISHED
# Count active connections
conntrack -C

# Check conntrack table limits
sysctl net.netfilter.nf_conntrack_max       # default ~65536
sysctl net.netfilter.nf_conntrack_count     # current usage

# Tune conntrack table size (for high-traffic NAT)
sysctl -w net.netfilter.nf_conntrack_max=1048576

# AWS: Check NAT Gateway metrics (CloudWatch)
# - ActiveConnectionCount
# - ConnectionAttemptCount
# - ConnectionEstablishedCount
# - PacketDropCount (conntrack overflow indicator)
```

### 🔄 The Complete Picture — End-to-End Flow

```
┌──────────────────────────────────────────────────┐
│  AWS Private Subnet → Internet via NAT Gateway   │
└──────────────────────────────────────────────────┘

Private EC2 (10.0.1.50)
  → Route table: 0.0.0.0/0 → nat-gw-id
  → Packet: src=10.0.1.50:54321, dst=1.2.3.4:443

NAT Gateway (10.0.2.1, EIP: 54.200.x.x)
  → SNAT: src=54.200.x.x:32100 (records mapping)
  → Packet: src=54.200.x.x:32100, dst=1.2.3.4:443
  → Route table: 0.0.0.0/0 → igw-id

Internet Gateway → Internet
  ← Response: src=1.2.3.4:443, dst=54.200.x.x:32100

NAT Gateway
  ← Lookup: 32100 → 10.0.1.50:54321
  ← SNAT reverse: dst=10.0.1.50:54321
  ← Delivers to private EC2
```

### 💻 Code Example

```python
import subprocess
import re

def get_nat_connections(nat_ip: str = None) -> list[dict]:
    """Parse conntrack table for NAT connections (Linux)."""
    result = subprocess.run(
        ['conntrack', '-L', '--proto', 'tcp'],
        capture_output=True, text=True
    )

    connections = []
    for line in result.stdout.splitlines():
        # Parse conntrack output: tcp ESTABLISHED src=x dst=y sport=N dport=N ...
        if 'ESTABLISHED' not in line:
            continue

        # Extract the original direction (pre-NAT)
        src = re.search(r'src=(\S+)', line)
        dst = re.search(r'dst=(\S+)', line)
        sport = re.search(r'sport=(\d+)', line)
        dport = re.search(r'dport=(\d+)', line)

        if src and dst and sport and dport:
            connections.append({
                'src': src.group(1),
                'dst': dst.group(1),
                'sport': int(sport.group(1)),
                'dport': int(dport.group(1)),
            })

    if nat_ip:
        connections = [c for c in connections
                      if c['src'] == nat_ip or c['dst'] == nat_ip]

    return connections

# Report NAT connection counts by destination
from collections import Counter
conns = get_nat_connections()
by_dst = Counter(c['dst'] for c in conns)
for dst, count in by_dst.most_common(10):
    print(f"{dst}: {count} connections")
```

### ⚠️ Common Misconceptions

| Misconception                       | Reality                                                                                                                                                                                               |
| ----------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| NAT provides firewall security      | NAT blocks unsolicited inbound by default, but that's a side-effect, not the purpose. A stateful firewall with no NAT provides the same inbound block. Don't rely on NAT as your only security layer. |
| NAT Gateway is free in AWS          | NAT Gateway costs ~$0.045/hour + $0.045/GB processed. High-traffic services pay significant amounts. Consider VPC endpoints for S3/DynamoDB traffic to avoid NAT costs.                               |
| NAT supports all protocols          | NAT breaks protocols that embed IP addresses in the payload (FTP active mode, SIP, IPsec AH). These require ALG (Application Layer Gateway) helpers.                                                  |
| Conntrack table size doesn't matter | At scale (100K+ connections), conntrack table overflow causes new connections to fail silently. Monitor `nf_conntrack_count` vs `nf_conntrack_max`.                                                   |

### 🚨 Failure Modes & Diagnosis

**Connection Failures: Conntrack Table Overflow**

**Symptom:**
New TCP connections fail intermittently. Error: `nf_conntrack: table full, dropping packet`. Occurs on high-traffic NAT gateways (self-managed, not AWS NAT GW).

```bash
# Check current conntrack usage
sysctl net.netfilter.nf_conntrack_count
sysctl net.netfilter.nf_conntrack_max

# Check for dropped packets due to conntrack overflow
dmesg | grep nf_conntrack
# "nf_conntrack: table full, dropping packet"

# Increase conntrack table size
sysctl -w net.netfilter.nf_conntrack_max=2097152
echo "net.netfilter.nf_conntrack_max = 2097152" >> /etc/sysctl.conf

# Reduce conntrack timeouts for UDP/idle TCP
sysctl -w net.netfilter.nf_conntrack_udp_timeout=30
sysctl -w net.netfilter.nf_conntrack_tcp_timeout_established=3600
```
