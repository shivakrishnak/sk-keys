# 185 — VPN

⚡ TL;DR — A VPN (Virtual Private Network) creates an encrypted tunnel between two endpoints over an untrusted network (the internet), making the remote machine appear to be on the local network — used for secure remote access, site-to-site connectivity, and (increasingly replaced by Zero Trust) as a perimeter security model.

---

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
A remote employee needs to access an internal company database at 10.0.1.5 (not internet-facing). The database can't be publicly exposed (security). The only option: fly the employee to the office, or move the database to the internet (insecure). For thousands of remote employees, neither works.

**THE BREAKING POINT:**
Pre-2000s: corporate networks were physical. Security was the office walls. Remote access meant a modem dial-up. Cloud and globalisation changed this: employees work from anywhere; branch offices need to communicate with HQ; engineers access production systems from laptops in airports. Sending database queries over raw internet: packets visible to any network between you and the server.

**THE INVENTION MOMENT:**
VPN tunneling: wrap every packet from the remote client in an encrypted envelope addressed to the VPN server. The VPN server unwraps the packet and routes it on the internal network. To the database, the packet came from the VPN server's internal IP — the client appears local. To the internet, only encrypted UDP/TCP packets to the VPN server IP are visible. The result: an employee in Tokyo can securely access a database in London as if they were sitting in the London office.

---

### 📘 Textbook Definition

A **VPN** (Virtual Private Network) is a technology that creates a secure, encrypted tunnel between a client and a VPN server (or between two networks) over an untrusted network (typically the internet). Types:

1. **Remote access VPN:** Individual client connects to corporate network; client gets a virtual private IP in the corporate IP range.
2. **Site-to-site VPN:** Two networks (e.g., AWS VPC ↔ on-premise datacenter) connected via persistent VPN tunnel; traffic between subnets routed transparently.
3. **Consumer VPN:** Encrypts traffic between user and VPN provider; hides activity from ISP; exits at VPN provider's IP.

Common protocols: **IPSec** (industry standard, complex, hardware-accelerated), **OpenVPN** (TLS-based, widely compatible, slower), **WireGuard** (modern, minimal, very fast), **SSL VPN / TLS VPN** (browser-based, port 443).

---

### ⏱️ Understand It in 30 Seconds

**One line:**
A VPN creates an encrypted tunnel through the internet that makes your remote computer appear to be physically present on a private network — all traffic between you and the VPN server is encrypted and invisible to anyone on the path.

**One analogy:**

> A VPN is like a secure glass tube (the tunnel) running through a busy public street (the internet). Your message travels through the tube — people on the street can see the tube exists, but can't see what's inside. At the other end, the message exits the tube into the private building (corporate network). The building's security doesn't see a visitor arriving from a street — it sees someone stepping in from the internal hallway.

**One insight:**
A VPN doesn't make you anonymous — the VPN server sees all your traffic. "Consumer VPNs for privacy" just shift your trust from your ISP to the VPN provider. For corporate use, the VPN server is a trust boundary: once inside, the traditional model trusts everything on the internal network — which is exactly what Zero Trust challenges (if an attacker gets VPN credentials, they're "inside" the trusted network with lateral movement capability).

### 🔩 First Principles Explanation

**HOW VPN TUNNELING WORKS:**

```
┌─────────────────────────────────────────────────────────┐
│  Remote Access VPN: Client → VPN Server → Internal Net  │
└─────────────────────────────────────────────────────────┘

Without VPN:
  Client (5.5.5.5) → [internet] → db.internal (NOT reachable)
  Client sends: src=5.5.5.5 dst=10.0.1.5 (private, non-routable)
  → Internet routers drop it (10.0.0.0/8 not routed publicly)

With VPN:
  1. Client establishes encrypted tunnel to VPN server (192.0.2.1)
  2. Client gets virtual IP: 10.0.100.5 (in corporate subnet)
  3. Client sends packet: src=10.0.100.5 dst=10.0.1.5
  4. VPN client encrypts packet, wraps in UDP/TCP to VPN server:
     Outer: src=5.5.5.5 dst=192.0.2.1 (public VPN server IP)
     Inner: encrypted(src=10.0.100.5 dst=10.0.1.5)
  5. VPN server receives outer packet
  6. VPN server decrypts inner packet
  7. VPN server routes inner packet on internal network
     → forwarded to 10.0.1.5 (database)
  8. Database responds: src=10.0.1.5 dst=10.0.100.5
  9. VPN server encrypts, wraps, sends to 5.5.5.5
 10. Client decrypts, delivers to application

Internet observer sees: encrypted packets between 5.5.5.5 and 192.0.2.1
Cannot determine: destination (10.0.1.5), payload (SQL query)
```

**WIREGUARD ARCHITECTURE:**

```
WireGuard: modern VPN protocol (kernel module in Linux 5.6+)
  - Uses Curve25519 key exchange, ChaCha20 encryption, Poly1305 MAC
  - ~4000 lines of code (vs 400,000 for OpenVPN/IPSec)
  - UDP-based, typically port 51820
  - Peers identified by public key (not IP, not certificates)
  - Roaming: client IP can change, connection persists (by public key)

# WireGuard config (wg0.conf on server)
[Interface]
PrivateKey = <server-private-key>
Address = 10.0.0.1/24
ListenPort = 51820

[Peer]
PublicKey = <client-public-key>
AllowedIPs = 10.0.0.2/32   # client's VPN IP

# WireGuard config (client)
[Interface]
PrivateKey = <client-private-key>
Address = 10.0.0.2/24
DNS = 10.0.0.1

[Peer]
PublicKey = <server-public-key>
Endpoint = 192.0.2.1:51820
AllowedIPs = 10.0.0.0/24   # route this subnet through VPN
PersistentKeepalive = 25
```

**SPLIT TUNNELING:**

```
Full tunnel: ALL client traffic goes through VPN
  → Corporate traffic: through VPN tunnel (encrypted)
  → Netflix, YouTube: also through VPN (slow, expensive for org)
  AllowedIPs = 0.0.0.0/0  (route everything)

Split tunnel: ONLY corporate traffic goes through VPN
  → Corporate traffic (10.0.0.0/8): through VPN
  → Internet traffic: directly to internet (no VPN overhead)
  AllowedIPs = 10.0.0.0/8  (route only corporate subnet)

Security trade-off: full tunnel = more secure (all traffic inspected
by corporate firewall), split tunnel = better performance.
```

**SITE-TO-SITE VPN (AWS):**

```
AWS VPC (10.0.0.0/16) ↔ On-premise (192.168.0.0/16)

AWS Side:
  - Virtual Private Gateway attached to VPC
  - Route table: 192.168.0.0/16 → vgw-xxxxx

On-premise Side:
  - Customer Gateway (router/firewall with public IP)
  - IPSec tunnel to AWS VGW

Result:
  EC2 instance at 10.0.1.5 can ping on-prem server at 192.168.1.10
  Traffic: encrypted IPSec tunnel
  Bandwidth: up to 1.25 Gbps per tunnel
  HA: two tunnels per connection (different AWS AZs)
```

### 🧪 Thought Experiment

**SETUP:**
A startup has 50 engineers, all remote. They need to access internal services: Jenkins (CI/CD), internal Grafana, PostgreSQL (direct access for DBA), internal wikis. Compare: traditional VPN vs Zero Trust.

**TRADITIONAL VPN MODEL:**

- All 50 engineers connect to VPN
- Once connected: access to entire internal network
- If Alice's laptop is compromised and VPN credentials stolen: attacker has access to Jenkins, Grafana, PostgreSQL, everything
- VPN server = single point of failure
- Performance: all traffic goes through VPN server

**ZERO TRUST MODEL:**

- No VPN (or VPN only for specific infra access)
- Each service is independently accessible from internet via HTTPS
- Authentication: SSO + MFA for every service
- Authorisation: Alice can access Jenkins and Grafana (her role), but NOT PostgreSQL (only DBAs)
- Device posture: must have corp-managed device with MDM enrolled
- If Alice's credentials stolen: attacker only gets her specific permissions
- No lateral movement: compromising Jenkins doesn't grant PostgreSQL access

**CONCLUSION:**
VPN is a perimeter model (all-or-nothing trust inside the network). Zero Trust is a per-resource model (authenticate and authorise each access individually). The trend is moving from VPN toward Zero Trust — but VPN remains practical for small teams, infrastructure access, and site-to-site cloud connectivity.

### 🧠 Mental Model / Analogy

> A VPN is like a secure diplomatic pouch. A diplomat (client) needs to send sensitive documents to their embassy (internal network) through foreign territory (the internet). The documents are sealed in an encrypted diplomatic pouch and given a new outer envelope addressed to the border crossing (VPN server). Foreign customs (network observers) can see the outer envelope going to the border crossing but cannot open the diplomatic pouch. At the border, the pouch is unsealed and the documents are delivered via the internal embassy courier (private network routing).

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
A VPN creates a secret tunnel through the internet. When you use one, your internet traffic is encrypted (no one can read it) and sent through the VPN server. Your computer appears to be in the same location as the VPN server. Companies use VPNs so employees working from home can access company systems as if they were in the office.

**Level 2 — How to use it (junior developer):**
Set up WireGuard (modern, fast): install `wireguard` package, generate key pairs (`wg genkey | tee private.key | wg pubkey > public.key`), create config file, run `wg-quick up wg0`. For AWS site-to-site: configure Virtual Private Gateway, Customer Gateway, VPN Connection in the console — AWS provides config templates for common routers (Cisco, Palo Alto, pfSense). For Kubernetes: don't use VPN for pod-to-pod — use CNI (Calico, Cilium). Use VPN for external access to the Kubernetes API server.

**Level 3 — How it works (mid-level engineer):**
IPSec internals: two protocols — AH (Authentication Header, integrity only) and ESP (Encapsulating Security Payload, encryption + integrity). Two modes: transport (encrypts payload only, L4+) and tunnel (encrypts entire original packet, creates new IP header — used for site-to-site VPN). IKE (Internet Key Exchange) v2 handles key negotiation: uses X.509 certificates or pre-shared keys for authentication, Diffie-Hellman for key agreement, AES-GCM for encryption. WireGuard is simpler: no IKE, no certificates (just public keys), handshake uses Noise protocol framework. VPN routing: a virtual network interface (tun0/wg0) is created; routing rules send specific subnets through it; the VPN daemon reads from the tun interface, encrypts, sends via UDP.

**Level 4 — Why it was designed this way (senior/staff):**
IPSec was designed for host-to-host and network-to-network security at the IP layer — it works transparently below the application layer. This was the 1990s approach: secure the network, and all applications benefit without code changes. The complexity comes from flexibility — IPSec supports dozens of algorithms, two protocols, multiple modes, and certificate-based authentication for enterprise PKI integration. WireGuard deliberately rejects this flexibility: one cipher suite (Curve25519 + ChaCha20-Poly1305 + BLAKE2), no certificate management, pure public key identity. This makes WireGuard cryptographically agile-free but far simpler to audit (4K vs 400K lines). The Zero Trust architecture challenge to VPN is architectural, not cryptographic: VPN provides network-level access; Zero Trust provides per-resource access with identity-aware proxies (BeyondCorp, Cloudflare Access, Tailscale).

### ⚙️ How It Works (Mechanism)

```bash
# WireGuard setup (server)
# Install
apt-get install -y wireguard

# Generate key pair
wg genkey | tee /etc/wireguard/private.key | \
  wg pubkey > /etc/wireguard/public.key
chmod 600 /etc/wireguard/private.key

# Server config
cat > /etc/wireguard/wg0.conf << 'EOF'
[Interface]
PrivateKey = $(cat /etc/wireguard/private.key)
Address = 10.8.0.1/24
ListenPort = 51820
PostUp = iptables -A FORWARD -i wg0 -j ACCEPT; \
  iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
PostDown = iptables -D FORWARD -i wg0 -j ACCEPT; \
  iptables -t nat -D POSTROUTING -o eth0 -j MASQUERADE

[Peer]
# Add each client peer
PublicKey = <client-public-key>
AllowedIPs = 10.8.0.2/32
EOF

# Start WireGuard
wg-quick up wg0
systemctl enable wg-quick@wg0

# Check status
wg show

# Client connection test
ping 10.8.0.1

# View current connected peers
wg show wg0 peers
wg show wg0 transfer
```

### 🔄 The Complete Picture — End-to-End Flow

```
┌──────────────────────────────────────────────────────┐
│  WireGuard Packet Flow                               │
└──────────────────────────────────────────────────────┘

Client (5.5.5.5, VPN IP: 10.8.0.2)
  → App sends TCP packet: src=10.8.0.2 dst=10.0.1.5
  → Routing: 10.0.0.0/8 → wg0 interface
  → WireGuard: encrypt with shared session key (ChaCha20)
  → Wrap in UDP: src=5.5.5.5:54321 dst=192.0.2.1:51820
  → [internet: only sees UDP port 51820, encrypted payload]

VPN Server (192.0.2.1, VPN IP: 10.8.0.1)
  → Receives UDP packet
  → Authenticates peer (by public key)
  → Decrypts inner packet: src=10.8.0.2 dst=10.0.1.5
  → Routes to 10.0.1.5 (internal network)
  → [network firewall sees src=10.8.0.2 — VPN client range]

Database (10.0.1.5)
  → Receives packet from 10.8.0.2
  → Responds: src=10.0.1.5 dst=10.8.0.2
  → VPN server intercepts (routes for 10.8.0.0/24 → wg0)
  → Encrypts, sends to 5.5.5.5:54321
  → Client decrypts, delivers to app
```

### 💻 Code Example

```python
import subprocess
import ipaddress
from pathlib import Path

def generate_wireguard_config(
    server_ip: str,
    server_port: int,
    vpn_subnet: str,
    n_clients: int
) -> dict:
    """Generate WireGuard server and client configs.
    Returns dict with server_config and list of client_configs.
    """
    def gen_keypair() -> tuple[str, str]:
        priv = subprocess.check_output(
            ["wg", "genkey"], text=True
        ).strip()
        pub = subprocess.run(
            ["wg", "pubkey"], input=priv, capture_output=True, text=True
        ).stdout.strip()
        return priv, pub

    net = ipaddress.IPv4Network(vpn_subnet, strict=False)
    hosts = list(net.hosts())

    server_priv, server_pub = gen_keypair()
    server_vpn_ip = str(hosts[0])

    # Build server config
    server_config = f"""[Interface]
PrivateKey = {server_priv}
Address = {server_vpn_ip}/{net.prefixlen}
ListenPort = {server_port}
PostUp = iptables -A FORWARD -i wg0 -j ACCEPT; \\
  iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
PostDown = iptables -D FORWARD -i wg0 -j ACCEPT; \\
  iptables -t nat -D POSTROUTING -o eth0 -j MASQUERADE
"""

    client_configs = []
    for i in range(1, n_clients + 1):
        client_priv, client_pub = gen_keypair()
        client_vpn_ip = str(hosts[i])

        # Add peer to server config
        server_config += f"""
[Peer]
# Client {i}
PublicKey = {client_pub}
AllowedIPs = {client_vpn_ip}/32
"""

        # Client config
        client_config = f"""[Interface]
PrivateKey = {client_priv}
Address = {client_vpn_ip}/{net.prefixlen}
DNS = {server_vpn_ip}

[Peer]
PublicKey = {server_pub}
Endpoint = {server_ip}:{server_port}
AllowedIPs = {vpn_subnet}
PersistentKeepalive = 25
"""
        client_configs.append({
            "id": i,
            "vpn_ip": client_vpn_ip,
            "config": client_config
        })

    return {
        "server_config": server_config,
        "server_vpn_ip": server_vpn_ip,
        "clients": client_configs
    }
```

### ⚠️ Common Misconceptions

| Misconception                              | Reality                                                                                                                                                                            |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| VPN = anonymity                            | The VPN provider sees all your traffic. You shift trust from ISP to VPN provider. VPNs are for security (encryption), not anonymity.                                               |
| VPN tunnels protect against all attacks    | VPN encrypts transit traffic. Malware on your endpoint can still exfiltrate data through the VPN tunnel. Application-layer attacks (SQLi, XSS) are not affected by VPN             |
| Site-to-site VPN gives unlimited bandwidth | AWS VPN tunnels max at 1.25 Gbps per tunnel. AWS Direct Connect (dedicated link, not VPN) offers 1-100 Gbps without the overhead                                                   |
| VPN is being replaced by Zero Trust        | They're complementary. Zero Trust replaces VPN for user-to-application access. Site-to-site VPN for network-to-network connectivity (VPC↔on-premise) is not replaced by Zero Trust |

### 🚨 Failure Modes & Diagnosis

**VPN Split-Tunnel DNS Leak**

**Symptom:**
Employee connected to corporate VPN, but DNS queries for internal hostnames (`db.internal`) resolve to public IPs or fail. External sites load fine. Internal services unreachable.

**Root Cause:**
DNS is not routed through the VPN tunnel. Client sends DNS queries to the local ISP resolver (not the corporate DNS server). ISP resolver doesn't know `db.internal`.

```bash
# Check current DNS resolver
cat /etc/resolv.conf
# nameserver 192.168.1.1 ← ISP/home router (wrong)
# Should be: nameserver 10.8.0.1 ← VPN server (corporate DNS)

# Check WireGuard DNS config
cat /etc/wireguard/wg0.conf | grep DNS
# DNS = 10.8.0.1 ← should be set

# Check if DNS requests go through VPN
dig db.internal @10.8.0.1   # should work if VPN is up
dig db.internal              # if fails: DNS not routed through VPN

# Test for DNS leak
# https://dnsleaktest.com/ → should show VPN provider's DNS servers

# Fix: ensure DNS is set in WireGuard client config
# [Interface]
# DNS = 10.8.0.1    ← corporate DNS server VPN IP
# Restart: wg-quick down wg0 && wg-quick up wg0
```
