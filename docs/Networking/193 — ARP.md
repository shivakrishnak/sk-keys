# 193 — ARP

⚡ TL;DR — ARP (Address Resolution Protocol) resolves IP addresses to MAC addresses on a local network segment (Layer 2). When a device wants to send a packet to a local IP, it broadcasts "Who has 192.168.1.5?" and the owner replies with its MAC address. The result is cached in an ARP table. ARP spoofing (poisoning) is a critical network attack vector — a rogue device claims someone else's IP, intercepting their traffic.

---

### 🔥 The Problem This Solves

IP routing delivers packets to the correct network (subnet). But on the local Ethernet segment, packets are actually transmitted using MAC addresses (the hardware address of the NIC). How does a device know the MAC address for a given IP? ARP is the glue between IP addressing (Layer 3) and Ethernet addressing (Layer 2) — it answers "What MAC address corresponds to this IP on my local network?"

---

### 📘 Textbook Definition

**ARP (Address Resolution Protocol):** An L2/L3 protocol (RFC 826) that maps IP addresses to MAC (Media Access Control) addresses on a local network segment. ARP is only used for same-subnet communication — for cross-subnet communication, the device ARPs for its default gateway's MAC (not the destination IP's MAC). The gateway then handles routing.

**ARP Cache:** A kernel-maintained table of {IP → MAC} mappings, with a timeout (default Linux: 60 seconds stale, 20 second reachable). Shows via `arp -n` or `ip neigh`.

**Gratuitous ARP:** An unsolicited ARP reply — a device announces its own IP→MAC mapping to the network without being asked. Used for: notifying of IP change (DHCP renewal), cache refresh, and network failover (VRRP/HSRP — virtual router sends gratuitous ARP on failover to update neighbor caches).

---

### ⏱️ Understand It in 30 Seconds

**One line:**
ARP asks "Who has this IP? Tell me your MAC address" — because Ethernet needs MAC addresses to actually deliver packets on the local segment, not IP addresses.

**One analogy:**

> ARP is like a PA announcement in an office building: "Will whoever sits at desk 192.168.1.5 please come to reception?" The person (device) stands up: "That's me — my name is aa:bb:cc:dd (MAC address)." Everyone nearby hears the response and notes it down (ARP cache). Next time you need to find that person, you check your notes instead of broadcasting again.

### 🔩 First Principles Explanation

**WHY ARP IS NEEDED:**

```
IP layer:  delivers packets to the right subnet (via routing)
Ethernet:  delivers frames to the right NIC (via MAC address)

Gap: you know the destination IP; you don't know its MAC
     Ethernet cannot deliver without MAC

Solution: ARP query before first Ethernet frame

Packet to 192.168.1.50 from 192.168.1.10:
  Is 192.168.1.50 on my subnet? YES (same /24)
  Do I have 192.168.1.50 in my ARP cache?
    YES: use cached MAC → send directly
    NO: ARP request →
      ARP Request (broadcast):
        "Who has 192.168.1.50? Tell 192.168.1.10 (MAC: aa:bb:cc:11:22:33)"
        dst MAC: ff:ff:ff:ff:ff:ff (Ethernet broadcast)
        All devices on segment receive this
      ARP Reply (unicast):
        192.168.1.50: "I have it, my MAC is dd:ee:ff:44:55:66"
        dst MAC: aa:bb:cc:11:22:33 (only back to asker)
      Cached: 192.168.1.50 → dd:ee:ff:44:55:66 (60s)
```

**CROSS-SUBNET ARP BEHAVIOUR:**

```
Device 192.168.1.10 wants to reach 10.0.0.1 (different subnet):
  NOT in local subnet → route to default gateway (192.168.1.1)
  ARP for: 192.168.1.1 (gateway), NOT 10.0.0.1
  Result: frame sent to gateway's MAC
  Gateway handles routing to 10.0.0.1

Key insight: ARP is purely local (Layer 2 broadcast domain)
  Never ARPs for IPs outside the local subnet
  That's what routing tables and gateways are for
```

**ARP SPOOFING / POISONING:**

```
Attack: Rogue device sends unsolicited ARP replies:
  "I am 192.168.1.1 (gateway), my MAC is rogue:mac:addr"

  ALL devices cache: 192.168.1.1 → rogue MAC

  Now: all devices send their traffic to the rogue MAC
  Rogue forwards to real gateway (MITM) or drops (DoS)

ARP has NO authentication: anyone can claim any IP

Real ARP cache of victim after attack:
  ip neigh show
  192.168.1.1 dev eth0 lladdr rogue:mac:addr STALE  ← WRONG!

Defences:
  - Dynamic ARP Inspection (DAI): switch validates ARP against DHCP snooping table
  - Static ARP entries: manually set gateway MAC (impractical at scale)
  - 802.1X port authentication: only authenticated devices on switch ports
  - IPv6 uses NDP (Neighbor Discovery Protocol) instead of ARP, with ICMPv6
    (not inherently more secure, but supports SEND — Secure NDP with crypto)
```

### 🧪 Thought Experiment

**ARP AND VRRP (Virtual Router Redundancy Protocol):**
Two routers share a virtual IP (192.168.1.1). The active router broadcasts a gratuitous ARP: "192.168.1.1 is at master:mac:addr." All hosts cache this. When the active router fails, the standby router becomes active and immediately sends a gratuitous ARP: "192.168.1.1 is now at standby:mac:addr." All hosts update their ARP caches within seconds. Traffic resumes through the new active router. Failover time: typically 1-3 seconds (gratuitous ARP propagation + switch flooding).

### 🧠 Mental Model / Analogy

> ARP is the office phone directory lookup. You have someone's extension number (IP) but need their desk phone number (MAC) to physically ring them. You ask the receptionist (broadcast): "Does anyone know the desk number for extension 192.168.1.50?" The person at that extension answers: "That's me, my desk is dd:ee:ff:44:55:66." You write it in your notebook (ARP cache) for future calls. ARP spoofing = a bad actor responds to all directory queries claiming every extension — now all your calls go to them.

### 📶 Gradual Depth — Four Levels

**Level 1:** ARP translates IP addresses to MAC addresses. Without it, Ethernet can't deliver packets on the local network. When you send a packet to a local IP, your computer broadcasts "Who has that IP?" and the owner replies with its MAC address.

**Level 2:** View your ARP table: `arp -n` or `ip neigh`. If you can't reach a local device, check its ARP entry — `FAILED` means ARP didn't get a reply (device down or wrong IP). Clear stale entries: `ip neigh flush dev eth0`. Send gratuitous ARP (announce your IP): `arping -U -I eth0 192.168.1.10`.

**Level 3:** ARP cache management: Linux tracks states (INCOMPLETE, REACHABLE, STALE, DELAY, PROBE, FAILED). REACHABLE: recently confirmed. STALE: cached but not recently confirmed — will probe before next use. Tuning: `net.ipv4.neigh.eth0.base_reachable_time_ms` (ARP cache validity). For high-traffic servers: increase ARP cache size (`net.ipv4.neigh.default.gc_thresh3`) to prevent ARP cache overflow (which causes packet drops). Proxy ARP: a router answers ARP requests on behalf of devices on other networks — used by cloud providers to make VMs appear local.

**Level 4:** Cloud networking implications: AWS uses virtual MAC addresses and never actually sends ARP over the physical fabric (virtualised networking — hypervisor intercepts and handles ARP internally). The "MAC address" seen by a VM is virtual; actual traffic uses an overlay network. This is why in AWS, you can reassign an Elastic IP instantly — no physical ARP propagation needed; the overlay network simply updates its mapping. Kubernetes: pods communicate via overlay networks (VXLAN, IP-in-IP); ARP is either handled by the CNI plugin's per-pod MAC allocation or bypassed entirely (Cilium with eBPF routes directly at L3 without L2 ARP for pod-to-pod).

### ⚙️ How It Works (Mechanism)

```bash
# View ARP cache
arp -n
# Address    HWtype  HWaddress           Flags  Iface
# 192.168.1.1  ether   aa:bb:cc:dd:ee:ff  C      eth0

# Modern way (ip neighbor)
ip neigh show
# 192.168.1.1 dev eth0 lladdr aa:bb:cc:dd:ee:ff REACHABLE

# ARP cache states: REACHABLE, STALE, DELAY, PROBE, FAILED, PERMANENT

# Flush ARP cache
ip neigh flush dev eth0

# Send ARP request (manually)
arping -I eth0 192.168.1.1
# ARPING 192.168.1.1 from 192.168.1.10 eth0
# Unicast reply from 192.168.1.1 [aa:bb:cc:dd:ee:ff]  1.234ms

# Detect ARP spoofing (check for duplicate MACs)
arp -n | awk '{print $3}' | sort | uniq -d
# If any MAC appears twice → possible ARP spoofing

# Capture ARP traffic
tcpdump -i eth0 arp -n
# ARP, Request who-has 192.168.1.1 tell 192.168.1.10
# ARP, Reply 192.168.1.1 is-at aa:bb:cc:dd:ee:ff

# Gratuitous ARP (announce your IP to update neighbour caches)
arping -U -I eth0 192.168.1.10 -c 3

# Check for ARP cache overflow (Linux)
cat /proc/net/stat/arp_cache
# Or: watch -n 1 'ip -s neigh show | tail -5'
```

### 🔄 The Complete Picture — End-to-End Flow

```
Full packet journey (same subnet):

Application: send data to 192.168.1.50

IP layer:
  src=192.168.1.10, dst=192.168.1.50
  Is dst local? YES (same /24)
  Check ARP cache for 192.168.1.50 → MISS

ARP:
  Broadcast: "Who has 192.168.1.50?"
  Wait for reply (timeout: ~1s if no reply → ARP FAILED)
  Reply: "192.168.1.50 is at dd:ee:ff:44:55:66"
  Cache updated

Ethernet frame:
  src MAC: aa:bb:cc:11:22:33
  dst MAC: dd:ee:ff:44:55:66  ← from ARP
  payload: IP packet

Switch: forward to port with MAC dd:ee:ff:44:55:66

Receiver: extract IP packet, deliver to application
```

### 💻 Code Example

```python
import subprocess
import re
from typing import Optional

def get_mac_for_ip(ip: str, interface: str = "eth0") -> Optional[str]:
    """Get MAC address for a local IP via ARP.
    Triggers ARP request if not cached.
    """
    # Try to ping first to populate ARP cache
    subprocess.run(
        ["ping", "-c", "1", "-W", "1", ip],
        capture_output=True, timeout=3
    )

    # Read ARP cache
    result = subprocess.run(
        ["ip", "neigh", "show", ip, "dev", interface],
        capture_output=True, text=True
    )

    mac_match = re.search(
        r'lladdr ([0-9a-f:]+)',
        result.stdout, re.IGNORECASE
    )

    return mac_match.group(1) if mac_match else None

def detect_arp_conflicts() -> list[dict]:
    """Detect potential ARP spoofing: multiple IPs with same MAC."""
    result = subprocess.run(
        ["ip", "neigh", "show"],
        capture_output=True, text=True
    )

    mac_to_ips: dict[str, list[str]] = {}

    for line in result.stdout.splitlines():
        ip_match = re.match(r'^(\S+)', line)
        mac_match = re.search(r'lladdr ([0-9a-f:]{17})', line, re.I)

        if ip_match and mac_match:
            ip = ip_match.group(1)
            mac = mac_match.group(1).lower()
            mac_to_ips.setdefault(mac, []).append(ip)

    # Flag MACs appearing for multiple IPs (possible ARP spoofing)
    conflicts = [
        {"mac": mac, "ips": ips}
        for mac, ips in mac_to_ips.items()
        if len(ips) > 1
    ]

    return conflicts

# Usage
mac = get_mac_for_ip("192.168.1.1")
print(f"Gateway MAC: {mac}")

conflicts = detect_arp_conflicts()
if conflicts:
    print("⚠️  Possible ARP spoofing detected:")
    for c in conflicts:
        print(f"  MAC {c['mac']} claims IPs: {c['ips']}")
```

### ⚖️ Comparison Table

| Protocol | ARP (IPv4)                    | NDP/ICMPv6 (IPv6)            |
| -------- | ----------------------------- | ---------------------------- |
| Purpose  | IP → MAC resolution           | IP → MAC + other functions   |
| Method   | Broadcast                     | Multicast (solicited-node)   |
| Scope    | Same subnet only              | Same subnet only             |
| Security | No authentication (spoofable) | Optional SEND (crypto, rare) |
| Cache    | ARP cache (ip neigh)          | Neighbor cache (ip neigh)    |

---

### ⚠️ Common Misconceptions

| Misconception            | Reality                                                                                                                                            |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| ARP works across routers | ARP is strictly Layer 2 (broadcast domain). You never ARP for an IP on another subnet — you ARP for the gateway                                    |
| ARP only happens once    | ARP entries expire (Linux default: 30-90 seconds reachable). Every expiry causes a new ARP probe. High-traffic environments need larger ARP caches |
| Cloud VMs use real ARP   | AWS, GCP, Azure use virtual networking — ARP is handled by the hypervisor/SDN fabric. Physical ARP broadcasts never leave the compute host         |

### 🚨 Failure Modes & Diagnosis

**ARP Cache Poisoning: Traffic Intercepted by Rogue Device**

```bash
# Detection: check for suspicious ARP entries
ip neigh show
# 192.168.1.1 dev eth0 lladdr 02:00:00:00:00:01 REACHABLE  ← changed MAC!

# Compare gateway MAC to known value
# Normal: aa:bb:cc:dd:ee:ff
# After poisoning: 02:00:00:00:00:01 (rogue device)

# Check if two IPs share same MAC (clear indicator)
ip neigh show | awk '{print $1, $5}' | sort -k2 | \
  awk 'prev_mac == $2 {print "CONFLICT:", $0} {prev_mac=$2}'

# Immediate mitigation: add static ARP entry for gateway
arp -s 192.168.1.1 aa:bb:cc:dd:ee:ff
# Now rogue ARP replies won't overwrite this entry

# Flush bad entry and add static
ip neigh flush dev eth0
ip neigh add 192.168.1.1 lladdr aa:bb:cc:dd:ee:ff dev eth0 nud permanent

# Network level: enable Dynamic ARP Inspection on switch
# (Cisco IOS)
# ip dhcp snooping
# ip arp inspection vlan 1
```
