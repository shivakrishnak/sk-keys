# 0322 — Records (Java 16+)

⚡ TL;DR — Records are Java's concise immutable data classes: declare the fields once in the header, and the compiler auto-generates a constructor, accessors, `equals()`, `hashCode()`, and `toString()` — replacing 50+ lines of boilerplate with 1.

### 🔥 The Problem This Solves

WORLD WITHOUT IT:
A simple data holder class in Java required enormous boilerplate. For a `Point(int x, int y)` class: private final fields, a constructor, `getX()`, `getY()`, `equals()` comparing both fields, `hashCode()` combining both fields, `toString()` listing them, and possibly `withX(int)` copy methods. Fifty-plus lines of code for two integers. This boilerplate is not just verbose — it is a source of bugs (forgetting a field in `equals`), maintenance burden (every field addition requires updating all methods), and noise that obscures the actual data shape.

THE BREAKING POINT:
A data pipeline processes 30 types of business events. Each event is a small DTO with 3–8 fields. Writing and maintaining correct `equals`/`hashCode`/`toString` for 30 classes = 1,500+ lines of boilerplate. Lombor `@Data` helps but requires an annotation processor dependency. Every code review flags "did you update `equals` when you added the new field?"

THE INVENTION MOMENT:
This is exactly why **Records** were created — to provide a language-native concise declaration of a "data class" where the compiler automatically generates all the standard tedium, with immutability as the default.

---

### 📘 Textbook Definition

A **Record** is a restricted class kind introduced as a preview in Java 14 and made final in Java 16 (JEP 395). A record declaration `record Point(int x, int y) {}` implicitly declares: (1) private final fields for each component; (2) a canonical constructor; (3) public accessor methods named after each component (`x()` and `y()`, not `getX()`); (4) `equals()` and `hashCode()` that compare all components; (5) `toString()` listing all components. Records cannot extend other classes (they implicitly extend `java.lang.Record`), cannot declare instance fields beyond the header, and all components are effectively final.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
Declare your data fields once; the compiler writes all the boilerplate for free.

**One analogy:**
> A printed form with labelled boxes versus writing the same information from scratch every time. Records are the pre-printed form — declare the boxes (fields) once, and every repetitive task (filling in equals, hashCode, toString) is already done.

**One insight:**
Records express an intent beyond "here is a class with these fields" — they say "this class IS its data." A Record has no hidden state, no mutable fields, and no identity beyond its component values. This is a semantic contract the compiler can exploit: two `Point(3, 5)` records are always equal because records are defined by their values, not their identity.

---

### 🔩 First Principles Explanation

CORE INVARIANTS:
1. A record's components are its entire observable state — no instance fields outside the header.
2. Records are shallowly immutable — the record's own fields are final; referenced objects may still be mutable.
3. The canonical constructor is the only way to set component values (though you can add compact constructor validation).

DERIVED DESIGN:
Given invariant 1, `equals()` and `hashCode()` can be correctly generated by comparing all components — there are no hidden fields to forget. Given invariant 2, a `record Address(List<String> lines)` is still a record, but the `lines` list is mutable. True deep immutability requires immutable component types. Given invariant 3, the compact constructor allows validation without repeating the assignment boilerplate:

```java
record Range(int min, int max) {
    Range { // compact constructor: params already named
        if (min > max) throw new IllegalArgumentException(
            "min > max: " + min + " > " + max
        );
        // assignments happen automatically after this block
    }
}
```

**Generated methods (equivalent Java class):**
```java
// This record:
record Point(int x, int y) {}

// Generates (approximately):
final class Point extends Record {
    private final int x;
    private final int y;
    public Point(int x, int y) { this.x=x; this.y=y; }
    public int x() { return x; }
    public int y() { return y; }
    public boolean equals(Object o) {
        return o instanceof Point p
            && x==p.x && y==p.y;
    }
    public int hashCode() {
        return Objects.hash(x, y);
    }
    public String toString() {
        return "Point[x=" + x + ", y=" + y + "]";
    }
}
```

THE TRADE-OFFS:
Gain: Dramatic reduction in boilerplate; enforced immutability; correct `equals`/`hashCode` by default; improved readability and fewer bugs.
Cost: No inheritance from other classes (records extend `Record` implicitly); no mutable fields; accessor naming convention (`x()` not `getX()`) breaks some older frameworks expecting JavaBean convention; requires Java 16+.

### 🧪 Thought Experiment

SETUP:
A `Coordinate` class is used as a `HashMap` key in a geospatial service. Two `Coordinate(lat=37.7, lon=-122.4)` objects should be equal and hash identically so map lookups work correctly.

WITHOUT RECORDS (manual class):
Programmer writes `equals()` and `hashCode()` manually. In version 1, adds `lat` and `lon`. In version 2, adds `altitude`. Forgets to update `hashCode()` to include `altitude`. `hashCode()` is now inconsistent with `equals()` — same coordinates hash differently if altitude differs — breaking `HashMap`: some lookups return null even when the key is in the map.

WITH RECORDS:
```java
record Coordinate(double lat, double lon, double altitude) {}
```
`equals()` and `hashCode()` are auto-generated including all three fields. Adding any new field automatically updates both. The bug simply cannot occur — the compiler guarantees consistency.

THE INSIGHT:
The record compiler guarantee isn't just convenience — it's a correctness guarantee. The set of fields in `equals` and `hashCode` is always exactly the set of declared record components. There is no possibility of drift between field declarations and comparison logic.

---

### 🧠 Mental Model / Analogy

> A record is like a value on a number line. The number `42` IS its value — two `42`s are identical because they ARE the same value, not because they happen to be equivalent objects. A record's identity is its data, just as a number's identity is its value.

"Number identity is its value" → record identity is its component values.
"42 == 42 always" → `new Point(3,5).equals(new Point(3,5))` is always true.
"You can't have a 'special' version of 42" → no mutable state hidden in a record.

Where this analogy breaks down: Numbers are deeply immutable; records are only shallowly immutable — a `record Bag(List<Item> items)` has a mutable list inside even though the record reference is final.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
Instead of writing a class with fields, constructor, and 5 repetitive methods, you write one line: `record Point(int x, int y) {}`. Java writes all the boring parts for you. The object is read-only — you can't change its values after creation.

**Level 2 — How to use it (junior developer):**
Declare records in their own file or as nested classes. Access components with method syntax: `point.x()`, not `point.getX()`. Use compact constructors to validate input. Implement `Serializable` for network transmission. Records work as DTOs, value objects, method return types holding multiple values, and as part of sealed class hierarchies (Java 17+).

**Level 3 — How it works (mid-level engineer):**
The compiler generates a canonical constructor (all components as parameters) and all five standard methods. You can override any generated method: write your own `toString()` to override the generated one. Compact constructors (body without parameter list) run before the auto-generated assignments — good for validation. Records can have static fields, static methods, and instance methods. Records can implement interfaces. Records can be generic: `record Pair<A, B>(A first, B second) {}`.

**Level 4 — Why it was designed this way (senior/staff):**
Records are a deliberate restriction: the "restriction" (no mutable state, no inheritance, no extra instance fields) IS the feature. By restricting what a record can be, the compiler can guarantee what it IS — a pure data class. This is the distinction between "support data classes" and "be a data class." The JEP explicitly states: "the goal is not simply to reduce boilerplate, but to express a semantic constraint." Records pair powerfully with sealed classes (sum types) and pattern matching to enable algebraic data type patterns common in functional languages. The accessor naming convention (`x()` not `getX()`) is intentionally different from JavaBeans to signal that records are not beans and not subject to beans-centric frameworks' conventions.

### ⚙️ How It Works (Mechanism)

**Basic record:**
```java
// Everything in one line:
record Person(String name, int age) {}

Person p = new Person("Alice", 30);
System.out.println(p.name());         // "Alice"
System.out.println(p.age());          // 30
System.out.println(p);                // Person[name=Alice, age=30]
System.out.println(p.equals(
    new Person("Alice", 30)));        // true
```

**Compact constructor for validation:**
```java
record EmailAddress(String value) {
    EmailAddress {
        // 'value' is the parameter name
        if (!value.contains("@")) {
            throw new IllegalArgumentException(
                "Not an email: " + value
            );
        }
        // assignment: this.value = value happens after
    }
}
```

**Generic record:**
```java
record Pair<A, B>(A first, B second) {
    // Can add utility methods:
    public Pair<B, A> swap() {
        return new Pair<>(second, first);
    }
}

Pair<String, Integer> p = new Pair<>("hello", 42);
Pair<Integer, String> swapped = p.swap();
```

**Record with interface:**
```java
sealed interface Shape permits Circle, Rectangle {}

record Circle(double radius) implements Shape {
    double area() { return Math.PI * radius * radius; }
}
record Rectangle(double w, double h) implements Shape {
    double area() { return w * h; }
}
```

**Record serialization (safe):**
```java
// Records implement Serializable correctly — uses
// canonical constructor (not java.io.Serializable bypass)
record Order(Long id, String status) implements Serializable {
    // Serialization uses canonical constructor — safe!
    // No ObjectInputStream.readObject() bypass
}
```

### 🔄 The Complete Picture — End-to-End Flow

NORMAL FLOW (record as HTTP response DTO):
```
[Service creates: new OrderSummary(id, total, status)]
    → [Jackson detects record components via reflection]  ← YOU ARE HERE
    → [Serializes to JSON: {"id":1,"total":99.99,...}]
    → [HTTP response sent]
    → [Jackson deserializes response JSON]
    → [Uses canonical constructor: new OrderSummary(1,99.99,...)]
    → [Immutable DTO available to caller]
```

FAILURE PATH:
```
[Old framework expects getX() accessor naming]
    → [Framework calls getters: getTotal() — not found]
    → [NullPointerException or silent empty binding]
    → [Fix: add explicit accessor alias OR switch to Jackson]
    → [Jackson 2.12+ natively supports records]
```

WHAT CHANGES AT SCALE:
Records are lightweight and create no overhead beyond equivalent manual classes — the bytecode is identical. At scale, records improve codebase quality by preventing the silent `equals`/`hashCode` drift that causes subtle cache and map bugs in large systems. Using records as value objects in `Stream` pipelines is idiomatic Java 16+ style — they are naturally compatible with functional-style processing.

### 💻 Code Example

Example 1 — Before and after records:
```java
// BEFORE (50+ lines for 3 fields):
public final class Money {
    private final BigDecimal amount;
    private final String currency;
    private Money(BigDecimal amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }
    public BigDecimal getAmount() { return amount; }
    public String getCurrency() { return currency; }
    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Money m)) return false;
        return amount.equals(m.amount)
            && currency.equals(m.currency);
    }
    @Override
    public int hashCode() {
        return Objects.hash(amount, currency);
    }
    @Override
    public String toString() {
        return "Money[" + amount + " " + currency + "]";
    }
}

// AFTER (1 line + validation):
record Money(BigDecimal amount, String currency) {
    Money {
        Objects.requireNonNull(amount);
        Objects.requireNonNull(currency);
        if (amount.signum() < 0)
            throw new IllegalArgumentException("Negative");
    }
}
```

Example 2 — Record in a stream pipeline:
```java
record Product(String name, BigDecimal price) {}

List<Product> products = List.of(
    new Product("Widget", new BigDecimal("9.99")),
    new Product("Gadget", new BigDecimal("24.99")),
    new Product("Doohickey", new BigDecimal("4.99"))
);

// Stream using record components naturally:
var expensive = products.stream()
    .filter(p -> p.price().compareTo(
        new BigDecimal("10")) > 0)
    .map(Product::name)
    .toList();
// ["Gadget"]
```

Example 3 — Pattern matching with records (Java 21):
```java
sealed interface Shape permits Circle, Rect {}
record Circle(double r) implements Shape {}
record Rect(double w, double h) implements Shape {}

double area(Shape s) {
    return switch (s) {
        case Circle(var r) -> Math.PI * r * r;
        case Rect(var w, var h) -> w * h;
    };
}
// Record deconstruction pattern: extracts components directly
```

### ⚖️ Comparison Table

| Approach | Boilerplate | Immutable | equals/hashCode | Java Version | Best For |
|---|---|---|---|---|---|
| **Record** | Minimal | Yes (shallow) | Auto-generated | 16+ | DTOs, value types, data classes |
| POJO (manual) | High | Optional | Manual | All | Mutable domain objects |
| Lombok @Data | Low | Optional | Auto-generated | All + plugin | Legacy code, mutable DTOs |
| Lombok @Value | Low | Yes | Auto-generated | All + plugin | Immutable value objects pre-16 |

How to choose: Use records for all new immutable data classes on Java 16+. Use `@Value` Lombok for the same purpose on older Java. Use POJOs for domain objects that need mutation or JPA entities (JPA requires no-arg constructor and mutable fields, incompatible with records).

### ⚠️ Common Misconceptions

| Misconception | Reality |
|---|---|
| Records are deeply immutable | Records are shallowly immutable — the record's own fields are final, but referenced objects (lists, sets, other classes) can still be mutable. `record Bag(List<Item> items)` — the list is mutable |
| Records work as JPA entities | JPA entities require a no-arg constructor, mutable fields, and setter methods. Records have none of these. Use regular POJOs for JPA entities |
| Records cannot have methods | Records can have any number of instance methods, static methods, static fields, and interfaces. Only additional instance fields (beyond header components) are prohibited |
| The accessor is `getX()` like JavaBeans | Record accessors are named after the component: `x()`, not `getX()`. This is intentional to signal that records are NOT beans. Jackson 2.12+ supports record accessor naming natively |
| Records are a Kotlin data class equivalent | Records are similar but not identical. Kotlin data classes generate `copy()` methods; Java records do not (by design — mutation is not the record contract). Kotlin data classes can extend other classes; records cannot |

### 🚨 Failure Modes & Diagnosis

**JPA Entity Fails with Record**

Symptom:
`javax.persistence.PersistenceException: No default constructor for entity`.

Root Cause:
JPA requires mutable POJOs with no-arg constructors. Records cannot have additional instance fields and have no settable properties.

Diagnostic:
```bash
# Error at Hibernate init:
# ERROR o.h.t.s.i.ExceptionHandlerLoggedImpl -
#   SchemaManagementException: export identifier [...] encountered exception
# Or: No default constructor for entity: Record
```

Fix:
```java
// BAD: using record as JPA entity
@Entity
record Order(Long id, String status) {}  // COMPILATION ERROR

// GOOD: use regular class for JPA entities
@Entity
@Table(name = "orders")
public class Order {
    @Id @GeneratedValue
    private Long id;
    private String status;
    // No-arg constructor, getters/setters required
}
```

Prevention: Never use records as JPA entities. Use records for DTOs that transport data out of the persistence layer.

---

**equals/hashCode Bug from Mutable Component**

Symptom:
Using a record as `HashMap` key — key exists in map, but lookup returns `null` after modifying a mutable component of the key.

Root Cause:
Record's `hashCode()` is computed from component values. If a component is a mutable object and its state changes, `hashCode()` changes, but the map's bucket doesn't update.

Diagnostic:
```java
record UserGroup(String name, Set<String> members) {}

Map<UserGroup, Integer> map = new HashMap<>();
Set<String> members = new HashSet<>(Set.of("Alice"));
UserGroup group = new UserGroup("Admin", members);
map.put(group, 1);

members.add("Bob"); // mutates the component!
map.get(group);     // returns null — wrong bucket
```

Fix:
```java
// Use immutable collections as record components:
record UserGroup(String name, Set<String> members) {
    UserGroup {
        members = Set.copyOf(members); // defensive copy
    }
    // members() returns unmodifiable view
}
```

Prevention: Use immutable types (`List.copyOf`, `Set.copyOf`, `Map.copyOf`) for all collection components in records that will be used as map keys or in sets.

---

**Older Framework Fails on Record Accessor Name**

Symptom:
REST framework or object mapper returns `{}` or blank response. No error, but fields are missing.

Root Cause:
Framework uses JavaBean introspection expecting `getName()` accessor pattern. Records use `name()` — the framework finds no getters and serializes nothing.

Diagnostic:
```bash
# Check Jackson version — pre-2.12 doesn't support records:
mvn dependency:tree | grep jackson-databind

# Test with explicit ObjectMapper:
String json = new ObjectMapper().writeValueAsString(myRecord);
System.out.println(json); // {} if accessors not found
```

Fix:
```java
// Upgrade Jackson to 2.12+ for native record support
// Or: add explicit jackson-module-parameter-names module
// Or: override accessors with JavaBean naming:
record Person(String name, int age) {
    // Override with getters if needed for legacy frameworks
    public String getName() { return name; }
    public int getAge() { return age; }
}
```

Prevention: Verify framework record support before adopting records broadly. Spring Boot 2.7+ and Jackson 2.12+ have native record support.

### 🔗 Related Keywords

**Prerequisites (understand these first):**
- `Generics` — records support generic type parameters (`record Pair<A,B>(...)`); understanding generics is needed for generic records
- `Serialization / Deserialization` — records have different serialization semantics than regular classes; understanding serialization explains why records are safer to serialize

**Builds On This (learn these next):**
- `Sealed Classes (Java 17+)` — records pair with sealed interfaces to create type-safe algebraic data types; the combination enables exhaustive pattern matching
- `Pattern Matching (Java 21+)` — record patterns allow direct deconstruction of record components in `switch` expressions; records are the data counterpart to pattern matching

**Alternatives / Comparisons:**
- `Sealed Classes (Java 17+)` — the control-flow counterpart to records' data declaration; together they form algebraic data types
- `Serialization / Deserialization` — records use the canonical constructor during serialization (safe); regular class serialization bypasses the constructor (unsafe)
