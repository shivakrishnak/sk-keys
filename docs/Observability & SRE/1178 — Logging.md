# 1178 — Logging

`#observability` `#logging` `#logs` `#slf4j` `#logback` `#log4j` `#debug`

⚡ TL;DR — **Logging** records discrete events that occur in an application — what happened, when, and with what context. Logs are the first pillar of observability. In Java: **SLF4J** (logging facade) + **Logback** or **Log4j2** (implementation). Best practices: use log levels appropriately, prefer structured JSON logging in production, include context (user ID, request ID, trace ID) in every log line.

### 📘 Textbook Definition

**Application logging**: the practice of recording timestamped events and messages generated by an application during its execution. Each log entry (log record) typically contains: timestamp, log level (TRACE/DEBUG/INFO/WARN/ERROR), logger name (class/component), message, and contextual data (user ID, request ID, exception). Logging frameworks provide: (1) **Logging API/facade**: SLF4J (Simple Logging Façade for Java) — defines the logging interface independently of the implementation; (2) **Logging implementation**: Logback (default in Spring Boot), Log4j2 (high-performance, async), java.util.logging (JDK built-in — avoid in production). Log output targets (appenders): console (development), file with rotation, remote log aggregation (Logstash, Fluentd). Log formats: unstructured text (human-readable, hard to parse at scale) vs **structured logging** (JSON, machine-parseable, enables log aggregation and querying — see #1179). Log correlation: each log line should include a correlation ID (trace ID from OpenTelemetry) so logs from multiple services for the same request can be found together. Java standard logging stack: **SLF4J + Logback** (Spring Boot default), **SLF4J + Log4j2** (higher performance, async logging), **Lombok `@Slf4j`** (reduces boilerplate). Production logging best practices: JSON format, log levels set correctly (INFO in production, DEBUG only during troubleshooting), include trace ID for correlation, avoid logging sensitive data (passwords, PII, payment card numbers), use structured key-value fields.

### 🟢 Simple Definition (Easy)

Logging is your application's diary: "User 123 logged in at 14:23:01. User 123 placed order 456 for $49.99 at 14:23:45. Payment for order 456 failed: Card declined at 14:23:46." When something goes wrong, you read the diary to find out what happened. Without logs, debugging production issues is guesswork.

### 🔵 Simple Definition (Elaborated)

Logs are the oldest and most fundamental observability tool. Every application generates logs; the challenge is generating useful ones.

**What makes a log line useful**:

1. **Timestamp** with milliseconds (or microseconds): when did this happen exactly?
2. **Log level**: how severe is this? (DEBUG/INFO/WARN/ERROR)
3. **Correlation ID**: which request/user triggered this? (to correlate logs across services)
4. **Structured fields**: not "User 123 placed order 456" but `{"userId": 123, "orderId": 456, "event": "order.created"}` — machine-parseable
5. **Actionable message**: what happened in business terms? (not "NullPointerException" alone — add context: "Failed to fetch inventory for productId=prod-123")

**Text logs vs structured logs**:

```
TEXT LOG (bad):
INFO  2024-03-15 14:23:01 OrderService - Order placed for user 123: $49.99

STRUCTURED LOG (good):
{"timestamp":"2024-03-15T14:23:01.456Z","level":"INFO","service":"order-service",
 "traceId":"abc123","userId":"123","orderId":"456","total":49.99,"event":"order.created"}
```

The structured version can be queried: "find all orders placed by user 123 in the last hour" or "find all orders above $100 that failed."

### 🔩 First Principles Explanation

```java
// SPRING BOOT LOGGING SETUP (Logback + SLF4J — the standard Java logging stack)

// 1. SLF4J facade — code against the abstraction, not the implementation
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

// OR with Lombok (preferred — less boilerplate):
@Slf4j  // generates: private static final Logger log = LoggerFactory.getLogger(OrderService.class);
@Service
public class OrderService {

    public OrderResponse createOrder(OrderRequest request) {
        // INFO: business-relevant events (always logged in production)
        log.info("Creating order for user {} with {} items",
            request.getUserId(), request.getItems().size());

        try {
            // DEBUG: detailed flow (only logged in dev/troubleshooting)
            log.debug("Validating inventory for {} items", request.getItems().size());

            inventoryService.validateAll(request.getItems());
            OrderResponse response = orderRepository.save(buildOrder(request));

            // INFO: successful business outcome with key fields
            log.info("Order created successfully: orderId={}, userId={}, total={}",
                response.getOrderId(), request.getUserId(), response.getTotal());

            return response;

        } catch (InventoryException e) {
            // WARN: expected business error (not a system failure)
            log.warn("Order creation failed: insufficient inventory for userId={}, productId={}",
                request.getUserId(), e.getProductId());
            throw e;

        } catch (Exception e) {
            // ERROR: unexpected system failure — include stack trace
            log.error("Unexpected error creating order for userId={}: {}",
                request.getUserId(), e.getMessage(), e);  // ← last param = exception → logs stack trace
            throw new OrderServiceException("Failed to create order", e);
        }
    }
}
```

```java
// MDC (Mapped Diagnostic Context): add context to ALL log lines in a request
// Without MDC: you add userId to every individual log.info() call (verbose)
// With MDC: set once per request; all log lines in the request automatically include it

@Component
public class LoggingFilter implements Filter {

    @Override
    public void doFilter(ServletRequest req, ServletResponse resp, FilterChain chain)
            throws IOException, ServletException {

        HttpServletRequest request = (HttpServletRequest) req;

        // Set context fields in MDC — available to ALL log lines for this request
        MDC.put("traceId",   request.getHeader("X-Trace-Id") != null
                             ? request.getHeader("X-Trace-Id") : UUID.randomUUID().toString());
        MDC.put("requestId", UUID.randomUUID().toString());
        MDC.put("userId",    extractUserId(request));
        MDC.put("path",      request.getRequestURI());

        try {
            chain.doFilter(req, resp);
        } finally {
            MDC.clear();  // CRITICAL: clear MDC after request (thread pool reuse)
        }
    }
}

// Now EVERY log line in the request automatically includes traceId, userId, path:
// {"level":"INFO","message":"Creating order","traceId":"abc123","userId":"456","path":"/orders"}
// {"level":"DEBUG","message":"Validating inventory","traceId":"abc123","userId":"456","path":"/orders"}
// {"level":"INFO","message":"Order created","orderId":"789","traceId":"abc123","userId":"456"}
//   ↑ traceId appears in every line → search in Loki/ELK by traceId to see all events for one request
```

### ❓ Why Does This Exist (Why Before What)

Software systems run autonomously — unlike a human task, there's no one watching the code execute step by step. When something goes wrong (and it always will), logs are the record of what the system was doing. Without logs: debugging production issues requires either reproducing the problem (often impossible) or guessing. With logs: a production bug leaves a trail — the exact time, the exact user, the exact inputs, the exact error — enabling diagnosis and fix within minutes rather than hours.

---

### 🧠 Mental Model / Analogy

> **Logs are the audit trail for your application**: like a bank's transaction ledger that records every financial event (who, what, when, how much), an application's logs record every significant event. Just as a bank auditor can reconstruct exactly what happened in any account by reading the ledger, a developer can reconstruct exactly what happened during any user's session by reading the logs. The ledger is useless if it's incomplete or ambiguous; likewise, logs are useless if they lack context, timestamps, or correlation IDs.

### 💻 Code Example

```java
// WHAT NOT TO LOG (security and privacy pitfalls)

// BAD: logs sensitive data
log.info("User {} authenticated with password {}", username, password);  // ← NEVER log passwords
log.info("Payment processed for card {}", cardNumber);  // ← NEVER log payment card numbers
log.debug("Request body: {}", requestBody);  // ← request body might contain sensitive data

// GOOD: log identifiers, not sensitive values
log.info("User {} authenticated successfully", username);   // username OK; password never
log.info("Payment processed for card ending {}", cardLast4); // last 4 digits only
log.debug("Request received: path={}, contentLength={}", path, contentLength); // no body

// GOOD: log the correlation ID for all log lines
log.info("Order created: orderId={}, userId={}, traceId={}", orderId, userId, traceId);

// GOOD: log exceptions with stack trace (pass exception as last parameter to SLF4J)
log.error("Failed to process payment for orderId={}", orderId, exception); // stack trace logged
```

### ⚠️ Common Misconceptions

| Misconception                      | Reality                                                                                                                                                                                                                                                                                                                                                                |
| ---------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| More logging is always better      | Excessive logging (e.g., logging at DEBUG level in production) causes: increased storage costs, I/O overhead slowing down the application, noise drowning out important signals, and security risk (verbose logs may contain sensitive data). Log at INFO in production for important business events; use DEBUG only during active troubleshooting with a time limit. |
| System.out.println() is acceptable | `System.out.println()` has no log levels, no timestamps, no logger name, no structured format, no configurable output destination, no performance optimization (always synchronous), and no way to filter noise from signal. Always use a proper logging framework (SLF4J + Logback/Log4j2).                                                                           |
| Log aggregation makes logging slow | Logging to a remote aggregation system (ELK, Loki) can be slow if done synchronously on the request thread. Solution: use async logging (Logback's `AsyncAppender` or Log4j2's `AsyncLogger`) — log events are written to a queue and sent to the appender on a background thread, keeping the request thread fast.                                                    |

### 🧠 Think About This Before We Continue

**Q1.** Cardinality in logging: logs are high-cardinality by nature (millions of events per day with unique user IDs, request IDs, order IDs). Log aggregation systems (ELK, Loki) handle this well — they're designed for full-text search over high-cardinality data. But compare to metrics systems (Prometheus): Prometheus stores time-series indexed by label values. If you add `user_id` as a Prometheus metric label, you get millions of time-series (one per user) — this is the "high-cardinality problem" in metrics, causing Prometheus memory exhaustion. This is why: user-specific data belongs in logs (and traces); aggregate counts and rates belong in metrics. Design the boundary: what data should be in logs vs metrics? How do you decide?

**Q2.** Log retention and compliance: production logs contain business events (who ordered what, who logged in when) and may contain PII (names, email addresses, IP addresses). Compliance regulations (GDPR, HIPAA, PCI-DSS) impose requirements: (a) GDPR right to erasure — if a user requests deletion, must their log entries be deleted? (b) PCI-DSS — payment-related logs must be retained for 1 year but must NOT contain full card numbers; (c) HIPAA — health data logs must be encrypted at rest and retained for 6 years. Design a logging strategy for a healthcare application subject to HIPAA: what goes in logs, what's redacted, how long is it retained, where is it stored, who can access it?
