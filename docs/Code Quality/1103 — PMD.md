# 1103 — PMD

⚡ TL;DR — PMD is a Java (and multi-language) static analysis tool that detects code smells, inefficient code, overly complex logic, duplicate code, and potential bugs through rule-based AST analysis.

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
A Java codebase has a method with 12 nested `if` statements. Cyclomatic complexity: 14. The method has 4 different `if (x == null)` null checks scattered across it. There's a dead `catch (Exception e) {}` block that swallows all exceptions silently. There's a string comparison with `==` instead of `.equals()`. None of these are naming violations (Checkstyle won't catch them). None are null pointer dereferences (SpotBugs looks at runtime behaviour). These are quality patterns: code that is *structurally bad* — hard to understand, likely to contain bugs, difficult to test.

**THE BREAKING POINT:**
Style checkers like Checkstyle catch *how code looks*. Bug detectors like SpotBugs catch *specific bug patterns*. But there's a middle ground: code that is structurally problematic without being obviously buggy. This code accumulates silently, making future bugs more likely, future maintenance more expensive, and future developers more frustrated.

**THE INVENTION MOMENT:**
This is exactly why **PMD** was created: to catch code quality issues that are neither style violations nor known bugs — the structural patterns that indicate poor design, hidden complexity, and maintenance debt.

---

### 📘 Textbook Definition

**PMD** (originally "Programming Mistake Detector," now just PMD) is a Java-first, multi-language (Apex, Groovy, JavaScript, Kotlin, XML, PLSQL) static code analyser that detects code quality violations through rule-based analysis. PMD analyses source code ASTs and applies rules organised into rule sets: **Best Practices** (avoiding empty catch blocks, not using raw types, using `iterator.remove()` not `collection.remove()` in loops), **Code Style** (field naming, unnecessary modifiers), **Design** (avoid deeply nested `if`, GodClass, ExcessiveMethodLength, TooManyMethods), **Error Prone** (broken equals/hashcode contract, string comparison with `==`, suspicious integer casting), **Multithreading** (non-thread-safe singleton, avoid synchronized on `this`), **Performance** (unnecessary `String` creation in loops, `BigDecimal` from floats), and **Security** (hardcoded credentials, insecure random). PMD also includes **CPD** (Copy-Paste Detector) — a language-agnostic duplicate code finder. PMD integrates with Maven (maven-pmd-plugin), Gradle, Ant, and major IDEs.

### ⏱️ Understand It in 30 Seconds

**One line:**
A code quality inspector that finds poor structure, dead code, and subtle bug patterns that style checkers miss.

**One analogy:**
> PMD is like a building inspector checking for code violations beyond aesthetics. A paint inspector (Checkstyle) checks that the walls are the right colour. An electrical inspector (PMD) checks that the wiring meets code: no exposed wires, no overloaded circuits, no dead ends. The walls can look perfect while the wiring is a fire hazard. PMD finds the fire hazards in code that look fine on the surface.

**One insight:**
Checkstyle tells you your code *looks* like Java. PMD tells you your code *behaves* like good Java — that you're using the language idiomatically, not fighting it. The combination of Checkstyle (style) + PMD (quality) + SpotBugs (bugs) covers the three distinct layers of Java code analysis.

### 🔩 First Principles Explanation

**CORE INVARIANTS:**
1. Code complexity correlates with bug density: higher cyclomatic complexity → more branches → more potential defect paths.
2. Silent exception swallowing (`catch (Exception e) {}`) hides runtime errors — any exception in a `try` block is silently discarded, including bugs.
3. String comparison with `==` in Java compares object identity, not string value. Two distinct `String` objects with identical content are `==`-unequal. This is a well-known Java footgun.

**DERIVED DESIGN:**
These patterns (excessive complexity, empty catch blocks, incorrect String comparison) are structurally identifiable in the AST without executing the code. A rule that checks if `catch` blocks have empty bodies catches empty catch blocks. A rule that checks if `==` is applied to `String` types catches incorrect string comparison. Rules encode expert knowledge about Java anti-patterns.

**THE TRADE-OFFS:**
Gain: Catches structural quality issues and subtle Java-specific bugs that neither style checking nor runtime testing catches. Especially valuable for catching Java footguns.
Cost: Higher false positive rate than Checkstyle (structural rules are more context-dependent than style rules). PMD requires more careful rule selection and threshold tuning than Checkstyle.

### 🧪 Thought Experiment

**SETUP:**
A Java service processes payments. A developer writes:
```java
// Catches all exceptions, discards them silently
try {
    processPayment(order);
} catch (Exception e) { }
```

**WITHOUT PMD:**
The code compiles. Tests pass (they test the happy path; `processPayment` doesn't throw in tests). Code ships. In production, a payment gateway is temporarily unreachable. `processPayment` throws `IOException`. The exception is caught and silently discarded. The payment order is recorded as "processed" even though it failed. Customers are charged but payments are never sent. The bug manifests as financial corruption, not an error in logs.

**WITH PMD:**
PMD Rule `EmptyCatchBlock` fires:
```
PaymentService.java:42: EmptyCatchBlock:
Avoid empty catch blocks
```
Developer is forced to handle the exception — at minimum, log it. The silent failure is caught before it reaches production.

**THE INSIGHT:**
An empty catch block is not a bug the compiler can detect (it compiles) and not a style issue Checkstyle addresses (it's about behaviour, not appearance). It is a structural pattern that PMD is designed to detect.

### 🧠 Mental Model / Analogy

> PMD is like a code archaeologist who reads your code looking for civilisation-ending patterns. An archaeologist doesn't just describe what they see — they recognise patterns that historically correlate with collapse: "here, debt was accumulated and ignored; here, single points of failure were built without fallback; here, complexity was added without design." PMD reads your code and recognises patterns that historically correlate with bugs and maintenance failure: empty catch blocks, god classes, string comparisons with `==`, non-thread-safe singletons.

- "Civilisation-ending patterns" → anti-patterns that cause production failures
- "Archaeologist" → PMD rule engine
- "Debt accumulated" → cyclomatic complexity exceeding threshold
- "Single point of failure" → empty catch block swallowing all exceptions

Where this analogy breaks down: an archaeologist finds patterns after the fact; PMD finds them before deployment.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
PMD is a tool that reads your Java code and finds problematic patterns — not style issues like naming, but structural problems: code that's too complex, exception handling that silently swallows errors, loops that create too many objects in memory, comparisons that use the wrong operator for Strings. It finds patterns that developers know are wrong but sometimes write anyway.

**Level 2 — How to use it (junior developer):**
Add the PMD Maven plugin to `pom.xml`. Run `mvn pmd:check`. PMD will list violations by rule name, file, and line number. For each violation: look up the rule name in PMD's documentation (pmd.github.io) to understand what it's detecting. Examples to learn from: `EmptyCatchBlock` (add logging or re-throw), `UseEqualsToTestStringEquality` (use `s1.equals(s2)` not `s1 == s2`), `ExcessiveMethodLength` (break the method into smaller methods). Use `@SuppressWarnings("PMD.RuleName")` only for genuine false positives, with a comment.

**Level 3 — How it works (mid-level engineer):**
PMD builds an AST from Java source and applies rule visitors. Rules are Java classes implementing the `AbstractJavaRule` interface with `visit()` methods for specific AST node types. Example: `EmptyCatchBlock` visits `ASTCatchStatement` nodes and checks if the body is empty. PMD supports both Java-coded rules and **XPath rules** — rules written as XPath expressions over the AST. XPath rules are powerful and don't require Java code: `//CatchStatement[count(Block/BlockStatement)=0]` matches all empty catch blocks. XML rulesets reference rules by class name or XPath expression; teams selectively enable/disable rules. CPD (Copy-Paste Detector) uses a token-based suffix tree algorithm to find duplicate code blocks across files — it's language-agnostic and configurable for minimum token duplication threshold.

**Level 4 — Why it was designed this way (senior/staff):**
PMD's XPath rule capability was unusual at the time of its creation and reflects a key insight: allowing users to write rules as data (XPath expressions over the AST) rather than code dramatically lowers the barrier to custom rule creation. A team can write a PMD rule to enforce their project-specific patterns ("services may not call repositories directly; they must go through a use-case layer") as an XPath expression without Java programming. This is the same philosophy as CSS selectors — a query language for tree-structured data. PMD's integration of CPD alongside syntactic rules reflects the practical observation that code duplication is a major source of maintenance problems and bugs (fixing a bug in a duplicated code block, then forgetting the duplicate still exists). Combining duplication detection with quality analysis in one tool reduces the number of tools teams need to manage.

### 🔄 The Complete Picture — End-to-End Flow

**NORMAL FLOW:**
```
Developer writes Java code
  → mvn pmd:check
  → PMD: 0 violations [← YOU ARE HERE]
  → Build succeeds
  → No structural quality debt introduced
  → Code review: reviewer focuses on logic,
    not structural anti-patterns
```

**FAILURE PATH:**
```
Developer writes:
  catch (Exception e) {}  // swallows all exceptions
  → PMD: EmptyCatchBlock violation
  → Build fails
  → Developer adds logging: LOG.error("...", e)
  → Production: exceptions are now visible in logs
  → Incident detection time reduced from "never"
    to "next log scan"
```

**WHAT CHANGES AT SCALE:**
At large scale, CPD becomes critical: in large codebases, copy-paste code accumulates across hundreds of files. Fixing a bug in one copy requires finding all copies — often missed. PMD/CPD reports: "These 3 blocks across 5 files are 95% identical — refactor into a shared method." Consistent CPD scanning prevents duplicate bug patterns.

### 💻 Code Example

**Example 1 — PMD Maven plugin:**
```xml
<!-- pom.xml -->
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-pmd-plugin</artifactId>
  <version>3.21.2</version>
  <configuration>
    <!-- Custom ruleset -->
    <rulesets>
      <ruleset>config/pmd-rules.xml</ruleset>
    </rulesets>
    <failOnViolation>true</failOnViolation>
    <printFailingErrors>true</printFailingErrors>
    <!-- Skip generated code -->
    <excludeRoots>
      <excludeRoot>
        target/generated-sources
      </excludeRoot>
    </excludeRoots>
  </configuration>
  <executions>
    <execution>
      <goals>
        <goal>check</goal>
        <goal>cpd-check</goal>
      </goals>
    </execution>
  </executions>
</plugin>
```

**Example 2 — Custom PMD ruleset:**
```xml
<!-- config/pmd-rules.xml -->
<?xml version="1.0"?>
<ruleset name="Team Ruleset"
  xmlns="http://pmd.sourceforge.net/ruleset/2.0.0">
  
  <description>Team custom ruleset</description>

  <!-- Error-prone patterns -->
  <rule ref="category/java/errorprone.xml/EmptyCatchBlock"/>
  <rule ref="category/java/errorprone.xml/UseEqualsToTestStringEquality"/>
  <rule ref="category/java/errorprone.xml/BrokenNullCheck"/>

  <!-- Performance -->
  <rule ref="category/java/performance.xml/BigDecimalInstantiation"/>
  <rule ref="category/java/performance.xml/StringInstantiation"/>
  
  <!-- Design -->
  <rule ref="category/java/design.xml/CyclomaticComplexity">
    <properties>
      <property name="methodReportLevel" value="10"/>
    </properties>
  </rule>
  <rule ref="category/java/design.xml/ExcessiveMethodLength">
    <properties>
      <property name="minimum" value="50"/>
    </properties>
  </rule>
  
  <!-- Best practices -->
  <rule ref="category/java/bestpractices.xml/UnusedPrivateField"/>
  <rule ref="category/java/bestpractices.xml/UnusedLocalVariable"/>
</ruleset>
```

**Example 3 — Key violations and fixes:**
```java
// BAD: EmptyCatchBlock — exception swallowed silently
try {
    processPayment(order);
} catch (PaymentException e) {
    // nothing — PMD EmptyCatchBlock fires
}

// GOOD: at minimum, log the exception
try {
    processPayment(order);
} catch (PaymentException e) {
    log.error("Payment processing failed: {}", 
               order.getId(), e);
    throw new ServiceException("Payment failed", e);
}

// BAD: UseEqualsToTestStringEquality
if (status == "ACTIVE") {  // compares reference, not value

}

// GOOD: use equals()
if ("ACTIVE".equals(status)) { // null-safe + correct

}

// BAD: BigDecimalInstantiation from float (precision issue)
BigDecimal price = new BigDecimal(19.99); 
// 19.99 stored as imprecise float

// GOOD: use String constructor
BigDecimal price = new BigDecimal("19.99");
```

### ⚠️ Common Misconceptions

| Misconception | Reality |
|---|---|
| PMD and Checkstyle do the same thing | Checkstyle checks style (naming, formatting). PMD checks quality (complexity, anti-patterns, error-prone code). They are complementary, not overlapping. |
| PMD catches all bugs | PMD catches structural patterns that correlate with bugs. SpotBugs targets actual bug patterns (null dereferences, concurrency errors). Both are needed. |
| CPD false positives shouldn't be suppressed | CPD sometimes flags legitimately identical code (boilerplate configuration, domain constants). Suppression with explanation is appropriate where duplication is genuinely intentional. |
| PMD rules are fixed | PMD supports custom XPath rules — teams can write rules for their specific code patterns. |

### 🚨 Failure Modes & Diagnosis

**1. PMD Running on Test Code with Wrong Rules**

**Symptom:** PMD fails on test code: methods that should be short in production are long in tests (a `@Test` method with 100 lines of test scenarios is acceptable).

**Root Cause:** Same ruleset applied to both `src/main` and `src/test` without test-appropriate thresholds.

**Diagnostic:**
```bash
mvn pmd:check 2>&1 | grep "Test.*MethodLength\|test.*ExcessiveMethod"
# Violations in test code? Thresholds too strict for tests
```

**Fix:** Create a separate PMD ruleset for test code with relaxed complexity thresholds.

**Prevention:** Maintain `pmd-rules.xml` (production) and `pmd-test-rules.xml` (test) with different thresholds.

---

**2. CPD Reports 1,000 Duplication Violations — All Valid**

**Symptom:** CPD runs and reports massive duplication in JPA entity classes, DTO builders, and Spring configuration. Most are boilerplate that looks similar but serves different purposes.

**Root Cause:** CPD minimum token threshold is too low (default 100 tokens flags small blocks). Framework boilerplate looks structurally similar.

**Diagnostic:**
```bash
# Increase minimum tokens for CPD
mvn pmd:cpd-check -Dcpd.minimumTokens=200
# Check: does the count drop significantly?
```

**Fix:** Increase `minimumTokens` to 150–200 for most projects. Exclude generated code directories.

**Prevention:** Calibrate CPD threshold against your codebase before enabling as a gate.

### 🧠 Think About This Before We Continue

**Q1.** PMD's CPD (Copy-Paste Detector) identifies that 30% of your Java service's code is duplicated. Upon investigation, 40% of the flagged duplication is intentional: Spring configuration boilerplate, entity-to-DTO mapping methods that must be separate per type, and JPA repository patterns that look structurally identical. How would you configure CPD to catch genuinely harmful duplication (copied business logic across two payment processors) while not flagging legitimate structural repetition?

**Q2.** A new developer joins and runs PMD on the legacy codebase. It reports 2,000 violations. The developer says: "We should fix all of these before writing any new features." A senior developer says: "No, we should continue feature development and fix PMD violations incrementally." Who is right? Design a specific process for managing 2,000 PMD violations in a production system that is still in active feature development.
