### 📘 Textbook Definition

A **JDK Dynamic Proxy** is a proxy class created at runtime by `java.lang.reflect.Proxy.newProxyInstance()`, implementing one or more specified interfaces. All method calls on the proxy are dispatched to a single `InvocationHandler.invoke(Object proxy, Method method, Object[] args)` callback. Spring uses JDK dynamic proxies when a target bean implements at least one interface and `proxyTargetClass` is `false`. The proxy class is generated by the JVM's built-in proxy mechanism (no bytecode library required) and is a direct implementation of the specified interfaces — it does not extend the target class. Mockito and Spring Data's repository interfaces are prominent users of JDK dynamic proxies.

---

### 🟢 Simple Definition (Easy)

A JDK proxy is a fake implementation of an interface that the JVM creates automatically. Every method call goes through a single handler you provide — perfect for adding behaviour like transactions or mock responses around real code.

---

### 🔵 Simple Definition (Elaborated)

Java's reflection API can generate a class at runtime that implements any interface. This generated class routes every method call to a single `InvocationHandler` — a callback you provide. Spring uses this for AOP: the handler opens a transaction, calls the real method on the actual bean, then commits. The limitation: only interface methods can be intercepted. If you call `orderService.place(req)` through a `PaymentGateway` interface reference, every method on that interface goes through the handler. But methods on `OrderService` that aren't in any interface are unreachable through the proxy.

### 🔩 First Principles Explanation

**The JDK Proxy API:**

```java
// The entire JDK proxy mechanism:
Object proxy = Proxy.newProxyInstance(
    classLoader,              // to define the proxy class
    new Class[]{MyInterface.class}, // interfaces to implement
    (proxyObj, method, args) -> {   // InvocationHandler
      System.out.println("Before: " + method.getName());
      Object result = method.invoke(realTarget, args);
      System.out.println("After: " + method.getName());
      return result;
    }
);

// proxy IS-A MyInterface — can be cast and used anywhere
// MyInterface is used
```

**JDK Proxy vs CGLIB — key structural difference:**

```
┌───────────────────────────────────────────────────┐
│  JDK DYNAMIC PROXY                                │
│                                                   │
│  Proxy class hierarchy:                           │
│    $Proxy0 implements PaymentGateway              │
│    (no extends → not a subclass of anything)      │
│                                                   │
│  Only methods in PaymentGateway interface         │
│  can be intercepted                               │
│                                                   │
│  CGLIB PROXY                                      │
│                                                   │
│  Proxy class hierarchy:                           │
│    OrderService$$CGLIB$$0 extends OrderService    │
│    (subclass → overrides all non-final methods)   │
│                                                   │
│  All public non-final methods can be intercepted  │
└───────────────────────────────────────────────────┘
```

### ❓ Why Does This Exist (Why Before What)

**WITHOUT JDK Dynamic Proxy:**

```
Without JDK proxy (for interface-based AOP):

  Manual delegation boilerplate per interface + method:
    class TransactionPaymentGateway
        implements PaymentGateway {
      public Receipt charge(PaymentRequest r) {
        openTx(); try {
          return delegate.charge(r); commit();
        } catch(e) { rollback(); throw e; }
      }
      // Repeat same boilerplate for each of the 5 methods
    }
    → 5 methods × 5 interfaces × cross-cutting: 125 classes

  Runtime AOP impossible:
    Can't add @Transactional to existing classes at runtime
    → Compile-time code generation required for every addition
```

**WITH JDK Dynamic Proxy:**

```
→ One InvocationHandler implements AOP for all interfaces
→ Framework-generated: zero boilerplate in your code
→ Spring Data repositories: no implementation needed
   interface UserRepository extends JpaRepository<User,Long>{}
   → JDK proxy generates the full implementation at runtime
→ Mockito.mock(UserRepository.class) → JDK proxy
→ Works with any Java interface
```

### 🧠 Mental Model / Analogy

> A JDK dynamic proxy is like a **receptionist who handles all your calls through a switchboard**. You give the receptionist a script (InvocationHandler) — "for every call, check if the caller is authorised, connect them to me, then log the outcome." Callers think they're talking directly to you (the interface), but every call passes through the receptionist first. The limitation: only calls that go through the main switchboard (interface methods) are intercepted — direct office visits (non-interface method calls) bypass the receptionist.

"Caller via switchboard" = method call through interface reference
"Receptionist with script" = InvocationHandler
"Direct office visit" = method call not on the interface (skips proxy)
"Switchboard" = Proxy.newProxyInstance machinery

### ⚙️ How It Works (Mechanism)

**How Spring uses JDK proxy for AOP (pre-Spring Boot default):**

```java
// Spring creates this at BPP.postProcessAfterInitialization:
PaymentGateway proxy = (PaymentGateway)
    Proxy.newProxyInstance(
        target.getClass().getClassLoader(),
        target.getClass().getInterfaces(), // [PaymentGateway]
        new JdkDynamicAopProxy(advised)    // Spring's handler
    );

// JdkDynamicAopProxy.invoke() chains:
// 1. Transaction interceptor (if @Transactional)
// 2. Security interceptor (if @Secured)
// 3. Cache interceptor (if @Cacheable)
// 4. method.invoke(target, args) — real method call
```

**Spring Data Repository implementation:**

```java
// YOU define only the interface:
public interface UserRepository
    extends JpaRepository<User, Long> {
  List<User> findByEmailDomain(String domain);
}
// Spring Data generates a JDK proxy at startup
// InvocationHandler: SimpleJpaRepository + query derivation
// findByEmailDomain → "SELECT u FROM User u
//                      WHERE u.email LIKE ?1"
```

**When Spring chooses JDK vs CGLIB:**

```java
// JDK proxy used (default in plain Spring when interface exists):
@Bean
@Transactional
OrderGateway orderGateway() {
  return new OrderGatewayImpl(); // implements OrderGateway
}
// Proxy == JDK proxy implementing OrderGateway

// CGLIB used (Spring Boot default proxyTargetClass=true):
// → Always CGLIB even if interface exists
// @EnableAspectJAutoProxy(proxyTargetClass = true)
```

### 💻 Code Example

**Example 1 — Manual JDK proxy for AOP:**

```java
public interface OrderService {
  Order place(OrderRequest req);
  void cancel(long orderId);
}

public class OrderServiceImpl implements OrderService {
  public Order place(OrderRequest req) { /* ... */ }
  public void cancel(long orderId) { /* ... */ }
}

// Create a timing proxy manually:
OrderService proxy = (OrderService) Proxy.newProxyInstance(
    OrderServiceImpl.class.getClassLoader(),
    new Class[]{OrderService.class},
    (proxyObj, method, args) -> {
      long start = System.nanoTime();
      try {
        return method.invoke(new OrderServiceImpl(), args);
      } finally {
        long ms = (System.nanoTime() - start) / 1_000_000;
        log.info("{} took {}ms", method.getName(), ms);
      }
    });

proxy.place(new OrderRequest(...)); // logs: "place took 12ms"
```

**Example 2 — Spring Data JDK proxy in action:**

```java
// No implementation class needed:
@Repository
public interface ProductRepository
    extends JpaRepository<Product, Long> {
  // Spring Data generates JDK proxy at runtime
  List<Product> findByCategoryAndPriceBelow(
      String category, BigDecimal maxPrice);
  // Derived query: SELECT p FROM Product p
  //   WHERE p.category = ?1 AND p.price < ?2

  @Query("SELECT p FROM Product p WHERE p.stock = 0")
  List<Product> findOutOfStock();
}

// Usage — proxy intercepts every call:
productRepository.findByCategoryAndPriceBelow("books", new BigDecimal("30"));
```

**Example 3 — Checking JDK proxy at runtime:**

```java
@Autowired OrderService orderService; // interface type

// Check proxy type:
AopUtils.isJdkDynamicProxy(orderService);  // true if JDK proxy
AopUtils.isCglibProxy(orderService);       // true if CGLIB proxy
AopUtils.getTargetClass(orderService);     // OrderServiceImpl.class

// Casting to implementation class (common mistake):
OrderServiceImpl impl = (OrderServiceImpl) orderService;
// ClassCastException if JDK proxy (proxy is NOT a subclass)
// Works with CGLIB proxy (is a subclass)
```

### ⚠️ Common Misconceptions

| Misconception | Reality |
|---|---|
| JDK proxy can intercept any method on the target class | JDK proxy can only intercept methods declared in the proxied interfaces — methods only on the implementation class are unreachable through the proxy |
| Casting a JDK proxy to the implementation class works | JDK proxy implements the interface but does NOT extend the implementation class — casting throws ClassCastException |
| Spring always uses JDK proxy when an interface is present | Spring Boot defaults to `proxyTargetClass=true` (CGLIB always). Only plain Spring with default settings uses JDK proxy for interface-bearing beans |
| JDK proxy is always faster than CGLIB | A long-lived JDK proxy's InvocationHandler dispatch via reflection can be slower than CGLIB's overridden method calls — JIT optimizes CGLIB subclass calls better |

### 🔥 Pitfalls in Production

**1. ClassCastException when injecting by implementation type**

```java
// OrderService is a JDK proxy (implements interface)
@Autowired
OrderServiceImpl orderService; // FAILS: wants concrete class
// Spring throws: Cannot cast $Proxy21 to OrderServiceImpl

// FIX: always inject by interface type
@Autowired
OrderService orderService;     // WORKS: proxy IS-A OrderService

// If you truly need the impl (rare), use AopTestUtils:
OrderServiceImpl impl =
    AopTestUtils.getTargetObject(orderService);
```

**2. Calling non-interface methods on JDK proxy**

```java
// Interface has only charge() and refund()
interface PaymentGateway {
  Receipt charge(PaymentRequest r);
  Receipt refund(String txId);
}

class StripeGateway implements PaymentGateway {
  // Not in interface — invisible through proxy
  public void syncWebhooks() {...}
}

// Via interface reference — proxy only:
PaymentGateway gw = ctx.getBean(PaymentGateway.class);
gw.syncWebhooks(); // COMPILE ERROR — method not on interface
// Even if called via reflection: not intercepted, not proxied
// Fix: add syncWebhooks() to interface
```
