### 📘 Textbook Definition

A **URL Shortener** is a web service that transforms a long URL into a short, fixed-length alias (typically 7–8 characters) and redirects users from the short URL to the original long URL via HTTP 301 (permanent) or 302 (temporary) redirects. The system has two distinct traffic patterns: **writes** (URL creation) are rare (~100 per second at scale), while **reads** (redirects) are extremely heavy (~100K–1M per second). The core design challenge is implementing a fast, globally unique short code generation mechanism (typically a distributed ID generator such as Snowflake ID + Base62 encoding, or pre-generated key pool) and a read-optimised architecture (Redis caching + CDN) to handle high redirect volume at minimal latency. Expiry management, custom aliases, analytics tracking, and abuse prevention (malicious URL blocking) are common extension requirements.

---

### 🟢 Simple Definition (Easy)

URL Shortener: a machine that converts `https://www.example.com/very/long/path?param=value` into `bit.ly/abc1234`. User visits `bit.ly/abc1234`, the machine looks it up in a table (short_code → long_url), and instantly redirects the user to the long URL. The short code `abc1234` is generated by encoding a unique number in Base62. The lookup must be extremely fast (cache in Redis) because millions of clicks happen per second.

---

### 🔵 Simple Definition (Elaborated)

Two main API operations: `POST /shorten` takes a long URL and returns a short code (writes: 100/second). `GET /{short_code}` looks up the short code and returns HTTP 302 redirect to the long URL (reads: 100,000/second — 1000× more reads than writes). Scale challenge: billions of short URLs in the database, millions of redirects per second. Solution: Redis caches the hot 20% of URLs (which get 80% of redirects), sharded MySQL stores the full mapping table, a unique ID generator creates collision-free short codes without coordination between servers.

### 🔩 First Principles Explanation

**URL shortener: capacity estimation, ID generation, and redirect flow:**

```
BACK-OF-ENVELOPE ESTIMATION:

  Assumptions (for 10-year system):
    Write QPS: 100 URL creations/second
    Read/Write ratio: 10,000:1 (100K redirects/second peak)
    URL expiry: none (permanent)
    Average URL length: 100 bytes

  Calculations:
    URLs created per day: 100 × 86,400 = 8.64M/day
    URLs created per year: 8.64M × 365 ≈ 3.15B/year
    Total URLs over 10 years: 3.15B × 10 = 31.5B

    Short code length needed:
      Base62: 62 characters (a-z, A-Z, 0-9)
      7 characters: 62^7 = 3.5 trillion → sufficient for 31.5 billion URLs

    Storage:
      Per record: 8 bytes (id) + 7 bytes (short_code) + 500 bytes (long_url) = ~515 bytes
      Total: 31.5B × 515 bytes ≈ 16 TB over 10 years
      Sharded across 4 MySQL nodes: 4 TB per shard → manageable

    Cache sizing:
      Redirects/day: 100K × 86,400 = 8.64B redirects/day
      Pareto: 20% of URLs = 80% of traffic
      "Hot" URLs in cache: 20% of daily creates = 1.73M URLs
      Cache memory: 1.73M × 515 bytes ≈ 1 GB → easily fits in Redis

SHORT CODE GENERATION STRATEGIES:

  STRATEGY 1: MD5/SHA-256 hash of URL (BAD):
    short_code = MD5(long_url)[0:7]
    Problem: hash collisions (two different URLs → same short code)
    Problem: same URL shortened twice → same code (OK for dedup, bad for analytics)

  STRATEGY 2: UUID → Base62 (simpler but large):
    UUID: 128-bit → Base62 needs 22 characters. Too long.

  STRATEGY 3: Auto-increment ID → Base62 (GOOD, with coordination issue):
    id = AUTO_INCREMENT in MySQL
    short_code = base62(id)

    id=1 → "1" (1 char), id=62 → "10" (Base62), id=62^7=3.5T → "aaaaaaa" (7 chars)

    Coordination issue: multiple write servers = multiple sequences = ID collision
    Solution: single sequence service (bottleneck) or pre-generated key pools

  STRATEGY 4: SNOWFLAKE ID → Base62 (PRODUCTION CHOICE):
    Snowflake ID: 64-bit = timestamp(41) + datacenter(5) + machine(5) + sequence(12)
    Globally unique without coordination. ~4096 IDs/ms per machine.

    short_code = base62(snowflake_id)
    Base62(snowflake_id) = ~11 chars (snowflake IDs are large ~1.8×10^18)

    For 7-char codes: use only lower 43 bits of Snowflake (less unique but sufficient for URL shortener scale)

  STRATEGY 5: KEY GENERATION SERVICE (KGS) — Pre-generated keys (BEST for URL shorteners):
    KGS: background job pre-generates millions of unique 7-char Base62 codes.
    Stores in "unused_keys" table.

    URL creation:
      1. Fetch one pre-generated key from KGS (mark as used)
      2. Store: (short_code → long_url) in database
      3. Return short_code to client

    No hash computation, no collision checking, no coordination.
    KGS generates keys offline in bulk → atomic mark-as-used via DB transactions.

    KGS can cache a batch of keys in memory per application server:
      Each server pre-fetches 1,000 keys from KGS on startup.
      Uses keys locally (no KGS call per request).
      Replenish when local pool drops below 100.
      Server crash: at most 1,000 keys lost. Acceptable.

DATABASE SCHEMA:

  urls table:
  CREATE TABLE urls (
    short_code  VARCHAR(7)   PRIMARY KEY,
    long_url    VARCHAR(2048) NOT NULL,
    created_at  TIMESTAMP    DEFAULT NOW(),
    expires_at  TIMESTAMP    NULL,           -- NULL = permanent
    user_id     BIGINT       NULL,           -- creator (for analytics)
    custom_alias BOOLEAN     DEFAULT FALSE   -- was this a custom alias?
  );

  -- Shard by short_code (hash partitioning):
  -- 4 shards: short_code % 4 (or consistent hashing)
  -- Rationale: reads (redirect lookups) are by short_code → single shard lookup

  -- Index on long_url for deduplication check (optional):
  -- CREATE INDEX idx_long_url ON urls(long_url(255));  -- prefix index for long URLs

REDIRECT FLOW (the hot path):

  Client: GET /abc1234

  1. CDN (CloudFront/Cloudflare):
       Is this URL in CDN cache? → YES → return 301 redirect (permanent, cached at CDN edge)
       NO → forward to application server

  2. Application Server:
       Is this URL in Redis? → YES → return 302 redirect from Redis
       NO → query MySQL shard

  3. MySQL Shard (only on cache miss):
       SELECT long_url, expires_at FROM urls WHERE short_code = 'abc1234'
       Check expires_at: if expired → return 410 Gone

  4. Populate Redis cache (TTL = 24 hours):
       SET url:abc1234 "https://..." EX 86400

  5. Return HTTP 302 Location: https://...

  301 vs 302:
    301 Permanent: browser caches redirect → no future server requests → reduces load
    302 Temporary: browser always asks server → server tracks every click → analytics
    Choice: 302 for analytics (know how many times each link clicked); 301 for performance

ARCHITECTURE DIAGRAM:

  [Client] → [CDN Edge] → [Load Balancer]
                                │
                    ┌───────────┴───────────┐
                    │   App Server Fleet    │
                    │   (URL Servers)       │
                    └───────────┬───────────┘
                                │
              ┌─────────────────┼─────────────────┐
              │                 │                 │
         [Redis Cache]    [MySQL Shard 1]   [MySQL Shard 2]
          (hot URLs)      (short_code: a-m)  (short_code: n-z)
              │
         [KGS Service]   [MySQL Shard 3]   [MySQL Shard 4]
         (key pool)       (short_code: ...)  (short_code: ...)
```

### ❓ Why Does This Exist (Why Before What)

WITHOUT URL Shortener design considerations:

- Single database: 31.5 billion rows, 16 TB → table scans slow, single point of failure
- No caching: 100K lookups/second → MySQL overwhelmed (handles ~50K simple reads/sec)
- MD5 hash for short codes: collisions + no collision resolution = broken links

WITH proper URL shortener design:
→ KGS pre-generated keys: zero collision, O(1) code generation
→ Redis caching: 80% of reads served from memory, MySQL only for cold cache misses
→ Hash-sharded MySQL: 16 TB split across 4 nodes, each handles 25K redirects/sec

### 🧠 Mental Model / Analogy

> A coat check at a concert venue. You hand in your coat (long URL), receive a numbered ticket (short code). The ticket number is from a pre-printed book — no coordination needed, no collisions. To retrieve your coat: show your numbered ticket (short code lookup). The coat check attendant knows exactly which rack to go to (shard routing). Popular ticket numbers (frequently clicked links) are pinned to the front desk for instant retrieval (Redis cache). The full coat rack (database) is only consulted for unusual ticket numbers.

"Numbered ticket" = short code (Base62 encoded unique ID)
"Pre-printed ticket book" = KGS pre-generated key pool (no collision, no coordination)
"Coat retrieval from ticket" = redirect lookup (short_code → long_url)
"Front desk with popular items" = Redis cache (hot URLs served from memory)
"Full coat rack" = sharded MySQL (cold URL lookups on cache miss)

### ⚙️ How It Works (Mechanism)

**URL shortener: full implementation in Spring Boot:**

```java
@RestController
@RequestMapping
public class UrlShortenerController {

    @Autowired private UrlRepository urlRepository;       // MySQL (sharded)
    @Autowired private RedisTemplate<String,String> redis;
    @Autowired private KeyGenerationService kgs;

    // POST /shorten — create short URL:
    @PostMapping("/shorten")
    public ResponseEntity<ShortenResponse> shorten(@RequestBody ShortenRequest request) {
        String longUrl = request.getLongUrl();

        // Validate URL (security: prevent open redirect abuse):
        if (!isValidUrl(longUrl)) {
            return ResponseEntity.badRequest().build();
        }

        // Optional: check for malicious URL against blocklist:
        if (maliciousUrlChecker.isBlocked(longUrl)) {
            return ResponseEntity.status(422).build();
        }

        // Get pre-generated key from KGS:
        String shortCode = kgs.getNextKey();

        // Store in MySQL (shard by shortCode):
        UrlMapping mapping = new UrlMapping(shortCode, longUrl, request.getExpiresAt());
        urlRepository.save(mapping);

        // Cache in Redis immediately (avoid cold start):
        redis.opsForValue().set("url:" + shortCode, longUrl, 24, TimeUnit.HOURS);

        return ResponseEntity.ok(new ShortenResponse(
            "https://short.ly/" + shortCode,
            shortCode
        ));
    }

    // GET /{shortCode} — redirect:
    @GetMapping("/{shortCode}")
    public ResponseEntity<Void> redirect(@PathVariable String shortCode) {
        // 1. Check Redis cache first (O(1), ~0.1ms):
        String longUrl = redis.opsForValue().get("url:" + shortCode);

        if (longUrl == null) {
            // 2. Cache miss: query MySQL (O(log n), ~5ms):
            UrlMapping mapping = urlRepository.findByShortCode(shortCode)
                .orElse(null);

            if (mapping == null) {
                return ResponseEntity.notFound().build();
            }

            if (mapping.isExpired()) {
                return ResponseEntity.status(410).build();  // 410 Gone
            }

            longUrl = mapping.getLongUrl();

            // 3. Populate cache (future requests: served from Redis):
            redis.opsForValue().set("url:" + shortCode, longUrl, 24, TimeUnit.HOURS);
        }

        // 4. Return 302 redirect (302 = analytics; 301 = browser caches, no analytics):
        return ResponseEntity.status(302)
            .header("Location", longUrl)
            .build();
    }

    private boolean isValidUrl(String url) {
        try {
            URI uri = new URI(url);
            return "http".equals(uri.getScheme()) || "https".equals(uri.getScheme());
        } catch (URISyntaxException e) {
            return false;
        }
    }
}
```

### 💻 Code Example

**Base62 encoding/decoding for ID → short code conversion:**

```java
public class Base62 {

    private static final String ALPHABET =
        "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789";
    private static final int BASE = 62;

    // Encode a long ID to Base62 string:
    public static String encode(long id) {
        if (id == 0) return String.valueOf(ALPHABET.charAt(0));

        StringBuilder sb = new StringBuilder();
        while (id > 0) {
            sb.append(ALPHABET.charAt((int)(id % BASE)));
            id /= BASE;
        }
        return sb.reverse().toString();
    }

    // Decode Base62 string back to long ID:
    public static long decode(String shortCode) {
        long result = 0;
        for (char c : shortCode.toCharArray()) {
            result = result * BASE + ALPHABET.indexOf(c);
        }
        return result;
    }

    // Example:
    // encode(1)          → "a"
    // encode(62)         → "ba"  (62 = 1*62 + 0 → digits [1,0] → "ba")
    // encode(1_000_000)  → "4c92" (4 chars)
    // encode(62^7 - 1)   → "9999999" (7 chars, max 7-char code)
}
```

### ⚠️ Common Misconceptions

| Misconception                                      | Reality                                                                                                                                                                                                                                                                                                                                                         |
| -------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A URL shortener is trivial to build                | The redirect endpoint (100K req/sec) requires Redis caching, CDN integration, and sharded database. The creation endpoint needs a collision-free ID generation strategy. Handling 31.5 billion URLs at 16 TB with expiry management, analytics, and abuse prevention makes it a realistic medium-complexity system design problem                               |
| MD5 hash is a good short code generator            | MD5 collisions are possible (though rare for URL inputs). More importantly: MD5 doesn't guarantee unique short codes without collision checking (expensive at 31.5B rows), and produces the same code for the same URL every time (can't create multiple short codes for the same long URL for different campaigns). KGS pre-generated keys are strictly better |
| HTTP 301 is always better than 302 for performance | 301 (permanent) causes browsers to cache the redirect and never contact the server again — great for performance, terrible for analytics (can't count clicks). 302 (temporary/found) forces browser to always check the server — slower but enables click counting. Choose based on whether analytics matter                                                    |
| The database is the main bottleneck                | For a URL shortener, the database is rarely the bottleneck because Redis caches 80%+ of redirects. The actual bottleneck is usually network throughput (bandwidth for 100K responses/sec) or application server CPU (parsing HTTP, Redis calls). Properly sized Redis cluster handles the hot path entirely in memory                                           |

### 🔥 Pitfalls in Production

**Cache stampede on popular URL first click:**

```
PROBLEM: Viral URL created → millions of concurrent first-clicks before cache is warm

  10:00:00 AM: New viral link created → stored in DB, NOT YET in Redis (cache cold)
  10:00:01 AM: Tweet goes viral → 50,000 users click in first second

  All 50,000 requests: Redis MISS → all query MySQL simultaneously
  MySQL shard: 50,000 concurrent queries for same row → CPU 100% → timeouts

  This is CACHE STAMPEDE (thundering herd for a single key).

BAD: Simple cache-aside without stampede protection:
  String longUrl = redis.get("url:" + shortCode);  // all 50K get cache miss
  if (longUrl == null) {
    longUrl = db.findByShortCode(shortCode);       // all 50K hit DB simultaneously
    redis.set("url:" + shortCode, longUrl);
  }

FIX 1: WARM CACHE ON WRITE:
  When URL is created → immediately add to Redis.
  → First click is always a Redis HIT (no cold start).

  // In /shorten endpoint:
  urlRepository.save(mapping);
  redis.set("url:" + shortCode, longUrl, 24, TimeUnit.HOURS);  // pre-warm!

FIX 2: MUTEX LOCKING (for cache misses on existing URLs):
  String lockKey = "lock:url:" + shortCode;
  String longUrl = redis.get("url:" + shortCode);

  if (longUrl == null) {
    // Acquire lock: only ONE thread queries DB for this key:
    Boolean acquired = redis.setIfAbsent(lockKey, "1", 1, TimeUnit.SECONDS);
    if (acquired) {
      longUrl = db.findByShortCode(shortCode);
      redis.set("url:" + shortCode, longUrl, 24, TimeUnit.HOURS);
      redis.delete(lockKey);
    } else {
      // Another thread is fetching: wait briefly and retry:
      Thread.sleep(10);
      longUrl = redis.get("url:" + shortCode);  // should be populated now
    }
  }

FIX 3: PROBABILISTIC EARLY EXPIRY (PER — prevent expiry stampede):
  Before TTL expires, randomly start refreshing early:
  if (ttl < threshold && Math.random() < probability) {
    // This thread proactively refreshes before expiry
    // Other threads: still serve stale but valid data
  }
```

### 🧠 Think About This Before We Continue

**Q1.** The KGS (Key Generation Service) pre-generates 7-character Base62 codes and stores them in an `unused_keys` table. Multiple application servers each pre-fetch a batch of 1,000 keys to cache locally. Design the KGS: how do you prevent two application servers from receiving the same key? What database operations ensure this atomically? What happens to the ~1,000 cached keys in a server's memory when that server crashes? Is this acceptable? How do you monitor that the KGS pool doesn't run out of keys?

**Q2.** You want to add analytics to the URL shortener: for each short code, track total clicks, unique visitors (by IP hash), top 5 countries, and hourly click distribution. Describe the data pipeline: (a) how do you record click events without adding latency to the 302 redirect hot path? (b) where do you store the analytics data (not in the same MySQL shards as URL mappings)? (c) how do you handle 100K click events/second being ingested for analytics without overwhelming your storage system?
