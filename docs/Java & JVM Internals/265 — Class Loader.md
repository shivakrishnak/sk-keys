# 265 — Class Loader

⚡ TL;DR — The Class Loader is the JVM component that finds, loads, and links `.class` files into memory on demand at runtime.

### 🔥 The Problem This Solves

WORLD WITHOUT IT:
Imagine a JVM that had to load every class in the entire classpath before starting. A Spring Boot application with 500 dependencies would spend 30 seconds scanning and loading thousands of classes before running a single line of application code — even for classes never used in that execution. Worse, how would the JVM load a class defined only at runtime (e.g., a Groovy script compiled on the fly, or a CGLIB proxy generated by Spring)? There would be no mechanism for dynamic class introduction.

THE BREAKING POINT:
Two distinct problems collide: lazy loading (don't load what you don't need) and dynamic extensibility (load things that didn't exist at JVM startup). A static "load everything at launch" approach fails both. There needs to be a well-defined protocol for resolving class names to bytecode — one that can be customised.

THE INVENTION MOMENT:
The Class Loader system solves both: classes are loaded on first reference (lazy), and the loading mechanism is pluggable, allowing frameworks to define entirely new class sources at runtime. This is exactly why the Class Loader was created.

### 📘 Textbook Definition

A ClassLoader is a JVM component responsible for loading class definitions (from `.class` files, JARs, network, databases, or generated bytecode) into the JVM's Method Area. The JVM uses a delegation hierarchy: the Bootstrap ClassLoader → Extension (Platform) ClassLoader → Application ClassLoader → custom loaders. When a class is referenced, the JVM delegates upward to the parent loader first; only if the parent cannot find the class does the child attempt to load it. This parent-delegation model prevents user code from replacing core library classes.

### ⏱️ Understand It in 30 Seconds

**One line:**
The Class Loader finds and imports a class's bytecode into the JVM the first time it's needed.

**One analogy:**
> A library runs on a "fetch on demand" system. They don't put every book on every table — a librarian fetches a book only when a reader requests it. Once fetched, the book stays on the reading room shelf for reuse. The Class Loader is the librarian: it fetches class definitions on first request and caches them (in the Method Area) for subsequent use.

**One insight:**
The most non-obvious thing about Class Loaders: two loaded classes with identical names but different Class Loaders are considered different classes by the JVM. This is the mechanism behind OSGi, application server isolation, and hot reloading — and the root cause of `ClassCastException` when the same class is loaded by two loaders.

### 🔩 First Principles Explanation

CORE INVARIANTS:
1. A class is identified by its fully qualified name AND the Class Loader that loaded it — not by name alone.
2. Parent delegation: before loading, always ask the parent loader first.
3. Class loading is lazy — a class is loaded only when first referenced.

DERIVED DESIGN:
Invariant 1 creates the "class namespace" concept — different application modules can use different versions of the same library without conflict, because they use different Class Loaders. Invariant 2 ensures the JDK's `java.lang.String` can never be replaced by user code (the Bootstrap loader owns it; user loaders always defer to it). Invariant 3 enables fast startup — unused classes are never loaded.

THE TRADE-OFFS:
Gain: Lazy loading, dynamic extensibility, module isolation, security via parent-first delegation.
Cost: Two different Class Loaders loading the same class name create separate types — a source of subtle `ClassCastException` bugs. Class Loader leaks cause memory problems in application servers (PermGen/Metaspace exhaustion).

### 🧪 Thought Experiment

SETUP:
An application server (e.g., Tomcat) runs two web applications: `webapp-A` (uses Guava 28) and `webapp-B` (uses Guava 32). Both JARs contain `com.google.common.collect.ImmutableList`.

WHAT HAPPENS WITHOUT ISOLATED CLASS LOADERS:
A single Application ClassLoader loads Guava. Only one version can be loaded — the first one wins. `webapp-A` gets Guava 32's `ImmutableList` instead of 28's. One of the apps crashes with `NoSuchMethodError` because a method changed between versions. Deploying `webapp-A` can break `webapp-B`.

WHAT HAPPENS WITH ISOLATED CLASS LOADERS:
Tomcat gives each webapp its own ClassLoader. `webapp-A`'s ClassLoader loads Guava 28. `webapp-B`'s ClassLoader loads Guava 32. Both `ImmutableList` classes exist in the JVM simultaneously, under different ClassLoaders. Each webapp gets its own version. Neither can affect the other.

THE INSIGHT:
Class Loaders enable dependency isolation within a single JVM. This is the foundation of every Java application server and OSGi module system — the same principle behind containers, applied at the JVM level.

### 🧠 Mental Model / Analogy

> Think of Class Loaders like concentric security clearance zones at a government facility. The innermost zone (Bootstrap Loader) holds the most sensitive files (core JDK classes). To get a document, you always check the innermost zone first. Only if it's not there do you check the next zone outward. This means nobody in an outer zone can secretly replace a document in the inner zone.

"Inner zone" → Bootstrap ClassLoader (JDK core: java.lang.*)
"Middle zone" → Platform ClassLoader (JDK extensions)
"Outer zone" → Application ClassLoader (your app's classpath)
"Custom outer zones" → custom ClassLoaders (Spring, OSGi)
"Clearance check order" → parent-delegation: inner first

Where this analogy breaks down: unlike secure facilities, you can bypass parent delegation by overriding `loadClass()` directly — though doing so is almost always a mistake.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
When your Java program uses a class for the first time, something needs to find that class's compiled code and make it available. The Class Loader does this job — it finds the `.class` file, reads it, and prepares the class for use. It then remembers the class so future uses are instant.

**Level 2 — How to use it (junior developer):**
You rarely interact with Class Loaders directly. Use `Class.forName("com.example.MyClass")` to load a class by name. Use `Thread.currentThread().getContextClassLoader()` to get the context loader. Understanding ClassLoaders matters most when debugging `ClassNotFoundException`, `NoClassDefFoundError`, or `ClassCastException` with "already loaded by a different ClassLoader" messages.

**Level 3 — How it works (mid-level engineer):**
Loading a class goes through three phases: (1) Loading — reads bytecode into a byte array; (2) Linking — Verification + Preparation (allocate static field memory) + Resolution (replace symbolic references with direct references); (3) Initialisation — runs `<clinit>` static initialiser blocks. `Class.forName("X")` triggers all three phases. `ClassLoader.loadClass("X")` triggers only loading (not initialisation). This distinction matters for JDBC driver loading patterns.

**Level 4 — Why it was designed this way (senior/staff):**
The parent-delegation model was controversial — it prevents plugin systems from providing their own implementation of a "standard" interface. This is why OSGi and Tomcat both override it: a web app's ClassLoader checks itself first, not the parent, for application classes. Java 9's module system replaces the flat classpath model with named modules, changing ClassLoader behaviour: the Platform module loader now understands module boundaries, making the flat parent-delegation chain largely obsolete for modular applications.

### ⚙️ How It Works (Mechanism)

**The Three ClassLoaders (Java 8):**

```
┌─────────────────────────────────────────────┐
│      CLASS LOADER HIERARCHY (Java 8)        │
├─────────────────────────────────────────────┤
│  Bootstrap ClassLoader                      │
│  └─ Loads: java.lang.*, java.util.*,        │
│             java.io.* etc. (rt.jar)          │
│  └─ Written in: C/C++ (native)              │
│      ↓ parent of                            │
│  Extension ClassLoader                      │
│  └─ Loads: $JAVA_HOME/lib/ext/*.jar         │
│      ↓ parent of                            │
│  Application ClassLoader                    │
│  └─ Loads: everything on -classpath         │
│      ↓ parent of (optional)                 │
│  Custom ClassLoaders                        │
│  └─ Loads: plugins, hot-reloaded code,      │
│             dynamically generated classes   │
└─────────────────────────────────────────────┘
```

**Parent Delegation Flow:**

```
┌─────────────────────────────────────────────┐
│   loadClass("com.example.MyClass") called   │
│   on Application ClassLoader                │
├─────────────────────────────────────────────┤
│  1. Check cache: already loaded? → return   │
│     ↓ not cached                            │
│  2. Delegate to parent (Extension CL)       │
│     → Extension CL delegates to Bootstrap  │
│     → Bootstrap: found? → return           │
│     → Bootstrap: not found → back to Ext   │
│     → Extension: found? → return           │
│     → Extension: not found → back to App   │
│  3. Application CL tries to load from disk  │
│     → Found: load, verify, cache, return   │
│     → Not found → ClassNotFoundException   │
└─────────────────────────────────────────────┘
```

**Class Loading Phases (detailed):**
1. **Loading**: Find the bytecode (file, JAR, network). Call `defineClass()` to create a `Class<?>` object.
2. **Verification**: Bytecode verifier runs all safety checks.
3. **Preparation**: JVM allocates memory for static fields and initialises to default values (0, null, false).
4. **Resolution**: Symbolic references (class names as strings) in the constant pool are replaced with direct memory references to `Class<?>` objects.
5. **Initialisation**: `<clinit>` method runs static initialiser blocks and static field assignments. Thread-safe — guaranteed to run exactly once.

### 🔄 The Complete Picture — End-to-End Flow

NORMAL FLOW:
```
Code references com.example.MyService for first time
  → JVM calls loadClass on context ClassLoader
  ← YOU ARE HERE (ClassLoader resolves the name)
  → Parent delegation chain checked
  → .class file found on classpath
  → Bytecode loaded and verified
  → Class prepared and resolved
  → <clinit> runs (static init)
  → Class object cached in ClassLoader
  → Reference resolved; code continues
```

FAILURE PATH:
```
ClassNotFoundException
  → No ClassLoader in hierarchy found the class
  → Missing JAR on classpath, or wrong ClassLoader
  → Symptom: startup failure or NoClassDefFoundError
    at runtime if class was available at compile time
    but not at runtime
```

WHAT CHANGES AT SCALE:
In a large application server with many hot-deployed applications, each deployment creates new ClassLoader instances. If ClassLoaders are not properly garbage collected (because a static reference in a framework holds a reference to a class loaded by the old ClassLoader), the old class's Metaspace/PermGen can never be freed. This is the "ClassLoader leak" — the leading cause of Metaspace `OutOfMemoryError` in long-running application servers.

### 💻 Code Example

Example 1 — Check which ClassLoader loaded a class:
```java
// Check the ClassLoader for any class
Class<?> cls = String.class;
// null → loaded by Bootstrap ClassLoader (native)
System.out.println(cls.getClassLoader());

Class<?> appCls = MyApplication.class;
// sun.misc.Launcher$AppClassLoader@...
System.out.println(appCls.getClassLoader());
```

Example 2 — Load a class dynamically by name:
```java
// BAD: assumes class in current classpath only
Class<?> cls = Class.forName("com.example.Plugin");

// GOOD: use context ClassLoader for frameworks
ClassLoader cl =
    Thread.currentThread().getContextClassLoader();
Class<?> cls = Class.forName(
    "com.example.Plugin", true, cl
);
Object instance = cls.getDeclaredConstructor()
                     .newInstance();
```

Example 3 — Custom ClassLoader for hot reloading:
```java
public class HotReloadClassLoader
        extends ClassLoader {

    private final Path classDir;

    public HotReloadClassLoader(Path classDir,
            ClassLoader parent) {
        super(parent);
        this.classDir = classDir;
    }

    @Override
    protected Class<?> findClass(String name)
            throws ClassNotFoundException {
        // Load from disk on every call (bypasses cache)
        // enabling hot reload on file change
        Path file = classDir.resolve(
            name.replace('.', '/') + ".class"
        );
        try {
            byte[] bytes = Files.readAllBytes(file);
            // defineClass registers it with this loader
            return defineClass(name, bytes, 0,
                               bytes.length);
        } catch (IOException e) {
            throw new ClassNotFoundException(name, e);
        }
    }
}
```

Example 4 — Debugging ClassLoader issues:
```bash
# Verbose class loading — shows which loader loaded what
java -verbose:class -jar myapp.jar 2>&1 | \
  grep "Loaded com.example"

# Show class loading with loader name (Java 17+)
java -Xlog:class+load=info -jar myapp.jar 2>&1 | \
  grep "my.app"
```

### ⚖️ Comparison Table

| ClassLoader Type | Scope | Parent | Use Case |
|---|---|---|---|
| **Bootstrap** | JDK core (java.*) | None (native) | JVM internals |
| Platform (Java 9+) | JDK modules | Bootstrap | JDK extension modules |
| Application | App classpath | Platform | Normal applications |
| Custom (e.g., WebApp) | Plugin/module | Application | App servers, OSGi, hot reload |
| URL ClassLoader | Any URL | Configurable | Remote class loading |

How to choose: You almost never choose a ClassLoader — the JVM provides the right one. Create a custom ClassLoader only when you need isolation (multiple versions of the same library), hot reloading, or loading classes from non-standard sources (database, network).

### ⚠️ Common Misconceptions

| Misconception | Reality |
|---|---|
| "ClassCastException means incompatible types" | It can also mean the same class loaded by two different ClassLoaders. The JVM sees them as different types even though the bytecode is identical. |
| "Class.forName always uses the caller's ClassLoader" | By default yes, but in multi-ClassLoader environments (app servers), you need `Thread.currentThread().getContextClassLoader()` to get the right one. |
| "A class is loaded once and stays forever" | In app servers with hot deployment, old ClassLoaders can be GC'd when no references remain. A ClassLoader leak prevents this, causing Metaspace exhaustion. |
| "static fields are global across the JVM" | Static fields are per-ClassLoader. Two ClassLoaders loading the same class get separate static field instances — a common bug in plugin systems. |

### 🚨 Failure Modes & Diagnosis

**1. ClassNotFoundException vs NoClassDefFoundError**

Symptom: `ClassNotFoundException` at runtime — class not found. `NoClassDefFoundError` — class was found at compile time but not at runtime classpath.

Root Cause: Missing JAR at runtime, wrong ClassLoader used, or class excluded from the runtime classpath.

Diagnostic:
```bash
# Check runtime classpath
java -cp "lib/*:." -verbose:class \
  com.example.Main 2>&1 | grep "MyMissing"

# List JAR contents to verify class is there
jar tf mylib.jar | grep MyMissingClass
```

Prevention: Use dependency management (Maven/Gradle) with `compile` vs `provided` scopes correctly. Test with the exact classpath used in production.

**2. ClassCastException from Duplicate ClassLoaders**

Symptom: `ClassCastException: com.example.Plugin cannot be cast to com.example.Plugin` — same class name, different ClassLoader instances.

Root Cause: The same class was loaded by two different ClassLoaders (e.g., app ClassLoader and plugin ClassLoader both load `Plugin.class`).

Diagnostic:
```java
// Diagnose: print ClassLoader of both sides
System.out.println(plugin.getClass().getClassLoader());
System.out.println(Plugin.class.getClassLoader());
// If different: ClassLoader isolation is the cause
```

Fix:
```java
// BAD: cast directly
Plugin plugin = (Plugin) loadedObject;

// GOOD: use interface from shared ClassLoader
// Define API interfaces in a shared module loaded
// by parent ClassLoader; plugins implement them
```

Prevention: Put shared API interfaces in a module loaded by a common parent ClassLoader; keep implementations in child ClassLoaders.

**3. Metaspace OutOfMemoryError from ClassLoader Leak**

Symptom: `java.lang.OutOfMemoryError: Metaspace` in long-running application server after multiple hot deploys.

Root Cause: Old ClassLoader instances not GC'd because a static field, ThreadLocal, or JVM internal (JDBC driver) holds a reference to a class loaded by the old ClassLoader.

Diagnostic:
```bash
# Check Metaspace usage
jcmd <pid> VM.native_memory summary

# Heap dump + Eclipse MAT analysis:
# Search for ClassLoader instances with high
# retained heap — they are the leaked loaders
jmap -dump:format=b,file=/tmp/heap.hprof <pid>
```

Prevention: Deregister JDBC drivers on `contextDestroyed`. Avoid static references to classes from the webapp ClassLoader. Use `javaagent` memory leak detectors in CI.

### 🔗 Related Keywords

**Prerequisites (understand these first):**
- `JVM` — the runtime environment that uses ClassLoaders to load all classes
- `Bytecode` — the binary format that ClassLoaders read from `.class` files

**Builds On This (learn these next):**
- `Metaspace` — where loaded class metadata is stored; ClassLoader leaks exhaust Metaspace
- `Reflection` — uses ClassLoaders to load classes by name at runtime
- `Spring Core` — Spring's ApplicationContext uses custom ClassLoaders for component scanning and proxy generation

**Alternatives / Comparisons:**
- `Java Module System (JPMS)` — Java 9's module system partially replaces classpath-based ClassLoading with explicit module declarations
- `OSGi` — a full dynamic module system using per-bundle ClassLoaders for fine-grained isolation
