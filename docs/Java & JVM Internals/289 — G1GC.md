# 289 — G1GC

`#java` `#jvm` `#gc` `#memory` `#internals` `#deep-dive`

⚡ TL;DR — Region-based, concurrent garbage collector (default since Java 9) that balances throughput and pause time by evacuating live objects into fresh regions.

### 📘 Textbook Definition

**G1GC (Garbage First Garbage Collector)** divides the JVM heap into a set of equal-sized regions (1–32 MB each) instead of fixed Young/Old areas. Each region is dynamically assigned a role (Eden, Survivor, Old, Humongous) and can be independently collected. G1 performs concurrent marking to identify live objects and selects the regions with the most garbage ("Garbage First") for evacuation, copying live objects to free regions. This compacts memory while maintaining predictable pause-time targets set via `-XX:MaxGCPauseMillis`.

### 🟢 Simple Definition (Easy)

G1GC divides the heap into small equal chunks, figures out which chunks have the most garbage, and cleans those first — giving you predictable, short pauses.

### 🔵 Simple Definition (Elaborated)

Traditional collectors divide the heap into fixed Young and Old areas. G1 instead slices the entire heap into hundreds of equal regions. During a background concurrent phase, G1 identifies how much garbage each region contains. Then, during a pause, it picks the set of regions that yields the most freed memory within your pause-time budget (e.g., `-XX:MaxGCPauseMillis=200`). Live objects in chosen regions are copied elsewhere, and the original regions become free. This copy-compaction prevents fragmentation — a key improvement over CMS.

### 🔩 First Principles Explanation

**Problem with CMS:** CMS solves pause time for Old Gen but leaves fragmentation. Over time, the Old Generation becomes a Swiss cheese of free fragments — too many small holes and not enough contiguous space for large object promotions.

**Problem with Parallel GC:** Parallel GC compacts, avoiding fragmentation, but the entire Old Gen must stop the world during compaction — causing pauses proportional to live data size (seconds on large heaps).

**Insight:** If you break the heap into small regions, you can compact individual regions one at a time. Regions with mostly dead objects can be evacuated (live objects moved out) using copy-collection — which inherently compacts. You can bound the pause time by choosing only as many regions as can be evacuated within the time budget.

**Region sizing:** G1 auto-selects region size as heap-size / 2048, clamped to 1–32 MB. Regions > 50% of region size become "Humongous" regions allocated directly in Old Gen (contiguous regions for very large objects).

**Remembered Sets (RSets):** Each region maintains a RSet — a data structure recording which other regions have pointers into it. This allows G1 to collect individual regions without scanning the entire heap for incoming references. RSets are updated by write barriers at every reference store.

**Collection Sets (CSets):** The set of regions chosen for evacuation in a given GC cycle. The scheduler selects regions predicted to yield the most garbage within the pause budget using historical data (MMU — Minimum Mutator Utilisation).

### ❓ Why Does This Exist (Why Before What)

WITHOUT G1GC (using CMS or Parallel):

- CMS: no compaction → fragmentation → unpredictable Full GC pauses.
- Parallel GC: great throughput but 1–4 second STW pauses on large heaps.
- Neither can be tuned to a specific pause time target.

What breaks without it:
1. P99 latency SLAs are violated unpredictably due to fragmentation or long compaction pauses.
2. Heap sizing becomes a guessing game without pause-time targeting.

WITH G1GC:
→ `MaxGCPauseMillis` gives a soft pause-time target the JVM tries to honour.
→ Concurrent marking identifies garbage without stopping the application.
→ Evacuation compacts regions — no fragmentation accumulation.
→ Default collector since Java 9 — no flags needed for most workloads.

### 🧠 Mental Model / Analogy

> Imagine a city with hundreds of city blocks (regions). Each block has houses (objects), some occupied (live), some abandoned (garbage). Instead of trying to clean the entire city at once, urban planners identify the blocks with the most abandoned houses. They move the remaining residents from those blocks to other blocks, then demolish the whole block and build fresh (free region). The city never needs to bulldoze every block simultaneously — just the worst ones, one group at a time.

"City blocks" = G1 regions, "abandoned houses" = dead objects, "moving residents" = object evacuation, "demolish and rebuild" = region freed, "identifying worst blocks" = concurrent marking + reclaimable space tracking.

The genius: you can do as many or as few blocks per "pause" as will fit in the time budget.

### ⚙️ How It Works (Mechanism)

**Heap Layout:**
```
G1GC Heap (e.g., 4 GB, 2 MB regions = 2048 regions)
┌─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┐
│E│E│S│O│O│F│E│H│H│O│F│S│E│O│F│O│ ...
└─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┴─┘
E=Eden S=Survivor O=Old F=Free H=Humongous
```

**G1 GC Cycle:**

1. **Young GC (STW):** Evacuates all Eden and Survivor regions. Live objects copied to new Survivor/Old regions. All Eden regions become free.

2. **Concurrent Marking Cycle** (when Old Gen reaches `InitiatingHeapOccupancyPercent`, default 45%):
   - Initial Mark (STW, piggybacks on Young GC): marks GC roots.
   - Concurrent Root Region Scan: scans Survivor regions concurrently.
   - Concurrent Mark: traverses object graph concurrently.
   - Remark (STW): processes SATB (Snapshot-At-The-Beginning) buffers.
   - Cleanup (STW for accounting, concurrent for freeing regions): sorts by reclaimable space.

3. **Mixed GC (STW):** Evacuates all Young regions PLUS a selected subset of Old regions with highest garbage ratio. Continues until reclaimable Old Gen space drops below threshold.

4. **Full GC (last resort STW):** If evacuation fails (no free region to copy into), G1 falls back to serial full compaction of the entire heap.

**SATB Write Barrier:** G1 uses Snapshot-At-The-Beginning — at Remark, it processes objects that were grey at concurrent marking start. This is cheaper than CMS's card table scanning for Remark.

### 🔄 How It Connects (Mini-Map)

```
Young GC (Eden/Survivor evacuation)
           ↓
  Old Gen reaches IHOP (45%)
           ↓
  Concurrent Marking Cycle
  ┌─────────────────────────────┐
  │ Initial Mark → Concurrent   │
  │ Mark → Remark → Cleanup     │
  └─────────────────────────────┘
           ↓
    Mixed GC ← you are here
  (Young + selected Old regions)
           ↓
  Evacuation Failure?
    └── Yes → Full GC (fallback)
    └── No  → Continue normal cycle
```

### 💻 Code Example

Example 1 — Recommended G1GC configuration for a latency-sensitive service:

```bash
java -Xms8g -Xmx8g \
     -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \
     -XX:InitiatingHeapOccupancyPercent=35 \
     -XX:G1HeapRegionSize=4m \
     -XX:G1NewSizePercent=20 \
     -XX:G1MaxNewSizePercent=40 \
     -XX:G1MixedGCLiveThresholdPercent=65 \
     -Xlog:gc*:file=g1gc.log:time,uptime,level,tags \
     MyApp
```

Example 2 — Monitoring G1GC via JFR:

```bash
# Start JFR recording
jcmd <pid> JFR.start \
  duration=120s \
  settings=profile \
  filename=g1gc_profile.jfr

# Analyse with JDK Mission Control (jmc)
# or with jfr command:
jfr print --events GCHeapSummary,GarbageCollection \
  g1gc_profile.jfr
```

Example 3 — G1GC region size and Humongous allocation:

```java
// With G1HeapRegionSize=4m:
// Objects > 2 MB (50% of region) → Humongous region
// These bypass Young Gen entirely!

// BAD: Frequently allocating large strings or byte arrays
byte[] bigBuffer = new byte[3 * 1024 * 1024]; // → Humongous

// GOOD: Pool and reuse large buffers to avoid humongous churn
// Use BufferPool or off-heap (ByteBuffer.allocateDirect)
```

### 🔁 Flow / Lifecycle

```
App allocates objects in Eden regions
           │
           ▼
Eden regions fill → Young GC (STW)
           │ Live objects evacuated to Survivor/Old
           ▼
Old Gen reaches IHOP threshold (e.g. 45%)
           │
           ▼
Concurrent Marking begins (app running)
           │
Initial Mark (piggybacked on Young GC, STW)
           │
Concurrent Root Region Scan (app running)
           │
Concurrent Mark (app running)
           │
Remark (STW) - SATB processing
           │
Cleanup (mostly concurrent)
           │
           ▼
Mixed GC cycles (STW)
  - Evacuate all Young regions
  - Evacuate selected high-garbage Old regions
           │
Repeat until Old Gen reclaimable < threshold
           │
           ▼
Return to normal Young GC cycles
```

### ⚠️ Common Misconceptions

| Misconception | Reality |
|---|---|
| G1GC eliminates all long pauses | Evacuation failure (no free region) triggers a serial Full GC — potentially longer than CMS Concurrent Mode Failure. |
| MaxGCPauseMillis is a hard guarantee | It's a soft target. G1 makes a best-effort prediction; the actual pause can exceed it, especially with Humongous allocations or large live data. |
| G1GC is always faster than Parallel GC | For batch/throughput workloads with large heaps, Parallel GC can be faster because Parallel GC's throughput is higher with no concurrent marking overhead. |
| Humongous regions are handled efficiently | Humongous objects are collected at the end of Marking cycles, not in Young GC — they can accumulate and cause pauses if created frequently. |
| G1GC needs no tuning | For most apps yes, but for high-throughput or very-low-latency use cases, tuning IHOP, region size, and NewSize ratios is important. |
| G1GC compacts Old Gen on every cycle | Only the Mixed GC selected regions are compacted — unreachable but unselected regions may not be immediately collected. |
| Increasing heap always reduces G1 pauses | Larger heaps increase the number of regions and RSets, increasing concurrent marking and Young GC work — optimal heap sizing exists. |

### 🔥 Pitfalls in Production

**1. Humongous Allocation Churn**

```java
// BAD: Creating large objects in hot paths
// With G1HeapRegionSize=8m, any object > 4MB is Humongous
List<byte[]> cache = new ArrayList<>();
// loop:
cache.add(new byte[5 * 1024 * 1024]); // 5 MB Humongous

// GOOD: Use direct byte buffers for large I/O buffers
// and pool them to avoid repeated Humongous allocation
ByteBuffer buf = ByteBuffer.allocateDirect(5 * 1024 * 1024);
// Reuse buf across requests
```

Humongous objects trigger early concurrent marking and can cause fragmentation in Old Gen.

**2. IHOP Too High → Mixed GC Starts Too Late → Evacuation Failure**

```bash
# BAD: Default IHOP=45% with very fast allocation rate
-XX:InitiatingHeapOccupancyPercent=45
# → Concurrent marking may not complete before Old Gen fills
# → Evacuation failure → serial Full GC

# GOOD: Lower IHOP to give more headroom
-XX:InitiatingHeapOccupancyPercent=30
# Monitor gc.log for "Evacuation Failure" messages
```

**3. Mixed GC Not Running — Old Gen Growing Unboundedly**

```bash
# Symptom: GC log shows only Young GC cycles, Old Gen fills
# Root cause: G1MixedGCLiveThresholdPercent too high
# → Regions with too few dead objects never selected for mixed GC

# GOOD: Lower threshold so more Old Gen regions are eligible
-XX:G1MixedGCLiveThresholdPercent=65  # default 85
```

**4. RSet Refinement Overhead**

```bash
# Symptom: "concurrent refinement" threads use excessive CPU
# Root cause: High object reference mutation rate → many dirty cards

# Monitor with:
-Xlog:gc+remset*=trace:file=remset.log

# Mitigation: Reduce refinement thread count if CPU is constrained
# or restructure data access to reduce cross-region references
-XX:G1RSetUpdatingPauseTimePercent=5
```

### 🔗 Related Keywords

- `ZGC` — next-generation low-latency collector with sub-millisecond pauses.
- `Shenandoah GC` — Red Hat's concurrent compacting collector similar in goals to ZGC.
- `CMS (Concurrent Mark Sweep)` — G1's predecessor; concurrent but without compaction.
- `Stop-The-World (STW)` — the brief pauses G1 minimises via Evacuation and SATB.
- `GC Tuning` — optimising G1 parameters for specific latency or throughput goals.
- `GC Pause` — the metric G1's `MaxGCPauseMillis` targets.
- `Remembered Set` — per-region data structure enabling independent region collection.
