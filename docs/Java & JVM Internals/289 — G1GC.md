# 289 — G1GC

⚡ TL;DR — G1GC (Garbage First) divides the heap into equal-size regions instead of fixed generational zones, prioritizing collection of the regions with the most garbage first to meet a configurable pause-time target.

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
A trading platform runs on Java 6 with CMS GC. The heap is 12 GB. During a full GC, CMS stops the world for 3–8 seconds — a disaster for latency-sensitive financial calculations. The alternative, Parallel GC, stops the world for even longer on large heaps. Tuning generation sizes is guesswork: too-small old gen triggers frequent full GCs; too-large new gen wastes memory.

**THE BREAKING POINT:**
As Java applications grew from 256 MB to 8–64 GB heaps, the fixed-generation model of the classic collectors broke down. Young Gen and Old Gen sizes had to be sized before the application ran, and any mismatch caused either excessive GC frequency or long stop-the-world pauses. CMS's concurrent marking had fragmentation problems; Parallel GC had throughput-vs-latency conflicts.

**THE INVENTION MOMENT:**
G1GC (introduced as production-ready in Java 7u4, default in Java 9) redesigned the heap layout: instead of two large contiguous zones, the heap is divided into hundreds of equal-sized regions (1–32 MB each). Any region can be Young, Old, or Humongous (large object). G1 collects the regions with the most garbage first (hence "Garbage First"), reclaiming maximum space in minimum pause time. The key innovation: a configurable pause-time goal (`-XX:MaxGCPauseMillis`) that G1 attempts to meet by selecting the right set of regions to collect per cycle.

---

### 📘 Textbook Definition

**G1GC (Garbage-First Garbage Collector)** is a region-based, predominantly concurrent garbage collector introduced in Java 7 and made the default collector in Java 9. The heap is partitioned into equal-sized regions (configurable: 1–32 MB, ~2048 regions by default). Regions are dynamically assigned roles: Eden, Survivor, Old Generation, or Humongous (for objects larger than 50% of region size). G1 performs: (1) concurrent marking to identify live objects across the heap; (2) evacuation of selected regions (Young-only collections or Mixed collections) with a stop-the-world pause bounded by `-XX:MaxGCPauseMillis` (default 200ms); (3) full GC as fallback when heap space is exhausted. G1 aims to balance throughput and latency for heaps in the 4 GB–64 GB range.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
G1 divides the heap into regions, measures which regions are most garbage-filled, and collects those first within your pause-time budget.

**One analogy:**

> G1GC is like a smart janitor with a time limit. Instead of cleaning every office in the building at once (Full GC), they inventory all offices for how dirty they are, then use their 200-millisecond break to clean the five dirtiest ones. Next break: five more dirtiest. The building stays mostly clean without anyone being blocked for long.

**One insight:**
The "pause-time goal" is not a hard guarantee — it's a statistical target. G1 builds historical models of region collection cost and duration to predict which regions it can evacuate within the budget. It can and will miss the target during Mixed Collections or when the heap is nearly full.

### 🔩 First Principles Explanation

**CORE INVARIANTS:**

1. Heap = array of equal-sized regions. Every object lives in exactly one region.
2. Regions are dynamically typed: Eden / Survivor / Old / Humongous — type changes at GC time.
3. G1 tracks the amount of live data and collection cost for each region via Remembered Sets (RSets).
4. The Remembered Set for region R holds all pointers from outside R into R — enables region-isolated collection.
5. Concurrent marking runs alongside the application to build the live-data map.

**DERIVED DESIGN:**

```
G1 HEAP LAYOUT:
┌────────────────────────────────────────────────────┐
│ E │ E │ S │ O │ O │ H │ H │ O │ E │ E │ F │ F │... │
└────────────────────────────────────────────────────┘
 E=Eden  S=Survivor  O=Old  H=Humongous  F=Free

Each region: 1-32 MB (determined at JVM start)
Total regions: ~2048 by default
Humongous: spans 1+ contiguous regions for large objects
```

```
G1 COLLECTION PHASES:
1. Young-Only Phase
   ├── Minor GC (STW evacuation of Eden + Survivor)
   └── Concurrent Marking initiated when heap fills

2. Concurrent Marking Cycle (overlaps with application)
   ├── Initial Mark (STW, piggybacks on minor GC)
   ├── Root Region Scanning (concurrent)
   ├── Concurrent Marking (concurrent)
   ├── Remark (STW, finalises marking)
   └── Cleanup (STW for accounting, concurrent for freeing)

3. Mixed GC Phase
   └── Evacuates Eden + Survivor + SELECTED OLD REGIONS
       (those with most garbage = "Garbage First")

4. Full GC (fallback only — indicates tuning needed)
   └── STW full compaction (very long pause)
```

**THE TRADE-OFFS:**

G1 provides: predictable pause times for large heaps; no CMS fragmentation; concurrent marking; automatic region assignment.

G1 costs: higher throughput overhead than Parallel GC (10–20%) due to write barriers for RSet maintenance; complex tuning; Full GC fallback is slow when triggered.

### 🧪 Thought Experiment

**SETUP:**
A Java microservice has a 4 GB heap. Default G1GC with `-XX:MaxGCPauseMillis=200`.

**WITHOUT G1 (using Parallel GC):**
Young GC: ~150ms STW. Old generation fills up. Full GC: 3–6 seconds STW. During this time the application is completely frozen. All in-flight requests time out. Kubernetes health check fails → pod restarts → cascading failure.

**WITH G1GC:**
Young GC: ~50ms STW (only Eden regions evacuated). Concurrent marking runs in background. Mixed GC: ~100–150ms STW (evacuates old regions with most garbage). Heap stays clean without multi-second pauses. Kubernetes health checks pass. Latency spikes are visible but manageable.

**THE INSIGHT:**
G1's value is not about making GC faster — it's about making pauses predictable. A 150ms pause every 200ms is infinitely better than a 4-second pause every hour when you need 99th-percentile latency to stay under 500ms.

### 🧠 Mental Model / Analogy

> G1GC's region model is like a city divided into equal-size blocks. Each block tracks how many empty lots (garbage) it has. The city's maintenance crew has a 30-minute window between rush hours. They don't attempt to clean the whole city — they look at the block map, pick the top-5 dirtiest blocks (most empty lots), clear those, and stop. Next window, repeat. The city is always mostly clean; the crew never blocks traffic for more than 30 minutes.

- "City blocks" → G1 heap regions
- "Empty lots" → garbage objects
- "Maintenance window" → MaxGCPauseMillis budget
- "Block map" → Remembered Sets + live data stats
- "Top-5 dirtiest blocks" → "Garbage First" region selection

Where this analogy breaks down: unlike city blocks, G1 regions can be compacted and relocated — objects are physically moved during evacuation, which requires updating all pointers (hence the STW component of evacuation).

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
G1GC is Java's main garbage collector for large-memory applications. It cleans up memory in small, predictable bursts instead of stopping everything for a long time. You can tell it "never pause for more than 200 milliseconds" and it does its best to respect that.

**Level 2 — How to use it (junior developer):**
G1GC is the default from Java 9 onwards — you don't need to enable it explicitly. Key flags: `-XX:MaxGCPauseMillis=200` (tune your pause target), `-Xms` and `-Xmx` (set heap size; for G1, set them equal to avoid resizing pauses), `-XX:G1HeapRegionSize=N` (set region size manually if needed for Humongous objects). Monitor with GC logs: `-Xlog:gc*:file=gc.log:time,uptime,level,tags`.

**Level 3 — How it works (mid-level engineer):**
G1's key internal data structure is the **Remembered Set (RSet)**: a per-region hash table tracking all cross-region references. When a pointer from region A to region B is written, a **write barrier** (added by JIT at every object reference write) updates B's RSet to record A's card. This enables G1 to evacuate region B in isolation — it knows all incoming pointers and can update them. RSet maintenance adds 10–20% throughput overhead. During evacuation (STW), G1 copies live objects out of selected regions to free regions, compacting as it goes. All pointers to evacuated objects are updated via forwarding addresses.

**Level 4 — Why it was designed this way (senior/staff):**
G1's region model directly addresses CMS's fragmentation problem: because evacuation compacts objects into fresh regions, Old Gen never fragments. The concurrent marking cycle (inspired by CMS's concurrent marking but with a full heap view) provides the data G1 needs to predict Mixed GC cost. The "Garbage First" heuristic maximises reclaimed bytes per unit of pause time — a greedy algorithm provably optimal under the assumption that collection cost is proportional to live data size. The pause-time model uses a historical regression against recent collection durations to select how many regions to include per mixed GC pause. G1's weakness — Full GC fallback using a single-threaded collector until Java 10 (Java 10+ uses parallel Full GC) — was a major pain point that drove adoption of ZGC and Shenandoah for sub-100ms requirements.

### ⚙️ How It Works (Mechanism)

```
G1GC COLLECTION CYCLE DETAIL:
┌─────────────────────────────────────────────────────────┐
│         YOUNG-ONLY COLLECTIONS                          │
│                                                         │
│  Application runs → Eden fills →                        │
│  STW: Evacuate Eden+Survivor to Survivor/Old regions    │
│  Duration: typically 10–100ms                           │
│  Triggered by: Eden full                                │
└─────────────────────────────────────────────────────────┘
              │ When heap occupancy > InitiatingHeapOccupancyPercent
              ▼
┌─────────────────────────────────────────────────────────┐
│         CONCURRENT MARKING CYCLE                        │
│                                                         │
│  1. Initial Mark (STW, ~10ms): mark GC roots;           │
│     piggybacks on minor GC                              │
│  2. Root Region Scan (concurrent): scan Survivor regs   │
│  3. Concurrent Mark (concurrent): trace live objects    │
│     using SATB (Snapshot At The Beginning) algorithm    │
│  4. Remark (STW, ~10ms): process SATB buffers           │
│  5. Cleanup (STW+concurrent): update region stats,      │
│     reclaim empty regions                               │
└─────────────────────────────────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────────────────┐
│         MIXED COLLECTIONS                               │
│                                                         │
│  STW: Evacuate Eden + Survivor + selected Old regions   │
│  Selection: Old regions with MOST garbage, ranked by    │
│  (garbage_ratio = dead/total), collected until pause    │
│  budget is consumed                                     │
│  Duration: bounded by MaxGCPauseMillis (best effort)    │
└─────────────────────────────────────────────────────────┘
              │ If heap exhausted during evacuation
              ▼
┌─────────────────────────────────────────────────────────┐
│         FULL GC (fallback — avoid in production!)       │
│                                                         │
│  STW: Full compaction of entire heap                    │
│  Duration: seconds (avoid by tuning G1)                 │
└─────────────────────────────────────────────────────────┘
```

```
KEY INTERNAL STRUCTURES:
┌────────────────────┬────────────────────────────────────┐
│ Remembered Set     │ Per-region: tracks cross-region    │
│ (RSet)             │ pointers into this region          │
├────────────────────┼────────────────────────────────────┤
│ Card Table         │ 512-byte granularity dirty-card    │
│                    │ tracking for write barrier         │
├────────────────────┼────────────────────────────────────┤
│ SATB Queue         │ Snapshot-At-The-Beginning write    │
│                    │ barrier buffers for concurrent mark│
├────────────────────┼────────────────────────────────────┤
│ CSet               │ Collection Set: regions selected   │
│                    │ for current GC pause               │
└────────────────────┴────────────────────────────────────┘
```

### 🔄 The Complete Picture — End-to-End Flow

```
NORMAL FLOW:
Object allocated in Eden region
→ Eden fills → Young GC triggered (STW evacuation)
→ Live objects copied to Survivor/Old regions
→ [G1GC Mixed Collection ← YOU ARE HERE when Old fills]
→ Garbage-first Old regions evacuated within pause budget
→ Heap reclaimed → application continues

FAILURE PATH:
Mixed GC can't keep up with allocation rate
→ Heap occupancy rises toward 100%
→ Evacuation failure: no free regions to copy into
→ G1 falls back to Full GC (long STW)
→ Observable: GC log shows "Pause Full (Allocation Failure)"
→ Fix: increase -Xmx, tune InitiatingHeapOccupancyPercent,
       reduce object allocation rate

WHAT CHANGES AT SCALE:
At >64 GB heaps, G1's RSet overhead grows significantly
because cross-region references multiply. G1 is designed
for 4–64 GB; beyond that, ZGC (near-pause-free via colored
pointers) or Shenandoah become better choices, scaling to
TBs of heap with <10ms pauses.
```

### 💻 Code Example

```bash
# Example 1 — Standard G1GC production configuration
java -server \
  -Xms8g -Xmx8g \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=200 \
  -XX:G1HeapRegionSize=16m \
  -XX:InitiatingHeapOccupancyPercent=45 \
  -XX:G1NewSizePercent=5 \
  -XX:G1MaxNewSizePercent=60 \
  -Xlog:gc*:file=/var/log/app/gc.log:time,uptime:filecount=10,filesize=50m \
  -jar application.jar

# Example 2 — Read G1GC logs to diagnose pause issues
# Look for Full GC in logs:
grep "Pause Full" /var/log/app/gc.log

# Example 3 — Humongous allocation check
# Objects > G1HeapRegionSize/2 go to Humongous regions
# (bypasses Young gen, causes GC overhead)
# Check in logs:
grep -i "humongous" /var/log/app/gc.log

# Example 4 — JVM flags to diagnose G1 tuning
java -XX:+PrintFlagsFinal -version | grep -E "G1|GC"
```

```java
// Example 5 — Avoid Humongous allocations
// BAD: allocating large byte array frequently
// triggers Humongous region allocation, skips Eden
byte[] LARGE_BUFFER = new byte[20 * 1024 * 1024]; // 20MB

// GOOD: pool and reuse large buffers
private static final ThreadLocal<byte[]> BUFFER_POOL =
    ThreadLocal.withInitial(() -> new byte[20 * 1024 * 1024]);

// Or increase G1HeapRegionSize so 20MB < region/2:
// -XX:G1HeapRegionSize=64m
// (max region size is 32m — so 20MB buffer will be Humongous
//  regardless; pool reuse is the correct fix)
```

### ⚖️ Comparison Table

| GC          | Pause Type              | Heap Range | Throughput        | Best For                    |
| ----------- | ----------------------- | ---------- | ----------------- | --------------------------- |
| Serial GC   | STW full                | <1 GB      | High (single CPU) | Tiny embedded apps          |
| Parallel GC | STW parallel            | 1–8 GB     | Highest           | Batch, throughput focus     |
| **G1GC**    | Concurrent + short STW  | 4–64 GB    | Good              | Large-heap, latency target  |
| ZGC         | Near-concurrent (<10ms) | 8 MB–16 TB | Good              | Ultra-low latency           |
| Shenandoah  | Near-concurrent (<10ms) | Any        | Good              | Ultra-low latency (OpenJDK) |

**How to choose:** G1GC is the right default for most server applications with 4–32 GB heaps and 200ms latency targets. If you need sub-50ms GC pauses, evaluate ZGC (Java 15+ production-ready) or Shenandoah. For batch jobs where throughput matters more than latency, Parallel GC may outperform G1 by 15–20%.

### 🔁 Flow / Lifecycle

```
G1 REGION STATE MACHINE:

[Free] ──alloc──► [Eden]
[Free] ──alloc──► [Humongous]

[Eden] ──YoungGC(copy)──► [Survivor]
[Eden] ──YoungGC(died)──► [Free]

[Survivor] ──YoungGC(tenured)──► [Old]
[Survivor] ──YoungGC(died)──► [Free]

[Old] ──MixedGC(copy)──► [Old] (compacted, different region)
[Old] ──MixedGC(fully dead)──► [Free]

[Humongous] ──concurrent mark(dead)──► [Free]

Normal path: Free → Eden → Survivor → Old → Free (via GC)
Error path: Old regions fill faster than Mixed GC reclaims
           → Evacuation Failure → Full GC
```

### ⚠️ Common Misconceptions

| Misconception                                | Reality                                                                                                                                                                                                              |
| -------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `MaxGCPauseMillis` is a hard pause guarantee | It is a target, not a contract. G1 uses historical stats to predict pause duration; it misses the target during high-allocation bursts or near-full heaps. ZGC/Shenandoah offer harder sub-10ms guarantees           |
| G1GC always outperforms Parallel GC          | For pure throughput (batch processing, no latency requirement), Parallel GC often achieves 15–20% more throughput because it has no write-barrier overhead for RSet maintenance                                      |
| Full GC means G1 failed                      | Full GC is G1's safety-net fallback, not a design failure. It indicates the application is allocating faster than G1 can reclaim — the fix is tuning heap size or reducing allocation rate, not switching collectors |
| G1 eliminates stop-the-world pauses          | G1 reduces and bounds STW pauses but does not eliminate them. Young GC and Remark/Cleanup phases are still stop-the-world. Only ZGC/Shenandoah approach near-zero STW via load-reference barriers                    |

### 🚨 Failure Modes & Diagnosis

**Frequent Full GC (Evacuation Failure)**

**Symptom:** Log shows `[GC pause (G1 Evacuation Pause) (to-space exhausted)]` followed by `[Pause Full (G1 Evacuation Failure)]`; application pauses for seconds.

**Root Cause:** During evacuation, G1 ran out of free regions to copy live objects into. Heap is too small for the application's working set, or allocation rate is too high for Mixed GC to keep up.

**Diagnostic Command:**

```bash
# Find Full GC events in G1 log:
grep "Pause Full" gc.log | awk '{print $1, $NF}'

# Check heap usage trend:
jstat -gcutil <pid> 1000 30
# OGCMX (old gen max) vs OGC (current old gen used)
```

**Fix:** Increase `-Xmx`; or decrease `-XX:InitiatingHeapOccupancyPercent` (triggers concurrent marking earlier, giving Mixed GC more headroom).

**Prevention:** Set `-Xms` == `-Xmx` to prevent heap-resize pauses; allocate 30–40% headroom above expected live data set size.

---

**Excessive GC Pause from Humongous Objects**

**Symptom:** Young GC pauses are longer than expected; heap shows large "H" (Humongous) region occupation in GC log.

**Root Cause:** Objects larger than `G1HeapRegionSize/2` bypass Eden and allocate directly in contiguous Humongous regions, triggering extra scanning during marking.

**Diagnostic Command:**

```bash
-Xlog:gc+humongous=debug:file=gc.log
# Shows each Humongous allocation with size and class
```

**Fix:** Increase `-XX:G1HeapRegionSize` (max 32m) to move objects below the Humongous threshold. If objects are truly large (>16 MB), pool/reuse them.

**Prevention:** Profile application for large array allocations; pool ByteBuffers and large arrays.
