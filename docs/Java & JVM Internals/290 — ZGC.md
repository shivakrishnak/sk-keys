# 0290 — ZGC

## 1. TL;DR
> **ZGC (Z Garbage Collector)** is a scalable, low-latency garbage collector that achieves **sub-millisecond STW pauses** regardless of heap size (from 8 MB to 16 TB). It performs all major GC work — marking, relocation, reference updating — **concurrently** with running application threads. ZGC became production-ready in Java 15 and is the go-to collector for latency-critical applications.

---

## 2. Visual: How It Fits In

```
ZGC Timeline — Almost Entire GC is Concurrent

Application:  ──────────────────────────────────────────────────────
              |<1ms|          concurrent work          |<1ms|
GC:                ──[Concurrent Mark]──[Concurrent Relocate]──

App threads CONTINUE running throughout all major phases.
Only 2–3 very short STW pauses: < 1ms each, regardless of heap size.

Contrast with G1GC:
Application:  ──────────────────┤ 50–200ms STW ├──────────────────
GC:                              ─────────────────

ZGC's secret: Colored Pointers + Load Barriers
  - References carry metadata (4 bits) about GC state
  - App threads "self-heal" stale references via load barriers
  - GC can relocate objects while app uses them
```

---

## 3. Core Concept

ZGC's three innovations that enable sub-ms pauses:

---

### 1. Colored Pointers (Pointer Metadata)

ZGC stores GC state in the **upper bits of object references** (on 64-bit systems):
```
64-bit reference:
[ 18 bits padding | 4 bits GC metadata | 42 bits address ]
                     ^^^^^^^^^^^^^^^^^^^^
                  Marked0, Marked1, Remapped, Finalizable bits
```
GC threads change these bits concurrently. When app code reads a reference with "wrong" bits, the load barrier triggers a fix.

---

### 2. Load Barriers

Every time app code **reads an object reference** from the heap, a small code snippet (load barrier) is executed. If the reference's color bits are "wrong" (the object has been relocated), the barrier:
1. Finds the new location via forwarding table
2. Updates the reference in place
3. Returns the correct (new) reference

This distributes reference updating across all app threads — no single STW "update all references" phase needed.

---

### 3. Concurrent Relocation

ZGC can **move objects while the app is running** via load barriers. Every access to a relocated object automatically heals the stale reference. No STW phase is needed for "fix all pointers after moving objects."

---

## 4. Why It Matters

For latency-sensitive systems where GC pauses directly impact user experience:
- < 1ms STW pauses mean **GC is invisible** at the application level
- Scales to **multi-terabyte heaps** without pause time growth
- Eliminates the "GC pause SLA breach" class of incidents
- Available production-ready in Java 15+; generational ZGC in Java 21+

## 6. Real-World Analogy

> Traditional GCs are like a hospital that must close entirely to perform renovations (STW). ZGC is like a hospital with a sophisticated renovation system: they put up colored warning tags on rooms being renovated. When a patient (application thread) enters a tagged room, a guide (load barrier) automatically redirects them to the new location. The renovation happens seamlessly while patients continue receiving care. The hospital only pauses briefly (< 1ms) for the logistics of starting and ending the renovation process.

## 7. How It Works — Step by Step

```
ZGC Collection Cycle (Concurrent):

Phase 1: Pause Mark Start (STW < 1ms)
  - Mark GC Roots (thread stacks, global refs)
  - Flip the "current mark bit" in all references
  - Very short: only root scanning

Phase 2: Concurrent Mark (no STW)
  - Traverse all reachable objects from roots
  - Mark live objects by setting colored bits
  - App threads continue; load barriers handle any accessed refs
  - Write barriers track new allocations during marking

Phase 3: Pause Mark End (STW < 1ms)
  - Drain final mark queues
  - Process weak references
  - Short: only finalization of marking data

Phase 4: Concurrent Prepare for Relocation (no STW)
  - Analyze which regions have most garbage (like G1's Garbage-First)
  - Build Relocation Set: regions to be evacuated
  - Prepare forwarding tables

Phase 5: Pause Relocate Start (STW < 1ms)
  - Mark GC Roots pointing into relocation set regions
  - These roots get immediate up-to-date addresses

Phase 6: Concurrent Relocate (no STW)
  - GC threads move live objects from relocation set regions to new regions
  - Old locations replaced with forwarding pointers
  - When app thread accesses relocated object:
    Load barrier detects stale reference → follows forwarding pointer → 
    updates reference → returns new address

Phase 7: Concurrent Remap (no STW)
  - Update all remaining stale references in heap
  - Piggybacked on next marking cycle for efficiency
```

## 8. Under the Hood (Deep Dive)

---

### Colored Pointers in detail

```
ZGC uses 42 bits for address (allows 4TB address space per color)
4 metadata bits:
  - Marked0: marked during even GC cycle
  - Marked1: marked during odd GC cycle (alternating prevents ABA)
  - Remapped: object has been relocated to new address
  - Finalizable: object only reachable through finalizer

When reference has wrong mark bits:
  - Load barrier: "This reference is stale"
  - Look up forwarding table for new address
  - Return new address to application
  - Optionally update the stale reference in heap
  
Cost: every object reference read has small overhead
  ~10-15% throughput reduction vs Parallel GC
  Worth it for sub-ms pauses
```

---

### Load Barrier injection

```java
// What your Java code looks like:
Object obj = someArray[i];

// What JIT compiles it to with ZGC load barrier:
Object obj = someArray[i];         // Load from heap
if (obj.colorBits != expectedBits) // Check color bits
    obj = loadBarrierSlowPath(obj); // If wrong, fix reference
use(obj);                          // Use correct reference
```

---

### ZGC vs G1GC: throughput tradeoff

```
Throughput benchmark (CPU-bound, high allocation rate):
  Parallel GC: 100% (baseline)
  G1GC:        ~90-95%
  ZGC:         ~83-88%  ← load barrier overhead

Latency (P99 pause):
  Parallel GC: 500ms – multi-second
  G1GC:        50–200ms
  ZGC:         < 1ms ← orders of magnitude better

Rule of thumb:
  If P99 latency matters → ZGC pays for itself
  If throughput matters exclusively → Parallel GC
```

---

### Generational ZGC (Java 21)

```
-XX:+UseZGC -XX:+ZGenerational

Improvements:
  - Separate Young/Old generations within ZGC regions
  - Young Gen collected more frequently (short-lived objects)
  - Reduces floating garbage from current ZGC
  - Better throughput (~20-30% improvement over non-generational ZGC)
  - Still sub-ms STW pauses
  
Becoming default in future Java versions
```

## 10. When to Use / Avoid

| Scenario | Guidance |
|----------|----------|
| < 5ms latency P99 required | ✅ ZGC is the answer |
| Large heap (> 32 GB) | ✅ ZGC handles multi-TB with sub-ms |
| Latency-critical APIs (trading, gaming) | ✅ ZGC |
| Memory-constrained tiny containers (< 512MB) | Consider G1 (ZGC has overhead) |
| Pure batch/throughput workloads | Parallel GC may be better |

## 11. Common Pitfalls & Mistakes

```
❌ Using non-generational ZGC (Java 11–20) and expecting low overhead
   → Non-generational ZGC has higher floating garbage and lower throughput
   → Upgrade to Java 21+ and use -XX:+ZGenerational

❌ Expecting ZGC to eliminate ALL pauses
   → ZGC still has 2-3 STW pauses per cycle; they're just < 1ms
   → Application TTSP (time to safepoint) can inflate apparent pause

❌ Setting heap too small with ZGC
   → ZGC needs headroom for concurrent relocation; recommend 2–3x live data

❌ Not monitoring allocation rate
   → ZGC can fall behind on very high allocation rates
   → If allocation rate > GC throughput: OOM even with ZGC
```

## 13. Interview Q&A

**Q: How does ZGC achieve sub-millisecond STW pauses?**
> ZGC performs all major GC work (marking, relocation planning, object relocation, reference updating) concurrently with application threads. It uses colored pointers (4 metadata bits in references) and load barriers (injected at every reference load) to handle concurrent object movement without STW. The only STW phases are brief root-scanning steps (< 1ms each).

**Q: What are colored pointers and why are they central to ZGC?**
> Colored pointers store GC state (marked, remapped, finalizable) in the unused upper bits of 64-bit object references. This allows GC threads to atomically update the GC state of references across the heap. When app code loads a "wrong-colored" reference, the load barrier detects it and transparently updates the reference to point to the correct (relocated) object location.

**Q: What is the throughput tradeoff of ZGC vs G1GC?**
> ZGC has ~10-15% lower throughput than G1GC due to load barrier overhead — every object reference read incurs a conditional check and potential slow-path resolution. For latency-critical workloads, this is typically worth it. Generational ZGC (Java 21+) closes most of this gap.

**Q: What is Generational ZGC?**
> Introduced in Java 21 (`-XX:+ZGenerational`), Generational ZGC adds separate Young and Old generations to ZGC. Young Gen is collected more frequently (shorter-lived objects), reducing floating garbage and improving throughput by ~20-30%. It maintains ZGC's sub-ms pause goal across both generations.

## 16. Anti-Patterns

```
🚫 Anti-Pattern: Using ZGC on Java 11-14 ("experimental")
   Problem:  Non-production quality; missing key optimizations
   Fix:      Use Java 15+ for production ZGC; Java 21+ for generational ZGC

🚫 Anti-Pattern: Expecting zero GC overhead with ZGC
   Problem:  Load barriers add ~10-15% CPU overhead vs Parallel GC
   Fix:      Profile to confirm ZGC tradeoff is worthwhile for your workload

🚫 Anti-Pattern: Heap too tight for ZGC relocation
   Problem:  ZGC needs room to relocate objects; tight heap causes OOM
   Fix:      Provision heap at 2-3x live data size
```

## 19. Human Summary

ZGC is what you choose when you're serious about latency. Sub-millisecond GC pauses — not as a marketing claim but as an engineering guarantee that holds even on 1 TB heaps — represent a fundamental shift in how we think about GC. The secret is moving the work from "everyone stops while we fix things" to "everyone notices a slight overhead while things fix themselves in the background."

The tradeoff is real: load barriers add CPU overhead on every object read. For I/O-bound services, this is invisible. For CPU-bound hot code paths, it might matter. The practical answer for most latency-sensitive services in 2025 is: migrate to Java 21, use `XX:+UseZGC -XX:+ZGenerational`, and stop having GC-induced SLA breaches.
