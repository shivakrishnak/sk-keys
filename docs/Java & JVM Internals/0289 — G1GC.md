# 0289 — G1GC

⚡ TL;DR — G1GC (Garbage-First Garbage Collector) is the JVM's default GC since Java 9, designed to deliver predictable pause times by dividing the heap into equal-sized regions and prioritising collection of the regions with the most garbage, avoiding the full-heap collection that made older collectors unpredictable at large heap sizes.

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
A Java application runs on a 32GB heap with the old Parallel GC or CMS. As the heap fills, the GC must scan or collect the entire heap. A Full GC on a 32GB heap takes 15–30 seconds of Stop-The-World (STW) pause. During this pause, all application threads halt — no requests are served, no messages are processed. For a payment service, 15 seconds of STW is catastrophic. Users see timeouts. SLA is breached. Kubernetes restarts the pod.

**THE BREAKING POINT:**
As JVM heap sizes grew from 256MB (2000s) to 4GB, 16GB, 64GB (2020s), the "collect the whole heap" approach became unacceptable. GC pause time scales linearly (or worse) with heap size for traditional collectors. At 64GB, a Full GC could pause for minutes.

**THE INVENTION MOMENT:**
G1GC was designed with a single primary goal: **predictable, configurable pause targets** (default: 200ms) regardless of heap size. It achieves this by working in small increments — collecting one region at a time in STW pauses short enough to meet the target — rather than collecting the entire heap.

---

### 📘 Textbook Definition

**G1GC (Garbage-First Garbage Collector):** A server-oriented, mostly-concurrent garbage collector that divides the JVM heap into a configurable number of equal-sized regions (1–32MB each). It maintains a collection set (CSet) of regions with the highest garbage density, collecting them in bounded stop-the-world pauses to meet a user-defined pause target (`-XX:MaxGCPauseMillis`). It became the JVM default in Java 9 (replacing Parallel GC as default in Java 8 update for large heaps) and is the recommended choice for most production applications with heap sizes of 4GB–256GB.

**Evacuation:** G1GC's primary collection mechanism — live objects are copied out of a region into a free region, leaving the collected region empty (100% free). This compacts live objects without a separate compaction phase.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
G1GC collects the most garbage-filled parts of the heap first, in small time-bounded pauses, instead of stopping the world to collect everything at once.

**One analogy:**

> Imagine cleaning a large office building. The old approach: every Saturday night, nobody works while cleaners scrub every single floor — 4 hours of downtime. G1GC's approach: identify the 5 dirtiest rooms, clean only those in 15 minutes while the office stays mostly open, then repeat. The building never accumulates a catastrophic level of mess because the dirtiest parts are constantly targeted, and the cleaning windows are predictable and short.

**One insight:**
The key design insight is that most garbage collection time is spent on _empty space recovery_, not on live object tracing. By selecting only the regions with the most garbage (highest garbage-to-live-data ratio), G1GC maximises free space recovered per unit of STW pause time. "Garbage-First" is literally about picking the best region to collect first — the one that returns the most free memory per millisecond of pause.

### 🔩 First Principles Explanation

**CORE INVARIANTS:**

1. The heap is divided into equal-sized regions (1–32MB, determined by heap size, ~2,048 regions total).
2. Every region has a role: Eden, Survivor, Old, Humongous, or Free.
3. G1GC maintains a Remembered Set (RSet) per region — tracking which other regions hold pointers into this region, enabling independent collection.
4. G1GC aims to meet `-XX:MaxGCPauseMillis` (default: 200ms) by selecting a CSet small enough to evacuate within that budget.

**HEAP LAYOUT:**

```
G1GC HEAP (example: 8GB, 2048 regions × 4MB each):

[E][E][S][O][O][O][H][H][F][E][E][S]...
 E = Eden region (young gen allocation)
 S = Survivor region (promoted from Eden)
 O = Old region (long-lived objects)
 H = Humongous region (objects > 0.5 × region size)
 F = Free region (available for any role)

The "young generation" = all E+S regions combined.
The "old generation" = all O regions combined.
Regions can change role dynamically — no fixed boundaries.
```

**THE COLLECTION TYPES:**

```
YOUNG-ONLY GC (most frequent):
  - STW pause
  - Collect ALL Eden + Survivor regions
  - Live objects → promoted to new Survivor or Old regions
  - Freed regions → become Free (available again)
  - Triggered: Eden regions full
  - Typical pause: 10–50ms

MIXED GC (periodic):
  - STW pause
  - Collect ALL young regions + SOME old regions (CSet)
  - Old regions selected: highest garbage percentage first
  - Typical pause: 50–200ms (bounded by MaxGCPauseMillis)
  - Triggered: Old gen occupancy > InitiatingHeapOccupancyPercent

FULL GC (rare, emergency):
  - STW pause, single-threaded (pre-Java 10) or multi-threaded
  - Collects ENTIRE heap
  - Triggered: allocation failure, metadata space exhausted
  - Goal: never trigger this
```

**CONCURRENT MARKING (background, non-STW):**

```
Concurrent Root Scan → Concurrent Mark → Remark (STW, brief) →
Cleanup (STW, brief) → Concurrent Cleanup
  ↑                                                           ↑
runs while application threads run              frees empty regions
```

Concurrent marking builds the liveness information used to select Old regions for the CSet. It runs in the background, overlapping with application execution.

**THE TRADE-OFFS:**

**Gain:** Predictable, short STW pauses (default target: 200ms) even on 64GB+ heaps. Better throughput than CMS on large heaps. Automatic heap layout optimisation.

**Cost:** Higher CPU overhead than Parallel GC (concurrent marking threads run alongside application threads). More complex tuning. Remembered Sets consume memory (typically 20–30% overhead on object count). Humongous objects are not compacted and can cause premature Full GCs.

### 🧪 Thought Experiment

**SETUP:**
Two JVM apps, each 16GB heap. App A: Parallel GC. App B: G1GC. Both receive steady traffic that creates ~2GB of short-lived objects per minute.

**WHAT HAPPENS WITH Parallel GC:**
After 6–8 minutes, Old Gen fills. GC triggers Full GC. All 16GB must be scanned and compacted. STW pause: 8–12 seconds. Every in-flight request times out. Kubernetes health check fails (10s timeout). Pod restarts. Service goes offline for 30 seconds including restart.

**WHAT HAPPENS WITH G1GC (MaxGCPauseMillis=200):**
Young GC fires every 30 seconds: pause 40ms. Concurrent marking runs in background after ~8GB old gen occupancy. Mixed GCs fire every 2–3 minutes: pause 150ms. Old gen objects collected incrementally. Heap never exceeds 80% occupancy. No Full GC. Worst STW pause: 180ms. All within SLA.

**THE INSIGHT:**
G1GC converts an unpredictable "pause when full" model into a continuous incremental collection model. The pause budget is spent proactively on the highest-value regions rather than reactively on everything at once.

### 🧠 Mental Model / Analogy

> G1GC is like a continuous office paper recycling program. Old approach: once a year, every desk is cleared of all paper. 2-day shutdown. G1GC: a recycling assistant walks through the office every 30 minutes, identifies the 5 most paper-cluttered desks, clears just those desks (15-minute mini-pause). The office never has a catastrophic paper crisis because the worst piles are constantly targeted.

Explicit mapping:

- "office" → JVM heap
- "desk" → heap region (4MB unit)
- "paper" → dead/garbage objects
- "clearing a desk" → evacuating live objects from a region, marking it free
- "identifying the most cluttered desks" → remembered set + liveness data to pick CSet
- "30-minute walkthrough" → background concurrent marking
- "15-minute mini-pause" → STW evacuation pause (young or mixed GC)
- "annual shutdown" → Full GC (what G1 avoids)

Where this analogy breaks down: G1GC can still trigger Full GC (the annual shutdown) if the incremental collection falls behind allocation rate. This is the failure mode to avoid.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
G1GC is Java's default garbage collector. It cleans up unused memory in short, frequent bursts rather than one long pause, which makes Java applications more consistently responsive. You tell it "never pause longer than 200 milliseconds" and it does its best to honour that.

**Level 2 — How to use it (junior developer):**
G1GC is the default since Java 9 — you don't need to enable it explicitly. The most important setting: `-XX:MaxGCPauseMillis=200` (default). For a heap of 4–8GB, the defaults work well. Monitor GC logs with `-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m`. Look for "Full GC" events in the logs — they indicate a tuning problem.

**Level 3 — How it works (mid-level engineer):**
G1GC divides the heap into ~2,048 equal regions. Eden regions are allocated into until full, triggering a Young GC (STW, ~20–50ms). The Remembered Sets allow independent collection of regions by tracking cross-region pointers. When Old gen occupancy exceeds `InitiatingHeapOccupancyPercent` (default: 45%), concurrent marking starts. After marking completes, Mixed GCs collect young + selected old regions. The CSet selection algorithm chooses old regions with the highest garbage percentage that can be evacuated within `MaxGCPauseMillis`. If evacuation fails (no free region to evacuate into), G1 falls back to Full GC.

**Level 4 — Why it was designed this way (senior/staff):**
G1's remembered set design is the key engineering trade-off: each region maintains a set of cards (512-byte memory chunks) that point into it from other regions. This enables independent region collection at the cost of per-write barrier overhead (every object reference write updates the appropriate card table) and RSet memory overhead. The alternative — scanning the entire heap on every collection — doesn't scale to 64GB heaps. The "Garbage First" naming encodes the CSet selection heuristic: collect the regions that return the most free memory per unit of pause time. This greedy heuristic is correct for the goal of maximising freed memory within a pause budget, though it can leave fragmented Old regions if large live objects occupy otherwise high-garbage regions. G1's biggest evolution over CMS was eliminating the concurrent sweep phase — evacuation (copy-compact) leaves no fragmentation within collected regions.

### ⚙️ How It Works (Mechanism)

```
G1GC COLLECTION CYCLE:

1. NORMAL OPERATION (application running):
   Threads allocate into Eden regions.
   Write barriers update card tables (Remembered Sets).
   Background: Concurrent Marking runs (if triggered).

2. YOUNG GC (STW, ~20–50ms):
   Triggered: Eden regions full.
   All Eden + Survivor regions → CSet.
   Live objects evacuated → new Survivor/Old regions.
   Eden regions → Free.
   Update RSets for moved objects.

3. CONCURRENT MARKING (parallel, background):
   Triggered: old gen > InitiatingHeapOccupancyPercent (45%).
   Phase 1: Initial Mark (piggybacks on Young GC STW)
   Phase 2: Concurrent Root Scan (background)
   Phase 3: Concurrent Mark (background, traces object graph)
   Phase 4: Remark (STW, ~5–15ms, finalises marks)
   Phase 5: Cleanup (STW, ~1ms, reclaims empty regions)
   Phase 6: Concurrent Cleanup (background)

4. MIXED GC (STW, ~50–200ms):
   After concurrent marking completes.
   CSet: ALL young regions + N old regions (highest garbage%).
   Evacuation: live objects copied to free regions.
   Repeat mixed GCs until old gen below target.

5. FULL GC (emergency):
   Triggered: evacuation failure, Metaspace OOM.
   Single/multi-thread STW. Full heap compaction.
   Very slow — design to avoid.
```

### 🔄 The Complete Picture — End-to-End Flow

```
Application allocates object
    ↓
Object lands in Eden region
    ↓
Eden regions fill up
    ↓
[G1GC YOUNG GC ← YOU ARE HERE]
STW pause ~20ms: evacuate Eden+Survivor
    ↓
Live objects → new Survivor or Old regions
    ↓
Eden regions freed → available for allocation
    ↓
Repeat until Old gen > 45% occupancy
    ↓
[CONCURRENT MARKING ← BACKGROUND]
Traces live objects in Old gen (no STW)
    ↓
[MIXED GC ← YOU ARE HERE]
STW pause ~150ms: evacuate young + selected old regions
    ↓
Old gen occupancy drops
    ↓
Application continues
```

**FAILURE PATH:**
Allocation rate exceeds evacuation rate → Young GC cannot keep up → Old gen fills → No free region for evacuation → Evacuation failure → Full GC → Long STW pause → Kubernetes health check timeout → Pod restart.

**WHAT CHANGES AT SCALE:**
At 256GB heap sizes, G1GC still works but concurrent marking duration increases (more objects to trace). Remembered Set overhead also scales with object reference density. At extreme heap sizes (>256GB), ZGC or Shenandoah are better choices because they perform concurrent evacuation, eliminating almost all STW pauses.

### 💻 Code Example

**Example 1 — G1GC JVM flags for a production service:**

```bash
# Recommended G1GC flags for a 16GB heap service
java -Xms8g -Xmx16g \
  -XX:+UseG1GC \                    # explicit (default Java 9+)
  -XX:MaxGCPauseMillis=200 \        # pause target (ms)
  -XX:InitiatingHeapOccupancyPercent=45 \ # when to start marking
  -XX:G1HeapRegionSize=16m \        # region size (auto or explicit)
  -XX:+G1UseAdaptiveIHOP \          # auto-tune IHOP
  -XX:G1ReservePercent=10 \         # reserve 10% for evacuation
  -Xlog:gc*:file=/var/log/gc.log:time,uptime:filecount=5,filesize=20m \
  -XX:+HeapDumpOnOutOfMemoryError \
  -XX:HeapDumpPath=/var/dump/ \
  MyApp
```

**Example 2 — Reading G1GC logs:**

```
# Good: young GC within target
[0.532s][info][gc] GC(12) Pause Young (Normal) (G1 Evacuation Pause)
  8192M->4096M(16384M) 42.152ms

# Warning: Mixed GC approaching target
[45.1s][info][gc] GC(89) Pause Young (Mixed) (G1 Evacuation Pause)
  12288M->9216M(16384M) 187.441ms  ← near 200ms target

# BAD: Full GC — something is wrong
[120s][info][gc] GC(200) Pause Full (Evacuation Failure)
  15360M->8192M(16384M) 8423.221ms  ← 8 seconds!
# Root cause: humongous allocation / IHOP too high / region exhaustion
```

**Example 3 — Diagnosing G1GC with JVM tools:**

```bash
# Live GC monitoring
jstat -gc <pid> 1000  # print GC stats every 1 second
# YGC = young GC count, YGCT = young GC time
# FGC = full GC count (should be 0 or near 0)
# FGCT = full GC time (should be 0)

# GC log analysis with GCViewer / GCEasy
# Upload gc.log to https://gceasy.io for visual analysis

# Check region distribution
jcmd <pid> VM.native_memory summary  # memory breakdown

# Heap dump for OOM analysis
jmap -dump:format=b,file=heap.hprof <pid>
```

### ⚠️ Common Misconceptions

| Misconception                                 | Reality                                                                                                                                                               |
| --------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| "G1GC guarantees MaxGCPauseMillis"            | It is a goal, not a hard guarantee. G1 tries to meet it but can exceed it. Setting it too low (e.g., 10ms) causes frequent small GCs that actually reduce throughput. |
| "G1GC eliminates Full GC"                     | Full GC is still possible: evacuation failure (no free region), Metaspace exhaustion, System.gc() call. Design to avoid Full GC, don't assume G1 prevents it.         |
| "Bigger heap = better with G1"                | Excessively large heaps (> available RAM) cause OS page swapping, which destroys GC performance. Heap should be ≤ 70% of available RAM.                               |
| "G1GC is always better than ZGC for Java 21+" | For most workloads with < 500ms latency requirement, G1GC has higher throughput than ZGC. ZGC wins only when sub-10ms pauses are required.                            |
| "IHOP at 45% is always optimal"               | For allocation-heavy workloads, lower IHOP (30–35%) triggers marking earlier, preventing evacuation failures. Use `-XX:+G1UseAdaptiveIHOP` to let G1 auto-tune.       |

### 🚨 Failure Modes & Diagnosis

**1. Frequent Full GC (Evacuation Failure)**

**Symptom:** GC logs show `Pause Full (Evacuation Failure)`. Pause duration > 1 second. Application latency spikes periodically. `FGC` counter in `jstat` incrementing.

**Root Cause:** Allocation rate exceeds G1's ability to evacuate regions in time. Old gen fills because mixed GCs aren't keeping up. No free regions available when evacuation needs a destination.

**Diagnostic:**

```bash
jstat -gcutil <pid> 1000
# Watch O (old gen %) column — if consistently > 85%, evacuation failing
# Watch FGC — should be 0; > 0 means Full GC occurred

# Check logs for:
grep "Evacuation Failure\|Pause Full" gc.log
```

**Fix:** Lower `InitiatingHeapOccupancyPercent` (e.g., 35) to start concurrent marking earlier. Increase `-XX:G1MixedGCCountTarget` (default: 8) to collect more old regions per cycle. Increase heap size or reduce object allocation rate.

**Prevention:** Set `InitiatingHeapOccupancyPercent` to 35–40 for allocation-heavy services. Enable adaptive IHOP: `-XX:+G1UseAdaptiveIHOP`.

---

**2. Humongous Object Allocation Causing Premature Collection**

**Symptom:** Young GC triggered much more frequently than expected. `H` regions in heap dump. GC log shows humongous allocations.

**Root Cause:** Objects larger than 50% of the region size are "humongous" — allocated directly in contiguous Old gen regions. These regions are never included in normal collection until they become entirely garbage. They can cause region exhaustion and premature Full GC.

**Diagnostic:**

```bash
# G1GC logs humongous allocations at debug level
-Xlog:gc+humongous=debug

# Check region size
-XX:G1HeapRegionSize=<N>m
# Humongous threshold = regionSize / 2
# e.g., 16MB region → objects > 8MB = humongous
```

**Fix:** Increase region size: `-XX:G1HeapRegionSize=32m` (max). This raises the humongous threshold. Alternatively, reduce the size of large objects (e.g., split large byte arrays, avoid holding large intermediate byte buffers).

**Prevention:** Profile allocation with async-profiler to identify humongous allocations at their source.

---

**3. Long Remark Pause**

**Symptom:** GC log shows `Remark` STW pause > 100ms. Happens during concurrent marking phase.

**Root Cause:** Concurrent marking cannot finish tracing all live objects before the Remark STW pause. This happens when: (a) very high allocation rate creates many new references during marking, or (b) complex object graphs require deep tracing.

**Diagnostic:**

```bash
grep "Pause Remark" gc.log
# Normal: < 20ms
# Problem: > 50ms

# Check SATB (Snapshot-At-The-Beginning) buffer overflow:
-Xlog:gc+phases=debug | grep "Update RS\|Scan RS"
```

**Fix:** Lower `InitiatingHeapOccupancyPercent` to start marking earlier (more time to complete before Remark). Increase concurrent marking threads: `-XX:ConcGCThreads=<n>` (default: ~1/4 of available CPUs).

**Prevention:** For services with very high allocation rates (> 1GB/s), consider ZGC which has near-zero Remark pauses.

The pause is within the `MaxGCPauseMillis` target but the end-to-end SLA is still being breached.
