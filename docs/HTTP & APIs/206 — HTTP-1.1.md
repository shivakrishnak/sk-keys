# 206 — HTTP/1.1

⚡ TL;DR — HTTP/1.1 is the text-based request-response protocol that powers the web, where a client sends a structured request and the server replies with a status code and body over a TCP connection.

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
Imagine the early internet: every application that needed to transfer data between
a browser and a server had to invent its own communication protocol from scratch.
How does the client say "give me this page"? How does the server say "I found it"
or "it moved" or "you're not allowed"? How do they agree on what format the data
is in? Without a standard, every website would require a custom client, and the
world-wide web could not exist.

**THE BREAKING POINT:**
HTTP/1.0 existed but had no persistent connections — every request opened a new
TCP connection and then closed it immediately. Loading a page with ten images
meant ten full TCP handshakes: slow, resource-hungry, and broken at any scale.
Teams needed a durable, efficient, and universal transfer protocol.

**THE INVENTION MOMENT:**
This is exactly why HTTP/1.1 was created. Defined in RFC 2616 (1997) and refined
in RFC 7230–7235 (2014), it introduced persistent connections, chunked transfer
encoding, virtual hosting, and a rich header system that turned a simple file
transfer protocol into the backbone of the programmable web.

---

### 📘 Textbook Definition

**HTTP/1.1** (HyperText Transfer Protocol version 1.1) is an application-layer,
stateless, text-based request-response protocol operating over TCP (port 80) or
TLS (port 443 for HTTPS). Each transaction consists of a client sending an HTTP
request message with a method (GET, POST, etc.), a request URI, headers, and an
optional body; the server responds with a status line (version + status code),
response headers, and an optional body. HTTP/1.1 introduced persistent connections
(keep-alive by default), chunked transfer encoding, the `Host` header (enabling
virtual hosting), content negotiation, and conditional requests. It is stateless:
each request carries all information needed for the server to respond — no prior
request context is remembered by the server.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
HTTP/1.1 is the standard language browsers and servers use to ask for and send
resources over the internet.

**One analogy:**

> Imagine ordering a meal at a fast-food counter. You walk up (TCP connection),
> say "I'd like a burger" (GET /burger), and the server says "200 OK, here it is"
> and hands you the food (response body). You can order again immediately without
> leaving the counter (persistent connection). Every order starts fresh — the
> cashier doesn't remember your last visit (stateless).

**One insight:**
HTTP/1.1's most important design decision is statelessness: the server never
remembers previous requests. This makes servers trivially scalable (any server
can handle any request), but it forces clients to re-send identity (cookies,
auth headers) on every request. Understanding this trade-off explains 90% of
why session management, JWT, and cookies exist.

---

### 🔩 First Principles Explanation

**CORE INVARIANTS:**

1. Every interaction is a single request followed by a single response —
   strictly sequential over one TCP connection (in HTTP/1.1 without pipelining).
2. The protocol is stateless: neither party stores context between requests.
3. Everything is text: both request and response are human-readable ASCII
   (with a binary-safe body). This makes debugging trivially easy.

**DERIVED DESIGN:**
Given these constraints, the designers arrived at a simple wire format:

```
┌─────────────────────────────────────────────────────┐
│               HTTP/1.1 Request Structure            │
├─────────────────────────────────────────────────────┤
│ START LINE:  METHOD  URI  HTTP/1.1                  │
│ HEADERS:     Key: Value (one per line)              │
│ BLANK LINE:  (signals end of headers)               │
│ BODY:        (optional, for POST/PUT)               │
└─────────────────────────────────────────────────────┘
```

The blank line separating headers from body is the parser's boundary.

**Persistent connections** (keep-alive) were added because TCP setup is expensive:
a 3-way handshake costs at least one full round-trip time (RTT). Reusing the same
TCP connection for multiple requests amortises that cost. HTTP/1.1 makes keep-alive
the default; you opt out with `Connection: close`.

**`Host` header** was added because a single IP address can serve many domain names.
Without `Host: example.com`, the server cannot tell which virtual site the request
targets.

**THE TRADE-OFFS:**

- Gain: simplicity, universality, debuggability, cacheability
- Cost: head-of-line blocking — if three requests are pipelined, request 3 waits
  for request 1 to complete. This spawned HTTP/2's multiplexing solution.

### 🧪 Thought Experiment

**SETUP:**
You're building a browser. You need to load `example.com/index.html`, which
references 10 images. You're using HTTP/1.0, which closes the TCP connection
after every response.

**WHAT HAPPENS WITHOUT HTTP/1.1 KEEP-ALIVE:**

1. Browser opens TCP connection (3-way handshake: ~100ms RTT)
2. Browser sends GET /index.html
3. Server sends response, closes connection
4. For each of 10 images: open new TCP connection (100ms), send GET,
   receive response, close connection
5. Total: 11 × 100ms handshakes = 1,100ms just in TCP setup overhead
   before a single byte of content is displayed.

**WHAT HAPPENS WITH HTTP/1.1 KEEP-ALIVE:**

1. Browser opens one TCP connection to example.com
2. Sends GET /index.html — receives response — connection stays open
3. Sends GET /img/1.png — receives response — still open
4. Sends GET /img/2.png … and so on for all 10 images
5. Total: 1 × 100ms TCP setup, then 10 sequential request/response cycles
   over the same connection. Orders of magnitude faster.

**THE INSIGHT:**
Connection reuse turns a protocol whose per-request overhead was dominated by
TCP setup into one where the bottleneck is actual data transfer. This insight
directly led to the "connection pool" pattern in every HTTP client library today.

### 🧠 Mental Model / Analogy

> Think of HTTP/1.1 as a walkie-talkie conversation with strict turn-taking:
> one party talks (request), the other responds (response), and then the first
> can talk again. The channel (TCP connection) stays open between turns.
> Each message is a complete, self-contained statement — the responder has
> no memory of previous messages and reacts purely to what was just said.

**Mapping:**

- "walkie-talkie channel" → TCP connection
- "one party talks" → client sends HTTP request
- "the other responds" → server sends HTTP response
- "channel stays open" → keep-alive / persistent connection
- "no memory of previous messages" → statelessness
- "message format" → request-line + headers + body

**Where this analogy breaks down:**
With HTTP pipelining (rare), the client can queue multiple requests before
receiving the first response — more like mailing several letters simultaneously
than strict walkie-talkie turn-taking.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
HTTP/1.1 is the language your browser uses to ask websites for pages and files.
When you type a URL, your browser sends an HTTP request ("give me this page"),
and the server sends an HTTP response ("here it is, it's HTML"). Every website
you've ever visited uses this protocol.

**Level 2 — How to use it (junior developer):**
As a developer you work with HTTP/1.1 every day through frameworks (Spring MVC,
Express, Django) that abstract the wire format. What matters: choose the right
HTTP method (GET = read, POST = create, PUT/PATCH = update, DELETE = remove),
set appropriate headers (Content-Type, Accept, Authorization), and interpret
status codes (2xx = success, 4xx = client error, 5xx = server error). Always
close response streams to return the connection to the pool.

**Level 3 — How it works (mid-level engineer):**
Every HTTP/1.1 request travels as raw text over a TCP socket. The client writes
the request including all headers and body, then reads the response. The parser
must correctly handle: chunked transfer encoding (for streaming responses of
unknown length), content-length parsing (used when length is known), compression
via `Content-Encoding: gzip`, multipart bodies, and the `Transfer-Encoding`
vs `Content-Encoding` distinction. Connection pooling requires understanding
server-side limits (`max-keepalive-requests`) and idle timeout handling to avoid
stale socket errors.

**Level 4 — Why it was designed this way (senior/staff):**
HTTP/1.1's statelessness was a deliberate scalability choice: any request can
hit any server in a farm without session affinity. The cost is that developers
must reinvent state management (cookies, JWT, session stores). The text-based
format was intentional too — binary protocols are faster but opaque; text-based
protocols are debuggable with `curl` and Wireshark, which accelerated adoption.
Head-of-line blocking (one slow response blocks subsequent requests on the same
connection) was accepted as a known trade-off, not an oversight. HTTP/2 was
designed specifically to fix it.

### ⚙️ How It Works (Mechanism)

**Request Wire Format:**

```
GET /api/users/123 HTTP/1.1\r\n
Host: api.example.com\r\n
Accept: application/json\r\n
Authorization: Bearer eyJhbGc...\r\n
Connection: keep-alive\r\n
\r\n
```

The `\r\n` (CRLF) terminates each header line. A blank CRLF line signals the
end of headers. For POST/PUT, the body follows immediately after.

**Response Wire Format:**

```
HTTP/1.1 200 OK\r\n
Content-Type: application/json\r\n
Content-Length: 47\r\n
Connection: keep-alive\r\n
\r\n
{"id":123,"name":"Alice","role":"admin"}
```

**Step-by-step request lifecycle:**

```
┌───────────────────────────────────────────────────┐
│             HTTP/1.1 Request Lifecycle            │
├───────────────────────────────────────────────────┤
│ 1. DNS lookup: example.com → 93.184.216.34        │
│        ↓                                          │
│ 2. TCP 3-way handshake (SYN → SYN-ACK → ACK)     │
│        ↓                                          │
│ 3. (If HTTPS) TLS handshake (1-2 RTT)             │
│        ↓                                          │
│ 4. Client writes HTTP request text to socket      │
│        ↓                                          │
│ 5. Server reads request; parses method, path,     │
│    headers                                        │
│        ↓                                          │
│ 6. Server routes to handler; executes logic       │
│        ↓                                          │
│ 7. Server writes HTTP response text to socket     │
│        ↓                                          │
│ 8. Client parses status line, headers, body       │
│        ↓                                          │
│ 9. Connection kept alive for next request         │
│    (unless Connection: close)                     │
└───────────────────────────────────────────────────┘
```

**Chunked Transfer Encoding:**
When the server does not know the response length upfront (streaming), it uses
`Transfer-Encoding: chunked`. Each chunk is prefixed with its hex size:

```
HTTP/1.1 200 OK
Transfer-Encoding: chunked

7\r\n
Mozilla\r\n
9\r\n
Developer\r\n
0\r\n
\r\n
```

A chunk size of `0` signals the end.

**On failure path:**
If the TCP connection drops mid-response, the client receives a partial body
with no valid Content-Length match. A well-behaved HTTP client surfaces this
as a `ConnectionResetException` or `IncompleteRead`. Never assume partial data
is valid — always validate the complete response.

### 🔄 The Complete Picture — End-to-End Flow

**NORMAL FLOW:**

```
┌───────────────────────────────────────────────────┐
│         HTTP/1.1 End-to-End Flow                  │
├───────────────────────────────────────────────────┤
│ Browser/Client                                    │
│    │ DNS resolve example.com                      │
│    ↓                                              │
│ TCP Connect (3-way handshake)                     │
│    │                                              │
│    ↓ [HTTP/1.1 ← YOU ARE HERE]                   │
│ Write request text to socket                      │
│    │                                              │
│    ↓                                              │
│ Server OS TCP buffer → Read by server app         │
│    │                                              │
│    ↓                                              │
│ Server: parse method/path/headers → handler       │
│    │                                              │
│    ↓                                              │
│ Handler logic → write response to socket          │
│    │                                              │
│    ↓                                              │
│ Client reads response, parses, renders            │
└───────────────────────────────────────────────────┘
```

**FAILURE PATH:**
Server takes >30 s to respond → client read timeout fires →
`SocketTimeoutException` logged → user sees "Gateway Timeout" (504).

**WHAT CHANGES AT SCALE:**
At high concurrency, HTTP/1.1's per-connection limit (one in-flight request)
forces browsers and API clients to open many parallel connections (typically 6
per host). This exhausts server socket file descriptors at ~50k concurrent users.
HTTP/2 multiplexing solves this by carrying many parallel streams on one TCP
connection, reducing connection count by 6–10×.

### 💻 Code Example

**Example 1 — Raw HTTP/1.1 request with curl (observe the wire):**

```bash
# -v shows request/response headers verbatim
curl -v --http1.1 https://httpbin.org/get

# Output shows:
# > GET /get HTTP/1.1
# > Host: httpbin.org
# > User-Agent: curl/7.88
# > Accept: */*
# >
# < HTTP/1.1 200 OK
# < Content-Type: application/json
# < Content-Length: 304
```

**Example 2 — Java: reading HTTP/1.1 response correctly:**

```java
// BAD: Ignores response body — connection not returned to pool
HttpURLConnection conn = (HttpURLConnection)
    new URL("https://api.example.com/users").openConnection();
int code = conn.getResponseCode(); // leaves body unread
// BAD: The TCP socket is now stuck — pool exhaustion!

// GOOD: Always drain or close the response body
try (var client = HttpClient.newHttpClient()) {
    HttpRequest req = HttpRequest.newBuilder()
        .uri(URI.create("https://api.example.com/users"))
        .version(HttpClient.Version.HTTP_1_1)
        .build();
    HttpResponse<String> resp =
        client.send(req, BodyHandlers.ofString());
    System.out.println(resp.statusCode()); // 200
    System.out.println(resp.body());
} // connection returned to pool automatically
```

**Example 3 — Spring Boot: setting HTTP version explicitly:**

```java
// application.properties
// server.http2.enabled=false  ← forces HTTP/1.1

// Verify the protocol from a request:
@GetMapping("/info")
public String info(HttpServletRequest request) {
    // "HTTP/1.1" or "HTTP/2.0"
    return request.getProtocol();
}
```

### ⚖️ Comparison Table

| Version      | Connections  | Multiplexing         | Header Format | Best For             |
| ------------ | ------------ | -------------------- | ------------- | -------------------- |
| **HTTP/1.1** | Multiple TCP | None (pipeline only) | Text          | Legacy, simple APIs  |
| HTTP/2       | One TCP      | Yes (streams)        | HPACK binary  | Modern APIs, browser |
| HTTP/3       | QUIC (UDP)   | Yes (no HOL)         | QPACK binary  | High-latency, mobile |
| HTTPS/1.1    | Multiple TCP | None                 | Text + TLS    | Secure legacy APIs   |

**How to choose:** Use HTTP/1.1 when you need maximum compatibility with older
servers and clients, or when debugging requires human-readable wire traffic.
Upgrade to HTTP/2 for any modern, latency-sensitive API or web application.

### ⚠️ Common Misconceptions

| Misconception                                          | Reality                                                                                                            |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------ |
| HTTP/1.1 is stateful because cookies persist           | Cookies are stored client-side; the protocol itself is stateless — headers re-sent each request                    |
| HTTP and HTTPS are different protocols                 | HTTPS is HTTP/1.1 (or 2/3) transported over TLS — the application protocol is identical                            |
| HTTP/1.1 can only do one request per connection        | HTTP/1.1 supports pipelining (multiple queued requests), but most browsers disable it due to head-of-line blocking |
| Connection: keep-alive must be explicitly set          | In HTTP/1.1, keep-alive is the DEFAULT; you only need Connection: close to opt out                                 |
| HTTP/1.1 always sends the full response before closing | Chunked transfer encoding allows streaming responses of indefinite length                                          |

### 🚨 Failure Modes & Diagnosis

**Connection Leak (fd exhaustion)**

**Symptom:** Server stops accepting connections; `EMFILE: too many open files` in logs;
`curl` returns "connection refused" even though the process is running.

**Root Cause:** HTTP responses whose body is never fully read leave the TCP connection
in CLOSE_WAIT state. The socket fd is never released, eventually exhausting the
OS limit.

**Diagnostic Command / Tool:**

```bash
# Count CLOSE_WAIT sockets on the server
ss -s | grep CLOSE-WAIT
# Or per-process:
lsof -p <pid> | grep -c TCP
```

**Fix:**

```java
// BAD — body unread:
conn.getResponseCode(); // never reads body

// GOOD — always drain body:
try (InputStream body = conn.getInputStream()) {
    body.transferTo(OutputStream.nullOutputStream());
}
```

**Prevention:** Use an HTTP client library (OkHttp, Apache HC, Java 11 HttpClient)
that always drains response bodies automatically.

---

**Stale Connection / TCP Half-Open**

**Symptom:** Requests to a connection pool fail with `Connection reset by peer`
sporadically, especially after a period of idle.

**Root Cause:** HTTP/1.1 keep-alive connections have a server-side idle timeout
(e.g., Nginx defaults 75 s). If the client holds a pooled connection longer
than this timeout, the server closes it. The client's next request on that
socket hits a half-open connection.

**Diagnostic Command / Tool:**

```bash
# Check Nginx keepalive_timeout:
nginx -T | grep keepalive_timeout
# Check client-side pool settings (Java):
curl -v --http1.1 https://api.example.com/health
```

**Fix:** Configure client pool `idle-timeout` to be less than the server's
`keepalive_timeout`. For OkHttp: `connectionPool(new ConnectionPool(5, 60, SECONDS))`.

**Prevention:** Use a health-checking connection pool that validates idle connections
before reuse via a cheap HEAD request or TCP write-probe.

---

**Head-of-Line Blocking**

**Symptom:** Under load, p99 response time rises dramatically while p50 stays low;
requests for large resources delay small, fast requests on the same connection.

**Root Cause:** HTTP/1.1 processes requests sequentially per TCP connection. A slow
3 MB file download blocks a subsequent 1 KB API call waiting behind it.

**Diagnostic Command / Tool:**

```bash
# See concurrent connection count from Nginx access logs:
awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c
# Chrome DevTools Network tab: waterfall view shows blocking
```

**Fix:** Increase connection pool size (browsers open 6 parallel connections per host),
or upgrade to HTTP/2 which solves this with multiplexing.

**Prevention:** Serve static assets from a CDN over HTTP/2; avoid mixing large binary
and small API requests on the same connection when using HTTP/1.1.

### 🔗 Related Keywords

**Prerequisites (understand these first):**

- `TCP` — HTTP rides on a TCP connection; you must understand the three-way
  handshake to understand connection latency and keep-alive
- `DNS` — before any HTTP request, DNS resolves the hostname to an IP address
- `TLS/SSL` — HTTPS is HTTP/1.1 wrapped in TLS; understanding TLS explains the
  extra handshake latency of HTTPS

**Builds On This (learn these next):**

- `HTTP/2` — multiplexing solution to HTTP/1.1's head-of-line blocking
- `REST` — the dominant architectural style for HTTP APIs
- `WebSocket` — upgrades an HTTP/1.1 connection to full-duplex messaging
- `HTTP Headers` — the extensibility mechanism that makes HTTP/1.1 adaptable

**Alternatives / Comparisons:**

- `HTTP/2` — binary, multiplexed successor; same semantics, radically different
  wire format and performance
- `HTTP/3` — built on QUIC/UDP; eliminates TCP head-of-line blocking entirely
- `gRPC` — uses HTTP/2 as transport; adds IDL, code generation, and streaming
