### рҹ“ҳ Textbook Definition

**HTTP/2** (RFC 7540) is the second major revision of HTTP, standardised in 2015, that replaces HTTP/1.1's text-based serial request-response model with a binary framing layer supporting full multiplexing вҖ” multiple concurrent request/response streams over a single TCP connection. Key features include: binary framing (replacing text), stream multiplexing (no head-of-line blocking at HTTP level), HPACK header compression (eliminating redundant header overhead), server push (proactive resource delivery), and stream prioritisation. HTTP/2 is semantically identical to HTTP/1.1 (same methods, status codes, headers) but differs entirely in how data is encoded and transmitted.

### рҹҹў Simple Definition (Easy)

HTTP/2 sends many web requests and responses simultaneously over one connection, like a multi-lane highway instead of a single-lane road вҖ” making pages load faster.

### рҹ”ө Simple Definition (Elaborated)

HTTP/1.1 processes requests one at a time per connection (with pipelining support rarely used). When a browser needs 50 resources, it opens 6 parallel connections and queues requests вҖ” but a slow response on one connection blocks all subsequent requests on that connection. HTTP/2 fixes this: it splits communication into binary "frames" and labels each with a stream ID, so 50 requests can all be in-flight simultaneously over one connection. Responses arrive as frames order-independently; fast responses never wait behind slow ones. Additionally, HTTP/2 compresses redundant headers (auth tokens, cookies sent on every request) saving significant bandwidth.

### рҹ”© First Principles Explanation

**The core problems HTTP/2 solves:**

1. **Head-of-line blocking (HTTP level):** In HTTP/1.1 pipelining, response 3 must wait for responses 1 and 2 even if 3 is ready. HTTP/2's multiplexing delivers response frames from any stream as they're ready.

2. **Connection overhead:** Opening 6 TCP connections per host (browser limit) means 6Г— handshake overhead, 6Г— SSL sessions, 6Г— congestion windows starting from scratch.

3. **Header overhead:** In HTTP/1.1, request headers (Host, Cookie, Accept, Authorization) are sent as redundant plaintext on every request. A 2 KB cookie repeated across 50 requests = 100 KB of pure header overhead.

**Binary framing:**

HTTP/2 introduces a binary framing layer below the HTTP semantics but above TCP:

**Multiplexing via streams:**

Each stream is a bidirectional sequence of frames tagged by Stream ID. Stream IDs are odd for client-initiated, even for server push:

```
Connection (single TCP)
  Stream 1: GET /api/users вҶҗвҶ’ 200 OK JSON
  Stream 3: GET /static/app.js вҶҗвҶ’ 200 OK JS
  Stream 5: GET /api/products вҶҗвҶ’ 200 OK JSON
  (all concurrent, no ordering between streams)
```

**HPACK headercompression:**

HPACK maintains a dynamic table of header name-value pairs, indexed by position. Subsequent requests reference a header by index (e.g., `62`) instead of re-sending the string. Huffman coding for literal values. Result: headers shrink from 2 KB to 50вҖ“100 bytes per request for typical APIs.

### вқ“ Why Does This Exist (Why Before What)

WITHOUT HTTP/2:

- 6 parallel connections per host hit browser limits; 7th+ requests queue.
- Large cookies/auth headers repeated on every request consume bandwidth.
- Server push impossible вҖ” server can't proactively send CSS the browser will need.

What breaks without it:
1. API-heavy apps with 50+ requests per page have unacceptable load times on mobile.
2. gRPC's bidirectional streaming is impractical without HTTP/2's multiplexed frames.

WITH HTTP/2:
вҶ’ Single connection handles all streams; no per-resource connection overhead.
вҶ’ Header compression reduces mobile data usage by 30вҖ“40%.
вҶ’ Server push reduces first-meaningful-paint for critical resources.

### рҹ§  Mental Model / Analogy

> HTTP/1.1 is like a single-track train line: trains leave the station one at a time and must wait for the previous train to arrive before the next can depart (pipelining) вҖ” or you run 6 parallel tracks (6 connections). HTTP/2 is like a multi-lane motorway: many cars travel simultaneously, each in their own lane (stream), tagged with destination labels (stream IDs). The road (TCP connection) is shared, but lanes run independently.

"Single track" = HTTP/1.1 serial request handling, "multi-lane motorway" = HTTP/2 multiplexed streams, "lane labels" = stream IDs, "HPACK" = compressed toll tickets instead of printing a new one each time.

### рҹ’» Code Example

Example 1 вҖ” Java 11+ HttpClient with HTTP/2:

```java
HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_2) // prefer HTTP/2
    .build();

// Concurrent requests вҖ” all go over same connection
List<HttpRequest> requests = List.of(
    HttpRequest.newBuilder()
        .uri(URI.create("https://api.example.com/users"))
        .build(),
    HttpRequest.newBuilder()
        .uri(URI.create("https://api.example.com/products"))
        .build()
);

// Both use the same HTTP/2 connection (multiplexed)
CompletableFuture<?>[] futures = requests.stream()
    .map(req -> client.sendAsync(
        req, HttpResponse.BodyHandlers.ofString()))
    .toArray(CompletableFuture[]::new);

CompletableFuture.allOf(futures).join();
```

### вҡ пёҸ Common Misconceptions

| Misconception | Reality |
|---|---|
| HTTP/2 eliminates all head-of-line blocking | HTTP/2 eliminates HTTP-level HOL blocking. TCP-level HOL blocking remains: a lost TCP packet stalls ALL streams on the connection. HTTP/3 (QUIC) solves TCP-level HOL blocking. |
| HTTP/2 always requires TLS | Technically HTTP/2 can run over cleartext (h2c), but in practice all browsers require TLS for HTTP/2. Use TLS for h2 in production. |
| HTTP/2 server push is widely useful | Server push has largely fallen out of favour; browsers can cancel pushed resources if already cached, and the implementation complexity is high. `<link rel=preload>` achieves similar goals more simply. |
| HTTP/2 is automatically faster than HTTP/1.1 | For high-latency or low-bandwidth connections with many small requests, HTTP/2 wins. For a single large file download, HTTP/2 and HTTP/1.1 are equivalent. |
| HTTP/2 connections are unlimited streams | `SETTINGS_MAX_CONCURRENT_STREAMS` limits concurrent streams (typically 100вҖ“1000). Exceeding this causes stream resets, not connection drops. |

### рҹ”Ҙ Pitfalls in Production

**1. Not Enabling HTTP/2 for gRPC Services**

```bash
# BAD: gRPC over HTTP/1.1 вҖ” not supported by gRPC spec
# gRPC requires HTTP/2 and specifically uses HTTP/2 trailers

# GOOD: ensure server supports HTTP/2
# Spring Boot with Netty (WebFlux) enables HTTP/2 by default
# For Tomcat: set server.http2.enabled=true in application.yml
```

**2. Single HTTP/2 Connection with Many Slow Streams Causing HOL**

```bash
# HTTP/2 still has TCP-level HOL blocking
# A packet loss on the TCP connection stalls ALL streams
# Monitor: tcp_retransmissions on load balancer metrics

# For ultra-low-latency requirements:
# Migrate to HTTP/3 / QUIC which solves TCP HOL blocking
```

**3. Proxy/Load Balancer Not Supporting HTTP/2 End-to-End**

```bash
# Common: browser вҶ’ LB via HTTP/2, but LB вҶ’ backend via HTTP/1.1
# This is "HTTP/2 termination" вҖ” fine for most use cases
# But eliminates multiplexing benefit for backend calls

# To verify: check server-side received protocol
# In Spring: HttpServletRequest.getProtocol() = "HTTP/2.0"
# If showing "HTTP/1.1", the proxy is downgrading
```

### рҹ”— Related Keywords

- `HTTP/1.1` вҖ” the predecessor HTTP/2 replaces at the transport encoding level.
- `HTTP/3` вҖ” replaces TCP with QUIC, solving TCP-level head-of-line blocking.
- `gRPC` вҖ” the RPC framework built specifically on HTTP/2 streams.
- `Keep-Alive / Connection Pooling` вҖ” HTTP/1.1's workaround that HTTP/2 supersedes.
- `TLS/SSL` вҖ” required in practice for HTTP/2 browser support (ALPN negotiation).
- `API Gateway` вҖ” typically terminates HTTP/2 from clients and may re-issue HTTP/1.1 to backends.
