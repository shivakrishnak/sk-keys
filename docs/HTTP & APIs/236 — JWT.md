# 236 — JWT (JSON Web Token)

⚡ TL;DR — A JWT is a self-contained, cryptographically signed string that encodes claims (user ID, roles, expiry) in a Base64URL-encoded JSON payload; the server verifies the signature using a secret or public key — without a database lookup — enabling stateless authentication at scale.

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
Stateful server-side sessions require the server to store session data (userId, roles)
in a database or in-memory cache for every logged-in user. When you scale horizontally
to 10 servers, either every server must query the same session database (adding latency)
or you use sticky sessions (routing the same client to the same server, breaking
horizontal scaling). Additionally, microservice architectures need to pass identity
from service to service — passing a session ID to 10 different services means 10
different DB lookups.

**THE INVENTION MOMENT:**
JWT (defined in RFC 7519, 2015) solved the stateless token problem. The insight: put the
session data INTO the token itself and cryptographically sign it. The server doesn't need
to "look up" the session because the token IS the session data, and the signature proves
it wasn't tampered with. Any server with the right public key (or shared secret) can
verify and trust the token's claims independently. Microservices can extract identity
from the token without calling back to a central auth service.

---

### 📘 Textbook Definition

A **JWT (JSON Web Token)** is an open standard (RFC 7519) for representing claims
securely between parties. A JWT consists of three Base64URL-encoded parts separated
by dots: a **header** (algorithm and token type), a **payload** (claims: sub, exp, iat,
custom data), and a **signature** (cryptographic proof of integrity). JWTs can be
signed (JWS — JSON Web Signature) using HMAC-SHA256 (symmetric) or RSA/ECDSA
(asymmetric) algorithms. The signature allows any holder of the verification key to
independently confirm the token was issued by a trusted party and was not modified.
JWT tokens are NOT encrypted by default (use JWE for that) — the payload is readable
by anyone who has the token. They are machine-verifiable without a database lookup,
enabling stateless authentication.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
A JWT is a tamper-proof, self-describing token: it carries claims about who you are
in JSON, signed so the server can verify it without looking you up in a database.

**One analogy:**

> A JWT is like a digitally signed pay stub. Anyone who knows the company's seal
> (public key) can verify it's genuine and read your salary/role/name from it.
> You carry it with you to each department. No department needs to call HR (auth DB)
> to check your identity — they verify the seal and trust the claims on the stub.
> The stub has a date range (exp) after which it's no longer valid.

**One insight:**
The critical insight is that JWTs move session state from the server to the client.
This inverts the trust model: instead of "I trust this session ID because I stored
it," the server says "I trust this token because its signature is valid." The
tradeoff: stateless = scalable but also means revocation is hard (you can't
"delete" a token that's cryptographically valid).

### 🔩 First Principles Explanation

**TOKEN ANATOMY:**

```
A real JWT (spaces added for clarity):
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6ImtleS0xIn0
.
eyJzdWIiOiJ1c2VyLTQyIiwibmFtZSI6IkFsaWNlIiwicm9sZXMiOlsiVVNFUiJdLCJpYXQiOjE2MzAwMDAwMDAsImV4cCI6MTYzMDAwMzYwMCwiYXVkIjoiYXBpLmV4YW1wbGUuY29tIiwiaXNzIjoiaHR0cHM6Ly9hdXRoLmV4YW1wbGUuY29tIn0
.
<signature>

DECODED:
HEADER:
{
  "alg": "RS256",          ← signing algorithm
  "typ": "JWT",            ← token type
  "kid": "key-1"           ← key ID (which key to use for verification)
}

PAYLOAD (claims):
{
  "sub": "user-42",          ← subject (user ID) — standard claim
  "name": "Alice",           ← custom claim (readable by anyone!)
  "roles": ["USER"],         ← custom claim
  "iat": 1630000000,         ← issued at (Unix timestamp) — standard claim
  "exp": 1630003600,         ← expiry (1 hour from iat) — standard claim
  "aud": "api.example.com",  ← audience (who this token is for) — standard claim
  "iss": "https://auth.example.com"  ← issuer — standard claim
}

SIGNATURE:
RS256(BASE64URL(header) + "." + BASE64URL(payload), privateKey)
```

**VERIFICATION PROCESS:**

```
Received token from client:
Header.Payload.Signature
  ↓
Step 1: Decode header → find "alg": "RS256", "kid": "key-1"
Step 2: Fetch public key for kid="key-1" from JWKS endpoint (cache this!)
Step 3: Verify signature: RS256_verify(header.payload, publicKey, signature) → TRUE?
Step 4: Decode payload claims:
  - exp > now? (not expired) → 1630003600 > 1630001200? ✓
  - iss == "https://auth.example.com"? ✓
  - aud contains "api.example.com"? ✓
Step 5: Extract sub="user-42", roles=["USER"]
Step 6: Authorization: can user-42+USER role access this endpoint? → proceed
```

**HMAC vs RSA:**

```
HS256 (HMAC-SHA256): symmetric
  - One shared secret: both issuer and verifier use the SAME key
  - Problem: if any verifier is compromised, attacker can forge tokens
  - Use when: you control all systems (monolith, single service)

RS256 (RSA-256): asymmetric
  - Private key: auth server signs (kept secret)
  - Public key: all services verify (safe to distribute)
  - Compromise of a service: can't forge tokens (no private key)
  - Use when: multiple services, third-party verifiers, OIDC providers

ES256 (ECDSA-256): asymmetric, smaller keys, faster
  - Same trust model as RS256, better performance
  - Prefer over RS256 in new systems
```

### 🧪 Thought Experiment

**JWT EXPIRY AND REVOCATION DILEMMA:**

```
Access token: exp = 15 minutes
Refresh token: exp = 30 days (stored in DB — stateful)

Scenario: User account compromised at T=0. User contacts support at T=5.
Support revokes account at T=5.

With 15-min JWT access token:
  - Attacker's existing access token valid until T=15 (10 more minutes)
  - After T=15: token refresh fails (refresh token invalidated in DB)
  - Maximum exposure: 10 minutes
  - No DB hit on every request during those 10 minutes

With 24-hour JWT access token:
  - Attacker's token valid for up to 23h55m more
  - Unacceptable for sensitive operations

With 5-minute JWT (tighter):
  - Exposure: max 5 minutes
  - Refresh overhead: client must refresh every 5 minutes
  - Acceptable tradeoff for most security requirements

With immediate revocation (stateful check per request):
  - Check "is this jti (JWT ID) revoked?" → Redis lookup on every request
  - Adds ~1ms per request
  - Eliminates revocation window entirely
  - BUT: defeats the stateless benefit of JWTs (you're doing DB lookups again)
  - Usually: keep short expiry + maintain Redis revocation list for emergency cases
```

### 🧠 Mental Model / Analogy

> A JWT is a driver's license for APIs.
>
> Your driver's license contains claims: your name, your date of birth, your license
> class, expiry date. It's signed by the DMV (the authority).
> Any police officer (server) can verify the license: check the hologram/watermark
> (signature), read the claims, check the expiry. They don't radio the DMV to
> confirm it's valid on every traffic stop.
>
> The limitation: if your license is stolen, it's valid until the expiry date and
> there's no easy way to invalidate just your license remotely. The DMV relies on
> short validity periods (1-4 years for a driver's license = 15 minutes for a JWT)
> to bound the damage window.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
A JWT is a digitally-signed identity card stored as a text string. It contains who
you are (user ID), what you're allowed to do (roles), and when it expires. The server
can trust it because of the signature, without looking you up in a database.

**Level 2 — How to use it (junior developer):**
Spring Security: configure `JwtDecoder` with JWKS URI. Client: store JWT in HttpOnly
cookie (not localStorage!). Include as `Authorization: Bearer <token>` header. Or via
cookie (needs CSRF protection). Never put passwords, sensitive PII, or secrets in JWT
payload — it's Base64-encoded, readable by anyone with the token.

**Level 3 — How it works (mid-level engineer):**
JWT validation must check: signature (using JWKS, cached), `exp` (not expired), `iss`
(trusted issuer), `aud` (intended recipient is you), optionally `nbf` (not before).
JWKS endpoint (`/.well-known/jwks.json`) returns public keys — cache this for
performance (TTL 5-15 minutes). Support multiple `kid` values in JWKS to enable key
rotation (new key added, old key stays valid during rotation window). Use `jti` claim
(JWT ID) for revocation blacklists if needed. Never disable signature verification
in tests — use test-signed tokens instead.

**Level 4 — Why it was designed this way (senior/staff):**
JWT's three-part structure enables the key separation: header carries metadata (algorithm,
key ID), payload carries claims (user data), signature carries the tamper-evidence.
This structure avoids re-fetching the signing key on every verification (kid in header
lets you cache by key ID). The Base64URL encoding (vs standard Base64) uses `-_` instead
of `+/`, making JWTs safe in URLs and HTTP headers without percent-encoding. The
`alg: none` attack (CVE-2015-9235 family) exploited implementations that trusted the
token's self-declared algorithm — always enforce algorithm in verification, never accept
what the token claims. The decision to make payload readable (not encrypted by default)
reflects JWT's use as an authorization token: downstream services need to read claims.
For sensitive data, use JWE (JSON Web Encryption) — less common in practice.

### ⚙️ How It Works (Mechanism)

```
TOKEN LIFECYCLE:

ISSUANCE (Auth Server):
  User authenticates →
  authServer.generateJWT({
    sub: "user-42",
    iss: "https://auth.company.com",
    aud: "api.company.com",
    exp: now + 900,   // 15 minutes
    iat: now,
    jti: UUID.randomUUID(),
    roles: ["USER", "PREMIUM"]
  }, privateKey)
  → eyJhbGci...Header.eyJzdWIi...Payload.Signature

TRANSMISSION:
  HTTP Request:
  Authorization: Bearer eyJhbGci...Header.eyJzdWIi...Payload.Signature

VERIFICATION (Resource Server):
  Parse → split by "." → 3 parts
  Base64URL decode header → extract alg + kid
  Fetch public key for kid from JWKS cache
  Verify signature cryptographically
  Decode payload → validate exp, iss, aud
  Extract claims → authorization

KEY ROTATION:
  1. Generate new key pair (kid="key-2")
  2. Add to JWKS endpoint (both key-1 and key-2 now published)
  3. Auth server starts signing with key-2
  4. Old tokens (key-1) still valid → services check JWKS for key-1, find it → verify ✓
  5. After old tokens expire: remove key-1 from JWKS
```

### 🔄 The Complete Picture — End-to-End Flow

```
1. POST /auth/login { email, password }
   Server: verify password hash → generate JWT (15min) + refresh (30d)
   Response: { access_token: "eyJ...", refresh_token: "...", expires_in: 900 }

2. Client stores: access_token (memory/HttpOnly cookie), refresh_token (HttpOnly cookie)

3. Client: GET /api/profile
   Authorization: Bearer eyJ...
   Server:
     - Spring Security JwtDecoder.decode() → validate sig + claims
     - Extract sub="user-42"
     - Load user context (maybe from JWT claims, not DB)
     - Return 200 { id: 42, name: "Alice" }

4. access_token expires → 401 from server
   Client: POST /auth/refresh { refresh_token }
   Server: validate refresh token (DB lookup) → issue new access_token
   Rotate refresh_token (invalidate old, issue new)
```

### 💻 Code Example

```java
// JWT issuance (signing) — using java-jwt or nimbus-jose-jwt

// Using Auth0 java-jwt library:
@Service
public class JwtService {

    private final RSAPrivateKey privateKey;   // loaded from secure storage
    private final RSAPublicKey publicKey;
    private final String issuer = "https://auth.company.com";

    public String generateAccessToken(User user) {
        return JWT.create()
            .withIssuer(issuer)
            .withAudience("api.company.com")
            .withSubject(user.getId().toString())
            .withClaim("roles", user.getRoles())
            .withIssuedAt(new Date())
            .withExpiresAt(Date.from(Instant.now().plusSeconds(900)))  // 15 min
            .withJWTId(UUID.randomUUID().toString())  // for revocation tracking
            .sign(Algorithm.RSA256(publicKey, privateKey));
    }

    public DecodedJWT verify(String token) {
        JWTVerifier verifier = JWT.require(Algorithm.RSA256(publicKey, null))
            .withIssuer(issuer)
            .withAudience("api.company.com")
            .build();
        return verifier.verify(token); // throws JWTVerificationException if invalid
    }
}

// Spring Security JWT Decoder with audience validation:
@Bean
public JwtDecoder jwtDecoder() {
    NimbusJwtDecoder decoder = NimbusJwtDecoder
        .withJwkSetUri("https://auth.company.com/.well-known/jwks.json")
        .build();

    OAuth2TokenValidator<Jwt> withIssuer =
        JwtValidators.createDefaultWithIssuer("https://auth.company.com");
    OAuth2TokenValidator<Jwt> withAudience = token ->
        token.getAudience().contains("api.company.com")
            ? OAuth2TokenValidatorResult.success()
            : OAuth2TokenValidatorResult.failure(
                new OAuth2Error("invalid_token", "Wrong audience", null));

    decoder.setJwtValidator(
        new DelegatingOAuth2TokenValidator<>(withIssuer, withAudience));
    return decoder;
}
```

### ⚖️ Comparison Table

| Aspect           | Session ID                  | JWT                      |
| ---------------- | --------------------------- | ------------------------ |
| **Server state** | Required (store session)    | None (stateless)         |
| **Verification** | DB lookup per request       | Cryptographic check      |
| **Revocation**   | Instant (delete session)    | Hard (wait for expiry)   |
| **Scale**        | Sticky session or shared DB | Any server validates     |
| **Payload**      | Opaque (server-side)        | Self-describing (Base64) |
| **Size**         | ~32 bytes                   | ~500B–2KB                |

### ⚠️ Common Misconceptions

| Misconception                               | Reality                                                                                                                              |
| ------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| JWT payload is encrypted                    | Base64URL-encoded only — anyone with the token can decode and read it. Use JWE for encryption                                        |
| A valid JWT means the request is authorized | A valid JWT means the identity is authenticated. Authorization (can they access this resource?) is a separate check                  |
| Store JWT in localStorage for SPAs          | LocalStorage is accessible to JavaScript → XSS steals tokens. Use HttpOnly cookies (CSRF risk → mitigate with SameSite + CSRF token) |
| `alg: none` in header is acceptable         | Never accept `alg: none` — it's a well-known attack. Hard-code the expected algorithm in your verifier                               |

### 🚨 Failure Modes & Diagnosis

**Algorithm Confusion Attack (alg:none)**

**Symptom:**
Security scanner reports "JWT algorithm confusion — alg:none accepted."

**Root Cause:**
JWT library configured to trust the algorithm declared in the token header.
Attacker crafts: header `{"alg":"none"}`, arbitrary payload, empty signature
→ library skips signature verification → attacker accepted as any user.

**Fix:**

```java
// WRONG: accept whatever alg the token claims
JWTVerifier verifier = JWT.require(Algorithm.none()).build();

// CORRECT: hardcode expected algorithm
JWTVerifier verifier = JWT.require(Algorithm.RSA256(publicKey, null))
    .withIssuer(issuer).build();

// Spring Security: NimbusJwtDecoder automatically rejects alg:none
// Always configure jwsAlgorithm explicitly if you override defaults
```

**Q.** You're building a financial trading platform where compliance requires an audit trail of every action taken by a user. JWTs are used for authentication. Token expiry is 15 minutes. A trade is executed at T=14:59 using a valid JWT; the trade settlement happens at T=15:01, but the JWT is now expired. Design the audit and authorization model that: correctly attributes the action to the authenticated user, handles the expiry-during-long-operation case, and produces an immutable audit record that a regulator can review 5 years later.
