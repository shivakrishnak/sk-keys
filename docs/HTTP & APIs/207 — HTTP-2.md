# 207 — HTTP/2

⚡ TL;DR — HTTP/2 multiplexes many requests over a single TCP connection using binary frames and streams, eliminating HTTP/1.1's head-of-line blocking and cutting page load times by reusing one encrypted connection for everything.

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
Web pages in 2012 loaded dozens to hundreds of resources: HTML, CSS, JavaScript,
fonts, images. All served over HTTP/1.1. Browsers worked around the one-in-flight-
request-per-connection limit by opening six parallel TCP connections per host.
Each connection required a TCP handshake plus a TLS handshake — adding hundreds
of milliseconds of pure setup overhead. Engineers resorted to ugly hacks: inlining
images as Base64 in CSS to avoid extra connections, domain sharding (splitting
assets across 4 subdomains to get 24 parallel connections), and CSS/JS bundling
into one giant file to reduce request count.

**THE BREAKING POINT:**
Google's research found that each additional HTTP/1.1 connection added ~70ms of
latency per roundtrip. A typical page needing 80 resources sent over 1.1 spent
more time in connection setup than in actual data transfer. The protocol itself
was the bottleneck.

**THE INVENTION MOMENT:**
This is exactly why HTTP/2 was created. Google's SPDY protocol (2009) proved the
binary multiplexed model. HTTP/2 (RFC 7540, 2015) standardised it: one connection,
many parallel streams, binary framing, header compression — same semantics as
HTTP/1.1 but radically different wire format.

---

### 📘 Textbook Definition

**HTTP/2** is a binary application-layer protocol (RFC 7540) that replaces HTTP/1.1's
text-based framing with a binary framing layer. All communication is divided into
small **frames** (HEADERS, DATA, SETTINGS, WINDOW_UPDATE, etc.) sent over
**streams** — independent bidirectional sequences of frames within a single TCP
connection. Multiple streams coexist concurrently on one connection, enabling true
request multiplexing. HTTP/2 introduces **header compression** (HPACK algorithm),
**stream prioritisation**, **flow control** at the stream and connection levels,
and optionally **server push** (proactively sending resources the client will
need). HTTP/2 requires TLS in practice (all major browsers enforce it). HTTP
semantics (methods, status codes, headers) remain unchanged from HTTP/1.1.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
HTTP/2 runs many simultaneous conversations over a single connection, where
HTTP/1.1 could only run one at a time.

**One analogy:**

> Think of HTTP/1.1 as a single-lane road: each car (request) must wait for
> the car ahead to clear before proceeding. HTTP/2 is a multi-lane highway:
> many cars travel simultaneously, a slow truck in lane 1 doesn't block the
> sports car in lane 3. The road itself (TCP connection) is the same; only
> the lane organisation changes.

**One insight:**
HTTP/2's critical insight is that head-of-line blocking in HTTP/1.1 is a
_protocol_ problem, not a network problem. By splitting responses into independent
frames assigned to numbered streams, a slow large response no longer blocks faster
small responses. This is why HTTP/2 dramatically reduces p99 latency even when
median latency is similar.

---

### 🔩 First Principles Explanation

**CORE INVARIANTS:**

1. All HTTP semantics (methods, status codes, headers) remain the same.
   HTTP/2 changes only the wire encoding.
2. Every interaction happens on **streams** — independent channels numbered
   within a single TCP connection.
3. All communication is broken into **frames** of a maximum negotiated size,
   each tagged with a stream ID. Frames from different streams can interleave
   freely.

**DERIVED DESIGN:**

```
┌──────────────────────────────────────────────────┐
│         HTTP/1.1 vs HTTP/2: Wire Model           │
├──────────────────────────────────────────────────┤
│ HTTP/1.1 (one request per connection):           │
│  TCP connection:                                 │
│  [Req1][Resp1][Req2][Resp2][Req3][Resp3]         │
│                                                  │
│ HTTP/2 (multiplexed streams):                    │
│  TCP connection:                                 │
│  [S1:HEADERS][S3:HEADERS][S1:DATA][S5:HEADERS]  │
│  [S3:DATA][S5:DATA][S1:DATA-END][S3:DATA-END]   │
│  (S1, S3, S5 = 3 parallel streams/requests)     │
└──────────────────────────────────────────────────┘
```

Every HTTP/2 frame has a 9-byte fixed header:

- Length (3 bytes) — payload size in bytes (max 16,383 by default)
- Type (1 byte) — HEADERS(0x1), DATA(0x0), SETTINGS(0x4), etc.
- Flags (1 byte) — END_STREAM, END_HEADERS, PADDED, etc.
- Stream Identifier (4 bytes) — which stream this frame belongs to

**HPACK Header Compression:**
HTTP/1.1 repeats headers verbatim on every request (`User-Agent`, `Cookie`,
`Accept-Encoding` — often 500–1000 bytes per request). HPACK maintains a dynamic
table of previously seen headers on both sides. Subsequent requests send integer
indices instead of full strings, compressing headers by 60–90%.

**THE TRADE-OFFS:**

- Gain: multiplexing, header compression, server push, one TLS handshake
- Cost: head-of-line blocking moves to TCP layer (one packet loss stalls all
  streams); HTTP/2 is more complex to implement and debug (binary, not text);
  server push is often misused and cache-wasted

### 🧪 Thought Experiment

**SETUP:**
A web page requires three resources: `/api/user` (fast, 5ms), `/api/dashboard`
(medium, 100ms), and `/reports/large.csv` (slow, 2000ms). A user loads this page
over HTTP/1.1 with one connection, then over HTTP/2.

**WHAT HAPPENS WITHOUT HTTP/2 (HTTP/1.1, one connection):**

1. Client sends GET /api/user → waits 5ms → receives response
2. Client sends GET /api/dashboard → waits 100ms → receives response
3. Client sends GET /reports/large.csv → waits 2,000ms → receives response
   Total wait time: 2,105ms sequential

**WHAT HAPPENS WITH HTTP/2:**

1. Client opens one TCP+TLS connection
2. Sends all three requests simultaneously on streams 1, 3, 5
3. Server responds: /api/user arrives at 5ms, /api/dashboard at 100ms,
   /reports/large.csv at 2,000ms — all in parallel
   Total wait time: 2,000ms (dominated by slowest, not sum)

**THE INSIGHT:**
In HTTP/1.1, total latency = sum of all response times when using one connection.
In HTTP/2, total latency = max of all response times. For a page with 50 resources,
this difference is enormous. Multiplexing converts serial latency into parallel
latency.

### 🧠 Mental Model / Analogy

> HTTP/2 is like a post-office sorting room that uses numbered labeled bins
> for every in-progress parcel (stream). Instead of one sorting clerk handling
> one parcel completely before touching the next, clerks interleave work across
> all parcels simultaneously — a piece of parcel #3 is processed, then a piece
> of parcel #7, then back to #3. The bin number (stream ID) always says which
> parcel each piece belongs to.

**Mapping:**

- "sorting room" → single TCP connection
- "numbered bin" → HTTP/2 stream (identified by stream ID)
- "parcel piece" → HTTP/2 frame
- "sorting clerk work" → server processing
- "all parcels simultaneously" → multiplexing
- "bin number on every piece" → stream ID in every frame header

**Where this analogy breaks down:**
The analogy suggests equal priority processing, but HTTP/2 has stream
prioritisation — streams can be weighted so critical resources (CSS blocking
page render) are delivered before lower-priority images.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
HTTP/2 is a faster version of the protocol your browser uses to download web pages.
The key improvement: instead of asking for one thing at a time, your browser can
ask for everything at once over a single connection.

**Level 2 — How to use it (junior developer):**
HTTP/2 is transparent to most application code — if your web server supports it
(Nginx, Apache, Tomcat 8.5+), it negotiates HTTP/2 automatically via the TLS ALPN
extension. Enable it in Spring Boot with `server.http2.enabled=true`. Use
HTTP/2-aware HTTP clients (Java 11+ HttpClient, OkHttp 3.14+, curl). Avoid
domain sharding (it defeats HTTP/2's single-connection multiplexing) and avoid
bundling all JS/CSS into one giant file (individual files can now be cached
independently).

**Level 3 — How it works (mid-level engineer):**
HTTP/2 negotiation happens via ALPN (Application-Layer Protocol Negotiation) in
the TLS ClientHello — the client lists `h2` as a supported protocol, and the
server confirms it. After the TLS handshake, both sides send a connection preface
(`PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n` from client, SETTINGS frame from server)
to establish stream parameters. Flow control uses WINDOW_UPDATE frames to prevent
fast senders from overwhelming slow receivers at both the stream and connection
level. Understanding SETTINGS negotiation (header table size, max concurrent
streams, initial window size) is essential for tuning under high load.

**Level 4 — Why it was designed this way (senior/staff):**
HTTP/2 was intentionally designed to preserve HTTP/1.1 semantics exactly — methods,
headers, status codes unchanged — to ensure backward compatibility through proxies
and intermediaries. The binary frame format was chosen over extending text because
a binary parser is unambiguous, fast, and zero-copy-friendly. Server push was
included to solve the latency problem of critical-path CSS/JS, but was underused
in practice because push races with client cache — pushing a resource the client
already has wastes bandwidth. Chrome removed server push support in 2022. The real
HTTP/2 weakness (TCP-layer head-of-line blocking) required moving to QUIC (UDP),
which became HTTP/3.

### ⚙️ How It Works (Mechanism)

**The HTTP/2 Frame:**
Every frame is prefixed with a 9-byte header:

```
┌──────────────────────────────────────────────────┐
│ HTTP/2 Frame Header (9 bytes, fixed)             │
├──────────────────────────────────────────────────┤
│ Length    (24 bits) — payload size               │
│ Type      (8 bits)  — HEADERS, DATA, SETTINGS..  │
│ Flags     (8 bits)  — END_STREAM, END_HEADERS    │
│ Reserved  (1 bit)   — always 0                   │
│ Stream ID (31 bits) — which stream               │
└──────────────────────────────────────────────────┘
```

**Connection Setup:**

```
┌──────────────────────────────────────────────────┐
│          HTTP/2 Connection Setup Flow            │
├──────────────────────────────────────────────────┤
│ Client      TLS ClientHello (ALPN: h2)           │
│     ──────────────────────────────────→          │
│ Server      TLS ServerHello (ALPN: h2)           │
│     ←──────────────────────────────────          │
│ Client      Connection preface + SETTINGS        │
│     ──────────────────────────────────→          │
│ Server      SETTINGS + SETTINGS_ACK              │
│     ←──────────────────────────────────          │
│ Client sends HEADERS frame (stream 1)            │
│ Server sends HEADERS+DATA frames (stream 1)      │
└──────────────────────────────────────────────────┘
```

**Stream States:**

```
┌──────────────────────────────────────────────────┐
│           HTTP/2 Stream State Machine            │
├──────────────────────────────────────────────────┤
│  idle ──HEADERS──→ open                          │
│          ↓                                       │
│  open ──END_STREAM──→ half-closed (local/remote) │
│          ↓                                       │
│  half-closed ──END_STREAM──→ closed              │
│  Any state ──RST_STREAM──→ closed                │
└──────────────────────────────────────────────────┘
```

**HPACK Compression:**

```
┌──────────────────────────────────────────────────┐
│          HPACK Header Compression                │
├──────────────────────────────────────────────────┤
│ Request 1: sends full header                     │
│ "content-type: application/json" → 34 bytes      │
│ Added to dynamic table at index 62               │
│                                                  │
│ Request 2: sends index only                      │
│ [62] → 1 byte instead of 34 bytes               │
│ 97% compression on repeated headers             │
└──────────────────────────────────────────────────┘
```

**Failure path (RST_STREAM):**
If a client cancels a request mid-flight (user navigates away), it sends a
`RST_STREAM` frame to immediately terminate that stream. The server stops
processing and discards buffered data for that stream ID. Other streams are
completely unaffected.

### 🔄 The Complete Picture — End-to-End Flow

**NORMAL FLOW:**

```
┌──────────────────────────────────────────────────┐
│        HTTP/2 Multiplexed Request Flow           │
├──────────────────────────────────────────────────┤
│ Client                                           │
│  │ TLS+ALPN negotiation → HTTP/2 agreed          │
│  │ SETTINGS exchange                             │
│  ↓ [HTTP/2 ← YOU ARE HERE]                      │
│  │ Send HEADERS frame (stream 1, GET /api/user) │
│  │ Send HEADERS frame (stream 3, GET /api/feed) │
│  │ Send HEADERS frame (stream 5, GET /img/logo) │
│  ↓                                              │
│ Server: routes each stream to parallel handler  │
│  │ Stream 1 completes first → DATA + END_STREAM │
│  │ Stream 5 completes second                    │
│  │ Stream 3 completes last                      │
│  ↓                                              │
│ Client: delivers responses as they arrive       │
└──────────────────────────────────────────────────┘
```

**FAILURE PATH:**
TCP packet loss in HTTP/2 stalls ALL streams (TCP head-of-line blocking):
one dropped packet blocks all stream frames behind it in the TCP buffer until
retransmission completes. Clients see all concurrent requests freeze together —
unlike HTTP/1.1 where only the affected connection freezes.

**WHAT CHANGES AT SCALE:**
At 10× load, the single TCP connection per client becomes a bottleneck if
stream concurrency is misconfigured. Default max concurrent streams is 100;
raising it too high causes server-side memory pressure (each stream buffers
headers + window state). At 1000× load, the TLS handshake cost disappears
(mostly amortised), but flow control window sizes must be tuned — a default
65,535-byte window throttles high-throughput large-file transfers.

### 💻 Code Example

**Example 1 — Verify HTTP/2 negotiation:**

```bash
# Check if a server supports HTTP/2:
curl -v --http2 https://api.example.com/health 2>&1
| grep "Using HTTP2"
# Output: "Using HTTP2, server supports HTTP/2"

# Check ALPN negotiation directly:
openssl s_client -alpn h2 -connect api.example.com:443 \
  < /dev/null 2>&1 | grep "ALPN"
# Output: ALPN protocol: h2
```

**Example 2 — Java 11 HttpClient with HTTP/2:**

```java
// Java 11+ HttpClient defaults to HTTP/2 with TLS fallback to HTTP/1.1
HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_2) // prefer HTTP/2
    .connectTimeout(Duration.ofSeconds(10))
    .build();

// Sends three requests in parallel over one connection:
var req1 = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users"))
    .build();
var req2 = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/orders"))
    .build();

// Both requests use the same HTTP/2 connection (different streams)
CompletableFuture<HttpResponse<String>> f1 =
    client.sendAsync(req1, BodyHandlers.ofString());
CompletableFuture<HttpResponse<String>> f2 =
    client.sendAsync(req2, BodyHandlers.ofString());

CompletableFuture.allOf(f1, f2).join();
System.out.println(f1.get().version()); // HTTP_2
```

**Example 3 — Spring Boot enabling HTTP/2:**

```yaml
# application.yml
server:
  http2:
    enabled: true # Requires TLS in production
  ssl:
    key-store: classpath:keystore.p12
    key-store-password: ${SSL_PASSWORD}
    key-store-type: PKCS12
```

**Example 4 — Nginx HTTP/2 configuration:**

```nginx
server {
    listen 443 ssl http2;  # ← http2 parameter enables it
    server_name api.example.com;
    ssl_certificate /etc/ssl/certs/cert.pem;
    ssl_certificate_key /etc/ssl/private/key.pem;

    # Tune HTTP/2 for API servers:
    http2_max_concurrent_streams 128;
    http2_recv_buffer_size 256k;
}
```

### ⚖️ Comparison Table

| Feature         | HTTP/1.1               | **HTTP/2**      | HTTP/3               |
| --------------- | ---------------------- | --------------- | -------------------- |
| Transport       | TCP                    | TCP             | QUIC (UDP)           |
| Multiplexing    | No (pipelining broken) | Yes (streams)   | Yes (QUIC streams)   |
| Header encoding | Text                   | HPACK binary    | QPACK binary         |
| HOL blocking    | Per-connection         | TCP-level       | Eliminated           |
| TLS required    | Optional               | De-facto yes    | Yes (always)         |
| Best For        | Legacy, simple APIs    | Modern web APIs | High-latency, mobile |

**How to choose:** Use HTTP/2 for any modern HTTPS API or web server — it is
the default in all major browsers and servers. Reserve HTTP/3 investigation for
environments with high packet loss rates (mobile, satellite, video streaming).

### ⚠️ Common Misconceptions

| Misconception                               | Reality                                                                                                                  |
| ------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| HTTP/2 eliminates all head-of-line blocking | HTTP/2 only eliminates HTTP-level HOL; TCP-level HOL (one dropped packet stalls all streams) remains. HTTP/3 fixes this. |
| HTTP/2 requires HTTPS                       | Technically optional per RFC 7540, but every major browser and many servers enforce TLS for HTTP/2                       |
| Server push makes pages faster              | In practice, server push races with client cache; Chrome removed push support in 2022 due to mis-use                     |
| HTTP/2 replaces domain sharding             | Domain sharding _hurts_ HTTP/2 performance by creating multiple TCP connections that defeat multiplexing                 |
| HTTP/2 always outperforms HTTP/1.1          | Over high-latency, high-packet-loss links, HTTP/2 can be _slower_ than HTTP/1.1 due to TCP-level HOL blocking            |

### 🚨 Failure Modes & Diagnosis

**SETTINGS_TIMEOUT / Connection Stuck**

Symptom: HTTP/2 connections established but no requests complete; logs show
"waiting for SETTINGS frame" or connection reset immediately after TLS.

Root Cause: Client and server failed to exchange the mandatory SETTINGS frame
within the connection timeout. Often caused by a proxy stripping the HTTP/2
preface or a load balancer not supporting HTTP/2 pass-through.

Diagnostic Command / Tool:

```bash
# Test if HTTP/2 frames are actually exchanged:
nghttp -v https://api.example.com/health
# Shows raw frame-level negotiation

# Check if a proxy is downgrading:
curl -v --http2 https://api.example.com/ 2>&1 | grep -E "HTTP/|version"
```

Fix: Configure the load balancer to pass TLS through (L4) or terminate TLS
and re-originate HTTP/2 upstream. Ensure Nginx/envoy `http2` directive is set.

Prevention: Always test HTTP/2 end-to-end in staging, including through all
proxies and load balancers.

---

**Flow Control Window Exhaustion**

Symptom: Large file downloads stall completely after ~65 KB; small requests
continue unaffected; `WINDOW_UPDATE` frames visible in packet traces.

Root Cause: Default HTTP/2 flow control window is 65,535 bytes per stream.
Receiver does not send WINDOW_UPDATE quickly enough; sender stalls waiting
for permission to send more data.

Diagnostic Command / Tool:

```bash
# Capture HTTP/2 frames and look for WINDOW_UPDATE:
wireshark -k -i any -Y "http2.type == 8"
# Or with nghttp:
nghttp -v -d /dev/null -u "https://api.example.com/large.csv"
| grep WINDOW_UPDATE
```

Fix:

```nginx
# Increase initial window size in Nginx:
http2_recv_buffer_size 1m;
# Or via SETTINGS_INITIAL_WINDOW_SIZE in application:
# Java 11 HttpClient tunes this automatically
```

Prevention: Increase initial window size for latency-sensitive or large-payload
APIs. Match window size to expected response sizes.

---

**Max Concurrent Streams Exhaustion**

Symptom: Requests hang waiting; `REFUSED_STREAM (0x7)` errors in logs;
client queues fill up under load bursts.

Root Cause: Server's `SETTINGS_MAX_CONCURRENT_STREAMS` limit reached (default
often 100 or 250). New stream creation is blocked until existing streams close.

Diagnostic Command / Tool:

```bash
# Check current stream count in Go/Nginx/Java server metrics:
curl http://localhost:9090/metrics | grep http2_concurrent_streams
```

Fix: Tune `http2_max_concurrent_streams` based on server memory and CPU load.
Each stream needs ~8–20 KB of server memory.

Prevention: Load-test with realistic concurrent stream counts. Implement client-
side queuing with backpressure rather than creating unbounded streams.

### 🔗 Related Keywords

**Prerequisites (understand these first):**

- `HTTP/1.1` — HTTP/2 replaces HTTP/1.1's wire format; its semantics are identical
- `TLS/SSL` — HTTP/2 requires TLS in practice; understanding ALPN negotiation
  is essential to understanding how HTTP/2 is established
- `TCP` — TCP's stream abstraction and head-of-line blocking are the core
  motivations for HTTP/2's design

**Builds On This (learn these next):**

- `HTTP/3` — solves TCP head-of-line blocking by replacing TCP with QUIC
- `gRPC` — uses HTTP/2 as its transport; multiplexing enables efficient
  bidirectional streaming RPCs
- `Server Push` — HTTP/2 feature for proactively sending resources the client
  will need

**Alternatives / Comparisons:**

- `HTTP/1.1` — text-based, per-connection serial; simpler to debug, worse latency
- `HTTP/3` — QUIC-based; eliminates TCP HOL blocking; higher adoption overhead
- `WebSocket` — uses HTTP/1.1 upgrade; for bidirectional real-time messaging,
  not multiplexed request-response

### 🧠 Think About This Before We Continue

**Q1.** HTTP/2 multiplexes streams over a single TCP connection. In a 1% packet
loss network (common on mobile), TCP retransmits the lost packet before delivering
any subsequent data. With 100 concurrent HTTP/2 streams, all 100 streams stall
during each retransmission event. Compare this to HTTP/1.1 with six connections:
which performs better in this scenario, and precisely why? What does this reveal
about the conditions under which HTTP/2 is actually _worse_ than HTTP/1.1?

**Q2.** HTTP/2's HPACK header compression maintains a shared dynamic table state
between client and server for each connection. If a load balancer routes two
consecutive requests from the same client to different backend servers, what
happens to HPACK decompression on the second server? How do modern HTTP/2
deployments solve this, and what constraint does it impose on infrastructure
topology?
