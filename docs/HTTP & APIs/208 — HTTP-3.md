# 208 — HTTP/3

⚡ TL;DR — HTTP/3 runs HTTP semantics over QUIC (a UDP-based transport) instead of TCP, eliminating the last source of head-of-line blocking and cutting connection setup time to near zero for repeat visitors.

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
HTTP/2 fixed HTTP/1.1's application-layer head-of-line blocking by multiplexing
streams. But it mounted those streams on a single TCP connection. TCP is byte-
stream protocol: if one packet is lost, TCP halts delivery of every subsequent
packet until the lost one is retransmitted. In HTTP/2, this means ALL concurrent
streams freeze — even those whose data arrived perfectly. On mobile networks where
1–5% packet loss is normal, this freezes an entire page load multiple times
per second. The protocol layer fixed HOL at layer 7 but left it equally dangerous
at layer 4.

**THE BREAKING POINT:**
Google measured: on 1% packet loss links, HTTP/2 was slower than HTTP/1.1 with
multiple parallel TCP connections because multiple TCP connections spread retransmit
penalties across independent streams. HTTP/2 concentrated all streams into one TCP
stream, making a single packet loss maximally disruptive.

**THE INVENTION MOMENT:**
This is exactly why QUIC (and HTTP/3 built on it) was created. QUIC implements
its own per-stream reliability at the QUIC layer, over UDP. A lost QUIC packet
for stream #3 only stalls stream #3 — streams #1, #2, #4, #5 continue
unaffected. HTTP/3 (RFC 9114) standardised HTTP semantics over QUIC.

---

### 📘 Textbook Definition

**HTTP/3** (RFC 9114, 2022) is the third major version of HyperText Transfer
Protocol. Unlike HTTP/1.1 and HTTP/2 which use TCP, HTTP/3 runs over **QUIC**
(Quick UDP Internet Connections, RFC 9000) — a multiplexed transport protocol
built on UDP. QUIC integrates TLS 1.3 at the transport layer (there is no
separate TLS handshake), supports per-stream flow control and retransmission
(preventing cross-stream head-of-line blocking), and enables **0-RTT connection
establishment** for repeat visitors (no round trips needed before sending data).
HTTP/3 uses **QPACK** (not HPACK) for header compression, which is designed
to work with QUIC's out-of-order stream delivery. HTTP semantics (methods,
status codes, headers) remain identical to HTTP/2.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
HTTP/3 transmits web requests over UDP with its own built-in reliability,
so a lost packet only delays its own stream, not every request in flight.

**One analogy:**

> Imagine HTTP/2 as a train with 100 passenger cars on a single track.
> If one car (TCP packet) derails, the entire train stops until the track
> is cleared. HTTP/3 is 100 separate cars, each on its own track — if
> car 37 derails, the other 99 cars keep moving. QUIC is the track-building
> technology that makes independent tracks possible over the same physical road
> (UDP).

**One insight:**
HTTP/3's deepest insight is that TCP's guarantee of ordered delivery is both
its greatest strength and its core weakness for multiplexing. QUIC solves this
by delivering streams in order only _within each stream_ — cross-stream ordering
is not imposed. A single lost packet cannot stall a stream that didn't lose it.

---

### 🔩 First Principles Explanation

**CORE INVARIANTS:**

1. QUIC streams are independently reliable: loss on stream A never delays stream B.
2. TLS 1.3 is integrated into QUIC — not layered on top. Every QUIC packet
   (except the handshake bootstrap) is encrypted.
3. Connection IDs decouple connections from IP addresses — a connection survives
   a network change (Wi-Fi → LTE) without reconnecting.

**DERIVED DESIGN:**

```
┌───────────────────────────────────────────────────────┐
│ Protocol Stack Comparison                             │
├───────────────────────────────────────────────────────┤
│ HTTP/1.1 & HTTP/2:          HTTP/3:                  │
│   HTTP                        HTTP/3                 │
│   TLS 1.3                     QUIC (+ TLS 1.3)       │
│   TCP                         UDP                    │
│   IP                          IP                     │
└───────────────────────────────────────────────────────┘
```

**Connection Setup Comparison:**

```
┌───────────────────────────────────────────────────────┐
│           Connection Latency Comparison               │
├───────────────────────────────────────────────────────┤
│ HTTP/1.1 (first visit):                              │
│  TCP 3-way: 1 RTT                                    │
│  TLS 1.3:   1 RTT                                    │
│  Request:   1 RTT = TOTAL: 3 RTT                     │
│                                                      │
│ HTTP/2 (first visit):                                │
│  TCP 3-way: 1 RTT                                    │
│  TLS+ALPN:  1 RTT (combined)                         │
│  Request:   1 RTT = TOTAL: 3 RTT                     │
│                                                      │
│ HTTP/3 (first visit):                                │
│  QUIC+TLS:  1 RTT (combined, no separate TCP)        │
│  Request:   1 RTT = TOTAL: 2 RTT (1 RTT better)     │
│                                                      │
│ HTTP/3 (repeat visit, 0-RTT):                        │
│  QUIC resumption + request in same datagram          │
│              = TOTAL: 0 RTT (data sent immediately!) │
└───────────────────────────────────────────────────────┘
```

**QPACK Header Compression:**
HTTP/2's HPACK assumed a shared, strictly-ordered header table between client
and server. QUIC streams can arrive out of order. QPACK solves this by using a
dedicated header-encoding stream (for table updates) plus independent per-stream
encoding, allowing streams to decode headers without waiting for other streams.

**THE TRADE-OFFS:**

- Gain: stream-independent HOL blocking removal, 0-RTT reconnection, connection
  migration (IP change survivability), integrated TLS
- Cost: UDP is blocked by many enterprise firewalls (QUIC falls back to TCP/HTTP/2);
  harder to debug (encrypted, binary, custom protocol); QUIC moves complexity
  from kernel (TCP) to userspace (increasing CPU usage by 2-5%)

### 🧪 Thought Experiment

**SETUP:**
A mobile user is streaming a live video dashboard with 10 concurrent requests
over HTTP/2. Network packet loss = 2%. We compare HTTP/2 (TCP) vs HTTP/3 (QUIC).

**WHAT HAPPENS WITHOUT HTTP/3 (HTTP/2 over TCP):**

1. 10 streams multiplexed on one TCP connection
2. A packet in stream #7's data is lost
3. TCP stops delivering ALL data for streams 1-10 until packet retransmit arrives
   (typically 100–300ms on mobile networks = full RTT)
4. With 2% packet loss on 10 requests, expected freezes = ~every 5 packets
5. User sees the entire dashboard freeze multiple times per second

**WHAT HAPPENS WITH HTTP/3 (QUIC):**

1. 10 streams on QUIC connection
2. A packet in stream #7's data is lost
3. QUIC stops delivering data only for stream #7 until retransmit
4. Streams 1-6 and 8-10 continue flowing uninterrupted
5. Only the one component associated with stream #7 briefly freezes;
   everything else remains smooth

**THE INSIGHT:**
HOL blocking in HTTP/2 is not theoretical — on any network with >0.5% packet
loss (all mobile networks, most real-world WiFi), it is a constant source of
pause and jitter. HTTP/3's stream-independent reliability converts worst-case
"everything freezes" into "one thing briefly pauses," which users experience
as a fundamentally smoother connection.

### 🧠 Mental Model / Analogy

> HTTP/3 over QUIC is like multiple independent phone calls over a radio
> frequency, where each call has its own error correction. In HTTP/2 (TCP),
> all calls share a single channel — static on the channel corrupts every
> call simultaneously. In HTTP/3, static only corrupts the specific call
> band affected; other calls continue crystal-clear. QUIC is the multi-band
> error-correction layer built into the radio itself.

**Mapping:**

- "radio frequency" → UDP (underlying transport)
- "phone call" → QUIC stream (independent reliability unit)
- "each call's error correction" → per-stream retransmission in QUIC
- "static on the channel" → packet loss
- "multi-band radio" → QUIC connection over UDP
- "crystal-clear other calls" → unblocked HTTP/3 streams

**Where this analogy breaks down:**
QUIC still runs over a single UDP address/port pair. If the UDP port itself is
blocked by a firewall, ALL streams fail — not a single one. The independence
is logical, not physical.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
HTTP/3 is the newest way browsers talk to websites. The key improvement is
that when one piece of data gets lost in transit, only that one thing is
delayed — unlike older versions where everything had to stop and wait for
the missing piece before anything else could continue.

**Level 2 — How to use it (junior developer):**
HTTP/3 is largely transparent to application developers. If your CDN or
web server supports it (Cloudflare, Google, Facebook already serve HTTP/3,
Nginx has QUIC support from 1.25+), it negotiates automatically via the
`Alt-Svc` header or QUIC Discovery. Java 21+ includes QUIC client support;
curl supports HTTP/3 with `--http3` flag. For backend-to-backend API calls,
HTTP/2 is typically sufficient.

**Level 3 — How it works (mid-level engineer):**
QUIC is implemented entirely in userspace (no kernel TCP stack). This lets
it evolve faster but means higher CPU usage per connection. The connection
migrates across network changes using **connection IDs**: a QUIC packet
contains the connection ID, not the IP/port 4-tuple, so switching from WiFi
to LTE keeps the same logical connection alive (unlike TCP which sees the
IP change as a connection reset). QUIC's packet number space is separate
from stream offset, meaning retransmitted packets get new packet numbers —
avoiding TCP's retransmission ambiguity problem that complicates RTT
measurement.

**Level 4 — Why it was designed this way (senior/staff):**
One of QUIC's key architectural decisions was to integrate TLS 1.3 into the
transport rather than layering it above. This means there is no cleartext QUIC
negotiation (except the very first Client Initial packet) — even the handshake
parameters are largely encrypted. This frustrates network middlebox vendors
(who could previously inspect TCP+TLS metadata) but closes a class of downgrade
attacks. The 0-RTT resumption feature (sending data before the handshake completes
using previously established session keys) introduces replay attack risk — QUIC
mitigates this by restricting 0-RTT to idempotent requests (GET, HEAD), requiring
application-level replay protection for POST.

### ⚙️ How It Works (Mechanism)

**QUIC Connection Establishment:**

```
┌───────────────────────────────────────────────────────┐
│        QUIC + HTTP/3 Connection Setup                 │
├───────────────────────────────────────────────────────┤
│ 1-RTT (First Visit):                                  │
│  Client → Initial packet (ClientHello, TLS 1.3)      │
│  Server → Handshake packet (ServerHello + certs)      │
│  Client → Handshake Complete + first HTTP/3 request  │
│                                                       │
│ 0-RTT (Repeat Visit):                                 │
│  Client → Initial packet with 0-RTT data             │
│           (HTTP/3 request embedded immediately)       │
│  Server → Accepts data before handshake completes     │
└───────────────────────────────────────────────────────┘
```

**QUIC Packet Structure:**

```
┌───────────────────────────────────────────────────────┐
│                  QUIC Packet                          │
├───────────────────────────────────────────────────────┤
│ Header (Long or Short form)                           │
│  - Connection ID (variable length, 0–20 bytes)       │
│  - Packet Number (1–4 bytes)                         │
├───────────────────────────────────────────────────────┤
│ QUIC Frames (multiple per packet):                    │
│  STREAM frame:  Stream ID, offset, data              │
│  ACK frame:     Which packets received               │
│  NEW_TOKEN:     0-RTT resumption token               │
│  PATH_CHALLENGE: Connection migration probe          │
└───────────────────────────────────────────────────────┘
```

**Head-of-Line Blocking Removal:**

```
┌───────────────────────────────────────────────────────┐
│  HTTP/2 (TCP): Packet Loss Stalls ALL Streams         │
│                                                       │
│  Packet sequence: [S1][S2][S3-LOST][S4][S5]         │
│  TCP delivers: [S1][S2]... STALL until S3 retransmit  │
│  All streams (S1,S2,S4,S5 data) blocked              │
│                                                       │
│  HTTP/3 (QUIC): Only S3's Stream Stalls              │
│                                                       │
│  Packet sequence: [S1][S2][S3-LOST][S4][S5]         │
│  QUIC delivers: S1-data, S2-data continue →          │
│  S4-data, S5-data continue →                         │
│  S3-data retransmit pending (only S3 stalled)        │
└───────────────────────────────────────────────────────┘
```

**Alt-Svc Discovery (how browsers find HTTP/3):**
A server advertising HTTP/3 sends:

```
Alt-Svc: h3=":443"; ma=86400
```

The browser notes this and on the next connection attempt, tries QUIC/UDP
to port 443. If QUIC is blocked by the firewall, it falls back to TCP/HTTP/2
gracefully (this fallback is called QUIC-happyeyeballs).

### 🔄 The Complete Picture — End-to-End Flow

**NORMAL FLOW:**

```
┌───────────────────────────────────────────────────────┐
│           HTTP/3 Request Flow                         │
├───────────────────────────────────────────────────────┤
│ 1. Browser checks Alt-Svc cache for h3 support        │
│         ↓                                            │
│ 2. QUIC Initial packet (UDP) → server                │
│         ↓                                            │
│ 3. TLS 1.3 handshake embedded in QUIC               │
│         ↓                                            │
│ [HTTP/3 ← YOU ARE HERE]                              │
│ 4. HTTP/3 HEADERS frame on stream 0                  │
│         ↓                                            │
│ 5. Server processes, sends HEADERS + DATA frames     │
│         ↓                                            │
│ 6. Browser renders as DATA frames arrive             │
│    (each stream independent — partial rendering)     │
│         ↓                                            │
│ 7. Connection kept alive for future requests         │
│    (0-RTT on reconnect)                              │
└───────────────────────────────────────────────────────┘
```

**FAILURE PATH:**
QUIC blocked by enterprise firewall → client falls back to TCP/HTTP/2 via
QUIC-happyeyeballs (races QUIC and TCP connections, uses whichever completes
first). Observable: `Alt-Svc` header present but requests served over HTTP/2.

**WHAT CHANGES AT SCALE:**
At 100,000 concurrent QUIC connections, CPU cost is significantly higher than
TCP: QUIC's userspace implementation performs work the kernel's TCP stack
handles at OS level. A single Nginx server handling 100k HTTP/2 TCP connections
might need 2× CPU for the same load with QUIC. Google solved this with custom
kernel-bypass (DPDK/AF_XDP) for QUIC processing.

### 💻 Code Example

**Example 1 — Test HTTP/3 with curl:**

```bash
# Requires curl with QUIC support (curl 7.66+ with quiche or ngtcp2):
curl --http3 https://cloudflare.com/ -v 2>&1 | grep "Using HTTP"
# Output: "Using HTTP/3 Stream ID: 0"

# Force HTTP/3 retry from Alt-Svc:
curl --alt-svc /tmp/altsvc.cache --http3 \
  https://www.google.com/ -I
```

**Example 2 — Nginx 1.25+ with QUIC/HTTP/3:**

```nginx
http {
    server {
        # Listen for HTTP/3 on UDP port 443:
        listen 443 quic reuseport;
        listen 443 ssl;         # HTTP/2 fallback (TCP)
        http2 on;

        ssl_certificate     /etc/ssl/cert.pem;
        ssl_certificate_key /etc/ssl/key.pem;
        ssl_protocols       TLSv1.3;

        # Advertise HTTP/3 support:
        add_header Alt-Svc 'h3=":443"; ma=86400';

        # QUIC-specific settings:
        quic_retry on;     # Source address validation
        ssl_early_data on; # Allow 0-RTT
    }
}
```

**Example 3 — Java (using Netty or quiche4j for QUIC):**

```java
// Java 21 HttpClient with QUIC (preview feature)
// Note: Standard API; QUIC backend varies by JVM build
HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_3)  // Prefer HTTP/3
    .connectTimeout(Duration.ofSeconds(5))
    .build();

HttpRequest request = HttpRequest.newBuilder()
    .uri(URI.create("https://cloudflare.com/cdn-cgi/trace"))
    .build();

HttpResponse<String> response =
    client.send(request, BodyHandlers.ofString());

// Check actual protocol used:
System.out.println(response.version()); // HTTP_3
```

### ⚖️ Comparison Table

| Property               | HTTP/2 (TCP)            | **HTTP/3 (QUIC)**              |
| ---------------------- | ----------------------- | ------------------------------ |
| Transport              | TCP                     | QUIC (UDP)                     |
| HOL blocking           | TCP-level (all streams) | None (per-stream only)         |
| Connection setup       | 2 RTT (first)           | 1 RTT (first), 0 RTT (repeat)  |
| Connection migration   | No (TCP 4-tuple)        | Yes (Connection ID)            |
| Firewall compatibility | Excellent               | Moderate (UDP often blocked)   |
| Debug tooling          | Wireshark, curl -v      | Wireshark w/keys, qvis, quiche |
| CPU overhead           | Low (kernel TCP)        | Higher (userspace QUIC)        |
| Best For               | Modern APIs, enterprise | CDN, mobile, video             |

**How to choose:** Serve HTTP/3 from CDN edge nodes where it provides the
most benefit (mobile users, high packet loss paths). Keep HTTP/2 for
backend-to-backend communication where TCP packet loss is rare and QUIC
UDP may be firewalled.

### ⚠️ Common Misconceptions

| Misconception                                    | Reality                                                                                               |
| ------------------------------------------------ | ----------------------------------------------------------------------------------------------------- |
| HTTP/3 uses UDP so it's unreliable               | QUIC provides its own reliability per stream on top of UDP — HTTP/3 is fully reliable                 |
| HTTP/3 is always faster than HTTP/2              | On low-loss enterprise networks, HTTP/2 often performs comparably or better due to lower CPU overhead |
| 0-RTT is always safe to use                      | 0-RTT data can be replayed by attackers; it must be restricted to idempotent operations               |
| HTTP/3 replaces HTTPS                            | HTTP/3 IS always encrypted (TLS 1.3 is integral to QUIC); it cannot be "downgraded" to cleartext      |
| Switching to HTTP/3 requires application changes | HTTP/3 is transparent to application code; only server/CDN configuration changes needed               |

### 🚨 Failure Modes & Diagnosis

**QUIC Blocked by Firewall**

Symptom: Users behind corporate networks see no HTTP/3 traffic; `Alt-Svc`
header present but protocol observed is `HTTP/2`; no performance improvement.

Root Cause: Enterprise firewalls commonly block or rate-limit UDP traffic to
port 443, because QUIC's encrypted UDP is opaque to deep packet inspection.

Diagnostic Command / Tool:

```bash
# Test if UDP is blocked to a known HTTP/3 server:
curl --http3-prior-knowledge https://cloudflare.com/ -v 2>&1
# If UDP is blocked: "QUIC connection establishment timeout"

# Check actual protocol in browser:
# Chrome DevTools → Network → Protocol column shows "h3" or "h2"
```

Fix: Fall back gracefully to HTTP/2 via QUIC-happyeyeballs. Ensure
`Alt-Svc: h3=":443"` header is set so browsers learn when QUIC succeeds.

Prevention: Always serve both HTTP/2 (TCP) and HTTP/3 (QUIC) on the same
port, letting clients negotiate the best supported version.

---

**0-RTT Replay Attack**

Symptom: POST requests arrive duplicated; idempotency violations in APIs;
write operations applied twice unexpectedly.

Root Cause: 0-RTT data can be replayed by a network attacker who captures
the 0-RTT packet and resends it to a new server. If the application processes
0-RTT POST requests, the action may execute twice.

Diagnostic Command / Tool:

```bash
# Verify 0-RTT is restricted in server config:
nginx -T | grep "ssl_early_data"
# ssl_early_data on; ← allows 0-RTT (check Early-Data header)

# In application: check Early-Data header:
curl --http3 -H "Early-Data: 1" https://api.example.com/checkout
```

Fix: Use `Early-Data` header to detect 0-RTT requests and reject
non-idempotent operations (POST, transactions). Respond with
`425 Too Early` status code.

Prevention: Restrict 0-RTT to GET/HEAD only. Never process monetary or
state-changing operations from Early-Data requests without replay protection.

---

**Connection Migration Failure**

Symptom: Long-lived API requests (file uploads, WebSocket-like flows) reset
when mobile device changes from WiFi to LTE; `CONNECTION_ID_LIMIT_ERROR`.

Root Cause: The client changes its source IP/port but the server's QUIC
implementation does not correctly validate the new path or exhausts the
retired connection ID limit.

Diagnostic Command / Tool:

```bash
# Inspect QUIC connection IDs in use:
# Use qvis (QUIC visualisation) to analyze qlog files
# Enable qlogs on server:
quic_qlog_path /var/log/nginx/qlogs;
```

Fix: Ensure server supports `NEW_CONNECTION_ID` and `PATH_CHALLENGE` frames
per RFC 9000. Increase `active_connection_id_limit` in QUIC configuration.

Prevention: Test mobile connection migration in staging by simulating network
switching. Use a QUIC implementation that actively validates migration paths
before acknowledging.

### 🔗 Related Keywords

**Prerequisites (understand these first):**

- `HTTP/2` — TCP-based predecessor; better enterprise firewall compatibility,
  lower CPU overhead
- `WebTransport` — HTTP/3-based API for bidirectional unreliable and reliable
  streams in browsers (replaces some WebSocket use cases)

### 🧠 Think About This Before We Continue

**Q1.** HTTP/3 eliminates TCP head-of-line blocking by using QUIC's per-stream
retransmission. But QUIC itself runs over a single UDP source port. If a network
middlebox performs per-flow load balancing by UDP 5-tuple (src IP, src port,
dst IP, dst port, UDP), all HTTP/3 streams for one client go to the same backend
server. When that server is taken down for maintenance, what happens to all active
HTTP/3 connections — and how does QUIC's connection migration compare to TCP's
behaviour in the same scenario?

**Q2.** HTTP/3's 0-RTT feature sends application data before the cryptographic
handshake completes, using keys derived from a previous session ticket. A payment
API serves both GET (safe, idempotent) and POST /transactions (non-idempotent)
over HTTP/3 with 0-RTT enabled globally. Design — at the protocol and application
levels — a complete defence strategy that prevents 0-RTT replay attacks from
causing duplicate charges, without disabling 0-RTT entirely.
