### 📘 Textbook Definition

**HTTP/3** (RFC 9114) is the third major version of HTTP, standardised in 2022, which replaces TCP as the transport layer with **QUIC** (RFC 9000) — a UDP-based, encrypted-by-default transport protocol that integrates TLS 1.3 into its handshake. HTTP/3 preserves HTTP/2's multiplexed stream semantics but implements them at the QUIC transport layer, ensuring that packet loss on one stream does not stall other streams (solving TCP-level head-of-line blocking). QUIC also enables 0-RTT and 1-RTT connection establishment, reducing latency compared to TCP's 1–2 RTT + TLS RTTs.

### 🟢 Simple Definition (Easy)

HTTP/3 sends data over a faster, smarter transport layer (QUIC over UDP) so that a lost packet only delays one request, not all of them simultaneously.

### 🔵 Simple Definition (Elaborated)

HTTP/2 fixed request-level queuing by multiplexing multiple requests over one TCP connection. But it couldn't fix TCP itself: when any one TCP packet is lost, ALL streams on that connection stall waiting for retransmission. HTTP/3 replaces TCP with QUIC — a new transport built on UDP. QUIC has the concept of independent streams built in at the transport level, so a lost UDP packet only stalls the specific QUIC stream that contained it. The other streams continue unimpeded. Additionally, QUIC merges TCP's connection handshake and TLS negotiation into a single round-trip (or zero round-trips for repeat connections), making new connections faster.

### 🔩 First Principles Explanation

**TCP's fundamental flaw for multiplexing:**

TCP is a single ordered byte stream. Packet loss causes a retransmission; until the retransmitted packet arrives, all bytes after the gap are held in the receive buffer. Since HTTP/2 maps all streams onto this single TCP byte stream, a lost packet anywhere stalls ALL streams — regardless of which stream's data was in the lost packet.

**QUIC's structural fix:**

QUIC implements the concept of independent streams at the transport layer. Each QUIC stream has its own flow control and sequencing. A lost packet belonging to stream 3 stalls only stream 3; streams 1, 2, 5, and 7 continue delivering their data unaffected.

**QUIC's additional improvements:**

1. **0-RTT connection:** QUIC stores connection parameters from prior sessions. On reconnect, client sends data in the first packet (0 round-trip time). TCP requires 1 RTT + TLS 1.3 requires 1 RTT = 2 RTT minimum.

2. **Connection migration:** QUIC connections are identified by a Connection ID, not by IP:port tuple. When a mobile device switches from WiFi to cellular (IP changes), the QUIC connection transparently migrates — TCP would have to reconnect.

3. **Built-in encryption:** QUIC integrates TLS 1.3; there is no option for unencrypted QUIC. This eliminates the class of attacks that exploit the TCP/TLS boundary.

```
HTTP/1.1:  1 TCP RTT + 2 TLS RTTs + HTTP request = 3–4 RTTs
HTTP/2:    1 TCP RTT + 1 TLS 1.3 RTT + HTTP = 2 RTTs (with 0-RTT TLS)
HTTP/3:    QUIC 1 RTT (merged) + HTTP = 1 RTT (0-RTT for repeat connections)
```

### ❓ Why Does This Exist (Why Before What)

WITHOUT HTTP/3 (HTTP/2 on TCP):

- 2% packet loss rate → TCP stalls ALL HTTP/2 streams during retransmission (~200ms).
- Mobile devices switching networks (WiFi → 4G) break TCP connections → reconnect overhead.
- TCP handshake + TLS handshake = 2+ RTT for new connections.

What breaks without it:
1. On lossy mobile networks, HTTP/2 performs only marginally better than HTTP/1.1.
2. Connection migration across network boundaries requires application-level retry logic.

WITH HTTP/3:
→ Packet loss isolated to affected stream only; others continue.
→ Connection survives network changes via Connection ID.
→ 0-RTT resumes prior connections instantly — critical for API latency.

### 🧠 Mental Model / Analogy

> HTTP/2 over TCP is like sending packages on a single conveyor belt — if one package jams (packet loss), the whole belt stops and every package waits. HTTP/3 over QUIC is like having separate pneumatic tubes for each parcel — a jam in tube 3 doesn't affect tube 1. And QUIC stamps each tube with a permanent parcel number (Connection ID) so if the sorting facility moves (IP change), the tube follows without cancelling the delivery.

"Conveyor belt" = TCP stream, "jam" = packet loss causing HOL blocking, "pneumatic tubes" = QUIC streams, "tube number" = Connection ID for migration.

### ⚙️ How It Works (Mechanism)

**QUIC/HTTP/3 connection establishment:**

```
Initial (new connection):
Client → Server: QUIC Initial packet + TLS ClientHello (1 RTT)
Server → Client: QUIC Handshake + TLS ServerHello + Certificate
Client → Server: TLS Finished
Total: 1 RTT for connection + TLS combined

0-RTT resumption (known server):
Client → Server: QUIC Initial + 0-RTT data (HTTP request!) (0 RTT)
Client gets response before connection formally complete
Total: 0 RTT for data start
```

**QUIC stream independence (packet loss isolation):**

```
QUIC streams (within 1 QUIC connection over UDP):
Stream 1: [ pkt 1 ][ pkt 2 ][ pkt 3 ] → all delivered
Stream 3: [ pkt 4 ][ pkt 5 ][ LOST ][ pkt 7 ] → stalled at LOST
Stream 5: [ pkt 8 ][ pkt 9 ] → fully delivered, unaffected
         ↑ Stream 5 NOT stalled by Stream 3's packet loss
```

**Connection migration:**
```
Device on WiFi:  Connection ID ABC, src IP: 192.168.1.5:1234
  → sends packets to server
Device switches to 4G: IP changes to 10.0.0.8:5678
  → sends QUIC PATH_CHALLENGE from new IP
  → same Connection ID ABC
Server recognises Connection ID → continues session
  (TCP would see new src IP:port as a different connection = broken)
```

### 💻 Code Example

Example 1 — Checking HTTP/3 support:

```bash
# Verify HTTP/3 support with curl
curl -v --http3 https://cloudflare.com 2>&1 | grep "HTTP/"
# HTTP/3 200

# Check if server advertises HTTP/3 via Alt-Svc header
curl -sI https://api.example.com/ | grep -i alt-svc
# alt-svc: h3=":443"; ma=86400
# ↑ Server says: "I support HTTP/3 on port 443, remember for 24h"
```

Example 2 — Spring Boot with HTTP/3 (experimental, Spring Boot 3.3+):

```yaml
# application.yml
server:
  http2:
    enabled: true
  ssl:
    enabled: true
    bundle: server  # TLS config
  # HTTP/3 via Netty (experimental)
  netty:
    h3-enabled: true
```

Example 3 — Alt-Svc header for HTTP/3 discovery:

```bash
# Server advertises HTTP/3 capability via Alt-Svc header
# First connection uses HTTP/1.1 or HTTP/2
# Browser remembers and uses HTTP/3 for subsequent connections

# Response header:
# alt-svc: h3=":443"; ma=3600, h3-29=":443"; ma=3600
# h3    = HTTP/3 (latest)
# h3-29 = HTTP/3 draft-29 (older Chrome versions)
# :443  = same port, QUIC
# ma=3600 = remember for 1 hour
```

### ⚠️ Common Misconceptions

| Misconception | Reality |
|---|---|
| HTTP/3 is based on TCP like previous versions | HTTP/3 runs on QUIC which is UDP-based. TCP is not used. This is the most fundamental change in HTTP's history. |
| QUIC eliminates all packet loss problems | QUIC isolates stream-level packet loss but still experiences network congestion and retransmission delays for each affected stream. It moves the stall from "all streams" to "one stream". |
| HTTP/3 is only for web browsers | APIs, CDNs, and any client-server communication benefit from HTTP/3's connection migration and 0-RTT features. |
| UDP being "unreliable" makes HTTP/3 unreliable | QUIC implements its own reliability (retransmission, ordering) on top of UDP — it's as reliable as TCP but with finer-grained control. |
| HTTP/3 always outperforms HTTP/2 | On low-latency, high-reliability networks (fast data centers), HTTP/2 over TCP performs comparably. HTTP/3's benefits are most pronounced on high-latency, lossy networks (mobile, intercontinental). |

### 🔥 Pitfalls in Production

**1. Enterprise Firewall Blocking UDP Port 443**

```bash
# Many corporate firewalls block UDP, causing HTTP/3 to fail silently
# QUIC uses UDP port 443, which many firewalls restrict

# GOOD: Implement proper fallback
# Browsers automatically fall back to HTTP/2 if UDP is blocked
# For custom clients: try HTTP/3, fall back to HTTP/2 on failure
# Monitor: what % of your traffic uses h3 vs h2?
curl -sI https://api.example.com/ | grep alt-svc
```

**2. Load Balancer Not Supporting QUIC**

```bash
# Many legacy load balancers terminate only TCP
# HTTP/3 connections go directly to backend (bypassing LB)
# or are downgraded at LB

# Check: modern LB support
# Envoy: native QUIC support (1.19+)
# NGINX: QUIC support (1.25+ main branch)
# HAProxy: QUIC support (2.6+)
# AWS ALB: HTTP/3 client-facing (as of 2023)
```

**3. 0-RTT Data Vulnerability to Replay Attacks**

```bash
# 0-RTT data can be replayed by an attacker
# A middlebox can capture and replay 0-RTT data
# NEVER use 0-RTT for non-idempotent requests (POST, DELETE)

# Safe 0-RTT use cases: GET requests for public content
# Unsafe: payment processing, checkout, auth flows
# Configure server to only accept 0-RTT for safe methods
```

**Q1.** QUIC's 0-RTT resumption allows a client to send application data in the very first packet of a reconnection, before the handshake completes. This creates a replay attack vulnerability: an attacker can capture 0-RTT packets and re-send them to the server. Given that HTTP/3 cannot prevent replay at the transport layer (by QUIC design), describe what application-layer mechanisms a REST API must implement to safely handle 0-RTT requests, and why the standard guidance is to only allow GET requests in 0-RTT mode.

**Q2.** HTTP/3 uses Connection IDs to separate connection identity from IP:port addresses, enabling connection migration. An attacker who can observe network traffic knows both the Connection ID and the new IP address. Explain what prevents this attacker from hijacking the migrated connection after a device switches networks, specifically identifying which QUIC mechanism ensures only the legitimate owner can continue the connection after an IP change.
