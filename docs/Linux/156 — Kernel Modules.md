# 156 — Kernel Modules

⚡ TL;DR — Kernel modules are loadable pieces of kernel code (drivers, filesystems, network protocols) that can be inserted and removed at runtime without rebooting — they are how Linux extends the kernel without recompiling it.

---

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
Every driver for every possible piece of hardware — network cards, RAID controllers, USB devices, specialised filesystems — would need to be compiled into a single monolithic kernel binary. Booting a server would load drivers for USB webcams, gaming GPUs, and Bluetooth headsets — hardware that doesn't exist on the server. The kernel binary would be gigabytes large. Adding a new driver requires recompiling the entire kernel and rebooting.

**THE BREAKING POINT:**
A new NVMe storage controller arrives. Without modules, you recompile the kernel, copy it to the boot partition, reboot the server (causing downtime), and hope the new kernel works. With 10,000 server configurations across a datacenter, this is operationally impossible.

**THE INVENTION MOMENT:**
Kernel modules solve this. The NVMe driver is a `.ko` file. `modprobe nvme` loads it at runtime — no reboot, no recompilation. The kernel binary stays small; drivers load on demand or at boot via `initramfs` and `modprobe`.

---

### 📘 Textbook Definition

A **kernel module** (`.ko` file — kernel object) is compiled machine code that can be dynamically loaded into and unloaded from the running Linux kernel. Modules run in kernel space (ring 0) with full hardware access and no memory protection from the rest of the kernel. They are typically used for device drivers, filesystem drivers (ext4, btrfs), network protocols, and security modules (SELinux, AppArmor). Modules are loaded via `insmod` (direct load) or `modprobe` (load with dependency resolution from `/lib/modules/$(uname -r)/`).

---

### ⏱️ Understand It in 30 Seconds

**One line:**
Kernel modules are plug-in extensions for the kernel — they add drivers and capabilities at runtime without a reboot.

**One analogy:**

> The kernel is a smartphone OS. Kernel modules are apps — except these apps run at the deepest level of the phone with access to all hardware. Loading a module is like installing an app that can reprogram the phone's touch screen or battery controller. Unloading it removes it completely. But unlike apps, a buggy module crashes the whole phone — there's no sandbox.

**One insight:**
Modules run in ring 0 (kernel space) with no memory protection. A bug in a kernel module causes a kernel panic. This is why the Linux kernel signs modules and recent distributions enforce signature verification — an unsigned or maliciously-crafted module would have complete control over the system.

---

### 🔩 First Principles Explanation

**CORE INVARIANTS:**

1. Modules run in kernel address space — no user-space protection boundary.
2. A module bug causes kernel oops or panic — there is no process isolation.
3. Modules can register hooks, interrupt handlers, character devices, and filesystem types.
4. Dependencies are tracked: module A may require module B to be loaded first.
5. The module's kernel version must match the running kernel (version checking enforced).

**DERIVED DESIGN:**
When a module is loaded, the kernel calls `module_init()` — a registered function pointer. This function registers the module's capabilities (a character device node, an interrupt handler, a filesystem type). When unloaded, `module_exit()` reverses this. Modules export symbols (functions and data) via `EXPORT_SYMBOL()`, making them available to other modules — this is how `dm-crypt` uses the crypto modules.

Dependencies are resolved from `/lib/modules/$(uname -r)/modules.dep` — generated by `depmod`. `modprobe` uses this to load the full dependency tree before the requested module.

**THE TRADE-OFFS:**
**Gain:** Runtime extensibility; small base kernel; no reboot for new drivers; clean loading/unloading.
**Cost:** Modules run with full kernel privilege — a malicious or buggy module owns the whole system; module interface stability is not guaranteed across kernel versions (in-tree modules are managed; out-of-tree modules break on kernel updates).

### 🧪 Thought Experiment

**SETUP:**
You're adding new Mellanox 100GbE NICs to a server. The OS is running. You cannot reboot (active production load).

**WITHOUT MODULES:**
The NIC would require kernel support baked into the kernel binary at compile time. Without it, the hardware is unrecognised. Options: reboot to a new kernel compiled with the driver, or use a live kernel patch (complex, risky). Neither is viable for the production server.

**WITH MODULES:**

```bash
# Check if the mlx5 driver is already available
ls /lib/modules/$(uname -r)/kernel/drivers/net/ethernet/mellanox/

# Load the driver
modprobe mlx5_core
modprobe mlx5_en

# Verify the NIC is detected
ip link show   # new interface appears
dmesg | tail   # kernel messages confirm detection
```

Hardware recognised, driver running, production traffic can be moved to the new NIC — all without rebooting.

**THE INSIGHT:**
`modprobe` resolves dependencies automatically. `mlx5_en` depends on `mlx5_core` which depends on `mlxfw` and several crypto modules. A single `modprobe mlx5_en` loads the entire tree in the correct order. This dependency graph is what makes the Linux driver model work at scale.

### 🧠 Mental Model / Analogy

> Kernel modules are like licensed electrical contractors for a building's power system. The building's core wiring (kernel) is fixed. A contractor (module) can be called in to wire a new piece of equipment (driver) directly to the building's electrical bus (kernel internals). While working, they have full access to all circuits — but if they make a mistake, they can trip the main breaker (kernel panic). Once finished, they pack up and leave (module unloaded), and the wiring they added stays.

- "Building's core wiring" → compiled kernel
- "Contractor" → kernel module
- "Direct electrical bus access" → kernel ring 0 privilege
- "Trip main breaker" → kernel panic from module bug
- "Wiring stays after contractor leaves" → registered callbacks persist until module unloads

Where this analogy breaks down: unlike contractors who leave after the job, a module stays loaded until explicitly unloaded. And unlike real contractors, a module can be loaded and unloaded hundreds of times without any physical work.

### 📶 Gradual Depth — Four Levels

**Level 1 — What it is (anyone can understand):**
Kernel modules are like plugins for Linux's core OS. They add support for hardware devices, new filesystem types, or networking features without changing the main OS and without rebooting. Think of plugging in a USB device and the driver loading automatically — that driver is a kernel module.

**Level 2 — How to use it (junior developer):**
`lsmod` lists loaded modules. `modinfo module_name` shows info about a module (author, parameters, dependencies). `modprobe module_name` loads a module (with dependencies). `modprobe -r module_name` unloads it (if no longer in use). `insmod /path/to/module.ko` loads a specific file. Automatic loading: add `module_name` to `/etc/modules` or drop a `.conf` file in `/etc/modules-load.d/`. Module parameters: `modprobe iwlwifi 11n_disable=1`.

**Level 3 — How it works (mid-level engineer):**
Modules are ELF (Executable and Linkable Format) object files with a `.modinfo` section containing the expected kernel version, module author, description, and license. On load, the kernel verifier checks the version magic and (if enforced) the cryptographic signature. The `init_module()` syscall copies the module into kernel memory and calls `module_init()`. The module registers its functions (character device methods, interrupt handlers, etc.) by writing function pointers into kernel data structures. `udev` autoloads modules for detected hardware using `/lib/modules/$(uname -r)/modules.alias` (maps hardware identifiers to module names).

**Level 4 — Why it was designed this way (senior/staff):**
The module system was a pragmatic solution to the "monolithic vs microkernel" debate. Linux Torvalds chose a monolithic kernel (all kernel code shares one address space) for performance, but added loadable modules to manage the driver complexity explosion. The cost of this pragmatism is that modules run with full kernel privilege — a deliberate trade-off of security for performance. The alternative (microkernel architecture: drivers in user space, like GNU Mach) was proven to be 2-5× slower in the microkernel wars of the 1990s. Modern mitigations: module signing (UEFI Secure Boot only allows signed modules), Kernel Lockdown (prevents unsigned code at all), and DKMS (Dynamic Kernel Module Support) for maintaining out-of-tree modules across kernel upgrades.

### ⚙️ How It Works (Mechanism)

**Module management commands:**

```bash
# List loaded modules
lsmod
# Output: Module, Size, Used by (dependent modules)

# Module information
modinfo ext4
modinfo --field=depends ext4  # just dependencies
modinfo --field=filename iwlwifi  # path to .ko file

# Load a module
modprobe ext4        # with dependency resolution
insmod /lib/modules/$(uname -r)/kernel/fs/ext4/ext4.ko

# Unload a module
modprobe -r ext4     # with dependency cleanup
rmmod ext4           # direct, no dependency cleanup

# Check if module is built-in (vs loadable)
cat /lib/modules/$(uname -r)/modules.builtin | grep ext4

# Load with parameters
modprobe iwlwifi 11n_disable=1 bt_coex_active=0

# Persistent module parameters
echo "options iwlwifi 11n_disable=1" > \
  /etc/modprobe.d/iwlwifi.conf

# Blacklist a module (prevent autoloading)
echo "blacklist nouveau" >> \
  /etc/modprobe.d/blacklist.conf
# (Used to prevent open-source GPU driver when
#  using proprietary NVIDIA driver)

# Rebuild module dependency cache
depmod -a

# View kernel ring buffer for module messages
dmesg | grep -i 'module\|driver\|error'
dmesg -w  # live view
```

**Module file locations:**

```bash
# All modules for current kernel
ls /lib/modules/$(uname -r)/kernel/

# By category
ls /lib/modules/$(uname -r)/kernel/drivers/
ls /lib/modules/$(uname -r)/kernel/fs/
ls /lib/modules/$(uname -r)/kernel/net/

# Module dependency map
cat /lib/modules/$(uname -r)/modules.dep

# Hardware → module mapping (used by udev)
cat /lib/modules/$(uname -r)/modules.alias

# Find module for a PCI device
lspci -k  # shows kernel driver in use
# OR
lspci -n | while read line; do
  id=$(echo "$line" | awk '{print $3}')
  grep "$id" /lib/modules/$(uname -r)/modules.alias
done
```

**DKMS (out-of-tree modules):**

```bash
# DKMS: automatically rebuilds external modules
# when the kernel is updated
dkms status             # show DKMS-managed modules
dkms install -m module_name -v 1.0.0  # install a module
dkms remove -m module_name -v 1.0.0 --all  # remove
```

### 🔄 The Complete Picture — End-to-End Flow

```
┌────────────────────────────────────────────────┐
│  AUTOLOAD: USB device plugged in               │
└────────────────────────────────────────────────┘

 USB device inserted
       │
       ▼
 Hardware interrupt → USB subsystem detects device
       │  Reads device VID:PID (e.g., 0x046d:0xc52b)
       ▼
 udev event generated (kernel → udev daemon)
       │
       ▼
 udev looks up modules.alias:
 "usb:v046DpC52B..." → "hid_logitech_dj"
       │
       ▼
 udev calls: modprobe hid_logitech_dj
       │
       ▼
 modprobe reads modules.dep:
 hid_logitech_dj depends: hid, usbhid
       │
       ▼
 Load in order: usbhid → hid → hid_logitech_dj
       │  Each: insmod → version check → init()
       ▼
 hid_logitech_dj.init() registers device driver
       │
       ▼
 /dev/input/eventX node created
 Device usable — all in < 500ms
```

**FAILURE PATH:**
If module has wrong kernel version magic: `insmod: ERROR: could not insert module: Invalid module format` → module was compiled for a different kernel version; need to recompile with `make -C /lib/modules/$(uname -r)/build M=$(pwd) modules`.

### 💻 Code Example

**Example 1 — Minimal kernel module (hello world):**

```c
// hello.c — minimal kernel module
#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/init.h>

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Engineer");
MODULE_DESCRIPTION("Hello world kernel module");

static int __init hello_init(void) {
    printk(KERN_INFO "Hello: module loaded\n");
    return 0;  // 0 = success; negative = error
}

static void __exit hello_exit(void) {
    printk(KERN_INFO "Hello: module unloaded\n");
}

module_init(hello_init);
module_exit(hello_exit);
```

```makefile
# Makefile
obj-m += hello.o

all:
	make -C /lib/modules/$(shell uname -r)/build \
	  M=$(PWD) modules

clean:
	make -C /lib/modules/$(shell uname -r)/build \
	  M=$(PWD) clean
```

```bash
# Build and test
make
sudo insmod hello.ko
dmesg | tail -1   # → Hello: module loaded
sudo rmmod hello
dmesg | tail -1   # → Hello: module unloaded
```

**Example 2 — Check and load required networking modules:**

```bash
#!/bin/bash
# Ensure required modules for container networking
REQUIRED_MODULES=(
  "overlay"          # Docker overlay filesystem
  "br_netfilter"     # Bridge netfilter (Kubernetes)
  "ip_vs"            # IPVS load balancing
  "ip_vs_rr"         # IPVS round-robin
  "nf_conntrack"     # Connection tracking
)

for mod in "${REQUIRED_MODULES[@]}"; do
  if ! lsmod | grep -q "^$mod "; then
    echo "Loading: $mod"
    modprobe "$mod"
  else
    echo "Already loaded: $mod"
  fi
done

# Persist across reboots
cat > /etc/modules-load.d/k8s.conf << 'EOF'
overlay
br_netfilter
ip_vs
ip_vs_rr
nf_conntrack
EOF
```

### ⚖️ Comparison Table

| Feature     | Kernel Module              | User-space Driver        | Built-in Driver       |
| ----------- | -------------------------- | ------------------------ | --------------------- |
| Performance | Best (ring 0)              | Worse (IPC overhead)     | Best (same as module) |
| Safety      | Low (ring 0 crash = panic) | High (process isolation) | Low                   |
| Loadability | Runtime                    | Runtime                  | Compile-time only     |
| Kernel ABI  | Unstable (version-tied)    | Stable (syscall API)     | N/A                   |
| Example     | ext4, e1000e, btrfs        | FUSE filesystems         | early-boot drivers    |

How to choose: use kernel modules for performance-critical drivers and core subsystems; use FUSE/user-space drivers for non-critical or experimental drivers where stability matters more than performance.

---

### ⚠️ Common Misconceptions

| Misconception                                        | Reality                                                                                                                                 |
| ---------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| Modules are sandboxed from the kernel                | Modules run in ring 0 — full kernel privilege; a buggy module crashes the entire system                                                 |
| `rmmod` safely removes any module                    | `rmmod` fails if the module is in use (use count > 0); forceful removal (`rmmod -f`) is dangerous and can corrupt kernel state          |
| Built-in drivers are faster than modules             | No performance difference; once a module is loaded, its code runs identically to compiled-in code                                       |
| Any module compiled for the kernel version will load | Modules also require the exact kernel build ID and CONFIG flags to match; distribution kernels add additional checks                    |
| Module parameters can be changed on a running module | Most parameters are set at load time; some expose `sysfs` attributes at `/sys/module/<name>/parameters/` that can be changed at runtime |

### 🚨 Failure Modes & Diagnosis

**Module Load Fails: Version Mismatch**

**Symptom:**
`modprobe: ERROR: could not insert 'module_name': Invalid module format`

**Root Cause:**
Module was compiled against a different kernel version (different `vermagic` string).

**Diagnostic Command:**

```bash
modinfo module_name | grep vermagic
uname -r  # compare: these must match exactly
```

**Fix:**
Recompile the module against the current kernel headers:

```bash
apt install linux-headers-$(uname -r)
# Then rebuild the module source
```

For DKMS modules: `dkms install -m module_name -v version`

### 🔗 Related Keywords

**Prerequisites (understand these first):**

- `Linux File System Hierarchy` — modules live in `/lib/modules/$(uname -r)/`; init systems use `/etc/modules-load.d/`
- `Operating Systems` — ring 0 vs ring 3, interrupt handlers, and device driver models are foundational concepts

**Builds On This (learn these next):**

- `/sys File System` — modules register their attributes in sysfs via kobjects; `/sys/module/<name>/` exposes each loaded module's parameters
- `Linux Security Hardening` — module signing, kernel lockdown, and UEFI Secure Boot build on the module system
- `Linux Namespaces` — Linux Namespaces are implemented as kernel modules; understanding modules explains how namespaces are registered

**Alternatives / Comparisons:**

- `eBPF programs` — modern alternative for extending kernel behaviour without loading a full kernel module; safer (verifier prevents unsafe code) but more limited in scope
- `FUSE (Filesystem in Userspace)` — user-space alternative for filesystem drivers; safer (crashes don't affect kernel) but higher overhead
