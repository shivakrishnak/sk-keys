### 📘 Textbook Definition

**ORC (Optimized Row Columnar)**: an open-source columnar storage format developed within the Apache Hive project (Facebook, 2013). Designed to overcome limitations of the earlier RCFile format. Key structure: file divided into **stripes** (analogous to Parquet row groups, default 256MB). Each stripe contains: **index data** (column statistics: min, max, sum, count, bloom filter per column), **row data** (actual column values, encoded and compressed), and **stripe footer** (stream locations). File **footer** contains stripe metadata, schema, and user metadata. **Postscript** at file end contains compression parameters and footer length. Supports complex types: struct, list, map, union. Compression: ZLIB (default), SNAPPY, LZO. Native ACID support in Hive: delete vectors, base + delta files, compaction. Used by: Apache Hive, Spark (full R/W support), AWS EMR, Apache Iceberg (ORC as alternative to Parquet), Presto/Trino.

---

### 🟢 Simple Definition (Easy)

ORC is to Hive what Parquet is to Spark. Both are columnar formats — store data column-by-column for fast analytics. ORC was developed first (2013) and is tightly integrated with Apache Hive. It includes built-in ACID support (Hive can UPDATE and DELETE rows in ORC tables). Parquet has broader ecosystem support (works natively with Spark, Pandas, Athena, BigQuery). If you're building on Hive, use ORC. If you're building on Spark / modern data lake tools, use Parquet.

---

### 🔵 Simple Definition (Elaborated)

ORC's key differentiators from Parquet:

1. **ACID in Hive**: Hive tables in ORC format support transactional INSERT/UPDATE/DELETE with ACID guarantees. Implemented via base files + delta files + compaction (Major compaction merges delta into base). Parquet lacks native ACID — that's why Delta Lake and Iceberg were created.

2. **Built-in bloom filters**: ORC has bloom filters per column per stripe, enabled by default for string columns. Parquet added bloom filter support later (v2). Both work similarly — ORC's bloom filters are slightly more integrated with the Hive query planner.

3. **Compression**: ORC defaults to ZLIB (better compression than Parquet's default SNAPPY, but slower). ORC files are typically 5-15% smaller than equivalent Parquet files with default settings.

4. **Ecosystem**: Parquet wins on ecosystem breadth. Athena, BigQuery external tables, Pandas/PyArrow — all have first-class Parquet support. ORC is supported but Parquet is the default for most non-Hive tools.

### 🔩 First Principles Explanation

```
ORC FILE STRUCTURE:

  ┌──────────────────────────────────────────────────────────────┐
  │ ORC FILE                                                     │
  │                                                              │
  │ STRIPE 1 (default 256MB):                                    │
  │ ┌──────────────────────────────────────────────────────┐    │
  │ │ Index Data:                                          │    │
  │ │   Column 0 (order_id): min=1001, max=262072, ...     │    │
  │ │   Column 1 (city): bloom_filter, min="Austin",...    │    │
  │ │   Column 2 (amount): min=9.99, max=2499.99, sum=...  │    │
  │ │   (row index every 10,000 rows for position lookup)  │    │
  │ ├──────────────────────────────────────────────────────┤    │
  │ │ Row Data (column streams, compressed):               │    │
  │ │   Stream: column 0 data (int64 delta-encoded)        │    │
  │ │   Stream: column 1 present (bitset: nulls)           │    │
  │ │   Stream: column 1 data (dict-encoded strings)       │    │
  │ │   Stream: column 1 length (string lengths)           │    │
  │ │   Stream: column 2 data (float64)                    │    │
  │ ├──────────────────────────────────────────────────────┤    │
  │ │ Stripe Footer:                                       │    │
  │ │   Stream positions (byte offsets for each stream)    │    │
  │ └──────────────────────────────────────────────────────┘    │
  │                                                              │
  │ STRIPE 2: (same structure)                                   │
  │                                                              │
  │ FILE FOOTER:                                                 │
  │   Schema (type tree: struct > field types)                   │
  │   Stripe information (offset, data length, index length,     │
  │                        footer length, row count)             │
  │   Statistics (per column, whole-file level)                  │
  │   User metadata (key-value pairs)                            │
  │                                                              │
  │ POSTSCRIPT:                                                  │
  │   Compression kind (ZLIB/SNAPPY/NONE)                        │
  │   Footer length, magic (ORC)                                 │
  └──────────────────────────────────────────────────────────────┘

ORC vs PARQUET STRUCTURAL COMPARISON:

  CONCEPT      │ ORC             │ Parquet
  ─────────────┼─────────────────┼────────────────────────────
  Horizontal   │ Stripe (256MB)  │ Row Group (128MB)
  partition    │                 │
  Column chunk │ Stream per col  │ Column Chunk
  Index data   │ Row Index       │ Column statistics (footer)
  Bloom filter │ Built-in        │ Optional (Parquet v2+)
  Null encoding│ Present stream  │ Definition levels (Dremel)
  Schema       │ File footer     │ File footer
  Compression  │ ZLIB default    │ SNAPPY default
  ACID         │ Native (Hive)   │ Via Delta/Iceberg layer
  Ecosystem    │ Hive-primary    │ Universal

ORC HIVE ACID OPERATIONS:

  CREATE TABLE orders (
    id BIGINT,
    amount DOUBLE,
    status STRING
  ) STORED AS ORC
  TBLPROPERTIES ('transactional'='true');

  INSERT INTO orders VALUES (1001, 149.99, 'COMPLETED');
  UPDATE orders SET status='REFUNDED' WHERE id=1001;
  DELETE FROM orders WHERE status='CANCELLED' AND amount < 10;

  Implementation:
  - Base files: full ORC snapshot at last major compaction
  - Delta files: INSERT/UPDATE/DELETE changes since last compaction
  - Delete vectors: bitmap of deleted rows (for MERGE/UPDATE/DELETE)

  Read: merge base + deltas (vectorized ACID reader)
  Compaction:
    Minor: merge multiple delta files into one delta file
    Major: merge base + all deltas → new base file (full rewrite)

  Hive runs automatic background compaction or:
  ALTER TABLE orders COMPACT 'MAJOR';

ROW-LEVEL STATISTICS (ORC advantage):

  Row index: every 10,000 rows within a stripe, ORC records:
  - Column statistics (min/max/sum/count/bloom filter)
  - Byte offset into compressed stream for seek

  This enables sub-stripe skipping (10K row granularity within a stripe):
  → Finer-grained skip than Parquet (which only does row-group-level skipping)

  For a 256MB stripe with 1M rows:
  Row index entries: 100 entries (1M/10K)
  If filter matches only rows 400K-500K: seek to index entry 40, skip to offset
  → Read only 100K rows instead of 1M (within the stripe)
```

### ❓ Why Does This Exist (Why Before What)

ORC was created at Facebook to replace RCFile (Row Columnar File), which stored data in row groups but with poor compression and no statistics. The goal: enable fast Hive queries on petabyte-scale data in HDFS. ORC achieved: 75% reduction in query time vs RCFile, 75% reduction in storage. It became the default Hive format. While Parquet later became more popular for the broader ecosystem, ORC remains the best choice for Hive-native workloads requiring ACID semantics without an additional Delta Lake/Iceberg layer.

---

### 🧠 Mental Model / Analogy

> **ORC vs Parquet are like two competing highway systems**: both get you from A to B (store and query columnar data). **ORC** is the older highway with more integrated features (built-in tolls, rest stops, emergency lanes = bloom filters, ACID, Hive statistics). **Parquet** is the newer highway with a cleaner design and more on-ramps from other cities (Spark, Athena, BigQuery, Pandas all have on-ramps). Both highways support the same speed limits; the choice is about which ecosystem you're already in and which features you need built-in.

### ⚙️ How It Works (Mechanism)

```
ORC BLOOM FILTER:

  Problem: WHERE order_id = 'specific-uuid'
  Min/max statistics: min=000..., max=fff... → useless (every stripe matches)

  Bloom filter: probabilistic membership structure
  For each stripe, for column "order_id":
    Insert all values into bloom filter (bit array of k hash functions)

  Query: does stripe contain order_id='ORD-12345'?
    Hash 'ORD-12345' k times → check k bits in bloom filter
    If ANY bit is 0 → definitely NOT in stripe → SKIP
    If ALL bits are 1 → MAYBE in stripe (false positive possible) → READ

  False positive rate configurable (default ~0.05% in ORC = 1 in 2000)
  Memory: ~1.2 bytes per distinct value for 0.05% FPR
  Result: 99.95% of non-matching stripes eliminated for high-cardinality equality queries

SPARK READING ORC:

  spark.read.orc("s3://bucket/orders/")

  Spark uses the ORC vectorized reader (since Spark 2.3):
  spark.conf.set("spark.sql.orc.impl", "native")      # use ORC native C++ reader
  spark.conf.set("spark.sql.orc.enableVectorizedReader", "true")

  Same optimizations as Parquet:
  - Partition pruning (directory-level)
  - Predicate pushdown (stripe statistics)
  - Column pruning (stream-level selection)
  - Vectorized batch reading (1024 rows per batch)
```

### 💻 Code Example

```java
// Hive: creating and using ORC ACID table

-- Create ACID-enabled ORC table
CREATE TABLE orders (
    order_id BIGINT,
    customer_id STRING,
    amount DOUBLE,
    status STRING,
    updated_at TIMESTAMP
)
CLUSTERED BY (order_id) INTO 16 BUCKETS
STORED AS ORC
TBLPROPERTIES (
    'transactional'='true',
    'orc.compress'='ZLIB',
    'orc.bloom.filter.columns'='order_id,customer_id',
    'orc.bloom.filter.fpp'='0.05'
);

-- ACID operations
INSERT INTO orders VALUES (1001, 'C001', 149.99, 'PENDING', CURRENT_TIMESTAMP);
UPDATE orders SET status='COMPLETED', updated_at=CURRENT_TIMESTAMP WHERE order_id=1001;
DELETE FROM orders WHERE status='CANCELLED' AND updated_at < '2024-01-01';

-- MERGE (upsert pattern)
MERGE INTO orders t
USING staging_orders s ON t.order_id = s.order_id
WHEN MATCHED THEN UPDATE SET t.status=s.status, t.amount=s.amount
WHEN NOT MATCHED THEN INSERT VALUES (s.order_id, s.customer_id, s.amount, s.status, CURRENT_TIMESTAMP);

-- Trigger manual compaction
ALTER TABLE orders COMPACT 'MAJOR';
SHOW COMPACTIONS;
```

```python
# Python/PySpark: reading ORC files
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .config("spark.sql.orc.impl", "native") \
    .config("spark.sql.orc.enableVectorizedReader", "true") \
    .getOrCreate()

# Read ORC with predicate pushdown
df = spark.read.orc("s3://bucket/orders/")
result = df.filter("status = 'COMPLETED' AND amount > 100") \
           .select("order_id", "amount", "customer_id") \
           .groupBy("customer_id") \
           .sum("amount")

# Write ORC with compression
df.write.mode("overwrite") \
        .option("compression", "zlib") \
        .orc("s3://bucket/output/")
```

### ⚠️ Common Misconceptions

| Misconception                                    | Reality                                                                                                                                                                                                                                                                                             |
| ------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ORC ACID works in Spark like it does in Hive     | Spark can READ Hive ACID ORC tables (with the ORC ACID reader), but full ACID write support in Spark (concurrent UPDATE/DELETE with Hive) requires careful configuration and is not the default. For ACID operations in Spark, Delta Lake or Apache Iceberg (which can use ORC) are better choices. |
| ORC is outdated / deprecated in favor of Parquet | ORC is actively maintained and is the native format for Apache Hive ACID. Apache Iceberg (a modern table format) supports both Parquet and ORC as underlying file formats. ORC remains the best choice for Hive-native workloads.                                                                   |
| ORC and Parquet have the same compression        | Different defaults: ORC defaults to ZLIB (good compression ratio, slower CPU). Parquet defaults to SNAPPY (worse ratio, fast CPU). Configure explicitly for your workload. ZSTD is available in both and is generally the best modern choice.                                                       |

### 🔥 Pitfalls in Production

```
PITFALL: reading Hive ACID ORC table from Spark without proper config

  # Hive ACID table: base + delta files
  # s3://warehouse/orders/base_000001/
  # s3://warehouse/orders/delta_000001_000010/
  # s3://warehouse/orders/delete_delta_000005_000007/

  # ❌ WRONG: raw Spark read doesn't understand ACID structure
  df = spark.read.orc("s3://warehouse/orders/")
  # → Reads ALL files including delta/delete files → duplicate/wrong records

  # ✅ CORRECT: use Hive metastore connector OR Spark HiveContext
  spark = SparkSession.builder \
      .config("spark.sql.hive.convertMetastoreOrc", "true") \
      .enableHiveSupport() \
      .getOrCreate()

  df = spark.sql("SELECT * FROM orders")
  # ← Uses Hive metastore to understand ACID file layout; reads merged view

PITFALL: missing bucket configuration for ACID ORC tables

  # Hive ACID requires bucketing (for concurrent writes):
  # ❌ Non-bucketed table with transactional='true' → Hive error

  # ✅ Always define bucketing for ACID tables:
  CLUSTERED BY (id) INTO 16 BUCKETS  -- 16 to 256 typical for large tables
  # Number of buckets = max concurrent writers (each writer owns some buckets)
  # Too few: contention; Too many: too many small files
```
