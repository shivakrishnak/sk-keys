# 1067 — pom.xml

`#maven` `#pom` `#xml` `#project-object-model` `#build-config`

⚡ TL;DR — `pom.xml` (Project Object Model) is Maven's project descriptor. It declares: project identity (groupId, artifactId, version), dependencies (what libraries to use), plugins (how to build), properties (version variables), and profiles (environment-specific config). Every Maven project has exactly one `pom.xml` at its root. In multi-module projects, child POMs inherit from a parent POM, reducing duplication.

### 📘 Textbook Definition

**pom.xml** (Project Object Model): the fundamental unit of Maven configuration. An XML file at the root of every Maven project that defines: (1) **Project coordinates** (`groupId`, `artifactId`, `version`, `packaging`); (2) **Parent**: optional parent POM to inherit from (Spring Boot parent POM is common); (3) **Properties**: reusable key-value pairs (version numbers, encoding settings); (4) **Dependencies**: direct dependencies with `groupId`, `artifactId`, `version`, and optional `scope`; (5) **Dependency Management**: version declarations for dependencies without adding them (used in parent POMs for BOM — Bill of Materials); (6) **Build/Plugins**: plugin configuration overriding defaults; (7) **Profiles**: activated by environment, property, or OS — allow conditional configuration. The POM inherits from the **Super POM** (Maven's built-in default) which defines default plugin versions, repository locations, and directory structure. Effective POM: the fully resolved POM including all inherited parent settings — `mvn help:effective-pom` shows it. A POM is both the build descriptor AND the published artifact descriptor (the `.pom` file deployed alongside the JAR to Maven Central tells consumers what this artifact's dependencies are).

---

### 🟢 Simple Definition (Easy)

`pom.xml` is like a `package.json` for Java. It says: "my app is called `com.example:my-api:1.0.0`, it needs Spring Boot and Jackson, and to build it, run `mvn package`." Maven reads the `pom.xml` and figures out everything else: download the dependencies, compile the code, run the tests, make the JAR.

---

### 🔵 Simple Definition (Elaborated)

`pom.xml` has three main roles:

1. **Identity card**: `groupId:artifactId:version` uniquely identifies this artifact in the Maven ecosystem. When someone depends on your library, they reference these coordinates.

2. **Shopping list**: `<dependencies>` lists what this project needs. Maven downloads them from Maven Central and puts them on the classpath.

3. **Build instructions**: `<build><plugins>` configures how to compile, test, and package. Most projects inherit from `spring-boot-starter-parent` and need almost no plugin configuration — it's pre-configured for Spring Boot.

**Parent POM inheritance**: `spring-boot-starter-parent` provides default versions for 300+ common dependencies (Jackson, SLF4J, JUnit 5, etc.). You declare `<dependency>spring-boot-starter-web</dependency>` with NO version — the parent POM manages it. This is dependency version management: the BOM (Bill of Materials) pattern.

### 🔩 First Principles Explanation

```xml
<!-- ANNOTATED pom.xml REFERENCE -->

<project>
  <modelVersion>4.0.0</modelVersion>  <!-- always 4.0.0; the POM schema version -->

  <!-- ══════════════════════════════════════════════
       PARENT POM (optional; inherit configuration)
       ══════════════════════════════════════════════ -->
  <parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.0</version>
    <!-- Parent provides:
         - dependency versions (Jackson, SLF4J, JUnit 5, etc.)
         - plugin versions (compiler, surefire, jar)
         - default compiler target: Java 17
         - resource filtering for application.properties
    -->
  </parent>

  <!-- ══════════════════════════════════════════════
       PROJECT IDENTITY (GAV)
       ══════════════════════════════════════════════ -->
  <groupId>com.example</groupId>         <!-- org/company (reverse domain) -->
  <artifactId>user-service</artifactId>  <!-- project name -->
  <version>2.1.0-SNAPSHOT</version>      <!-- SNAPSHOT=dev; release=no SNAPSHOT -->
  <packaging>jar</packaging>             <!-- jar|war|pom|ear (default: jar) -->

  <name>User Service</name>             <!-- human-readable name -->
  <description>REST API for user management</description>

  <!-- ══════════════════════════════════════════════
       PROPERTIES (reusable variables)
       ══════════════════════════════════════════════ -->
  <properties>
    <java.version>17</java.version>          <!-- used by compiler plugin -->
    <mapstruct.version>1.5.5.Final</mapstruct.version>
    <testcontainers.version>1.19.3</testcontainers.version>
    <!-- Reference with ${mapstruct.version} in dependencies -->
  </properties>

  <!-- ══════════════════════════════════════════════
       DEPENDENCY MANAGEMENT (version control, no actual deps added)
       ══════════════════════════════════════════════ -->
  <dependencyManagement>
    <dependencies>
      <!-- Import a BOM (Bill of Materials): adds many dependency version definitions -->
      <dependency>
        <groupId>org.testcontainers</groupId>
        <artifactId>testcontainers-bom</artifactId>
        <version>${testcontainers.version}</version>
        <type>pom</type>
        <scope>import</scope>  <!-- import: merges depManagement from this BOM -->
      </dependency>
    </dependencies>
  </dependencyManagement>

  <!-- ══════════════════════════════════════════════
       DEPENDENCIES (actual dependencies added to classpath)
       ══════════════════════════════════════════════ -->
  <dependencies>
    <!-- COMPILE scope (default): on classpath at compile + runtime + test -->
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
      <!-- no version: inherited from spring-boot-starter-parent -->
    </dependency>

    <dependency>
      <groupId>org.mapstruct</groupId>
      <artifactId>mapstruct</artifactId>
      <version>${mapstruct.version}</version>
    </dependency>

    <!-- TEST scope: only on classpath during test compile + run -->
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-test</artifactId>
      <scope>test</scope>
    </dependency>

    <!-- TESTCONTAINERS: version from imported BOM, no version needed -->
    <dependency>
      <groupId>org.testcontainers</groupId>
      <artifactId>postgresql</artifactId>
      <scope>test</scope>
    </dependency>

    <!-- PROVIDED scope: on compile classpath; NOT in runtime JAR -->
    <!-- (the container/server provides it at runtime) -->
    <dependency>
      <groupId>jakarta.servlet</groupId>
      <artifactId>jakarta.servlet-api</artifactId>
      <scope>provided</scope>
    </dependency>
  </dependencies>

  <!-- ══════════════════════════════════════════════
       BUILD CONFIGURATION
       ══════════════════════════════════════════════ -->
  <build>
    <!-- Override default source/output directories (rarely needed) -->
    <!-- <sourceDirectory>src/main/java</sourceDirectory> -->

    <plugins>
      <!-- Spring Boot fat JAR plugin -->
      <plugin>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-maven-plugin</artifactId>
        <configuration>
          <excludes>
            <exclude>
              <groupId>org.projectlombok</groupId>
              <artifactId>lombok</artifactId>
            </exclude>
          </excludes>
        </configuration>
      </plugin>

      <!-- Annotation processor for MapStruct + Lombok -->
      <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-compiler-plugin</artifactId>
        <configuration>
          <annotationProcessorPaths>
            <path>
              <groupId>org.projectlombok</groupId>
              <artifactId>lombok</artifactId>
              <version>${lombok.version}</version>
            </path>
            <path>
              <groupId>org.mapstruct</groupId>
              <artifactId>mapstruct-processor</artifactId>
              <version>${mapstruct.version}</version>
            </path>
          </annotationProcessorPaths>
        </configuration>
      </plugin>
    </plugins>
  </build>

  <!-- ══════════════════════════════════════════════
       PROFILES (conditional configuration)
       ══════════════════════════════════════════════ -->
  <profiles>
    <profile>
      <id>integration-tests</id>
      <activation>
        <property>
          <name>integration</name>   <!-- activated by: mvn verify -Dintegration -->
        </property>
      </activation>
      <build>
        <plugins>
          <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-failsafe-plugin</artifactId>
            <executions>
              <execution>
                <goals>
                  <goal>integration-test</goal>
                  <goal>verify</goal>
                </goals>
              </execution>
            </executions>
          </plugin>
        </plugins>
      </build>
    </profile>
  </profiles>

  <!-- ══════════════════════════════════════════════
       REPOSITORIES (additional artifact sources)
       ══════════════════════════════════════════════ -->
  <!-- Usually not needed; Maven Central is the default -->
  <!-- Add for private repos (Nexus/Artifactory) or jitpack -->
  <repositories>
    <repository>
      <id>company-nexus</id>
      <url>https://nexus.company.com/repository/maven-releases/</url>
    </repository>
  </repositories>
</project>
```

### ❓ Why Does This Exist (Why Before What)

The POM file externalizes build configuration from build scripts, making it declarative and machine-readable. Unlike Ant's imperative XML (write every command), the POM describes WHAT the project is. Maven (and IDEs like IntelliJ IDEA) can read the POM and infer everything: import dependencies, set up classpath, configure compilation. The POM is also a CONTRACT: when published to Maven Central, the `.pom` file tells consumers "here are my dependencies" — Maven can resolve transitive dependencies across the entire ecosystem without bundling them.

---

### 🧠 Mental Model / Analogy

> **`pom.xml` is like a birth certificate + resume + bill of materials combined**: the birth certificate (`groupId:artifactId:version`) uniquely identifies the artifact in the Maven world. The resume (description, developers, SCM URL) describes the project. The bill of materials (`<dependencies>`) lists exactly what the project needs to function. When you publish a JAR to Maven Central, the `.pom` file travels with it — consumers read it to know what else to download.

### 💻 Code Example

```bash
# View the effective POM (all inherited settings resolved):
mvn help:effective-pom

# Show all dependency versions (including inherited from parent):
mvn dependency:tree

# Validate pom.xml syntax:
mvn validate

# Show active profiles:
mvn help:active-profiles

# Run with a specific profile:
mvn clean package -P integration-tests
```

### ⚠️ Common Misconceptions

| Misconception                                   | Reality                                                                                                                                                                                                                                                    |
| ----------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| You must specify versions for all dependencies  | If inheriting from a parent that uses `<dependencyManagement>` (like `spring-boot-starter-parent`), versions are managed by the parent. You only need the version in `<dependencies>` if the parent doesn't manage it or you want to override it.          |
| `<dependencyManagement>` adds dependencies      | `<dependencyManagement>` only declares VERSIONS — it does NOT add dependencies to the classpath. A dependency must still appear in `<dependencies>` to be active. `dependencyManagement` is typically used in parent POMs to centralize version decisions. |
| `pom.xml` changes always require re-downloading | Maven caches everything in `~/.m2`. Adding a new dependency version that's already cached doesn't re-download. Only new/updated dependencies require network access. `mvn -o` (offline mode) uses only cached artifacts.                                   |

### 🧠 Think About This Before We Continue

**Q1.** Multi-module Maven projects use a parent POM with `<modules>` listing child modules. The parent POM has `<packaging>pom</packaging>` (not jar) and uses `<dependencyManagement>` to centralize all dependency versions. Child POMs declare `<parent>` and can override versions locally. In a microservices repo where 15 services share common dependencies, compare: (a) a monorepo with a shared Maven parent POM, vs (b) separate repos where each service declares its own dependency versions, vs (c) a published internal BOM artifact that each service imports. What are the trade-offs in version consistency, upgrade friction, and build independence?

**Q2.** Maven profiles can be activated by environment variables, system properties, OS, files, or JDK version. A common anti-pattern: different behavior in CI vs local development hidden in profiles, leading to "it works locally but not in CI." What criteria should guide profile design? How do you make profile activation explicit and discoverable? How does the `settings.xml` (user-level Maven config, not committed to the repo) interact with profiles, and what problems does it cause for reproducibility?
