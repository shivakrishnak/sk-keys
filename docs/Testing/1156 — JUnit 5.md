# 1156 — JUnit 5

⚡ TL;DR — JUnit 5 is the standard Java test framework: it provides test lifecycle annotations (`@Test`, `@BeforeEach`), parameterized tests, extensions, and a modular architecture — the foundation for all Java testing.

### 🔥 The Problem This Solves

Java needs a standard test runner that can: discover test methods automatically, manage test lifecycle (setup/teardown), provide extension points for Mockito, Spring, Testcontainers, run tests in parallel, parameterize tests with different inputs, and integrate with Maven/Gradle. JUnit 5 is that framework — the universal foundation that all other Java testing tools integrate with.

### 📘 Textbook Definition

**JUnit 5** (Jupiter) is the 5th major version of the JUnit testing framework for Java. It consists of three sub-projects: **JUnit Platform** (test engine SPI, used by Maven/Gradle to discover and run tests), **JUnit Jupiter** (new API and programming model — the annotations and assertions you write), **JUnit Vintage** (runs JUnit 3/4 tests on JUnit 5 Platform). Key improvements over JUnit 4: modular architecture, extension model (replaces `@Rule`), `@ParameterizedTest`, `@Nested` test classes, conditional execution, dynamic tests.

### ⏱️ Understand It in 30 Seconds

**One line:**
JUnit 5 = the annotations and runner that make your Java test methods executable.

**One analogy:**
> JUnit 5 is the **testing rulebook**: it defines what counts as a test (`@Test`), when setup runs (`@BeforeEach`), how to provide different inputs (`@ParameterizedTest`), and how to organize tests (`@Nested`). Maven and Gradle use the rulebook to find and run all tests automatically.

### 🔩 First Principles Explanation

CORE ANNOTATIONS:
```java
@Test                    // marks a method as a test case
@BeforeEach              // runs before each @Test method
@AfterEach               // runs after each @Test method
@BeforeAll               // runs once before all tests (must be static)
@AfterAll                // runs once after all tests (must be static)
@Disabled("reason")      // skip this test (with mandatory reason)
@DisplayName("...")      // human-readable test name
@Tag("integration")      // categorize tests for selective running
```

ASSERTIONS (JUnit 5 built-in):
```java
// Basic assertions
assertEquals(expected, actual);
assertNotNull(value);
assertTrue(condition);

// Better: use AssertJ instead (see keyword 1158)
assertThat(actual).isEqualTo(expected);

// Exception assertions
assertThrows(IllegalArgumentException.class, () -> service.call(null));

// Multiple assertions (don't stop at first failure)
assertAll(
    () -> assertEquals("Alice", user.getName()),
    () -> assertEquals("alice@test.com", user.getEmail()),
    () -> assertTrue(user.isActive())
);
```

PARAMETERIZED TESTS — THE KILLER FEATURE:
```java
@ParameterizedTest
@ValueSource(strings = {"", " ", "\t", "\n"})
void isBlank_trueForAllBlankStrings(String input) {
    assertThat(StringUtils.isBlank(input)).isTrue();
}

@ParameterizedTest
@CsvSource({
    "alice@example.com, true",
    "not-an-email, false",
    "user@, false",
    "@nodomain, false"
})
void emailValidation(String email, boolean expected) {
    assertThat(EmailValidator.isValid(email)).isEqualTo(expected);
}

@ParameterizedTest
@MethodSource("discountScenarios")
void calculateDiscount(double amount, double expectedDiscount) {
    assertThat(service.calculateDiscount(amount)).isEqualTo(expectedDiscount);
}

static Stream<Arguments> discountScenarios() {
    return Stream.of(
        Arguments.of(50.0, 0.0),    // below threshold
        Arguments.of(100.0, 10.0),  // at threshold
        Arguments.of(200.0, 30.0)   // above threshold (tiered)
    );
}
```

NESTED TESTS — ORGANIZE BY SCENARIO:
```java
@DisplayName("OrderService")
class OrderServiceTest {
    
    @Nested
    @DisplayName("when creating an order")
    class WhenCreating {
        @Test void withValidCart_createsOrder() { ... }
        @Test void withEmptyCart_throwsException() { ... }
    }
    
    @Nested
    @DisplayName("when cancelling an order")
    class WhenCancelling {
        @Test void pendingOrder_cancels() { ... }
        @Test void shippedOrder_throwsException() { ... }
    }
}
```

EXTENSIONS — THE MODULARITY MECHANISM:
```java
// Mockito uses @ExtendWith(MockitoExtension.class)
// Spring uses @ExtendWith(SpringExtension.class) (or @SpringBootTest)
// Testcontainers uses @Testcontainers

@ExtendWith(MockitoExtension.class)  // enables @Mock, @InjectMocks
@ExtendWith(TimingExtension.class)   // custom: logs test execution time
```

### 🧪 Thought Experiment

DYNAMIC TESTS — TESTS GENERATED AT RUNTIME:
```java
@TestFactory
Stream<DynamicTest> validPaymentAmounts() {
    return Stream.of(0.01, 1.00, 9999.99, 100000.00)
        .map(amount -> dynamicTest(
            "payment of " + amount + " should succeed",
            () -> assertThat(validator.isValid(amount)).isTrue()
        ));
}
// Tests are generated dynamically at runtime — useful when
// test data comes from an external source (database, file)
```

### 🧠 Mental Model / Analogy

> JUnit 5 is the **stage manager** of your test suite: it discovers which methods are tests, calls them in the right order (setup → test → teardown), reports results, and coordinates extensions (Mockito, Spring) that need to participate in the lifecycle. Your test methods are the performances; JUnit 5 is the production crew.

### 📶 Gradual Depth — Four Levels

**Level 1:** Put `@Test` on a method in a class ending in `Test` — Maven/Gradle find and run it. Use `@BeforeEach` for setup. Use `assertEquals()` or `assertThat()` for assertions.

**Level 2:** Use `@ParameterizedTest` with `@CsvSource` or `@MethodSource` to test multiple inputs. Use `@Nested` to group related tests and create descriptive test names. Add `@Tag("integration")` to mark slow tests and skip them in unit test runs (`-Dgroups=!integration`).

**Level 3:** JUnit 5 extension model: `@ExtendWith(YourExtension.class)` — implement `BeforeEachCallback`, `AfterEachCallback`, `ParameterResolver`. Custom extensions: database cleanup, test timing, custom assertions, custom argument resolvers. `@TestMethodOrder` to control test execution order (usually you want random to catch order dependencies). `@Timeout(5)` to fail tests that take more than 5 seconds.

**Level 4:** JUnit Platform architecture: `maven-surefire-plugin` uses JUnit Platform to discover tests via `TestEngine` SPI. This enables running non-JUnit test frameworks (Spock, KotlinTest, Cucumber) through the same Maven/Gradle test lifecycle. Parallel execution: `junit.jupiter.execution.parallel.enabled=true` in `junit-platform.properties`. Thread mode: `SAME_THREAD` (default, sequential), `CONCURRENT` (parallel within class), `CONCURRENT` for all classes. `@Isolated` annotation marks a test class for exclusive execution (no parallel).

### 💻 Code Example

```java
// Comprehensive JUnit 5 test class
@TestMethodOrder(MethodOrderer.Random.class)  // random order catches order dependencies
@DisplayName("CartService — add/remove items")
class CartServiceTest {
    
    @Nested @DisplayName("when adding items")
    class AddingItems {
        
        @ParameterizedTest(name = "price {0} → cart total {1}")
        @CsvSource({ "10.0, 10.0", "25.0, 25.0", "0.01, 0.01" })
        void addItem_updatesTotal(double price, double expectedTotal) {
            Cart cart = new Cart();
            cart.add(new Item("Widget", price));
            assertThat(cart.getTotal()).isEqualTo(expectedTotal);
        }
        
        @Test
        void addNullItem_throwsIllegalArgument() {
            Cart cart = new Cart();
            assertThrows(IllegalArgumentException.class, () -> cart.add(null));
        }
    }
    
    @Nested @DisplayName("discount calculation")
    class DiscountCalculation {
        
        @ParameterizedTest @MethodSource("discountCases")
        void appliesCorrectDiscount(double total, double expectedDiscount) {
            assertThat(Cart.calculateDiscount(total)).isEqualTo(expectedDiscount);
        }
        
        static Stream<Arguments> discountCases() {
            return Stream.of(
                Arguments.of(99.99, 0.0),
                Arguments.of(100.0, 10.0),
                Arguments.of(200.0, 25.0)
            );
        }
    }
}
```

### ⚖️ Comparison Table

| Feature | JUnit 4 | JUnit 5 |
|---|---|---|
| Extension model | `@Rule` / `@ClassRule` | `@ExtendWith` (cleaner) |
| Parameterized | `@RunWith(Parameterized.class)` | `@ParameterizedTest` (simple) |
| Nested tests | ✗ | `@Nested` |
| Dynamic tests | ✗ | `@TestFactory` |
| Architecture | Monolithic | Modular (Platform + Jupiter + Vintage) |

### ⚠️ Common Misconceptions

| Misconception | Reality |
|---|---|
| "@Test must be public" | JUnit 5 tests can be package-private (`void test()` without `public`) |
| "JUnit 5 includes AssertJ" | JUnit 5 has basic assertions; AssertJ is a separate library (strongly recommended) |
| "One @ExtendWith at a time" | Multiple `@ExtendWith` annotations are supported (composable) |

### 🧠 Think About This Before We Continue

**Q1.** JUnit 5's `@ParameterizedTest` dramatically reduces test duplication. But there's a design question: when should you use `@ParameterizedTest` vs. multiple separate `@Test` methods? The tradeoff: parameterized = concise but all cases use the same assertion structure; separate = verbose but each test can have a unique name and tailored assertion. Give three guidelines for choosing parameterized over separate tests, and describe a scenario where separate tests are clearly better (e.g., testing different error MESSAGE content for different invalid inputs — asserts differ by case).

**Q2.** JUnit 5 parallel execution (`junit.jupiter.execution.parallel.enabled=true`) can reduce test suite time by 2-4×. But it requires tests to be thread-safe. Describe the impact on: (1) tests using `ThreadLocal` state (Spring's `SecurityContextHolder`, Hibernate's session context), (2) tests using Mockito `@MockBean` in Spring Boot tests (shared application context with mock state), (3) Testcontainers with static containers (shared container, concurrent access), and (4) the `@ResourceLock` mechanism for declaring exclusive vs. concurrent access to shared resources. What is the "safe default" for a team migrating to parallel execution?
