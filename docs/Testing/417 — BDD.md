## 📘 Textbook Definition

Behaviour-Driven Development (BDD) is an extension of TDD where tests are written as human-readable specifications using a structured natural language format (Given/When/Then). BDD bridges the communication gap between technical and non-technical stakeholders — the same specification is both readable by a product owner and executable as an automated test. Key Java frameworks: Cucumber, JBehave, Serenity.

---

## 🟢 Simple Definition (Easy)

BDD writes tests as **plain English scenarios** — "Given a logged-in user, When they click checkout, Then they see an order confirmation." These sentences ARE the tests — readable by everyone, automated by the framework.

---

## 🔵 Simple Definition (Elaborated)

BDD shifts the focus from "testing code" to "specifying behavior from the user's perspective." The Given/When/Then format creates a shared language between developers, testers, and product owners. Specifications written in Gherkin can be used live in planning meetings and automatically verified by the CI pipeline — making the test suite a living specification of the system's behavior.

---

## 🔩 First Principles Explanation

**The core problem:**
TDD tests are code — only developers can read them. Product owners write requirements in tickets that may be interpreted differently. The same feature is specified multiple times (ticket, test, documentation) and they drift apart.

**The insight:**
> "Write the specification once, in language everyone understands. Use it as the test. The specification IS the test — always in sync with the code."

```
Given/When/Then structure:

  Given: preconditions (the starting state of the world)
  When:  the action being tested (the trigger)
  Then:  the expected outcome (what should be true after)

Example:
  Given the cart contains item "Java Book" costing $29.99
  When the user applies discount code "SAVE10"
  Then the cart total is $26.99
  And the discount label shows "10% off"
```

---

## ❓ Why Does This Exist (Why Before What)

Without BDD, requirements, tests, and documentation are three separate artifacts that drift apart over time. BDD forces them to be one: the Gherkin specification is reviewed by product owners, automated by developers, and provides living documentation that's always accurate.

---

## 🧠 Mental Model / Analogy

> BDD is like writing a recipe in both professional chef notation AND plain language. The plain-language version ("brown the onions until golden") is readable by anyone and IS the authoritative specification. The chef (developer) reads the same text and knows exactly how to execute it. No translation needed, no ambiguity.

## ⚙️ How It Works (Mechanism)

```
BDD toolchain (Cucumber):

  1. Write feature file (Gherkin — plain text):
     Feature: Discount Code
       Scenario: Valid discount code applied
         Given the cart total is 100.00
         When I apply discount code "SAVE10"
         Then the cart total is 90.00

  2. Implement step definitions (Java):
     @Given("the cart total is {double}")
     public void cartTotal(double amount) { cart.setTotal(amount); }

     @When("I apply discount code {string}")
     public void applyDiscount(String code) { cart.applyDiscount(code); }

     @Then("the cart total is {double}")
     public void assertTotal(double expected) {
         assertThat(cart.getTotal()).isEqualTo(expected);
     }

  3. Run: Cucumber executes step definitions → passes/fails
  4. Report: HTML report readable by product owner
```

## 🔄 How It Connects (Mini-Map)

```
[Product Owner writes scenarios in Gherkin]
       ↓
[Developer implements step definitions]
       ↓
[CI runs Cucumber → generates HTML report]
       ↓
[Passing scenarios = living specification]
[Failing scenarios = implementation gap]
```

## 💻 Code Example

```gherkin
# src/test/resources/features/checkout.feature
Feature: Checkout Process

  Scenario: Guest user can complete checkout
    Given I am on the product page for "Java Book"
    And the book costs 29.99
    When I add the book to my cart
    And I proceed to checkout
    And I enter my payment details
    Then I should see the order confirmation page
    And the order total should be 29.99

  Scenario: Premium user gets 15% discount
    Given I am logged in as a premium user
    And I am on the product page for "Java Book"
    And the book costs 100.00
    When I add the book to my cart
    Then the cart should show a discount of 15.00
    And the total should be 85.00
```

```java
// Step definitions — Cucumber + Spring Boot
@CucumberContextConfiguration
@SpringBootTest(webEnvironment = RANDOM_PORT)
public class CheckoutStepDefinitions {

    @Autowired Cart cart;
    @Autowired CheckoutService checkoutService;

    @Given("I am logged in as a premium user")
    public void loginAsPremiumUser() {
        cart.setCustomer(TestCustomers.premiumUser());
    }

    @Given("the book costs {double}")
    public void setBookPrice(double price) {
        cart.addItem(new Item("Java Book", price));
    }

    @Then("the cart should show a discount of {double}")
    public void verifyDiscount(double expectedDiscount) {
        assertThat(cart.calculateDiscount()).isEqualTo(expectedDiscount);
    }

    @Then("the total should be {double}")
    public void verifyTotal(double expectedTotal) {
        assertThat(cart.getTotal()).isEqualTo(expectedTotal);
    }
}
```

## ⚠️ Common Misconceptions

| ❌ Wrong Belief | ✅ Correct Reality |
|---|---|
| BDD = using Cucumber | BDD is a practice; Cucumber is one tool; you can do BDD with plain JUnit |
| BDD replaces unit tests | BDD specifies higher-level behavior; unit tests verify implementation details |
| Product owners write the step definitions | Product owners write/review Gherkin; developers write step definitions in code |
| BDD is only for acceptance tests | BDD mindset (specifying behavior) applies at all test levels |

---

## 🔥 Pitfalls in Production

**Pitfall 1: Gherkin Nobody Reads**
Feature files written by developers in developer language — product owners never read them.
Fix: involve product owners in writing Gherkin during refinement; review scenarios before implementation.

**Pitfall 2: One Step Per Line (Over-Verbose Scenarios)**
100-line scenarios that read like a UI test script — impossible to maintain.
Fix: use declarative (WHAT) not imperative (HOW) scenarios: "I am logged in" not "I enter email, click submit, wait for redirect."

**Pitfall 3: Cucumber as E2E Framework Only**
Cucumber scenarios drive full browser E2E tests — all the costs of E2E with none of the unit test speed.
Fix: BDD scenarios can drive unit or integration tests too; choose the lowest layer that verifies the behavior.
