# 1146 — Faking

⚡ TL;DR — A fake is a test double with a real, working implementation that is simpler or faster than the production version — like an in-memory database instead of PostgreSQL, or a local file system instead of S3.

### 🔥 The Problem This Solves

Some dependencies are too complex to stub method-by-method (hundreds of operations), but too slow or expensive to use the real version in tests. A fake is the middle ground: a lightweight working implementation that behaves correctly for the test domain, without the production infrastructure overhead.

---

### 📘 Textbook Definition

A **fake** is a test double that has an actual working implementation, but takes shortcuts that make it unsuitable for production use. Fakes implement the same interface as the real dependency and contain real logic — they are simplified, not scripted. Classic examples: H2 in-memory database (fake for PostgreSQL), in-memory HashMap-backed repository (fake for JPA repository), in-memory message queue (fake for Kafka). Fakes are typically hand-written (not generated by mocking frameworks) and reused across many tests.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
Fake = real working code, simplified — in-memory instead of disk/network.

**One analogy:**

> A fake is a **rehearsal stage set**: it looks real enough for the actors to perform correctly, but it's made of lighter materials and isn't weatherproof. It would collapse in a real storm (can't handle production load), but it works perfectly for rehearsals (tests).

### 🔩 First Principles Explanation

FAKE VS STUB VS MOCK:

```
Stub: when asked "findById(1)", always return Alice. No logic.
Mock: when asked "findById(1)", return Alice AND verify it was called.
Fake: implements full find/save/delete with a HashMap.
      findById(1) → looks up HashMap[1] → returns what was actually saved.

The fake KNOWS about the data saved in previous steps.
Stubs don't — they return pre-programmed responses regardless of state.
```

HAND-WRITTEN FAKE REPOSITORY:

```java
// Interface (shared by real and fake)
public interface UserRepository {
    Optional<User> findById(Long id);
    User save(User user);
    void deleteById(Long id);
    List<User> findAll();
}

// Fake: in-memory implementation for tests
public class FakeUserRepository implements UserRepository {
    private final Map<Long, User> store = new HashMap<>();
    private long nextId = 1L;

    @Override
    public User save(User user) {
        if (user.getId() == null) user.setId(nextId++);
        store.put(user.getId(), new User(user)); // defensive copy
        return user;
    }

    @Override
    public Optional<User> findById(Long id) {
        return Optional.ofNullable(store.get(id));
    }

    @Override
    public void deleteById(Long id) {
        store.remove(id);
    }

    @Override
    public List<User> findAll() {
        return new ArrayList<>(store.values());
    }
}

// Used in tests:
UserRepository repo = new FakeUserRepository();
UserService service = new UserService(repo);

service.createUser("Alice");
service.createUser("Bob");
assertThat(service.getAllUsers()).hasSize(2); // fake remembers both saves
```

WHY FAKES ARE VALUABLE:

```
Test scenario: create 3 users, get all, delete one, get all again

  With stubs: must stub findAll() to return 3 users, then 2 users
    → tedious, fragile, tests setup not behavior

  With fake: save 3 users, call getAll(), delete one, call getAll() again
    → fake naturally tracks state
    → test is readable and reflects real behavior
```

### 🧪 Thought Experiment

IN-MEMORY MESSAGE QUEUE FAKE:

```java
// Fake for Apache Kafka in tests
public class FakeMessageQueue implements MessageQueue {
    private final Map<String, Queue<String>> topics = new HashMap<>();

    @Override
    public void publish(String topic, String message) {
        topics.computeIfAbsent(topic, k -> new LinkedList<>()).add(message);
    }

    @Override
    public Optional<String> consume(String topic) {
        Queue<String> queue = topics.get(topic);
        return queue != null ? Optional.ofNullable(queue.poll()) : Optional.empty();
    }

    // Test helper
    public List<String> getMessages(String topic) {
        return new ArrayList<>(topics.getOrDefault(topic, new LinkedList<>()));
    }
}

// Test: verify order confirmation message published
FakeMessageQueue queue = new FakeMessageQueue();
OrderService service = new OrderService(queue);
service.placeOrder(testOrder);
assertThat(queue.getMessages("orders.confirmed")).hasSize(1);
```

### 🧠 Mental Model / Analogy

> A fake is the **test kitchen** version of a restaurant dish: it uses substitute ingredients (test data, in-memory storage), looks and tastes similar enough to verify the recipe (behavior), but isn't made with the restaurant's full industrial equipment (production database, real network). Chefs test recipes in the test kitchen before the main kitchen — not in front of diners.

---

### 📶 Gradual Depth — Four Levels

**Level 1:** A fake is a simple working version of a dependency (like a fake database that stores things in memory instead of on disk). It actually works, just isn't production-ready.

**Level 2:** Use fakes when: (1) the dependency has stateful behavior (reads see what was written); (2) stubbing every method would take longer than writing the fake; (3) multiple tests need the same simplified dependency. H2 in-memory database is a built-in fake for JDBC-based code. `MockMvc` is a fake HTTP server for Spring controllers.

**Level 3:** Fake maintenance: fakes must stay synchronized with the real dependency's contract. If the real `UserRepository` adds a new method, the fake must implement it too. Run the same test suite against both the fake and the real dependency (contract tests) to verify the fake is a faithful substitute. Fake drift: if the fake diverges from the real implementation, tests pass against the fake but fail against the real dependency in integration.

**Level 4:** The economics of fakes: writing a fake takes time upfront (100-200 lines for a repository fake), but saves time across hundreds of tests (no mocking boilerplate, faster tests, more readable tests). Martin Fowler's rule: "fakes are more expensive to create than stubs, but cheaper to maintain when tests require stateful behavior." The Google testing blog advocates "testing at the real layer of abstraction" using fakes rather than mocking everything — aligns with the Detroit school of TDD.

### 💻 Code Example

```java
// Using H2 in-memory database (built-in fake for JPA tests)
@DataJpaTest  // Spring Boot: sets up H2 in-memory DB automatically
class UserRepositoryTest {
    @Autowired
    private UserRepository repo;

    @Test
    void findByEmail_returnsUser() {
        repo.save(new User("alice@example.com", "Alice"));
        Optional<User> found = repo.findByEmail("alice@example.com");
        assertThat(found).isPresent().hasValueSatisfying(u ->
            assertThat(u.getName()).isEqualTo("Alice"));
    }
}

// Custom fake for service-layer tests (no Spring context)
class UserServiceTest {
    private FakeUserRepository fakeRepo = new FakeUserRepository();
    private UserService service = new UserService(fakeRepo);

    @Test
    void deactivateUser_userCanNoLongerLogin() {
        User user = service.createUser("bob@example.com");
        service.deactivateUser(user.getId());
        assertThat(service.canLogin("bob@example.com")).isFalse();
        // FakeRepo remembers the deactivation — no stubbing needed
    }
}
```

### ⚖️ Comparison Table

|                            | Stub      | Mock           | Fake    |
| -------------------------- | --------- | -------------- | ------- |
| Has real logic             | ✗         | ✗              | ✓       |
| Stateful (remembers saves) | ✗         | ✗              | ✓       |
| Hand-written               | Sometimes | No (framework) | Always  |
| Verifies calls             | ✗         | ✓              | ✗       |
| Reused across tests        | Rarely    | Rarely         | Usually |

### ⚠️ Common Misconceptions

| Misconception                         | Reality                                                                                          |
| ------------------------------------- | ------------------------------------------------------------------------------------------------ |
| "Fakes are only for databases"        | Fakes suit any stateful or complex dependency: queues, caches, filesystems, email servers        |
| "H2 is a perfect fake for PostgreSQL" | H2 dialect differences mean some PostgreSQL-specific SQL fails on H2; Testcontainers is safer    |
| "Fakes don't need tests"              | Fakes can have bugs too; run a contract test suite against both the fake and real implementation |

---

### 🚨 Failure Modes & Diagnosis

**1. Fake Diverges From Real — "Works on My Machine"**

**Symptom:** All tests pass with fake repository, but integration tests fail against real PostgreSQL.
Cause: Fake doesn't enforce unique constraints, NULL constraints, or transaction rollback behavior.
**Fix:** Contract test the fake against the real implementation. Use Testcontainers for tests that need real database behavior.

### 🧠 Think About This Before We Continue

**Q1.** H2 in-memory database is the most common Java fake for database tests. But H2 has dialect differences from PostgreSQL: `ILIKE` doesn't exist in H2, `::jsonb` type casting fails, `ON CONFLICT DO UPDATE` (upsert) syntax differs. Describe: (1) three specific H2 compatibility issues that cause tests to pass on H2 but fail on PostgreSQL, (2) how `spring.jpa.database-platform=H2Dialect` setting interacts with native queries (`@Query(nativeQuery=true)`), and (3) the Testcontainers alternative — at what test level you'd use Testcontainers vs H2, and the trade-off in build time.

**Q2.** Google's "Prefer Fakes Over Mocks" guideline (in their Testing Blog) argues that fakes produce more reliable and maintainable tests than mocks because: (a) fakes verify behavioral contracts (save and retrieve), not implementation details (method calls); (b) fakes produce tests that read as user stories, not interaction scripts; (c) fakes can be tested themselves (contract tests). Counter-argument: fakes require 10× more code to write and maintain. Design a decision matrix for when to write a fake vs. when to use mocks — include criteria: complexity of dependency's API, frequency of test reuse, whether behavior is stateful, and whether contract verification matters.
