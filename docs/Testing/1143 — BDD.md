# 1143 — BDD

⚡ TL;DR — Behaviour-Driven Development (BDD) extends TDD by writing tests in a human-readable language (Given-When-Then) that business stakeholders, QA, and developers can all understand, turning acceptance criteria directly into executable tests.

### 🔥 The Problem This Solves

WORLD WITHOUT IT:
Product manager writes a ticket: "Users should receive a confirmation email when they register." Developer implements it. QA tests it. Passes. Production: a user registers with an uppercase email (User@Example.com) — they never receive the confirmation email because the system treats it as a different address than user@example.com. The ticket said "users should receive email" — it didn't specify email addresses should be case-normalized. No one had the same mental model of the requirement. With BDD, that scenario would have been written as an explicit example: `Given a user registers with "User@Example.com" When they check their inbox Then they receive a confirmation email` — forcing the discussion before implementation.

THE TRANSLATION PROBLEM:
Requirements in English get translated to code by developers. In that translation, details are lost, assumptions are made, edge cases are overlooked. BDD creates a shared language — Gherkin (Given-When-Then) — that forces requirement authors to be explicit about examples, and turns those examples directly into tests that verify the code.

---

### 📘 Textbook Definition

**BDD (Behaviour-Driven Development)** is a software development methodology that extends TDD by describing system behaviour in concrete, human-readable scenarios using the Gherkin language. Gherkin scenarios follow the Given-When-Then structure: **Given** (the initial context/state), **When** (the action/event), **Then** (the expected outcome). These scenarios are simultaneously: (1) requirements documentation (readable by business), (2) acceptance criteria (agreed before development), (3) executable tests (automated via frameworks like Cucumber or JBehave). The goal is to eliminate the translation gap between business requirements and implementation by making both parties use the same language.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
BDD = write requirements as Given-When-Then examples that automatically run as tests.

**One analogy:**

> BDD is turning user stories into **contracts**: instead of "users should be able to log in" (vague), you write "Given I am a registered user, When I enter my correct password, Then I am directed to my dashboard" (specific, testable, executable). Both parties sign the contract before development begins.

### 🔩 First Principles Explanation

GHERKIN SYNTAX:

```gherkin
Feature: Shopping Cart Discount
  As a customer
  I want to receive a discount for large purchases
  So that I am rewarded for spending more

  Scenario: 10% discount applied when cart total exceeds $100
    Given my shopping cart contains:
      | Item    | Price |
      | Laptop  | 80.00 |
      | Mouse   | 30.00 |
    When I view my cart
    Then the discount should be 11.00
    And the total should be 99.00

  Scenario: No discount applied for small purchase
    Given my shopping cart contains:
      | Item    | Price |
      | Book    | 20.00 |
    When I view my cart
    Then the discount should be 0.00
    And the total should be 20.00
```

STEP DEFINITIONS (Java + Cucumber):

```java
@Given("my shopping cart contains:")
public void myShoppingCartContains(DataTable items) {
    items.asMaps().forEach(row ->
        cart.add(row.get("Item"), Double.parseDouble(row.get("Price")))
    );
}

@When("I view my cart")
public void iViewMyCart() {
    cartSummary = cart.getSummary();
}

@Then("the discount should be {double}")
public void theDiscountShouldBe(double discount) {
    assertThat(cartSummary.getDiscount()).isEqualTo(discount);
}
```

THE THREE AMIGOS:

```
Requirement review meeting with:
  Product Manager: defines the WHAT and WHY
  Developer: identifies technical constraints and edge cases
  QA: identifies boundary conditions, edge cases, error paths

Together they write:
  - Happy path scenarios
  - Edge cases (zero, negative, max values)
  - Error scenarios (what happens when input is invalid?)

Result: everyone agrees what "done" means before development starts
→ No "I didn't know that was a requirement" after implementation
```

### 🧪 Thought Experiment

THE SPECIFICATION WORKSHOP:

```
Requirement: "Premium users get early access to new features"

Without BDD:
  Developer assumes: "premium" = paying customers
  QA tests: paying customers see early access
  PM meant: users on the "Premium" plan specifically
  Trial users who pay aren't "Premium" plan users
  → Bug in production

With BDD (Three Amigos session):
  PM: "Given a user on the Premium plan, When a new feature is released..."
  Developer: "What about trial users? Annual subscribers?"
  QA: "What if a user upgrades mid-trial?"
  → Each case written as a scenario
  → No ambiguity when development starts
```

### 🧠 Mental Model / Analogy

> BDD is the **legal contract** approach to requirements: instead of agreeing verbally ("yeah, we'll add email notifications"), you write the exact conditions in advance ("Given user action X, When Y, Then Z must occur"). Both parties review and sign. The contract is then automatically enforced by the test suite — you can't ship code that violates the contract without failing the tests.

> The key insight: most requirement misunderstandings happen not in the abstract ("users should receive email notifications") but in the concrete examples ("what about a user with a Gmail alias? What about a user who disabled email?"). BDD forces concrete examples, which surfaces the misunderstandings BEFORE development.

---

### 📶 Gradual Depth — Four Levels

**Level 1:** BDD means writing requirements as "Given [setup], When [action], Then [result]" examples that both the business and developers understand. These examples automatically run as tests.

**Level 2:** Tool stack: Cucumber (Java/Ruby/JS) + Gherkin + JUnit 5. Feature files (.feature) contain Gherkin scenarios. Step definitions are Java methods that execute each step. Cucumber matches scenario steps to step definitions via regex/expression. Run Cucumber in CI: any scenario that fails means either the feature is broken or the feature doesn't yet exist.

**Level 3:** BDD at scale: feature files become documentation — "living documentation" because they're always synchronized with production behavior (they fail if behavior changes). Cucumber reports (HTML) can be published as documentation for business. The "Three Amigos" meeting is the key practice: the conversation that surfaces ambiguities is more valuable than the scenarios themselves. Scenario granularity: avoid UI-implementation-specific steps ("When I click the Submit button") — prefer behavior-level steps ("When I submit the registration form"). The former breaks when the UI changes; the latter doesn't.

**Level 4:** BDD vs TDD: TDD drives unit-level design; BDD drives feature-level acceptance criteria. They operate at different levels and complement each other. In the ATDD (Acceptance TDD) cycle: (1) write a failing BDD scenario (acceptance test); (2) drive down to failing unit tests (TDD) to implement it; (3) BDD scenario passes. This creates a two-level test harness: BDD scenarios verify business behavior, unit tests verify implementation correctness. BDD anti-patterns: (1) "Feature files maintained by developers, not shared with business" — defeats the purpose of the shared language; (2) "Every step is a UI interaction" — tightly coupled to UI, brittle; (3) "Scenarios test implementation, not behavior" — scenarios describe HOW not WHAT.

### 🔄 The Complete Picture — End-to-End Flow

```
Sprint planning: Feature = "Password reset via email"

Three Amigos session:
  Scenario 1: Happy path
    Given user exists with email "user@example.com"
    When they request a password reset
    Then they receive a reset email within 1 minute
    And the reset link expires after 1 hour

  Scenario 2: Unknown email
    Given no user exists with email "nobody@example.com"
    When they request a password reset
    Then no email is sent
    And they see message "If that email exists, you'll receive a reset link"

  Scenario 3: Expired link
    Given a reset token was generated 2 hours ago
    When the user clicks the reset link
    Then they see "This link has expired. Request a new one."

Development:
  Step definitions implement each step
  Each scenario drives specific implementation
  Edge cases (expired token, unknown email) are required by the scenario

QA validation:
  BDD scenarios ARE the acceptance tests
  QA confirms all scenarios pass in staging

Documentation:
  Cucumber HTML report published → product and business can read
  "Living documentation" always reflects current system behavior
```

### 💻 Code Example

```gherkin
# src/test/resources/features/password-reset.feature
Feature: Password Reset

  Scenario: Successful password reset request
    Given a registered user with email "user@example.com"
    When they request a password reset for "user@example.com"
    Then a reset email is sent to "user@example.com"
    And the reset link expires in 1 hour

  Scenario Outline: Edge cases
    Given a registered user with email "user@example.com"
    When they request a password reset for "<email>"
    Then "<outcome>"
    Examples:
      | email              | outcome                              |
      | unknown@test.com   | no email is sent                     |
      | USER@EXAMPLE.COM   | a reset email is sent to the address |
```

```java
// Step definitions
@CucumberContextConfiguration
@SpringBootTest
public class PasswordResetSteps {

    @Autowired
    private PasswordResetService resetService;
    @MockBean
    private EmailService emailService;

    @Given("a registered user with email {string}")
    public void givenRegisteredUser(String email) {
        userRepository.save(User.builder().email(email.toLowerCase()).build());
    }

    @When("they request a password reset for {string}")
    public void whenRequestReset(String email) {
        resetService.requestReset(email);
    }

    @Then("a reset email is sent to {string}")
    public void thenEmailSent(String email) {
        verify(emailService).sendResetEmail(eq(email.toLowerCase()), any());
    }

    @Then("no email is sent")
    public void thenNoEmailSent() {
        verifyNoInteractions(emailService);
    }
}
```

### ⚠️ Common Misconceptions

| Misconception                                   | Reality                                                                                                                     |
| ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| "BDD is just fancy unit testing"                | BDD is about the collaboration that produces the scenarios, not just the test format                                        |
| "BDD replaces TDD"                              | They operate at different levels; BDD for acceptance, TDD for unit design                                                   |
| "Feature files should be written by developers" | Feature files should be authored collaboratively with business; developer-only feature files lose the collaboration benefit |
| "Given-When-Then is the important part"         | The Three Amigos conversation is the important part; Given-When-Then is just the format                                     |

### 🚨 Failure Modes & Diagnosis

**1. Feature Files Become Developer Documentation, Not Business Documentation**

Symptom: Feature files describe implementation steps, not business behavior; business can't read them.
Fix: Feature file language must be domain language, not technical language. Replace "When I POST to /api/users" with "When I register a new account".

**2. BDD Scenarios Tightly Coupled to UI → Brittle Tests**

Symptom: Every UI change breaks 50 scenarios.
Fix: Step definitions should call the service/application layer directly, not the UI. Reserve full UI-driven BDD for the most critical end-to-end flows only.

### 🧠 Think About This Before We Continue

**Q1.** BDD's "living documentation" claim is that feature files + test results = always-current documentation. Compare this to three alternatives: (1) API documentation (OpenAPI/Swagger — accurate for the API contract but doesn't document business behavior), (2) Confluence wiki (written by humans — can document business behavior but drifts from code), (3) code comments (accurate if maintained, but developers don't read them for product understanding). In what scenario is BDD living documentation most valuable? In what scenario is it insufficient (e.g., documenting the WHY of a business decision, not just the WHAT)?

**Q2.** BDD at scale: a company with 50 microservices each has 200+ scenarios. Running all BDD scenarios in CI takes 2 hours. Describe the strategies to make BDD test suites fast enough for CI: (1) tagging (@smoke, @regression) to run subsets; (2) parallel scenario execution (Cucumber parallel mode); (3) service virtualization (mock dependencies instead of spinning up real services); (4) test data management (pre-built test data snapshots vs. per-scenario data setup). What's the trade-off between fast feedback and comprehensive coverage when you decide which scenarios to include in the PR check vs. nightly build?
