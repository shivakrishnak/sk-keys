### 📘 Textbook Definition

Test-Driven Development (TDD) is a software development practice where tests are written before the production code. The TDD cycle has three phases: **Red** (write a failing test), **Green** (write the minimal code to make it pass), **Refactor** (improve the code while keeping tests green). TDD results in code that is designed to be testable, well-specified through tests, and covered by a safety net from the start.

---

### 🟢 Simple Definition (Easy)

TDD means **write the test first, then write the code**. The test defines what the code should do; the code is written to satisfy the test.

---

### 🔵 Simple Definition (Elaborated)

TDD is a design practice as much as a testing practice. Writing a test first forces you to define the API of your code before implementing it — thinking from the caller's perspective. The Red-Green-Refactor cycle keeps progress incremental, each step verifiable, and the codebase always in a working state. Over time, TDD produces a comprehensive test suite and naturally well-structured code.

---

### 🔩 First Principles Explanation

**The core problem:**
Code written without tests is hard to test (not designed for it). Tests added after the fact often test implementation rather than behavior. Developers "gold-plate" code with features that have no defined requirements.

**The insight:**
> "Writing the test first forces you to define the requirement precisely before writing a line of production code. The test IS the specification."

```
Red-Green-Refactor cycle:

  Step 1 (Red):    Write a test that FAILS
                   (feature doesn't exist yet — test must be red)
       ↓
  Step 2 (Green):  Write the MINIMUM code to make the test pass
                   (no gold-plating, no extra features)
       ↓
  Step 3 (Refactor): Clean up the code — no behavior changes
                   Tests must stay GREEN throughout
       ↓
  Repeat for next behavior
```

---

### ❓ Why Does This Exist (Why Before What)

Without TDD, tests are often written after the fact (or not at all), code is designed without testability in mind, and the test suite lags behind the codebase. TDD closes these gaps by making tests a first-class design and development artifact.

---

### 🧠 Mental Model / Analogy

> TDD is like writing the answer key before writing the exam questions. You define the correct answer (test), then figure out how to get there (code). The answer key drives what you write — no ambiguity, no over-engineering. When the exam question (test) passes, you're done.

---

### ⚙️ How It Works (Mechanism)

```
TDD in practice — implementing a stack:

  Cycle 1:
    Red:    @Test void emptyStackHasSizeZero() { assertThat(new Stack().size()).isZero(); }
    Green:  class Stack { int size() { return 0; } }
    Refactor: nothing to refactor yet

  Cycle 2:
    Red:    @Test void pushIncreasesSize() { stack.push(1); assertThat(stack.size()).isEqualTo(1); }
    Green:  add list, push adds to list, size returns list.size()
    Refactor: nothing needed

  Cycle 3:
    Red:    @Test void popReturnsLastItem() { stack.push(1); assertThat(stack.pop()).isEqualTo(1); }
    Green:  pop removes and returns last element
    Refactor: extract helper if needed

  After 3 cycles: a working, tested Stack implementation.
```

### 💻 Code Example

```java
// TDD session: building a FizzBuzz function

// Cycle 1 — RED: write failing test
@Test
void returns_number_for_non_divisible() {
    assertThat(FizzBuzz.convert(1)).isEqualTo("1");  // FAILS - class doesn't exist
}

// Cycle 1 — GREEN: minimal code
class FizzBuzz {
    static String convert(int n) { return String.valueOf(n); }
}

// Cycle 2 — RED
@Test
void returns_Fizz_for_multiples_of_3() {
    assertThat(FizzBuzz.convert(3)).isEqualTo("Fizz");  // FAILS
}

// Cycle 2 — GREEN
static String convert(int n) {
    if (n % 3 == 0) return "Fizz";
    return String.valueOf(n);
}

// Cycle 3 — RED
@Test
void returns_Buzz_for_multiples_of_5() {
    assertThat(FizzBuzz.convert(5)).isEqualTo("Buzz");  // FAILS
}

// Cycle 3 — GREEN
static String convert(int n) {
    if (n % 15 == 0) return "FizzBuzz";
    if (n % 3 == 0)  return "Fizz";
    if (n % 5 == 0)  return "Buzz";
    return String.valueOf(n);
}

// Cycle 3 — REFACTOR: already clean. No changes.
// Result: complete FizzBuzz with 4 tests covering all cases.
```

### ⚠️ Common Misconceptions

| ❌ Wrong Belief | ✅ Correct Reality |
|---|---|
| TDD = writing tests before code always | The point is the design feedback; some skip TDD for exploratory code |
| TDD guarantees no bugs | TDD gives confidence for specified behavior; unspecified cases are unprotected |
| TDD slows development | Initial overhead; long-term faster due to confident refactoring and fewer regressions |
| The green step must be clever | Write the SIMPLEST thing that passes — "fake it till you make it" is valid |

---

### 🔥 Pitfalls in Production

**Pitfall 1: Writing Tests After Code (Test-After Development)**
You know the implementation — your tests end up testing implementation details, not behavior.
Fix: stick to the cycle; if you wrote code first, delete it and write the test first.

**Pitfall 2: Skipping the Refactor Step**
Making tests green and moving on — code quality degrades over time.
Fix: treat refactoring as mandatory; tests exist precisely to make refactoring safe.

**Pitfall 3: Over-Specifying in Tests**
Tests assert on too many details — they break on any refactoring.
Fix: test observable behavior (outputs, state), not internal implementation choices.
