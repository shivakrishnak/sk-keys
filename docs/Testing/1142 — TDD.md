# 1142 — TDD

⚡ TL;DR — Test-Driven Development (TDD) is a development practice where you write a failing test before writing the code to make it pass, then refactor. Red → Green → Refactor.

### 🔥 The Problem This Solves

**WORLD WITHOUT IT:**
Developer writes `calculateTax()` function. Tests it manually in the browser. It works. Ships it. Three months later, another developer changes a tax rounding rule. The function is now subtly wrong for negative amounts (refunds). No automated test catches it. Customer reports wrong refund amounts in production. Without TDD, tests are an afterthought — written after the fact, when the developer's mental model of the code is still fresh, leading to tests that verify what the code does rather than what it should do.

THE DESIGN INSIGHT:
TDD is not primarily a testing technique — it's a **design technique**. Writing the test first forces you to define the interface before the implementation. A class that's hard to test (requires complex setup, has many dependencies) is a class with poor design. TDD makes design problems painful immediately, driving better design.

---

### 📘 Textbook Definition

**TDD (Test-Driven Development)** is a software development process in which test cases are written before the production code they test. The TDD cycle: (1) **Red** — write a test for a small increment of new functionality; the test fails (red) because the code doesn't exist yet; (2) **Green** — write the minimal production code required to make the test pass; (3) **Refactor** — improve the code (remove duplication, improve names, extract methods) without changing behavior; the tests ensure no regression. Repeat for the next increment.

---

### ⏱️ Understand It in 30 Seconds

**One line:**
TDD = write test first, make it pass, clean it up. Repeat.

**One analogy:**

> TDD is writing the question before the answer. By forcing yourself to articulate what the correct behavior is (the test) before you implement it, you ensure you know what "done" means before you start. You can't accidentally widen scope because the test defines the exact target.

**One insight:**
The test suite produced by TDD is a living specification of the system's behavior. Unlike documentation, it can't drift — if the behavior changes, the tests fail.

### 🔩 First Principles Explanation

THE TDD CYCLE IN PRACTICE:

```java
// Step 1: RED — write a failing test
@Test
void shouldCalculateTaxForPositiveAmount() {
    TaxCalculator calc = new TaxCalculator();
    // TaxCalculator doesn't exist yet → compilation error (still "red")
    assertThat(calc.calculate(100.0)).isEqualTo(20.0);
}

// Step 2: GREEN — write minimum code to pass
public class TaxCalculator {
    public double calculate(double amount) {
        return amount * 0.20;  // hardcoded for now — minimum to pass
    }
}
// Test passes. ✓ Green.

// Step 3: Add next test (RED again)
@Test
void shouldCalculateTaxForZeroAmount() {
    assertThat(new TaxCalculator().calculate(0.0)).isEqualTo(0.0);
}
// Passes immediately (0 * 0.20 = 0). Still green.

@Test
void shouldApplyReducedRateForSmallAmounts() {
    // amounts < 50 → 10% tax
    assertThat(new TaxCalculator().calculate(30.0)).isEqualTo(3.0);
}
// FAILS: 30 * 0.20 = 6.0, expected 3.0. RED.

// GREEN: implement the rule
public double calculate(double amount) {
    if (amount < 50.0) return amount * 0.10;
    return amount * 0.20;
}

// REFACTOR: extract constants
private static final double STANDARD_RATE = 0.20;
private static final double REDUCED_RATE = 0.10;
private static final double REDUCED_THRESHOLD = 50.0;
```

WHY RED FIRST MATTERS:

```
If you write GREEN test first (write code, then write test):
  - Test might test what the code does, not what it should do
  - Test might always pass (even with bugs) because it was written to match the code
  - No proof the test can fail (test might not actually run the code path)

Red first proves:
  - The test actually exercises the code path
  - The test can detect the absence of the code
  - The assertion is testing the right thing
```

### 🧪 Thought Experiment

THE FIBONACCI SEQUENCE — TDD DEMO:

```
Test 1: fib(0) = 0  → RED → return 0 → GREEN
Test 2: fib(1) = 1  → RED → if n==1 return 1; else return 0 → GREEN
Test 3: fib(2) = 1  → RED → minimal change → GREEN
Test 4: fib(10) = 55 → RED → implement recursion → GREEN
REFACTOR: add memoization (tests confirm behavior preserved)
```

Each tiny step keeps you in control. No speculative implementation.

---

### 🧠 Mental Model / Analogy

> TDD is **navigation-by-compass**: instead of drawing the entire map before walking (design-first), you orient yourself (write test), take a step (write code), confirm your bearing (tests pass), and orient again (next test). You always know exactly where you are (tests pass = correct position) and where you're going (next failing test = next destination).

> "Make it work, make it right, make it fast" — Kent Beck's TDD mantra. Red = define "work". Green = make it work. Refactor = make it right. (Performance optimization comes much later.)

---

### 📶 Gradual Depth — Four Levels

**Level 1:** Write a test, see it fail, write the code, see it pass, clean up the code. Repeat. The test is always written BEFORE the code it tests.

**Level 2:** Increments should be tiny — TDD works best when each cycle is 2-5 minutes. If the cycle is 30 minutes, the increment is too large. If the test is hard to write (requires complex setup), that's a design signal: the class has too many dependencies or mixed responsibilities (violates SRP). Extract the dependency, pass it in the constructor (dependency injection), and it becomes easy to mock.

**Level 3:** TDD and design: following TDD strictly produces code with high cohesion (classes do one thing, so tests are simple) and loose coupling (dependencies are injected, so tests can substitute mocks). This isn't accidental — the pain of writing tests with tight coupling is the feedback signal that drives better design. "Test pain = design pain": if writing a test is hard, the production code needs redesign. ATDD (Acceptance Test-Driven Development): start with a failing acceptance test (end-to-end), drive down to failing unit tests, make them pass, confirm acceptance test passes.

**Level 4:** The economics of TDD: studies (Nagappan et al., Microsoft/IBM, 2008) found 15-35% longer initial development time with TDD, but 40-90% reduction in defect density. The net ROI depends on defect cost in your context (safety-critical: extremely high; internal tool: low). The "test suite as specification" property: in a well-practiced TDD codebase, the test suite can answer questions about behavior that code comments and documentation cannot, because tests are executable and cannot drift from reality. Kent Beck's "test until fear becomes boredom" heuristic: write tests for the things you're afraid of; stop writing tests for things that are obviously correct (like getters and setters).

### 🔄 The Complete Picture — End-to-End Flow

```
Feature: shopping cart discount
  requirement: 10% discount when cart total ≥ $100

TDD cycle 1: empty cart
  RED: assertThat(cart.getDiscount()).isEqualTo(0.0)
  GREEN: return 0.0
  REFACTOR: —

TDD cycle 2: cart below threshold
  RED: cart.add(item("book", 50.0)); assertThat(cart.getDiscount()).isEqualTo(0.0)
  GREEN: if total < 100 return 0.0
  REFACTOR: extract DISCOUNT_THRESHOLD constant

TDD cycle 3: cart at threshold
  RED: cart.add(item("laptop", 100.0)); assertThat(cart.getDiscount()).isEqualTo(10.0)
  GREEN: if total >= 100 return total * 0.10
  REFACTOR: extract DISCOUNT_RATE constant

TDD cycle 4: multi-item cart over threshold
  RED: multiple items, total 150 → discount 15.0
  GREEN: passes immediately (implementation already correct)
  REFACTOR: —

Result:
  - Cart discount implementation complete
  - 4 tests document all cases (zero, below, at, above threshold)
  - Tests are a specification of the discount rule
  - No production code without a test
```

### 💻 Code Example

```java
// Full TDD session: Money.add()
// Cycle 1: RED
@Test void canAddTwoDollarAmounts() {
    Money result = new Money(5, "USD").add(new Money(3, "USD"));
    assertThat(result.getAmount()).isEqualTo(8);
}
// GREEN: implement Money class with add()

// Cycle 2: RED
@Test void addingDifferentCurrenciesThrows() {
    assertThatThrownBy(() ->
        new Money(5, "USD").add(new Money(3, "EUR")))
        .isInstanceOf(IllegalArgumentException.class);
}
// GREEN: add currency check to add()

// Cycle 3: REFACTOR
// Both tests pass; extract validation to separate method
```

### ⚠️ Common Misconceptions

| Misconception                     | Reality                                                                |
| --------------------------------- | ---------------------------------------------------------------------- |
| "TDD means writing lots of tests" | TDD means writing the RIGHT tests, one at a time, before code          |
| "TDD is only about testing"       | TDD is primarily a design technique that produces tests as a byproduct |
| "100% TDD or nothing"             | Even partial TDD (for complex logic) provides value over zero tests    |
| "TDD is too slow"                 | Slower initially, but faster overall due to fewer debugging sessions   |

### 🚨 Failure Modes & Diagnosis

**1. Tests Written After the Fact ("Test After Development")**

**Symptom:** Test always passes; never catches regressions; test coverage is high but bugs still escape.
Cause: Tests written to match the code (they test what it does, not what it should do).
**Fix:** If you can't write the test first for new code, at least verify the test can FAIL by temporarily breaking the code (comment out a line) and confirming the test fails.

**2. Tests Too Large — Each Test Covers Too Much Behavior**

**Symptom:** TDD cycles take 20+ minutes; when a test fails, it's unclear which behavior is wrong.
**Fix:** One assertion per test (or one behavior per test). If the test requires 20 lines of setup, break the class into smaller pieces.

### 🧠 Think About This Before We Continue

**Q1.** Kent Beck says "I'm not a great programmer; I'm a good programmer with great habits." TDD is one of those habits. The claim is that TDD produces better-designed software because test pain maps directly to design pain (hard to test = bad design). Describe three specific code design anti-patterns that TDD would make painful to write tests for — and how following the "make it easy to test" principle would drive the developer away from each anti-pattern: (1) God Class (one class with 30 methods), (2) static utility methods with side effects, (3) new keyword inside a constructor (tight coupling to dependencies).

**Q2.** Database-backed code is notoriously hard to unit test. A `UserRepository.save(user)` method has external dependencies (database connection, transaction). TDD practitioners disagree: (a) the "London School" (mockist TDD) says mock the repository in tests — test the service layer in isolation; (b) the "Detroit School" (classicist TDD) says use real objects where possible — use an in-memory database (H2) instead of mocking. Compare the two schools: what does each school test at the service layer and at the repository layer? What are the false confidence risks of each approach? How does Testcontainers change the Detroit school's approach?
